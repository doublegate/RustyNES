# RustyNES v2.4.1 — VRC2a (mapper 22) register-select fix

**Release date:** 2026-06-11
**Type:** patch

## Summary

A follow-up to the v2.4.0 VRC2/VRC4 rendering fix. v2.4.0 rewrote `vrc_a_bits`
(the register-select address-line decode) per the nesdev "VRC2 and VRC4" board
table and swapped the A0/A1 pins for **VRC2c (mapper 25)**, but left **VRC2a
(mapper 22)** on the straight wiring. Per nesdev, VRC2a wires the chip's A0
register pin to **CPU A1** and A1 to **CPU A0** — the *same* swap as VRC2c.

With the wrong (straight) decode, every mapper-22 CHR-bank write landed in the
wrong CHR slot/nibble. The sprite slots happened to resolve correctly, so the
hash-only oracle saw a "coherent" frame and didn't flag it — but the background
tiles (TwinBee 3's ground / status bar) scrambled into a magenta/checker mess.

## The fix

`22 => (bit(1), bit(0))` (swapped), matching VRC2c. The VRC2a CHR `>> 1`
low-bit drop (also per nesdev) is unchanged and correct. **Visually verified:**
TwinBee 3 now renders its green status bar, score display, and ships.

## Scope

Isolated to mapper 22: the mapper-23 (Crisis Force, Akumajou Special, Wai Wai
World 2) and mapper-25 (Ganbare Goemon Gaiden) commercial-oracle games are
byte-identical. Only TwinBee 3's oracle snapshot + screenshot and the m22 VRC2a
CHR-banking test snapshot + screenshot are re-baselined, and the
`vrc24_a_bits_per_board_pin_rewiring` unit test is updated to the swapped m22
assertions. **AccuracyCoin 100.00% (139/139)** unchanged; the 60-ROM oracle +
39-title extended oracle pass; nes-mappers 336 tests; fmt + clippy (1.86 +
stable) clean.
