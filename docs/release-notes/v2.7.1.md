## Netplay: the root-cause desync fix

The deepest issue this release. Live two-instance sessions (native and browser) desynced with "desync at frame N (state — picture differs)" at a variable checksum frame, while every in-process harness passed.

The root cause: **`Bus::power_cycle()` was not a true cold boot.** It missed resetting several run-history-dependent fields — most critically **`ppu_clock`**, the master clock, whose residual value carried the old CPU/PPU phase into the "new" boot. Netplay power-cycles both peers at session start (so frame 0 is byte-identical), but each peer had run a *different* number of single-player frames first — so the two boots began with different master-clock phase and a timing-sensitive game (SMB) diverged from frame 0.

The fix resets every run-history field (`ppu_clock`, the DMA byte/page/total accounting, `dma_mc_consumed`, `last_read_addr`, the DMC-DMA latches, the deferred controller write) **and rebuilds the mapper** from the stored ROM bytes — fresh bank registers, cleared CHR-RAM + volatile PRG-RAM, cart metadata kept — so power-cycle is a true cold boot for **stateful-mapper games (MMC1/MMC3/…) too**, not just stateless ones.

The `power_cycle_result_is_independent_of_prior_history` determinism test (run-then-power-cycle vs cold-then-power-cycle must be byte-identical) failed at frame 0 before the fix and passes after; it is un-ignored as a regression guard covering NROM, MMC1, and MMC3.

*Note:* battery PRG-RAM is cleared on a power-cycle like a battery-pull. RustyNES does not persist standard battery saves to disk (only FDS), so nothing on-disk is lost. The FDS path keeps its mapper (FDS netplay is unsupported).

## Netplay reliability + driving

- **Input resend + cumulative ack.** Local inputs are redundantly resent every tick for the un-acked window (idempotent, capped), so a dropped `Input` on the unreliable+unordered transport recovers within a few frames instead of desyncing forever. The `InputAck` is now cumulative — it acks `last_confirmed_frame` (the highest contiguous, gap-free frame), so an out-of-order higher frame no longer suppresses resending a dropped lower one. New test: a 600-frame two-peer session over a link dropping **25% of messages with heavy reordering** still converges byte-identically to the no-loss reference.
- **One-frame-per-pace.** Both the native and wasm pacers previously stepped the rollback session in catch-up bursts; a netplay session now advances exactly one frame per pace (~60 Hz, snap-forward without a burst), leaving peer time-sync to the session's own stall / frame-advantage.
- **Self-diagnosing desync.** The checksum carries the framebuffer hash separately, so a desync now reports whether the *state* diverged ("picture differs") or only the *cycle* count ("same picture, different timing"). This was the diagnostic that localized the bug.

## >2-player browser WebRTC mesh (2-4 players)

The signaling relay now manages N-peer rooms (`max_players` 2..=4) and routes `offer`/`answer`/`candidate` to a specific peer by its `{from, to}` slot (lower slot offers to higher; legacy 2-peer clients still work, unit-tested for 2/3/4-peer rooms). A wasm-only `WebRtcMeshTransport` holds one `RtcDataChannel` per peer (broadcast `send`, merged `poll`; each `NetMessage` carries its `player`). `BrowserNetplay` drives the full mesh handshake and the lobby's 2-4 player selector is functional. A live N-browser session needs the hosted signaling/STUN stack (see `deploy/`).

## RetroAchievements

- **Login** was a dead chicken-and-egg (the session was only created when `[retroachievements] enabled = true`, but `enabled` only became true *after* a successful login). The session is now always created when the feature is built; the login state machine is pumped even with no ROM loaded; on success the token is persisted and the loaded ROM is (re-)identified.
- **Badge lock-state** now derives from the canonical `rc_client_achievement_t.unlocked` bitmask (state `3` is `DISABLED`, not unlocked — the old `state == 3` check was wrong), so unlocked achievements show the colour badge and unearned ones the greyed badge.
- **Badge images** now display: a native-only `badge_cache.rs` fetches each PNG off the UI thread (`ureq`), decodes it (`png`) to an `egui` texture, and caches by URL, with a text-badge fallback.
- The RA HTTP worker now sends a `RustyNES/<version> rcheevos` **User-Agent** — the prerequisite for RetroAchievements to allowlist the client server-side (which removes the "unknown emulator" notice and credits hardcore unlocks; the allowlisting itself is a request to the RA team — see `docs/ra-integration-request.md`).

All RA changes are native-only, behind the default-OFF `retroachievements` feature.

## Crash + correctness fixes

- **NTSC filter crash:** enabling the composite/RGB filter aborted with a wgpu shader-validation error — the WGSL post-pass indexed `let` (value) arrays with a dynamic loop counter, which naga rejects. They're now `var` (addressable). Added a naga shader-validation test for both embedded shaders so this class of bug fails CI instead of at runtime.
- **MMC6 / HKROM PRG-RAM:** the NES 2.0 byte-10 high (battery) PRG-RAM nibble is now honored. Carts that declare their save RAM only in the high nibble (StarTropics) previously got zero PRG-RAM, silently dropping `$6000-$7FFF` writes. `ram_size_from_shift(0) == 0` keeps the common case byte-identical.
- **Windows CI:** a portable `sizeof`-accessor ABI guard in `rustynes-cheevos` replaces the MSVC-incompatible `_Static_assert`.
- **Deploy:** the `Caddyfile` serves a friendly HTTP-200 health page for plain page visits (cert-acceptance shows "OK" instead of a confusing 502) while still proxying only real `wss://` upgrades.
- **Debugger:** opens with the CPU + PPU sub-windows closed (toolbar-only on `~`).

## Vs. DualSystem groundwork

A `vs_db` `dual_system` flag marks the four two-CPU/two-PPU carts (Tennis / Mahjong / Wrecking Crew / Balloon Fight); the frontend logs a clear "needs dual-system support" note instead of silently black-screening. The full two-system emulation is deferred as a documented multi-session feature — see `docs/audit/vs-dualsystem-design-2026-06-11.md`.
