## How rollback netcode works here

Each peer runs the deterministic emulator locally. Because of network latency you
can't wait for the remote input before advancing, so the session **predicts** it
(repeat-the-last-input), advances, and sends your input to the peer. When the real
remote input arrives and **differs from the prediction**, the session **rolls
back** — restores a save-state from the last confirmed frame and **re-simulates**
forward with the corrected inputs. RustyNES's determinism contract (same ROM +
inputs ⇒ bit-identical result) guarantees the re-simulation matches, so both peers
stay perfectly in sync.

This is proven, not asserted: the determinism harness runs two sessions over a
simulated-latency link for 600 frames of seeded-random input and asserts both
peers' gameplay state equals each other **and** a single no-rollback reference
run — plus a high-latency stress test, a desync-detection test, and a real-UDP
loopback test.

## Architecture (`rustynes-netplay`, a new crate)

- **`RollbackSession`** — the GGPO-style core: per-frame input history, a
  save-state ring, prediction, rollback-from-the-confirmed-**checkpoint** +
  re-simulation, and periodic checksum exchange for desync detection. Fully
  deterministic (seeded RNG only; no wall-clock).
- **`Transport` trait** — network-agnostic, so the algorithm is tested with an
  in-memory transport and runs over `UdpTransport` in production.
- **`UdpTransport` + `NetplayConnection`** — non-blocking UDP that drops
  malformed/foreign packets (never trusts network input), a direct host/join
  `Sync` handshake with a ROM-hash match check, and EMA ping/RTT measurement (the
  only place wall-clock is used — never in the deterministic session).
- **Subtlety:** the cross-peer digest is `framebuffer ^ cycle` (byte-deterministic
  across restore+replay), not the full save-state — which carries audio-drain
  transients that vary with drain history yet never affect future frames.

## Using it (native)

Open the debugger (`~`), tick **Netplay**:

- **Host:** enter a local port (default 7000) + the joiner's `IP:port`, click Host
  (you're Player 1).
- **Join:** enter the host's `host:port`, click Join (you're Player 2).
- The HUD shows connection phase, ping, current frame, and rollback/re-sim counts.
  Leave returns to single-player.

Both players must load the **same ROM** (the handshake verifies the SHA-256).

## Verification

AccuracyCoin 100.00% + the 60-ROM oracle 60/60 are byte-identical (netplay changes
no core/chip code; the single-player `produce_one_frame` path is untouched when
inactive). 22 `rustynes-netplay` tests (rollback-matches-reference, high-latency stress,
desync detection, real-UDP loopback) + 7 frontend netplay tests (incl. a
two-peer-over-loopback smoke). Native + wasm-winit + wasm-canvas all build;
`cargo fmt` / `clippy` (pinned 1.86) / `doc` clean; `#![forbid(unsafe_code)]` in
the netplay crate.

## Known / deferred

- **Native-only** — UDP / `std::net` doesn't exist on `wasm32`; the browser builds
  compile with netplay absent (the panel shows a "native-only" note). WebRTC-based
  browser netplay is a future option.
- **2-player; direct host/join** — the host pre-enters the joiner's address (fixed
  remote). Learning the joiner address from the first `Sync`, >2 players, NAT
  traversal, and matchmaking are future work.
- Real cross-machine play needs two running instances; the rollback + transport
  correctness is proven by the in-process determinism + loopback harnesses.
