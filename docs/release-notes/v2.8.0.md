# RustyNES v2.8.0 — "optimized performance"

**Release date:** 2026-06-13
**Type:** minor (frontend + build performance; no stock-NES accuracy or behaviour change)

## TL;DR

| Area | What landed |
|---|---|
| **Pacing (the headline feel-fix)** | Display-sync pacing matrix (`auto`/`display`/`vrr`/`wallclock`) — one emulated frame per display refresh on ~60 Hz panels, VRR + wall-clock for the rest. Kills the 60.0988-vs-display beat judder. |
| **Audio** | Lock-free SPSC ring + alloc-free cpal callback + dynamic rate control (Near's law, ±0.5%) — clean audio, no drift underruns/overruns. |
| **Latency** | Late input latch (immediately before `run_frame`) + run-ahead (default 1 frame) removing the game's own internal input lag. |
| **Throughput** | Mapper capability flags + a 512-entry pixel LUT + fat LTO + SIMD-shaped loops → **−26%** on the rendering-heavy bench. |
| **Threading** | A dedicated emulation thread (default-ON) so UI/render/file-I/O stalls never disturb emulation cadence, with best-effort Linux priority elevation. |
| **Browser** | AudioWorklet output (off the main thread) + rAF display-sync + a wasm Performance HUD. |
| **Instrumentation** | A debugger Performance panel (produced-vs-presented histograms, audio health, pacer counters) + an opt-in CSV "Logging" checkbox. |

**No accuracy or behaviour change to stock NES play.** AccuracyCoin **100.00%
(139/139)**, the 60-ROM `external_real_games` + 52-entry `external_extended`
commercial oracles **byte-identical**, nestest 0-diff. The determinism contract
holds because the dynamic rate control lives in a frontend resampler stage
(never the core's synthesis rate) and run-ahead is frontend orchestration of
the existing snapshot/restore — the core's per-frame framebuffer and audio are
untouched.

## Why this release exists

The accuracy program finished at v2.7.x, but frame output still "felt halted
and/or choppy." Four root causes, all addressed here:

1. **A 60.0988 Hz wall-clock producer beating against a fixed-Hz display** with
   no display-sync or VRR mode → a dup/drop every ~10 s at 60 Hz, drifting
   pull-down at high refresh.
2. **No audio dynamic rate control** → the queue occupancy walked until it
   dropped samples (pops) or ran dry (silence).
3. **Input latched a frame early** → ~2 frames of button-to-emulation latency
   before display latency.
4. **Everything on the winit event-loop thread** → any UI / egui / wgpu-submit /
   file-I/O stall cost a frame of emulation cadence.

## The six phases

- **Phase 0 — instrumentation.** Produced-vs-**presented** interval histograms
  (the presented row is where visible judder lives, invisible to a produced-FPS
  mean), the Performance panel, audio health counters, a `snapshot_restore`
  bench, present-mode-fallback surfacing.
- **Phase 1 — audio engine.** A hand-rolled lock-free SPSC ring
  (power-of-two, atomic f32-bit slots) with an allocation-free cpal callback,
  start-gating + >50 ms hard resync, and a 4-tap Hermite resampler running
  Near's dynamic-rate-control law (`ratio = (1−δ) + 2·fill·δ`, δ = 0.005)
  centred on the `[audio] latency_ms` target.
- **Phase 2 — display-sync pacing matrix.** `[graphics] pacing_mode`:
  `display` (Fifo = the clock, one frame per refresh, ≤0.5% bend), `vrr`
  (Fifo + wall-clock at the console rate), `wallclock` (the sleep-then-spin
  pacer), `auto` (display on ~60 Hz, wall-clock else). Occlusion watchdog +
  sustained-miss sticky fallback. The input latch moved late.
- **Phase 3 — snapshot fast path + run-ahead.** `snapshot_core_into`
  (36 → 14.6 µs, no thumbnail, reused buffer) + `restore_quiet`; run-ahead
  default 1 (persistent timeline byte-identical, unit-proven), median-cost
  budget-throttled.
- **Phase 4 — core micro-opts + build.** `MapperCaps` capability cache (skip
  the unused per-cycle virtual calls), a 512-entry `(emphasis, color) → RGBA`
  `emit_pixel` LUT, **fat LTO**, BLEP-scatter + rewind-XOR auto-vectorization,
  the per-render `to_vec` removed, a `release-native` profile + PGO recipe.
- **Phase 5 — emulation thread.** `App.emu` became a shared
  `Arc<Mutex<EmuCore>>`; single-player production moved onto a dedicated
  thread (default-ON `emu-thread`) that owns the pacer + run-ahead + a `Send`
  audio producer, reads inputs from a lock-free `SharedInput` (the late latch
  preserved), and pings the winit loop after each frame. Netplay stays
  synchronous (the thread pauses + fences on the lock; an under-lock re-check
  closes the TOCTOU so no stray frame can desync a rollback session).
  RetroAchievements stays on the winit thread. Best-effort Linux priority
  elevation (SCHED_RR → nice → timer-slack, per-thread, degrades silently
  without the `realtime` rlimit).
- **Phase 6 — browser.** AudioWorklet (off the main thread, JS embedded as a
  `Blob:` URL, postMessage sample + occupancy transport, the same Hermite DRC,
  `ScriptProcessorNode` fallback) + rAF display-sync (one frame per rAF on
  ~60 Hz panels) + the wasm Performance HUD.

## Performance

| Bench (i9-10850K) | Before | After | Δ |
|---|---|---|---|
| `flowing_palette` (rendering-heavy) | 2.659 ms | 1.958 ms | **−26.0%** |
| `nes_run_frame_nestest` | 3.988 ms | 3.315 ms | **−16.0%** |
| snapshot | 36 µs | 14.6 µs | −59% |

Both frame benches sit well inside the 16.639 ms NTSC budget (5–8× realtime) —
that headroom is what run-ahead and the pacing matrix spend on smoothness and
latency rather than raw speed.

## Configuration

```toml
[graphics]
pacing_mode = "auto"        # auto | display | vrr | wallclock
max_frame_latency = 2       # 1 = lowest latency, 2 = more slack

[audio]
latency_ms = 60             # DRC servo target (20..=250)
drc = true                  # dynamic rate control (false = bit-exact passthrough)

[input]
run_ahead = 1               # 0..=3; removes the game's internal input lag
```

### Lower-latency scheduling on Linux (optional)

The emulation thread elevates its own priority best-effort. To grant it the
real-time / nice rlimit so the elevation engages (no `sudo`, no `chrt`):

```bash
sudo pacman -S --needed realtime-privileges   # Arch / CachyOS
sudo usermod -aG realtime $USER
# log out and back in
```

Without it the thread runs at default priority exactly as before (a one-line
stderr hint tells you which tier engaged).

### A/B the emulation thread

```bash
cargo run --release -p nes-frontend -- rom.nes                       # threaded (default)
cargo run --release -p nes-frontend --no-default-features -- rom.nes # synchronous
```

## Notes

- The browser paths (AudioWorklet, rAF display-sync, the wasm HUD) are
  compile-verified through the full trunk pipeline (wasm-bindgen + wasm-opt;
  2.25 MiB gzip, within the 5 MiB budget) per the project's wasm convention;
  in-browser runtime confirmation on the deployed Pages site is the usual
  follow-up.
- The synchronous (`emu-thread`-off) path is retained for one release for A/B,
  then collapsed (the v2.0.0 → v2.0.1 pattern).
