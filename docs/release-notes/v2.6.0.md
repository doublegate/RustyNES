## Vs. System / PlayChoice-10 — RGB game-verified

v2.5.0 shipped the 2C03/2C04/2C05 RGB-PPU *device* but had no iNES Vs./PC10 ROMs to
game-verify it. v2.6.0 closes that:

- **Mapper 99 (Nintendo Vs. System)** — fixed PRG + `$4016` bit-2 8K CHR. A mapper-99
  cart is detected as Vs. System (mapper-driven, immune to the byte-7=`0x0A`
  corruption that falsely flags 24 home games) and routed to the **2C03** palette.
  **VS Excitebike** + **VS Clu Clu Land** render through genuine 2C03 RGB — visually
  confirmed. A `vs_system_rgb` integration test asserts every framebuffer colour is a
  2C03-palette colour and at least one is impossible under composite 2C02.
- **PlayChoice-10 + broader Vs. detection** — real arcade dumps are clean iNES-1.0
  headers (`byte7 == 0x02` PC10 / `0x01` Vs); the corruption is `0x0A` (console=2
  **plus** the NES-2.0 marker). Honoring only the *exact clean* `0x01`/`0x02`
  (0 home/oracle ROMs trip it) routes **PC10 dumps** (Power Blade, Rad Racer 2,
  Captain Skyhawk, Mario Open Golf) + **VS Castlevania** (m2) + **VS Pinball** (m3)
  to 2C03 RGB — all verified rendering.

> DIP-dependence: Vs. games need per-game DIP switches (`Nes::set_vs_dip`); the
> default DIPs boot some titles, others sit on a black screen. iNES-1.0 dumps lack
> byte-13, so the palette defaults to 2C03 (2C04-000x games would need byte-13).

## Netplay — N-peer + internet groundwork

- **N-peer UDP multi-joiner roster handshake** (`mesh_net.rs`): a host adopts up to
  `num_players-1` joiners from their `Sync`s, assigns player indices, and broadcasts
  a `NetMessage::Roster` so every peer forms the fully-connected mesh
  (`PROTOCOL_VERSION` 2→3). Loopback-verified: host + 2-3 joiners on `127.0.0.1`, full
  handshake + roster + 120 frames over real UDP == a no-rollback reference, for 3 AND
  4 players.
- **Reference signaling server** (`signaling.rs` `Relay` + a `tokio-tungstenite`
  example behind the non-default `signaling-server` feature) for brokering WebRTC
  SDP/ICE.
- **Wasm-frontend WebRTC wiring** (`wasm_netplay.rs`): a web-sys WebSocket signaling
  client + `RtcPeerConnection`/`RtcDataChannel` handshake driving a `RollbackSession`
  (both wasm builds compile). The live STUN probe is confirmed against
  `stun.l.google.com`.

> Full browser netplay still needs the signaling server deployed + two browsers
> (compile-verified); real cross-NAT UDP needs a STUN server + two NATs.

## FDS — real-BIOS boot now works

The v2.2.0 FDS device was unit-tested but **never booted a single disk with the real
BIOS** — it hung forever on "NOW LOADING" ("unverified in CI by design"). v2.6.0
makes it work, in two parts:

1. **Disk wire-format synthesis** — the `.fds` container stores only block payloads
   and omits the on-disk **gap → `$80` start-mark → CRC** framing the BIOS block-scan
   needs. The read engine now precomputes a per-side wire image (per block:
   `[gap][$80][block][crc16]`, CRC-16/KERMIT) and the head reads it with a gap-skip —
   the standard Mesen/FCEUX model. Plus 4 masked register bugs ($4025 bit0/1
   transfer-reset↔motor swap; $4030 byte-transfer flag is bit7 not bit1; $4030 read
   must not clear it; flag CRC-gated) + a drive spin-up.
2. **$4025 bit-3 mirroring inversion** — D3=1 is HORIZONTAL per nesdev (was Vertical),
   which aliased `$2000`↔`$2800`. The BIOS approval check loads the "©NINTENDO"
   licence tilemap to `$2800`, clears the message nametable at `$2000`, then
   self-verifies `$2800` — with the aliased mirroring, clearing `$2000` wiped the
   licence the same frame the verify ran → "DISK TROUBLE ERR.20". Correcting the bit
   fixed it.

**Verified across all 3 BIOS revisions** (FCD / FCD Rev 1 / Twin Famicom): Zelda
(title), Metroid (file-select), Bio Miracle, Otocky, Doki Doki Panic, Akumajou
Dracula. (Kid Icarus passes the licence but stalls later — a game-specific side-B
item.)
