# RustyNES v2.4.0 — Compatibility & Rendering-Accuracy

**Release date:** 2026-06-11
**Type:** minor (bug fixes + features + maintenance)

## TL;DR

| | |
|---|---|
| **Headline** | A 99-title commercial-ROM survey surfaced — and this release **fixes** — two real rendering bugs (VRC7 blank screen, VRC2 garbled tiles) the byte-identical oracle had silently locked into its baselines. |
| **Also** | Mapper 119 (TQROM → **39 mapper families**), netplay host learns the joiner's address from the first `Sync`, the 99-title survey + a regenerated/audited screenshot corpus, and CI maintenance. |
| **Accuracy** | **AccuracyCoin 100.00% (139/139)**; the ~95 unaffected commercial-oracle games stay byte-identical. |

## Why the oracle didn't catch these

The 60-ROM commercial oracle is a **regression gate**: each game's snapshot pins
the framebuffer/audio/cycle hash of *whatever the emulator currently produces*. It
catches *changes*, not *wrongness* — so a rendering bug present when a baseline was
first captured is invisible to it, and AccuracyCoin (100%) doesn't exercise these
specific games. This release added a **visual** survey on top of the hash gate: boot
99 commercial titles, dump a frame, and *look*. That's how these two were found.

## The two fixes

### VRC7 (mapper 85) — Lagrange Point: blank gray → renders

Lagrange Point booted to a flat gray screen for 80+ seconds despite its VRC7 FM
audio playing (so the CPU was clearly running). Instrumenting the CPU showed it
stuck in a `JMP $EC2F` self-spin from frame 30 on. The routine at `$EC32` is a
classic WRAM self-test — `STA ($00),Y` then `CMP ($00),Y` against `$00/$01 = $6000`.
VRC7's `$6000-$7FFF` WRAM had **no backing store** (reads returned 0, writes were
dropped), so the read-back always mismatched, the compare failed, and the boot
jumped to its lockup — never enabling rendering. This is the *same* bug class as the
VRC2/VRC4 WRAM fix from v0.9.x (T-60-003b), which had simply missed VRC7.

**Fix:** an 8 KiB `prg_ram` backing on `Vrc7`, wired into `cpu_read`/`cpu_write` for
`$6000-$7FFF`. Post-fix the CPU runs real code, writes CHR-RAM (4 KB) + nametables,
and turns on rendering — the Konami logo and title appear. (Also tightened:
`Vrc7::ppu_write` now stores CHR-RAM through the same banked `chr_offset` that
`ppu_read` uses, not the raw PPU address.)

### VRC2 / VRC4 (mappers 21/22/23/25) — garbled in-game tiles → coherent

TwinBee 3, Crisis Force, Akumajou Special, Wai Wai World 2, and Ganbare Goemon
Gaiden scrambled their in-game tiles (titles rendered fine). `vrc_a_bits` — which
maps a register write's address to the `(A0, A1)` selector pins — fell through to a
single default for every iNES-1.0 (submapper-0) ROM, i.e. *all* the commercial
ones. But the boards wire the selector to **different address lines per variant**
(register-write traces confirmed: m21 `A1|A6` / `A2|A7`; m22 `A0` / `A1`; m23 `A0|A2`
/ `A1|A3`; m25 `A1|A3` / `A0|A2`). Reading only bits 0/1 collapsed every mapper-23
CHR write to one slot/nibble. Plus VRC2a (m22) doesn't connect the CHR value's low
bit (bank = `value >> 1`).

**Fix:** `vrc_a_bits` rewritten per the nesdev "VRC2 and VRC4" board table + the m22
`chr_offset >> 1`, both trace-verified against the test ROMs. 6 affected snapshots +
7 screenshots + the m22 CHR-test snapshot were re-baselined (the old baselines were
the *buggy* output); 2 VRC unit tests updated to the corrected semantics.

## Also in this release

- **Mapper 119 (TQROM)** — Pin\*Bot, High Speed. An MMC3 variant with a mixed CHR
  space (64 KiB CHR-ROM + 8 KiB CHR-RAM, bit-6 of the bank selects ROM vs RAM),
  implemented as a `Tqrom` that embeds `Mmc3`. **39 mapper families** now.
- **99-title compatibility survey** — `external_extended.rs` locks 39 new
  tricky/best-seller titles (mappers 0/1/2/4/7/71/206) next to the 60-ROM oracle,
  plus a `coverage_smoke` diagnostic bin.
- **Netplay UX** — the host learns the joiner's address from the first `Sync`
  handshake (no pre-entry), only adopting the first peer (no hijack).
- **Screenshot corpus (107 frames)** — regenerated + audited; new `DoubleStartTap`
  / `RepeatStartTap` / short-idle input scripts drive long-intro games (Kid Icarus,
  Excitebike, Mega Man 4/6, Bandit Kings, Burai Fighter) to a representative frame.
- **CI maintenance** — all ~39 newer-stable clippy lints fixed (code clean on 1.86
  *and* current stable, so a future MSRV bump won't re-break CI); the `test-roms`
  job runs in `--release`; `actions/checkout@v6`.

## Verification

AccuracyCoin **100.00% (139/139)**; the 60-ROM `external_real_games` oracle (6 VRC
games re-baselined, 54 byte-identical) + the 39-title `external_extended` survey all
pass; full `--features test-roms` suite green; `cargo fmt` / `clippy` clean on both
1.86 and stable / `RUSTDOCFLAGS=-D warnings` doc clean; no_std cross-compile + native

- both wasm builds compile.

## Known / deferred

- **GxROM (mapper 66)** — the two corpus titles (Doraemon, Thunder & Lightning)
  render correctly and the mapper code is spec-correct; no bug reproduced. If a
  specific mapper-66 game garbles, please report which.
- **SMB3 sprite flashing** — the core places Mario in OAM + the framebuffer every
  frame in all headless tests (title and in-level); the reported flashing could not
  be reproduced headlessly. It may be a within-scanline sprite-fetch artifact at the
  MMC3 status-bar split (needs a PPU-dot-level trace) or a frontend present/pacing
  effect — under investigation.
