## What "master-clock-precise" means

The v1.x scheduler advanced the PPU in whole dots and stepped the CPU every
third dot — an integer 3:1 lockstep. It was fast and correct for the vast
majority of software, but a handful of cycle-exact behaviors (interrupt-sample
timing, sub-instruction DMA interaction, `$2007` rendering-time reads, the PAL
3.2:1 ratio) are not expressible on an integer-dot grid.

**R1 replaces that with a `u64` master clock**, the fundamental NES timebase
(NTSC ≈ 21.477 MHz). The CPU advances `master_clock` by the cartridge region's
`cpu_divider` per cycle (NTSC 12 / PAL 16 / Dendy 15); the PPU is caught up to
`master_clock − ppu_offset` at the regioned `ppu_divider` (NTSC 4 / PAL 5),
sampled in *both* halves of every access (the double catch-up). The APU's DMC
byte timer and the DMA engine run on the same shared timebase. This is the
TetaNES/Mesen2 "run-to-timestamp" model, and it is what makes the seven final
AccuracyCoin tests closeable.

## The 18-month accuracy program → 100%

The opt-in `mc-r1-full-cpu` umbrella reached **AccuracyCoin 100.00%** before
this release (the W3-Stage-4 work, `docs/audit/v2.0-stage4-promotion-2026-06-10.md`).
The decisive breakthroughs:

- **The unified DMA engine.** RustyNES historically had *two* DMA drivers — an
  emergent-span standalone path and a hardcoded-span overlap path — sharing one
  byte-timer phase, so no single parity could satisfy both. v2.0 ports TriCNES's
  single per-cycle DMA dispatch as **one** `unified_dma_cycle` that both
  standalone and overlap DMA ride. This **overturned the long-documented
  "inseparable across span types" wall.**
- **The breakthrough put_cycle end-flip parity.** The DMA get/put decision +
  the DMC byte-timer/reload-arm now run in the references' single
  "access → APU tick → get/put flip" order, with `put_cycle` flipped at the
  *end* of the cycle (after the bus access) so the arm is invisible to its own
  cycle. `apu_phase` stays cycle-start, preserving the C1 φ2 IRQ-sample
  invariant — the first Y=4 in 18 months.
- **The last seven, by axis:** Interrupt-flag-latency (discrete branch IRQ
  poll points), `$2007` Stress (a PPU-dot-scheduled PPUDATA reload, +6-dot
  constant), DMC+OAM / Explicit + Implicit DMA Abort / Delta-Modulation /
  Implied-Dummy-Reads (the unified engine + TriCNES `APU_DelayedDMC4015`
  delayed-`$4015` application + the `$540` consume-edge arm-suppression + BRK
  padding read), and APU Register Activation (the active-window OAM-DMA register
  mirror decode + the `Store(.., $2004)` OAM-put-drives-open-bus rule).

## Phase-7/F — what this release adds on top of the opt-in 100%

v2.0.0 is the **promotion** of that opt-in path to the default build, plus the
region work and cleanup required to ship it.

### 1. PAL/Dendy region parametrization of the R1 CPU loop

The R1 CPU loop previously hardcoded `CPU_DIVIDER_NTSC = 12`, so PAL ran at
12/5 = 2.4 dots/cycle instead of the hardware-true 16/5 = **3.2:1**. v2.0
introduces `Bus::cpu_divider()` (NTSC 12 / PAL 16 / Dendy 15); `start_cycle` /
`end_cycle` / `reset` derive the read/write master-clock split from it via
`read_split(div)` / `write_split(div)`. For NTSC (`div=12`) the splits are
exactly `(5,7)` / `(7,5)` — **byte-identical** to the prior constants. The
bus-side DMA-coherence fixup folds the region's true `cpu_divider` span into
`dma_mc_consumed`, so PAL's 1-mc/cycle deficit vs the 3 physically-ticked dots
is caught up by the next `run_ppu_to` (correct 3.2 average); NTSC/Dendy stay
byte-identical (12 = 12 / 15 = 15). The `region_timing` PAL gate now asserts the
hardware-true **33,247 CPU cyc/frame** (PAL CPU 1.662 MHz / 50 Hz / 312 lines),
not the legacy 3:1 approximation's 35,464.

### 2. R1 promoted to default-on (Option X placement)

`mc-r1-full-cpu` is enabled by the **shipping consumers'** defaults
(`rustynes-frontend` — native + both wasm flavours — and `rustynes-test-harness`), **not**
`rustynes-core`'s own default. This is deliberate: putting it in `rustynes-core`'s default
would make R1 un-strippable through a consumer's `--no-default-features` (the
dependency default leaks it). With this placement,
`cargo test -p rustynes-test-harness --no-default-features --features test-roms`
gives a **clean** v1.7.0 integer-lockstep A/B baseline (94.24%), while
`cargo test --workspace` and the frontends unify to R1.

### 3. Goldens re-baselined under R1 (visually verified)

Under R1 the 60-ROM commercial oracle shifts in cycle count + audio-sample
count + audio FNV (the DMC cadence + region ratios). **6 of 60** ROMs also shift
a framebuffer checkpoint; per the project's oracle-masking discipline, **every
framebuffer change was visually verified** (R1 dump vs the committed v1.7.0 PNG)
to render the correct frame before any golden was re-baselined — Donkey Kong's
25m screen, the Zelda / Kirby / Famista titles, Castlevania III's castle intro
(pixel-identical), Mr Gimmick's menu. The deltas are timing-dependent
animation-phase shifts, **not regressions**. The other 54 ROMs (including the
SMB / Kid Icarus sacred trio) are framebuffer **byte-identical** under R1 — Kid
Icarus's cycle count drops exactly ×30/32, the PAL 3.2:1 ratio applying. 57
oracle snapshots + 7 screenshot PNGs updated.

### 4. Retired 19 refuted master-clock experiment flags

The dead/refuted experiment flags (all default-OFF, none in the umbrella) and
their ~55 `cfg` sites were removed; `burst_dma.rs` was deleted. **Behavior-
neutral** — independently re-verified (AccuracyCoin 100% / 94.24% / oracle 60/60
unchanged). The umbrella chain and the diagnostic tooling (`mc-r1-dmc-abort-probe`,
the PPU-offset sweeps, the trace fixtures) were kept.

## Breaking changes

- **The default emulator scheduler changed.** v2.0.0's default
  timing/audio/cycle output differs from v1.7.0 (the DMC fetch cadence + the
  region-exact ratios). Save states, TAS `.rnm` movies, and any externally
  recorded golden hashes produced under the v1.7.0 default scheduler are **not
  bit-compatible** with the v2.0.0 default. The on-disk save-state *format* is
  forward-compatible (CPU snapshot v2 + the `ppu_clock`/`dma_mc_consumed` pair,
  PPU v3, APU/BUS trailing tails; old blobs upconvert), but a state captured on
  one scheduler should be replayed on the same scheduler.
- **PAL games run ~6.3% fewer CPU cycles per frame** under the default
  scheduler (the correct 3.2:1 ratio) than under v1.7.0's 3:1 approximation.
- **`rustynes-core` embedders:** `rustynes-core`'s own default is the conservative legacy
  scheduler. Enable the `mc-r1-full-cpu` feature explicitly to get the v2.0
  accurate scheduler in a downstream embedding (the RustyNES frontends do this
  via their defaults).

To pin the old behavior for comparison: build any consumer with
`--no-default-features` (re-add `std` + the PPU pivots as needed).

## Determinism & save state

Same seed + ROM + input ⇒ bit-identical framebuffer and audio, on the R1
scheduler as before. The gated R1 DMA/PPU state is fully serialized:
`Cpu::master_clock` + the load-bearing `ppu_clock` / `dma_mc_consumed` pair
(restoring `master_clock` without it desyncs `run_ppu_to`), the unified-OAM/DMC
engine state, the delayed-`$4015` latches, and the PPUDATA state machine. Hot
mid-DMA and mid-reload save/restore round-trips are tested.
