# RustyNES v2.2.0 — Famicom Disk System

**Release date:** 2026-06-10
**Type:** major feature (a new sub-platform)

## TL;DR

| | |
|---|---|
| **Headline** | **Famicom Disk System (FDS) support** — `.fds` images now play (drive, IRQs, writable disks, 2C33 wavetable audio) with a user-supplied `disksys.rom` BIOS. |
| **Accuracy** | **AccuracyCoin 100.00% (139/139)** + 60-ROM oracle **60/60 byte-identical** — unchanged (FDS is a separate path; the iNES cartridge path is untouched). |
| **Tests** | Workspace `--features test-roms`: 876 → **937 strict** (+61 FDS), +7 frontend FDS tests. |
| **BIOS** | `disksys.rom` is **user-supplied and never committed** (Nintendo copyright). |

## What FDS adds

v2.1.0 *detected* `.fds` images and refused them with a clean diagnostic; v2.2.0
**plays them**:

- **Container + device** — the `.fds`/fwNES parser (16-byte header or headerless
  `*NINTENDO-HVC*`, 65500-byte sides) and the FDS RAM-adaptor as a `Mapper`
  (iNES 20): 32 KiB PRG-RAM (`$6000-$DFFF`), 8 KiB CHR-RAM, the user-supplied
  8 KiB BIOS (`$E000-$FFFF`), the `$4020-$4026`/`$4030-$4033` register map, and
  the 16-bit per-CPU-cycle **timer IRQ** (all three documented ack paths), on the
  R1 master clock.
- **Disk drive** — read + **write** byte transfer (~149-cycle cadence,
  byte-transfer flag + optional `$4025`-bit-7 IRQ), **multi-side eject/insert**,
  and **writable-disk persistence** (games save back to the disk).
- **FDS audio (2C33)** behind the `mapper-audio` feature — the 64-entry 6-bit
  wavetable, the 32-step modulation unit with the nesdev pitch-bias formula,
  volume + mod envelopes, and master volume, mixed through the existing
  `mix_audio`→`Apu::tick_with_external` expansion-audio slot (same path as
  VRC6/VRC7/N163/5B).
- **API** — `Nes::from_disk(disk_bytes, bios_bytes)` (+ `_with_sample_rate`) and
  `LockstepBus::with_disk`, parallel to `from_rom` (unchanged).

## Using FDS (frontend)

1. Open or drag-drop a `.fds` file. On first FDS load you're prompted for the
   **`disksys.rom`** BIOS (8 KiB); the path is saved to a new `[fds]` config
   section.
2. Games that span multiple disk sides ask you to "insert side B" — press **F9**
   to cycle Eject → Side 1 → Side 2 → … The toolbar shows "Disk: Side N/M".
3. Disk writes (game saves) persist to `<data_dir>/fds-saves/<disk>.fds.sav`.
4. **Web build:** wasm-winit supports an in-browser BIOS upload (session writes
   only, no on-disk `.fds.sav`); the lightweight wasm-canvas embed does not run
   FDS.

The BIOS is never bundled — supply your own `disksys.rom`.

## Verification

AccuracyCoin 100.00% (139/139, RAM-direct) and the 60-ROM commercial oracle
60/60 are **byte-identical** (FDS is mapper 20, never in those corpora; the iNES
parse/construct path is untouched). 56 FDS device unit tests (parser, IRQ timer,
registers, disk read/write, audio synthesis with concrete asserted values,
eject/insert, save-state v3 round-trips) + harness construct/run tests + 7
frontend FDS tests. `cargo fmt`/`clippy -D warnings`/`doc` clean; no_std
cross-compile green; native + wasm-winit + wasm-canvas all build.

## Known / deferred

- **Real-BIOS FDS boot is unverified in CI by design** — `disksys.rom` is
  non-distributable, so the device + audio are unit-tested for correctness but
  in-game FDS boot requires a user-supplied BIOS. An env-gated
  (`RUSTYNES_FDS_BIOS`) `fdsirqtests.fds` test path exists for local validation.
- Simplified (documented): disk CRC/gaps are not synthesized on write; no analog
  seek-time model; the wiki's output low-pass is left to the host filter chain.
- Next on the roadmap: **netplay** (rollback netcode) — see
  `~/.claude/plans/toasty-noodling-scroll.md` and `to-dos/ROADMAP.md`.
