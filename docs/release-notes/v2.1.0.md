## Mappers (25 → 38 families)

Thirteen new mapper families, spec-implemented from the nesdev wiki with
register/IRQ/nametable unit tests + boot-smoke:

| iNES | Board | Notable titles |
|---|---|---|
| 16 / 159 | Bandai FCG (+ minimal I2C EEPROM) | Dragon Ball Z III/外伝, Famicom Jump II, Datach |
| 18 | Jaleco SS88006 | Ganbare Goemon Gaiden, Magical Kids Doropie |
| 64 | Tengen RAMBO-1 (dual-mode IRQ) | Klax, Skull & Crossbones |
| 65 | Irem H3001 | Daiku no Gen-san, Spartan X 2 |
| 67 | Sunsoft-3 | Fantasy Zone 2 |
| 68 | Sunsoft-4 (CHR-ROM nametables) | After Burner, Maharaja |
| 70 | Bandai discrete | Kamen Rider Club, Family Trainer |
| 73 | Konami VRC3 | Salamander |
| 78 | Holy Diver / Cosmo Carrier | Holy Diver, Uchuusen Cosmo Carrier |
| 88 / 206 | Namco 118 / DxROM | Dragon Spirit, Quinty, Family Circuit |
| 118 | TxSROM / MMC3-TLSROM | Armadillo, NES Play Action Football |
| 210 | Namco 175 / 340 | Famista variants, Chibi Maruko-chan |

**Verification level:** no redistributable behavioral fixtures exist for these
boards, so each is verified by spec-faithful implementation + register/IRQ unit
tests + a 60-frame no-panic boot-smoke — not cross-emulator behavioral
verification. The Bandai FCG EEPROM and the SS88006 `$F003` ADPCM are
spec-faithful but unverified without commercial dumps; dropping a mapper-16/18
dump into `tests/roms/external/` would give them real oracle coverage. Dynamic
`Box<dyn Mapper>` dispatch is preserved (ADR-0001).

## Input devices (opt-in per-port overlay)

A new `InputDevice` overlay attaches per port; `read_port` branches on it only
when a device is set, so the default standard-controller and Four Score paths
are **byte-identical** (no behavior change unless you explicitly attach a
device). Device state is appended to the save-state tail with trailing-default
back-compat.

- **Arkanoid Vaus paddle** (`$4017`): 8-bit serial pot position + fire button.
  `Nes::set_paddle(port, pos, fire)`. **Game-ROM-verified** against the in-tree
  `vaus-test` (the ROM reads our serial knob data and renders position-dependent
  output).
- **Zapper light gun** (`$4017`): light-detect via Rec.601 luma threshold sampled
  at the aim point from the framebuffer, + trigger. `Nes::set_zapper(port, x, y,
  trigger)`. **Unit-verified only** (synthetic-framebuffer light-detect tests; no
  redistributable pass/fail Zapper ROM exists).
- Frontend (native): a Port-2 device selector (Standard / Zapper / Vaus) + mouse
  → aim/trigger/position.

## Test-ROM coverage (+195 strict)

Wired previously-unrun `nes-test-roms/` accuracy suites (all pass on R1 unless
noted): blargg PPU (`2005.09.15b`), `vbl_nmi_timing`, blargg APU
(`2005.07.30`), `apu_reset`, `pal_apu_tests` (PAL-region), `oam_read`,
`read_joy3`, `sprdma_and_dmc_dma`, `dmc_tests`, `ppu_read_buffer`,
`cpu_exec_space`, plus the re-wired expansion-audio corpus
(`audio_expansion.rs`). Each ROM's real reporting mechanism was verified to
avoid deceptive `$6000 == 0` false passes (several report via audio tones or
on-screen text and are wired as deterministic audio/framebuffer-FNV-1a smokes).
Two new harness runners (`run_nes_blargg_pal`, `run_nes_blargg_reset`); the
latter repairs a dead reset path (the old magic check used `"DEB"`, never
matching the real blargg magic `$DE $B0 $61`).

## Fixes

- **`ppu-state-trace` diagnostic compiles again.** The v1.3.1 BG-pipeline fix
  widened `at_shift_lo/hi` to u16 (+2 bytes) without bumping `RECORD_SIZE`
  (111 → 113), failing a const-assert. Reconciled the constant + the Mesen2
  reference emitter.

## FDS groundwork

`.fds` disk images (fwNES `"FDS\x1A"` + raw `*NINTENDO-HVC*`) are detected with
a clear `RomError::FdsUnsupported` ("planned for v2.2.0") instead of a generic
bad-magic error. Full FDS support — `.fds` parser, `disksys.rom` BIOS, disk
drive + IRQ/transfer timing, writable-disk persistence, FDS wavetable audio — is
the v2.2.0 program.

## Known / deferred

- Documented expected-fails: `apu_reset` `len_ctrs_enabled` + `4017_written`
  (nes-apu warm-reset frame-counter semantics; a byte-identical-safe
  `$4017`-retain fix was attempted and reverted because it did not flip them),
  and `mmc3_test_2/4` #3 (the ADR-0002 IRQ-sample axis, 18 prior rollbacks).
- FDS, full input-device UX, and Vs. System / PlayChoice-10 remain out of scope.

## Verification

AccuracyCoin 100.00% (139/139, RAM-direct); 60-ROM commercial oracle 60/60
byte-identical; nestest 0-diff; workspace `--features test-roms` 876 strict + 16
ignored, 0 failed; `cargo fmt`/`clippy -D warnings`/`doc` clean; `no_std`
cross-compile green. No chip hot-path behavior changed.
