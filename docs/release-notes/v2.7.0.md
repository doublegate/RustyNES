## RetroAchievements

The single biggest user-facing feature still missing. RetroAchievements has no Rust
library — every RA-capable emulator (RetroArch, Dolphin, Mesen, SkyEmu) FFIs the
canonical C `rcheevos`. RustyNES does the same, contained in a new opt-in,
native-only crate.

- **`crates/nes-cheevos`** — a safe `RaClient` over the vendored MIT `rcheevos`
  sources (commit 9ade739). `build.rs` compiles them with `cc` (no bindgen/libclang;
  hand-written `extern "C"` declarations guarded by a `_Static_assert`/`size_of` ABI
  check); the entire crate is `cfg(not(wasm32))`, so wasm builds never touch C, and
  the feature is **default-OFF** so the default `cargo build` is unchanged and needs
  no C compiler.
- **The three callbacks.** rc_client hashes the ROM itself (it strips the 16-byte
  iNES header and MD5s the rest), so RustyNES just passes the raw bytes. The
  read-memory callback maps RA's flat NES address space → CPU `$0000-$07FF` /
  `$6000-$7FFF` via the side-effect-free `Nes::cpu_bus_peek` (a scoped thread-local
  closure — rc_client is single-threaded and reads synchronously within each call).
  HTTP runs on a `ureq` worker thread, with rc_client's completion callbacks invoked
  on the main thread. C events become a safe `RaEvent` enum.
- **Frontend.** Login (password → a persisted token), an achievements panel (list /
  points / measured progress), a hardcore toggle, unlock toasts, and a rich-presence
  + leaderboard-tracker HUD. A `[retroachievements]` config section; per-game RA
  progress persists to `<data_dir>/ra-progress/<sha256>.rap`, deliberately **outside**
  the deterministic `.rns` save-state. **Hardcore mode** disables save-state load,
  rewind, cheats (Game Genie + raw), frame-advance, and the debugger memory view
  (save / reset stay allowed).

> Build it with `cargo run -p nes-frontend --features retroachievements`. Real login
> + unlocks need a live RetroAchievements account (a documented manual
> release-checklist item; the FFI linkage, memory map, and hardcore gating are
> unit-tested + an offline FFI smoke test). RA is native-only — the C library would
> need emscripten for the browser, which is deferred. Achievement badge *images* are
> text-only for now.

## Vs.-System per-game database

Closes two v2.6.0 known gaps with one SHA-256-keyed table (`nes-core/src/vs_db.rs`):

- **DIP-switch presets** (from MAME `vsnes.cpp` factory defaults) so single-CPU Vs.
  games boot instead of black-screening on the default DIPs.
- **Exact 2C04-000x palettes** per game (vs the 2C03 default for byte-13-less iNES-1.0
  dumps) — VS Castlevania, VS Super Mario Bros, and VS Excitebike verified in correct
  RGB. A new `Nes::set_vs_ppu_type` re-resolves the palette; precedence is explicit
  `[vs] dip` > DB > 0.

> **Finding:** the still-black Vs. games (Balloon Fight / Tennis / Mahjong / Wrecking
> Crew VS) are genuine Vs. **DualSystem** titles — two CPUs and two PPUs — which this
> single-system core cannot boot past attract. A DIP sweep confirmed it is not a DIP
> issue. Documented in `docs/compatibility.md`.

## Browser (WebRTC) netplay — deployable + wired

The signaling relay, mesh, STUN, and `BrowserNetplay` existed but weren't
wired/configurable/deployable. v2.7.0:

- A **`deploy/` bundle** — a Dockerfile + `docker-compose.yml` running the signaling
  server behind a Caddy `wss://` reverse proxy with a coturn STUN/TURN service, plus
  a deployment guide.
- **Configurable** `signaling_url` + `stun_servers` (`DEFAULT_STUN_SERVERS` public
  default) plumbed into `BrowserNetplay`.
- A **wired wasm lobby** — the `BrowserNetplay` driver runs in the wasm frame loop
  behind a minimal egui lobby (signaling URL / room code / host-join / status).

> 2-player WebRTC is fully wired. A live two-browser session needs the `deploy/`
> stack hosted + two browsers (the wasm path is compile-verified); cross-NAT UDP
> needs a STUN server + two NATs; >2-player browser netplay (the N-peer mesh
> handshake) is a follow-up.

## Screenshots

The commercial-game corpus in `screenshots/external/` is regenerated for all current
games (iNES + FDS + Vs/PC10) via the `coverage_smoke`/`fds_smoke` bins and re-sorted
to mirror `tests/roms/external/` (`mapper-NNN-FAMILY/`, `fds/`, `vs-system/`,
`pc10/`; ~170 PNGs), plus a new `screenshots/montage.png` showcase.

## Best-effort residuals (investigated, documented)

- **Mito Koumon (mapper 89)** renders backdrop-only — a PPU rendering-enable
  dependency (a mirroring-invert experiment ruled out a banking cause; ties to the
  deferred PPU-timing axis).
- **Kid Icarus (FDS)** boots to its name-registration screen (further than the v2.6.0
  report); full play-through past registration needs interactive input that the
  headless harness can't drive.
