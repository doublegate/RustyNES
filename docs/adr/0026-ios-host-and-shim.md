# 26. An iOS host + the `rustynes-ios` shim over the shared mobile bridge

Date: 2026-06-25

## Status

Accepted (v1.9.0 "Sunrise", Workstream A — foundation & build). Mirrors ADR 0024
(the Android host) for iOS.

## Context

The v1.8.0 "Android" plan factored a shared, platform-agnostic
**`rustynes-mobile`** crate that exposes a UniFFI-generated typed control surface
over the deterministic `#![no_std]`+alloc `rustynes-core` chip stack (ADR 0024).
That ADR explicitly anticipated a v1.9.0 iOS port reusing the same bridge. v1.9.0
"Sunrise" begins the iOS/iPadOS app as the first release of the **v1.9.0 → v1.9.9**
TestFlight train (see
[`../../to-dos/plans/v1.9.x-ios-train-plan.md`](../../to-dos/plans/v1.9.x-ios-train-plan.md)).

The core compiles to Apple ARM and produces **byte-identical** framebuffer +
audio (the same property AccuracyCoin 139/139 proves on Android ARM64 CI), so the
entire emulator value — accuracy, shaders, save-states, TAS, and the v1.8.6
bridge additions (Lua / RetroAchievements / netplay) — ports for free. Only the
host shell is new: the render surface, the audio sink, lifecycle, file access, and
the UI.

Two iOS-specific questions had to be settled:

1. **How does Swift call the Rust core?** Re-use the shared `rustynes-mobile`
   UniFFI surface (UniFFI generates Swift exactly as it generates Kotlin), or hand
   a fresh FFI per platform.
2. **What renders the NES image, and where does the hot, hand-rolled glue live?**
   iOS withholds **JIT / W^X**, the one capability the hardest console emulators
   need — so whether RustyNES can ship at all hinges on confirming it needs none.

## Decision

**Bindings — reuse the shared `rustynes-mobile` crate, add a thin
`crates/rustynes-ios` shim** (the iOS analog of `rustynes-android`).

`rustynes-mobile` is **unchanged**: `uniffi-bindgen-swift` / `cargo swift package`
emits the Swift bindings, packaged as an **`.xcframework`** (lipo the simulator
arches, then `xcodebuild -create-xcframework`). The typed surface — `nes_new`,
`run_frame`, `drain_audio`, `set_controller`, `snapshot`/`restore`, plus the
v1.8.6 `load_script` / `ra_*` / `np_*` — is Swift-ready verbatim and type-checked,
identical to the Kotlin surface. No new determinism surface: every method forwards
directly into `rustynes_core::Nes` with no timing feedback or wall-clock
dependence.

`crates/rustynes-ios` (a `staticlib`) carries only the narrow, hot glue UniFFI
cannot express:

- **The `CAMetalLayer` → wgpu-Metal surface handoff** — the iOS analog of the
  Android `ANativeWindow` → wgpu handoff. The SwiftUI shell hosts an
  `MTKView` / `CAMetalLayer`; its layer pointer is handed to wgpu via
  `raw-window-handle`. v1.9.0 ships a ~30-line hand-Metal nearest-blit of the
  256×240 RGBA texture (lighter for a single textured quad on old devices); v1.9.4
  graduates to the full wgpu→Metal path reusing the WGSL `rustynes-gfx-shaders`
  stack (NTSC / CRT / Bisqwit) and the 8:7-PAR / overscan / letterbox blit.
- **The CoreAudio / AVAudioSession audio sink** — cpal's CoreAudio host driving
  the lock-free SPSC ring + Hermite DRC resampler (`resampler.rs`) **verbatim**;
  AVAudioSession (`.playback`, interruption + route-change notifications, the
  background-audio capability) via `objc2-avf-audio`.

Everything iOS-specific is behind `#[cfg(target_os = "ios")]`, so on a host build
the crate is a near-empty shell that exists to be linted; the real staticlib +
xcframework are produced by the iOS targets. `unsafe` is confined here (each site
with a `// SAFETY:` note), the same exemption `rustynes-cheevos`,
`rustynes-frontend`, and `rustynes-android` already take.

**The SwiftUI shell (`ios/`)** — the iOS analog of `android/` — owns all chrome:
the scene / lifecycle (`ScenePhase` pause/resume), the Metal host view, the
on-screen controller + GameController input, `UIDocumentPicker` ROM import,
settings, and the per-version feature surfaces. Touch / gamepad events converge on
the same `set_controller` **late-latched `Buttons` mask** as desktop / wasm /
Android — so TAS / netplay / rollback see input identically and there is **no new
determinism surface**.

**The determinism boundary holds.** The core's `run_frame()` output is untouched;
DRC + run-ahead are frontend stages (timing-only); the `.rns` save-state format is
platform-independent. A state saved on desktop / Android loads on iOS and a `.rnm`
replays bit-identically, so desktop ⇄ Android ⇄ iOS cross-play stays valid.

**JIT — confirmed NON-ISSUE.** RustyNES needs no JIT: the CPU/PPU/APU core is a
pure cycle-stepped **interpreter** (no dynamic codegen); `mlua` is **interpreted**
Lua 5.4 (not LuaJIT); the experimental `piccolo` backend is pure Rust; wgpu→Metal
shaders are compiled by Apple's toolchain at build/load, not generated by the app
at runtime. The headline iOS-emulator restriction is irrelevant to an NES
interpreter.

**Toolchain.** Rust targets `aarch64-apple-ios` (ship) + `aarch64-apple-ios-sim`
(Simulator / CI), built into the staticlib → xcframework. The macOS CI job is
**tag-gated + manual-dispatch only** (macOS minutes bill ~10×, mirroring the PGO /
release gating), running cargo iOS targets → `cargo swift` → xcframework →
`xcodebuild` archive/export → fastlane match → fastlane pilot.

## Consequences

- **Positive.** The hand-rolled `unsafe` FFI shrinks to the surface/audio handoff;
  the entire control surface is type-checked and shared verbatim with Android (one
  `rustynes-mobile`, two shims). The shader / PAR / overscan render crown jewels
  port unchanged (v1.9.4). Host CI still owns accuracy; the iOS CI proves the build
  links + a smoke boot. Cross-play / cross-device save portability stays valid. The
  no-JIT confirmation means the App-Store policy posture is feasible (ADR 0027).
- **Negative.** UniFFI's Rust-lib ↔ generated-Swift version coupling must be kept
  in lockstep (the iOS analog of the wasm-bindgen pin discipline); xcframework
  arch / lipo mistakes are a recurring footgun; macOS CI is ~10× the cost (hence
  the tag-gating); provisioning / signing (fastlane match) adds setup. The
  `CAMetalLayer` surface-loss lifecycle (drop the drawable on background, recreate
  on foreground) must be handled correctly — the keystone risk v1.9.4 retires.
- **Rejected — a winit + wgpu + egui port verbatim on iOS.** winit's mobile story
  is weak and an egui-only UI feels alien (no native navigation / share-sheet /
  document-picker / Dynamic-Type / Stage-Manager / VoiceOver); App Review and
  users penalize non-native chrome. Kept only as a throwaway M0 "does the core run
  on-device" spike.
- **Rejected — a no-shared-crate per-platform FFI.** Duplicates the binding layer
  across Android / iOS and is not type-checked; the shared `rustynes-mobile`
  bridge (ADR 0024) already solved this.
