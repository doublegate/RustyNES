## Hypothesis under test

The prior cross-diff (`golden/tricnes/implicit_abort_xdiff_2026-06-07.txt`)
located the last cycle of the Implicit-DMA-Abort residual as the φ2
*open-bus visibility* axis: the abort-context DMC reload GET drives `$00`
on the external open bus and the immediately-resumed `LDA $4000` reads that
`$00` one CPU cycle too early (`CalculateDMADuration` Y=3 where
`TEST_ImplicitDMAAbort_Key3` wants Y=4). The proposed fix: hold the GET's
external-bus drive back so the resumed read sees the OLD bus (`$40`).

This was implemented CLEANLY (not via the contaminating `mc-phi-subpos`
alternate DMA-step model) as a new default-off flag
**`mc-r1-dmc-get-obus-phi2`** that composes with the working
`mc-r1-dmc-load-get-entry` / `mc-r1-dmc-abort-timer-phase` span:

* `dmc_dma_read` captures the pre-GET φ1 open-bus latch, lets `raw_cpu_read`
  drive the GET byte, then restores the φ1 latch and stashes the GET value
  as pending.
* `cpu_clock` re-applies the pending GET value after a tunable number of
  `cpu_clock` boundaries (`OBUS_DEFER`, env `RUSTYNES_OBUS_DEFER`, so the
  depth sweeps without a rebuild; `255` = disabled = floor-identical).

## Why open-bus cannot be the lever (root cause)

`$540` (Loop3) is the steady-state looping-DMA timing array (NO implicit
aborts — see `AccuracyCoin.asm:13009-13011`). Key3 = `$00×10, $04×6`; the
residual is `$540[10,11] = $03` (want `$04`). `CalculateDMADuration` reads
`$4000` (pure open bus) in a loop, catching `$00` when a steady-state reload
GET halts the CPU exactly at the `LDA $4000` data read; Y counts iterations
to the catch.

The CATCHING GET halts at `$4000` for BOTH the wrong entries (`[10,11]`) AND
the correct ones (`[12..15]`) — they are indistinguishable at the open-bus
layer (same address, same drive value, same φ relationship to the read). The
only thing that differs is the ABSOLUTE byte-timer phase coming out of each
iteration's `$4015=$00 -> $10` re-enable transient (the "cumulative
byte-timer-fire boundary"). A *uniform* open-bus shift therefore moves ALL
entries together (confirmed: it shifts `[12..15]` off `$04` too, and at
depth≥1 collapses the whole measurement), and there is NO local open-bus
discriminator that could scope the shift to `[10,11]` only.

`TriCNES` (the oracle) does NOT defer open bus at all — its `Fetch()` drives
`dataBus` synchronously on the GET. It reads `$40` at the measuring `LDA
$4000` because its DMC DMA halts the CPU at the *instruction boundary* (the
GET runs BEFORE the `LDA $4000` opcode/operand fetch, so the operand
high-byte fetch `$40` re-drives the bus AFTER the GET). RustyNES halts at the
`$4000` data read (GET adjacent to the read, no intervening operand fetch).
Closing the gap means moving the GET earlier — which changes the 576-cycle
A->B reload interval that the `mc-r1-dmc-abort-timer-phase` fix just got
byte-locked to TriCNES. That is the master-clock byte-timer-phase collapse,
not a surgical open-bus shift.
