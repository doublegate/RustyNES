# Phase 1 — DMC-DMA span rewrite (toward 139/139)

> Working config: `mc-r1-substrate,mc-r1-dmc-idle-halt,mc-r1-apu-unified-clock,mc-r1-one-clock,cpu-stack-dummy-reads`.
> Entry baseline (Phase 0): **92.81%**, 10 fail, DMA 7/10, C1 trio 2/3.
> Goal: `CheckDMATiming` Y=3→4 (the abort/OAM cluster precondition) without breaking the 6/10.

## The pinned divergence (measured, not inferred)

`CalculateDMADuration` (AccuracyCoin.asm §12899) measures the DMC-DMA span via the `LDA $4000` spin.
The Implicit-Abort test uses a **looping** sample (`$4E` rate) → its DMAs are **reload** DMAs.

Per-span parity analysis of `rusty_r1stack_0478.csv` (`in_dmc` runs, `put_cycle_post` column):

```
len=2 halt_put_post=0 get_put_post=1   x2     (1-byte implicit-abort edge)
len=3 halt_put_post=1 get_put_post=1   x196   (the CheckDMATiming reload DMAs)
```

`put_cycle_post=1` at the halt ⇒ the halt executes while `put_cycle=0` ⇒ **the reload DMA halts on a
GET cycle** ⇒ span 3 (halt(get)→put→GET(get)). Mesen's reload DMA **halts on a PUT cycle** ⇒ span 4
(halt(put)→dummy(get)→align(put)→GET(get)).

## Why this is (potentially) surgical — the GET-preservation hypothesis

Span 3, halt on get C1:  `halt(get C1) · put(C2) · GET(get C3)`.
Span 4, halt on put C0:  `halt(put C0) · dummy(get C1) · align(put C2) · GET(get C3)`.

**The GET lands at the same absolute cycle (C3) in both.** If the reload halt is moved one cycle
EARLIER (onto the preceding put), the span becomes 4 *without moving the GET*. The 6/10 currently-
passing DMA tests observe the GET *landing*, so a GET-preserving span extension should not regress
them. This is the key distinction from the refuted `arm-delay1` lever, which moved the arm one cycle
**later** (delaying the GET too) → broke the 6/10. The correct direction is **earlier**, not later.

> Oracle confirmation pending (Phase 0 Mesen DMC-fetch trace, `m_dmc_full.csv`): verify Mesen's reload
> GET for the same CheckDMATiming DMA lands at the same absolute cycle as RustyNES's span-3 GET (i.e.
> Mesen's extra cycle is a *leading* halt on the put, not a trailing cycle that moves the GET). If
> confirmed, the fix is surgical; if Mesen's GET is one cycle later, the coupling is real and the fix
> must come from the coherent arm-phase (the deep core).

## GOLD-ORACLE CONFIRMATION (Mesen DMC+RDY trace, `m_rdy_full.csv`, 2171 spans)

Ran the Mesen2 oracle over the full AccuracyCoin run (`MESEN_DMA_RDY_TRACE_OUT`, kind 0=halt/1=dummy/
2=align/3=get). Measured by **master clock** (the reliable axis — the `cpu_cycle`/CycleCount column has
a half-cycle logging artifact where align+get share a value; mc increments +12/cycle cleanly):

- **Mesen's CheckDMATiming `$4000` DMAs = span 4** (×368): halt, dummy, align, get at mc +0/+12/+24/+36.
  The **align is inserted before the get**, so the span-4 GET lands one cycle *later* than a span-3 GET
  would (corrects the earlier "GET-preservation" hypothesis — matching Mesen DOES push the GET later).
- **Mesen produces span-4 UNIFORMLY** for the looping-sample DMAs — the readout-cluster addresses
  (FB66/FB67/FD71/FD72/FF50/…, all the tests that already PASS in RustyNES) are *also* len=4.
- RustyNES: span-4 for the readout cluster (those pass), but span-**3** for CheckDMATiming.

**Conclusion (root cause, oracle-grounded):** RustyNES's reload-DMA **arm parity is inconsistent across
code paths** — it matches Mesen (span 4) for the readout cluster but is one arm-cycle off (span 3) in
the CheckDMATiming / abort-test DMASync setup. This is NOT a span-side bug (the get-parity gating adds
cycles in units of 2, so no clean span-side +1 exists — confirming the 17+ refuted span levers). It is
the **cumulative-counter / arm-phase coherence** defect: somewhere in the abort-test control flow the
DMC arm phase desyncs from the get/put counter by one cycle. The fix is to find and remove that desync
so the arm phase is consistent everywhere (→ span 4 everywhere, matching Mesen), NOT to force span 4 on
the span side (which would move the GET and break the already-correct readout cluster).

**Next concrete step:** per-cycle oracle cross-diff (Phase-0 unified cell trace) of the abort-test
DMASync entry — find the exact cycle where RustyNES's `put_cycle`/DMC-arm phase diverges from Mesen's,
working backward from the span-3 halt. Candidates: the `dma_mc_consumed` fold (`cpu.rs:617`), the OAM
513/514 alignment, or a write-cycle/RMW halt-delay the arm doesn't account for (`DMA.xhtml:30`:
write-cycle halts delay the DMA by 2 (RMW) or 3 (interrupt) cycles). RW-2 refuted the OAM fold as the
Y-mover, so prioritize the write-cycle/RMW halt-delay and the abort-setup `$4010` rate-change path.

## MAJOR CORRECTION (2026-06-05, same session) — the span is 4, NOT 3

The `in_dmc`-run histogram (`{3:196}`) was an **artifact**: R1 clears `in_dmc` BEFORE the GET cycle's
trace row, so counting `in_dmc` runs undercounts the span by 1. Measuring the **true span through the
GET** (the `r $FFC0` read that follows): the `$4000`-halt CalculateDMADuration DMAs are **span 4**
(×75) — identical to Mesen. **The prior 17+ "span 3→4" levers (and this doc's first draft) were all
chasing the `in_dmc` undercount.** RustyNES's DMC-DMA span is already correct.

**Y measured DIRECTLY (not inferred):** the ROM's `STY <$50` in the DMC+OAM test stores Y. Trace of
`W $0050` → **Y = $03** (cyc 48633695). So the test genuinely computes Y=3, wants 4 — but NOT because
the span is short.

## What Y actually measures (re-derived)

`CalculateDMADuration` syncs the DMC DMA "to occur in 50 cycles," then walks the `LDA $4000` spin 1
cycle earlier per 575-cycle iteration vs the 576-cycle DMA period. Y = iterations until `LDA $4000`
reads `$00` (the GET driving `$FFC0=$00` onto the bus). **Y is the absolute phase distance from the
DMASync anchor to the GET's `$00`**, in cycles. Y=3-vs-4 = the GET's `$00` is 1 cycle EARLIER (closer
to the anchor) in RustyNES than in Mesen.

Two candidate sources for the 1-cycle, both phase (not span):
1. **The GET lands 1 cycle early** — but the global DMC timer is seeded once at power-on + counts
   continuously (`dmc.rs:215`, not reset by `$4010`/`$4015`), and the readout-cluster DMAs land
   correctly, so a path-specific GET shift is unlikely.
2. **The DMASync anchor is 1 cycle off** (LEAD HYPOTHESIS). `DMASync_50CyclesRemaining` exits its
   `LDA $4000;BNE` spin when the DMC conflict drives `$00`. If RustyNES's DMC-conflict-during-spin
   phase is 1 cycle off, DMASync exits 1 cycle early → the whole CheckDMATiming measurement is
   anchored 1 cycle off → Y=3. This matches the prior R1-1 finding ("R1's spin exits at a different
   phase") and the DC-2(b) "DMASync un-wedge / spin-exit alignment" thread.

## Redirected Phase 1 plan

The fix is NOT a span edit (span is correct) and NOT an arm-parity shift (global timer is consistent).
It is the **DMASync spin-exit / DMC-conflict-phase alignment** (DC-2(b)): align the cycle at which
RustyNES's DMC DMA drives `$00`/`$40` open-bus onto `$4000` *during the DMASync spin* to Mesen, so the
spin exits at Mesen's phase. **Next step:** unified per-cycle oracle cross-diff (Phase-0 cell trace,
NOT the artifact-prone `in_dmc`/RDY traces) of the DMASync `LDA $4000;BNE` spin exit — RustyNES vs
Mesen — to pin whether the spin exits 1 cycle early and why (the DMC halt's open-bus `$40`→`$00`
transition timing on `$4000` during the conflict). Validate any fix on the BATTERY (the 7/10 must hold).

## Per-cycle cross-diff progress (RustyNES side, definitive)

Contiguous trace of the DMC+OAM ($0477) CheckDMATiming, anchored on the `$4010=$4E` landmark (the
`STA $4010` in CycleClockBegin, cyc 48630742):

- **DMASync spin exit looks CORRECT** (off −387..−383 from landmark): DMA halt(−387)/dummy(−386)/
  align(−385)/GET `$FFC0=$00`(−384), then CPU `LDA $4000` reads `$00`(−383) → spin exits. Span 4, the
  `$00` open-bus from the GET persists 1 cycle to the CPU read. So the DMASync exit is NOT the 1-cycle
  source (it matches the expected structure).
- **The walk (CalculateDMADuration, after the landmark) is where Y=3 is set:** the sparse `LDA $4000`
  (`R`) reads are at **+1204 `$40`, +1779 `$40`, +2354 `$40`, +2929 `$00`** (575 apart) → Y=3. The
  ROM's `STY <$50` at +2953 stores `$03`.
- The +1204 first-read offset is fixed clockslide code (same in Mesen). Y=3-vs-4 ⇒ RustyNES's DMA GET
  (the `$00` the walk coincides with) is **1 cycle off relative to the landmark**, amplified by the
  575-cycle/iteration walk into a one-iteration (one extra `$40`) difference.

**Cross-diff target:** Mesen's walk-`$00` offset from its own `$4010=$4E` landmark (expected +3504 if
Y=4), and more precisely the GET DMA's offset from the landmark — the direct 1-cycle measurement. (NB:
Mesen must run ~46s wall to reach the abort tests at cyc ~48.6M; the early windows used DMC Bus
Conflicts' `$4F` rate, not CheckDMATiming's `$4E`.) The 1-cycle is the DMC-DMA-GET position vs the
landmark = the DMC timer phase at the landmark; if global, it is the boot-phase residual the long walk
uniquely amplifies (readout tests observe the DMA immediately, no accumulation → tolerant).

## Fix attempt (b): `mc-r1-dmc-phase-minus1` — WEDGES (re-confirmed in the new context)

Measured the targeted DMC-phase shift on the FULL working config
(`...,mc-r1-one-clock,cpu-stack-dummy-reads,mc-r1-dmc-phase-minus1`): **battery WEDGES** — only 83/144

**This definitively re-confirms the binding coupling:** the DMC GET phase that makes DMASync exit
correctly (the working 7/10) is the SAME phase that puts the walk's GET 1 cycle off. A global DMC-phase
shift cannot separate them. The targeted-phase-fix path (b) is refuted.

## Remaining levers (narrowed)

1. **Open-bus `$00` persistence at the CPU's `LDA $4000` read** — INDEPENDENT of the DMC GET *fire*
   phase. The GET drives `$FFC0=$00`; how many cycles that `$00` persists on the open-bus latch before
   the CPU's resumed read refreshes it (`$40` from the `LDA $4000` operand fetch) decides which walk
   iteration first reads `$00`. The DASMync spin exit shows R1's `$00` persists exactly 1 cycle
   (GET at −384, CPU reads `$00` at −383). If default persists it differently, the walk catches it a
   different iteration WITHOUT moving the GET fire (so DMASync may not wedge). NEXT TO TEST: the
   open-bus update timing in the GET path (`dmc_dma_read` / `replay_dma_noop_read`, `bus.rs`).
2. **The full cumulative master-clock rewrite** — if the open-bus persistence is also coupled to
   DMASync, the 1-cycle is intrinsic to R1's master_clock/run_ppu_to split and needs the RW-style
   single-counter rewrite (multi-week, the deferred core).

Tier-1 alternative (per the plan): **NMI Overlap BRK** (Phase 4) is CPU-only / lockstep-safe and does
NOT depend on this DMC core — a guaranteed-tractable win that can land independently while the DMC
core remains the long grind.

## REWRITE-START FINDING (2026-06-05): the DMA is IDENTICAL default-vs-R1 — the 1-cycle is cumulative CPU/master-clock

Compared the DMASync spin exit between the default build (passes, Y=4) and R1 (fails, Y=3), both
captured with the legacy/R1 trace (default-as-oracle, same boot):

- **Default spin exit:** `r $4000=$40`(halt) / `$40`(dummy) / `$40`(align) / `r $FFC0=$00`(GET) /
  `R $4000=$00`(CPU read, GET+1). Span 4, GET-to-read = 1 cycle.
- **R1 spin exit:** byte-identical structure — span 4, GET-to-read = 1 cycle.

**The DMA itself — span (4), GET landing, open-bus `$00` persistence (1 cyc), DMASync exit shape — is
identical between default and R1.** So the Y=3-vs-4 1-cycle is **NOT in the DMC scheduler** (not span,
not GET, not persistence, not arm). It is a **cumulative CPU/master-clock timing difference** that
accumulates over the ~2900-cycle CheckDMATiming sequence (DMASync exit → fixed clockslides → walk),
shifting where the walk's `LDA $4000` samples the DMA by 1 cycle by the time Y is measured.

This explains every refuted DMC lever (the DMA is already correct) AND the phase-minus1 wedge (shifting
the correct DMA breaks DMASync). It re-frames the rewrite target: **away from the DMC scheduler, toward
the R1 master_clock/start_cycle/end_cycle CPU-cycle accounting** — specifically a sub-cycle (φ1/φ2)
phase difference in how the CPU read samples the bus relative to the DMA, or a tiny per-cycle
master_clock drift the long sequence amplifies. nestest is zero-diff (CPU cycle COUNTS correct), so the
drift is sub-cycle / DMA-cycle-specific, not an integer count error nestest would catch.

**Rewrite next step:** an AUTOMATED default-vs-R1 per-cycle drift harness — align both on a robust
landmark (the DMASync exit `R $4000=$00`, which is byte-identical) and walk forward counting the cycle
at which the two streams' bus accesses first diverge through to the walk's GET. Hand-matching is
intractable (the boot-flow wall + many identical-looking CheckDMATiming calls). The drift harness pins
the exact CPU/master-clock cycle where R1 and default diverge — the precise structural fix site.

## Mechanism (current code)

- `crates/nes-apu/src/apu.rs` `dmc_step_reload_arm` (~799): arms a reload (`dmc_dma_is_load=false`,
  `dmc_need_halt=dmc_need_dummy_read=true`) when `self.dmc.needs_dma()`. Fires in `tick_dmc` at END of
  cycle after `clock_timer` (on `!put_cycle`).
- `crates/nes-core/src/bus.rs` `dmc_dma_step_impl` default-R1 path (~2288-2320): `dmc_halt` single
  preamble; GET on `get_cycle(=!put_cycle) && !dmc_halt`; the halt cycle clears `dmc_halt` on the
  first get-parity cycle (so the halt doubles as the get-alignment cycle → span = start-parity).

## Candidate fix (to test against the full battery)

A reload-only **leading halt on the preceding put cycle**: when a reload arms and its halt would land
on a get cycle, start the halt one cycle earlier (put) so the span is 4 with the GET preserved. Must
distinguish load (keep span 3, halt-on-get) from reload (span 4, halt-on-put) — exactly Mesen's
load-vs-reload split (`DeltaModulationChannel` `_transferStartDelay` vs the output-unit `Run()` arm).

Gate: full battery — `CheckDMATiming` Y=4 (Instruction Timing flips), the 6/10→7/10 (DC-6) HOLD, C1
trio holds, nestest zero-diff, no wedge. Validate on the BATTERY, never the isolated sub-test
(`project_c1_trace_loop_ceiling` hard rule). ASK before any rollback (`feedback_adr0002`).
