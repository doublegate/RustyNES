## TL;DR

* The residual is a **1-CPU-cycle DMC byte-timer phase error**, NOT an open-bus
  or recognition-latency bug. RustyNES's looping-reload byte-timer wraps
  (fires `clock_output` + arms the reload DMA) one CPU cycle LATER than TriCNES
  relative to the CPU instruction stream, so the reload GET preempts the *next*
  instruction-fetch slot instead of the current one.
* It is **NOT uniform**: the Loop3 X=10/11 iterations are 1 cycle wrong
  (Y=3, want 4) but X=12..15 already match TriCNES exactly (Y=4). A global
  byte-timer phase shift therefore trades [10,11] for [12..15] *and* shatters
  the rest of the chip (SH*, the `$500` loop, DMASync).
* The X=10/11-vs-X=12..15 distinction is the **sub-cycle byte-timer phase the
  `$4015` re-enable establishes at the plateau edge** — the documented
  18-rollback master-clock axis (Phase 2). No simple discriminator scopes the
  correction to only [10,11] (their `$4015` re-enable reads a *different*
  byte-timer value per X; in the breakthrough config X=10 reads `dmc.timer==25`,
  not the `{10,8}` the older "budget-OFF base" measured).

## The authoritative HEAD cross-diff (the real, current picture)

`golden/tricnes/implicit_abort_head_xdiff_2026-06-07.txt` is the per-cycle
TriCNES↔RustyNES diff at HEAD with the breakthrough config, anchored on the
`CalculateDMADuration` entry `$DA03`, first loop iteration (Y=0). Reproduction
recipe is at the bottom of that file. The divergence (entry-relative `e`):

```
        RustyNES                       TriCNES
 e=45   R 01E2   (CPU read)            R 01E2   (CPU read)
 e=46   R DA07   (CPU read)            r 01E2   DMA halt (re-read 01E2)  <<<
 e=47   r DA08   DMA halt              r 01E2   DMA halt
 e=48   r DA08   DMA halt              r 01E2   DMA halt
 e=49   r DA08   DMA halt              r FFC0   GET ($FFC0 -> open bus $00)
 e=50   r FFC0   GET                   R DA07   (CPU resumes)
 e=51   R DA08   (CPU resumes opcode)  R DA08
 ...                                    ...
 e=54   R 4000   = $40 (NO catch)      R 4000   = $40 (NO catch)
```

Both read `$4000 = $40` at Y=0 (no catch). The structural difference is WHERE
the reload DMA halts:

* **TriCNES** halts at `e=46`, preempting the `DA07` instruction-fetch slot.
  Its byte-timer fire is attributed to `e=45`'s APU tick (`_EmulateAPU` runs
  AFTER `_6502` in the same cycle), recognised at `e=46`'s `_6502`.
* **RustyNES** reads `DA07` at `e=46` and only halts at `e=47` (the `DA08`
  slot). Its byte-timer wraps in `cpu_clock` at the START of `e=46`
  (`dtimer 0->35, dbits 1->8, dmc_pending 0->1`, `apu_phase=1`), recognised at
  `e=47`'s `read1` top-check.

The byte-timer fires on `apu_phase=1` in RustyNES (`e=46`); TriCNES's equivalent
fire is on the `e=45` (the previous, opposite-parity) cycle. Per-cycle DMC-state
columns in both logs confirm the byte-timer COUNTDOWN itself is 1 cycle out of
phase (RustyNES `dtimer` hits 0 one CPU cycle later than TriCNES, relative to the
identical CPU instruction stream). This is a byte-timer PHASE difference, not a
recognition-order difference: the recognition latency is 1 cycle in BOTH
emulators (a fire attributed to cycle N is serviced at N+1 in both).

## Why it is plateau-edge-specific (X=10,11 wrong; X=12..15 correct)

`CalculateDMADuration` walks the measuring `LDA $4000` read 1 cycle earlier per
iteration (loop = 575 cyc vs the 576-cyc DMC period). Y = the iteration where the
read coincides with the reload GET's `$00`. Measured per-iteration `read - GET`
offset (cyc):

| iter (Y) | X=10 offset | X=12 offset |
|---------:|:-----------:|:-----------:|
| 0        | **+4**      | **+5**      |
| 1        | +3          | +4          |
| 2        | +2          | +3          |
| 3        | +1 (catch)  | +2          |
| 4        | —           | +1 (catch)  |

* X=10 starts the walk at offset **+4** -> catches at Y=3. (TriCNES starts +5
  -> Y=4.)
* X=12 starts at offset **+5** -> catches at Y=4, **matching TriCNES**.

The X=10 and X=12 *catch windows are byte-identical* (same instruction stream,
same arm at the `LDA` operand-high fetch, same GET at rel −17, same `$00` read at
rel −16) — only the iteration COUNT differs. So the defect lives entirely in the
**starting phase the `$4015` re-enable sets**, which is 1 cycle off for X=10/11
but correct for X=12..15. (The `Implicit DMA Abort` answer-key `Key3` is
`$00×10, $04×6` — a Y=4 *plateau* for X=10..15 created by the "a DMA cannot occur
within 2 cycles of a previous DMC DMA" exclusion; RustyNES reproduces the
plateau for X=12..15 but is 1 cycle short at its low edge.)

### Why no surgical Phase-1 fix exists

Every available lever IS the shared DMC byte-timer phase:

1. The byte-timer phase drives the DMC audio output, the DMASync `$4000`
   open-bus conflict (the un-wedge the whole R1 substrate depends on), the SH*
   unstable-store open-bus reads, and the `$500`/`$520` abort loops. A global
   shift breaks all of them (83.45%/23).
2. Scoping requires a discriminator that selects exactly the X=10/11 reloads and
   nothing else across the whole 139-test battery. No such discriminator exists
   as a simple byte-timer value: each X parameterises the `$4015` re-enable
   timing, so the byte-timer reads a different value per X at the re-enable
   (X=10 -> 25), and any fixed value collides with unrelated tests' enables.
3. The plateau-edge phase is a sub-cycle relationship between the `$4015`
   re-enable, the load DMA delay, the `CannotRunDMCDMARightNow`/`dmc_dma_cooldown`
   exclusion, and the byte-timer wrap. Only a model that derives all of those
   from one master-clock sub-position reproduces it universally — that is Phase 2.

## Phase 2 specification (the remaining work)

The residual requires modelling the DMC byte-timer phase that a `$4015`
re-enable establishes, at sub-cycle (master-clock) resolution, to match TriCNES
across the whole X sweep. Concretely, port TriCNES's re-enable model
(`Emulator.cs`):

* `APU_DelayedDMC4015 = APU_PutCycle ? 3 : 4` on every `$4015` write
  (`Emulator.cs:9512`), and `DMCDMADelay = 2` for the silent-start load
  (`:9521`) — both parity-exact and decremented on the put->get transition
  (`:1202`), versus RustyNES's `dmc_dma_delay = apu_phase ? 4 : 3`.
* The implicit-abort arm at `APU_ChannelTimer_DMC == 10` (get) / `== 8` (put)
  (`:9540`) reading the LIVE byte-timer at the enable.
* `CannotRunDMCDMARightNow = 2`, decremented by 2/cycle (`:1186`, `:4168`),
  gating the reload arm (`:1165`) — the "2-cycle exclusion" that builds the
  Y=4 plateau, versus RustyNES's `dmc_dma_cooldown = 4`.

The binding constraint that makes this hard (and the reason for the 18
rollbacks): the byte-timer phase MUST stay byte-identical for the un-wedge
(DMASync), SH*, the C1 IRQ trio, the `$500`/`$520` abort loops, X=12..15, and
the 60-ROM commercial oracle while the X=10/11 re-enable phase is corrected.
That is a single coherent sub-position model, not a poke.
