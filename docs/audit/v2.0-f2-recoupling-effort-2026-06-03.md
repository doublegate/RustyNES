## The problem (evidence-backed, this session)

`trace_dma_4015` + per-fetch parity analysis established:

- Under `mc-r1-substrate` the DMC-DMA **period jitters ±3 cyc** (inter-start interval 429..435) where
  the default lockstep DMA is **exact** (clean 432 / 450 / 576). The jitter accumulates across the
  landing-sensitive tests' long clockslides, drifting the DMA 30-80 cyc off the test-positioned event
  (`BIT $4015`, `STA $4015` abort, OAM overlap) — so the abort never truncates the DMA, the BIT never
  gets delayed, etc. The 6/10 pass because they observe the DMA immediately after DMASync (no drift).
- The jitter is the **DMA-arm parity variance**, NOT the span length: even a CONSTANT span=4
  (fixed-span experiment) still oscillates ±3, with the period correlating to `put_cycle@arm`
  (`=1` → 432, `=0` → 429..431) in a net-zero limit cycle (a 435 paired with a 429).
- `put_cycle` is **locked to CPU-cycle parity** (`putcyc=1 ↔ cyc&1=1` every fetch). The DMC timer is
  clocked on `!put_cycle` (F-2). A variable-length DMA stall shifts the CPU **instruction stream**
  relative to the CPU **cycle count**; the timer is on the cycle count, the arm-service is on the
  instruction stream → they drift → arm-parity variance → period jitter.

## Refuted localized levers (DO-NOT-RETRY)

1. **dummy-read preamble** (`mc-r1-dmc-dummy-read`) — INERT. The R1 span was never short (an `in_dmc`
   counting artifact); adding a preamble cycle does nothing under parity-gated GET.
2. **abort-cancel in `drain_dma` / `dmc_dma_step`** — INERT. The matured `pending_dmc_abort` never
   bites; the aborts don't even reach the DMA (period drift).
3. **fixed-span count** (`mc-r1-dmc-fixed-span`) — INSUFFICIENT. Constant span=4 still jitters; the
   jitter is the arm-parity variance, upstream of the span.
4. **arm-parity align** (`mc-r1-dmc-arm-align`) — REGRESSION. Gating the reload arm to a fixed
   `put_cycle` parity defers ALL DMA arms 1 cyc → breaks the 6/10 (the 6/10 are arm-cycle-sensitive).

**Conclusion forcing this effort:** any arm-phase/span manipulation that cancels the jitter ALSO
shifts the DMA timing the 6/10 rely on. The arm-parity variance is intrinsic to F-2 (the DMC timer on
a phase that the interleaved variable stall lets drift). Only re-coupling the timer to the integer
clock can make the period exact while keeping the arm cycles where the 6/10 expect them.

## Central open question (Phase 1 answers this)

**Why is the default lockstep DMC period EXACT while R1 jitters, when BOTH clock the DMC timer once
per CPU cycle (incl. during the DMA span)?** Hypotheses:
- (H1) Default services the DMA INLINE at the buffer-empty read (`drain_dma(Some)` from `cpu_read`),
  so the arm-to-service latency is 0; R1 arms at end-of-cycle (`tick_dmc`) and services in the next
  `read1` loop iteration, adding a variable instruction-stream-dependent latency.
- (H2) Default's DMC timer counts on `apu_phase` tied to `tick_one_cpu_cycle` (one integer clock with
  the span); R1's `!put_cycle` is the same parity BUT the arm/service split re-introduces drift.
- (H3) The buffer-empty CPU cycle itself is exact in both; only the SERVICE (arm→read1) jitters.

Phase 1 instruments the DMC timer/bits + arm/service cycles in `trace_dma_4015` and diffs default vs
R1 cycle-by-cycle across one fetch to pin which of H1/H2/H3 holds. The re-coupling design follows from
the answer (e.g. if H1: drive the DMA service inline at buffer-empty under R1, not via the deferred
`read1` loop; if H2/H3: drive the timer from `master_clock` deltas).

## Phase 1 RESULTS (2026-06-03) — the timer is EXACT; the effort REDIRECTS to span-constancy

Measured from the existing `trace_dma_4015` data (no new instrumentation): split the period into the
**arm** (where `dmc_dma_pending` goes 0→1 = buffer-empty, fired the same cycle `needs_dma` becomes
true) vs the **service** (where `in_dmc_dma` goes 0→1 = the `read1`-loop DMA start).

```
                 ARM-to-ARM (timer/buffer-empty)   SERVICE-to-SERVICE (read1 start)
R1 baseline:     {432: 175}   <- EXACT             {429..435}   <- ALL the jitter
DEFAULT:         {432, 450}   <- exact             {432 ±1}     <- nearly exact
arm->service latency:  DEFAULT {1:417,2:32,3:16}   R1 {1:183,2:8,3:7,4:6}
```

**Finding (pivots the effort):** the DMC timer / buffer-empty is **already EXACT (432) under R1** —
H2 (timer jitters) is REFUTED. `service_period = arm_period(432) + (latency_delta)`, so the jitter is
**entirely the arm→service latency** (H1/H3 confirmed). Mechanism: of the 432 CPU cycles between
arms, the DMA span (~4) is CPU-halted, so the CPU executes `432 − span` INSTRUCTION cycles between
arms. A **variable span** (3 vs 4) varies that instruction advance → the next arm lands at a different
instruction offset → the "wait for next read" latency jumps → service period jitters (and a single
odd span shifts the stream for a RUN of subsequent fetches until corrected — why 7 short-span-3s in
the fixed-span experiment produced ~35 off-period fetches).

**Effort REDIRECT:** do NOT re-couple the timer (it's exact). The lever is **span-constancy** — make
R1's per-fetch DMA span consume the SAME number of CPU cycles as default's at that fetch, so the
inter-arm instruction advance matches default → the arm→service latency is constant → service period
exact → the DMA lands on the test-positioned events. The fixed-span experiment approximated this
(short?3:4) but still emitted odd short spans at points that diverge from default's pattern.

**Revised phases:**
- **Phase 2 — SPAN MATCH (behind `mc-r1-dmc-spanmatch`, default-off):** instrument the per-fetch span
  + `dmc_dma_short` in BOTH default and R1, diff them, and drive R1's span to default's per-fetch
  value. The 6/10 must hold (they observe right after DMASync; matching default's span keeps the arm
  cycles where they expect). Gate: R1 service-period histogram collapses to default's; 6/10 + C1 hold.
  NOTE: the arm-align regression showed any change to arm CYCLES breaks the 6/10 — span-match must
  change span LENGTH only, leaving the arm cycle (buffer-empty, already exact) untouched.
- **Phase 3 / 4:** unchanged (abort truncation + OAM overlap; converge + promote).

Phase 2 first step = a per-fetch span+short diff (default vs R1) to find exactly which fetches
diverge, before touching the span logic.

### Phase 2 first-diagnostic (2026-06-03) — span-constancy ALSO refuted; jitter is a get-alignment oscillation

Before touching span logic, checked WHERE the fixed-span residual off-periods fall. Result: they are
**all `len=4`** and occur **far from any span-3** (no preceding span-3 within 99 fetches), in net-zero
**±k pairs** (435↔429, 434↔431, …). So the ±3 jitter is an **intrinsic period-2 oscillation in the
arm→service latency that PERSISTS with a constant span** — it is NOT caused by span variation.
**Span-constancy is refuted as the fix** (would not have helped; the fixed-span experiment already
showed it).

**New suspect (next investigation):** the R1 DMA service in `Cpu::read1` runs `dmc_dma_step` whose GET
is gated on a **get cycle** (`!put_cycle`). So the arm→service latency depends on the arm's parity vs
the next get-cycle read — alternating ±1-2 by parity = the period-2 oscillation. Default's burst
service (`service_dmc_dma`) is NOT get-cycle-gated (fixed `noop_cycles` then get), so its latency is
constant. **Refined hypothesis:** the jitter is the get-cycle alignment of the interleaved DMA
service, NOT the timer and NOT the span. BUT — `arm-align` proved that forcing the SERVICE/arm to a
fixed parity shifts the DMA cycle and breaks the 6/10. So the fix must make the latency constant
WITHOUT moving the DMA cycle the 6/10 observe — the same tension, now localized to the get-alignment
of the read1-loop service.

**Status:** four localized levers + two re-coupling-premise hypotheses (timer-jitter, span-constancy)
now refuted. Next: trace the next-read parity after each arm.

### Phase 2 next-read-parity trace (2026-06-03) — jitter = a ~1-cycle DMA-SERVICE PHASE offset

Traced the arm→service window (access type + parity) for every arm, default vs R1, with REAL CPU-cycle
parity (`cyc&1`; note `put_cycle` is meaningless on the default build — only toggled under R1's
`dmc_driven_externally`):

```
            latency dist             arm cyc&1   service cyc&1
DEFAULT:    {1:417, 2:32, 3:16}      ~even       ~odd
R1:         {1:183, 2:8, 3:7, 4:6}   ~even       ~odd
```

Both put arms on EVEN cycles and services on ODD, latency mostly 1 — the parity STRUCTURE is identical
(so it is NOT a simple get-gating offset; the earlier get-alignment guess is too coarse). The decisive
difference: **R1 has `lat=4` events (×6) that default NEVER has** (default caps at `lat=3`). The window
shows these are arms landing right before a **3-idle-cycle run** (e.g. `[R Ip Ig Ip r*]` — the DMA waits
3 CPU internal cycles for the next read). Since `service_period = 432 + (lat[i+1] − lat[i])`, a
`1→4` jump is +3 and the `4→1` return is −3 = exactly the observed net-zero ±3 pairs.

**Pinned mechanism:** the jitter is a **~1-cycle DMA-service PHASE offset** between R1 and default. The
arm period is exact (432) in both, but R1's service lands ~1 cycle off default's, so the 432-periodic
arm occasionally falls just before a 3-idle-cycle instruction (a multi-cycle internal op) where default's
arm falls on/next-to a read → R1 gets `lat=4`, default doesn't → ±3 jitter. NOT the timer, NOT the span,
NOT get-gating — the service PHASE.

**The tension, now precisely scoped:** the fix is to align R1's DMA-service phase to default's (remove
the ~1-cycle offset) so the arm lands where default's does. But `arm-align` proved that moving the
service/arm cycle to a fixed parity shifts ALL DMA cycles and breaks the 6/10 (they observe the DMA
right after DMASync). So the fix must shift R1's service phase to match default's EXACT phase (not an
arbitrary fixed parity) — i.e. replicate default's `drain_dma`-from-`cpu_read` service timing under the
interleaved R1 model, rather than the `read1`-loop's "service before the next instruction read." That is
a service-model alignment (Phase 2 proper), the genuine remaining work.

**Candidate Phase-2 designs (next session):**
- (i) Service the DMC DMA at the SAME access-point default does (inline in the bus read, the cycle the
  buffer-empty read occurs) instead of the `read1` pre-read loop — match default's phase exactly.
- (ii) Capture default's exact arm→service offset per fetch (trace) and reproduce it as a deterministic
  service delay under R1, validated to leave the 6/10's DMA cycles unmoved.
Both must be gated + 6/10-verified per the gates above. The arm period being exact means this is a
PHASE alignment, not a period/timer fix — a meaningfully smaller target than the original "re-couple
the timer" framing.

**Hypotheses refuted to date (7):** dummy-read, abort-cancel, fixed-span, arm-align, timer-jitter,
span-constancy, coarse-get-gating. The jitter is a ~1-cycle service-phase offset producing occasional
`lat=4` idle-waits. Banked 6/10 (`5ceab83`) untouched throughout; default 94.24% byte-identical.

## Phase B (abort truncation) — BLOCKED: the abort never lands on the DMA (~35-cycle setup divergence)

Taking on Phase B (build on `a8e482e`) revealed the truncation logic is **moot** — the abort never
coincides with a DMC DMA under R1, so it never schedules. Evidence (Explicit-abort trace, default vs
R1+idle-halt):

| | abort writes | abort-schedule flag fires | aborts with DMA within 4 cyc | nearest-DMA offset range |
|---|---|---|---|---|
| **DEFAULT** | 28 | **13** | **8** | `-6..+3` (the X-sweep catches the DMA) |
| **R1 + idle-halt** | 9 | **0** | **0** | `-35 .. -446` (DMA never in sweep range) |

So `schedule_explicit_dmc_abort_if_needed` (which needs the DMC mid-DMA at the `$4015` write) NEVER
fires under R1 — there is no DMA to abort. The abort-truncation port (Mesen `processCycle::if
(_abortDmcDma)`) cannot be exercised until the DMA LANDS on the abort.

**Crucially this is NOT the steady-state ~1-cycle jitter** (idle-halt fixed that — R1+idle-halt latency
`{1:183,2:3,3:1}` is actually tighter than default's `{1:417,2:32,3:16}`; both long-waits are `RWWr`
write-pairs). It is a **~35-cycle absolute-position divergence specific to the abort test's setup**:
after `DMASync`, the test does `LDA #$4E; STA $4010` (rate change to looping rate `$E`) + a long
clockslide chain calibrated for default's DMA timing; R1's DMA ends up ~35-450 cyc off the abort write
where default's lands within the ±6 sweep. `DMA + Open Bus` (also DMASync-based) PASSES because it
observes the DMA immediately after DMASync (no long clockslide to accumulate the divergence).

**Phase B refinement:** the blocker is the **abort-test-setup DMA position** (the ~35-cycle offset),
NOT the truncation logic and NOT the steady-state period. Next-step for B = trace the abort-test setup
(DMASync exit → `$4010` rate change → the setup DMAs) default-vs-R1 to pin where the ~35-cycle offset
accumulates (candidate: the rate-change `$4010` handling or a load/setup DMA whose span/arm R1 still
