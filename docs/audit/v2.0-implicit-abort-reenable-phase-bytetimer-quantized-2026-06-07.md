## What was ported (the TriCNES re-enable exclusion)

`Emulator.cs` `DMCDMA_Get` (`:4168`) sets `CannotRunDMCDMARightNow = 2` after
EVERY DMC GET; it is decremented by 2 on each get cycle (`:1186`) and gates the
looping-reload arm (`:1165`, blocked while `== 2`). This is the canonical "a DMA
cannot occur within 2 cycles of a previous DMC DMA" rule the Implicit-DMA-Abort
Loop3/`$540` Key3 plateau explicitly brackets (`AccuracyCoin.asm:13082`).

The faithful RustyNES port (gated on `mc-r1-dmc-reenable-phase`):

* `cannot_run_dmc_dma: u8` set to 2 in `complete_dmc_dma` /
  `complete_dmc_dma_before_get_tick` (the GET delivery), decremented by 2 on
  each `apu_phase` (get) cycle in `tick_with_external` after the reload arm,
  gating `dmc_step_reload_arm` while `== 2`.

### Probe result: the exclusion SCOPES EXACTLY to X=10/11

Gated `cannot_run_block` probe over the full battery: the exclusion blocks a
would-be reload arm **6 times** — precisely the X=10/11 entries of the three
Implicit-abort loops (`$500`/`$520`/`$540`), and **nowhere else**. The boundary
is therefore correctly identified: the steady-state 576-cycle-spaced reloads
always see the exclusion at 0; it only ever bites at a LOAD→reload boundary.

## Why no scalar realignment closes it — the EXHAUSTIVE sweep (the proof)

The bare exclusion alone is **absorbed** (RustyNES re-checks the persistent
`needs_dma()` every cycle, so it re-arms 2 cycles later → `$540[10,11]=$03`
unchanged). A full-byte deferral **overshoots** (`$540[10,11]=$00`, Y=0). To
test every realignment magnitude, the boundary applies a swept one-time
byte-timer phase shift (`REENABLE_BUMP`, env `RUSTYNES_REENABLE_BUMP`, in
APU-rate timer units; default 0 = bare gate). The sweep of `$540[10,11]`
(other entries unchanged — the realignment is correctly scoped):

| `REENABLE_BUMP` | `$540[10,11]` (= Y) |
|---:|:---|
| -5 | `0D 0D` (13) |
| -4 | `0B 0B` (11) |
| -3 | `09 09` (9) |
| -2 | `07 07` (7) |
| -1 | `05 05` (5) |
| **0** | **`03 03` (3)** — baseline |
| +1 | `49 49` (73) |
| +2 | `47 47` (71) |

**Y is quantized to ODD values (3, 5, 7, 9, 11, 13 …) and never lands on the
even Y=4 the answer key wants.** Each byte-timer unit = 2 CPU cycles, so the
realignment moves the X=10/11 catch in steps of 2 walk iterations. Positive
bumps wrap the catch around the byte period (Y≈73). **Y=4 (even) is
fundamentally unreachable on the APU-rate (every-other-CPU-cycle) byte-timer.**

This is the cycle-level proof of the pin's conclusion: the residual is a 1-CPU-
cycle (half-APU-cycle) sub-position effect — the GET-vs-resumed-`LDA $4000`
alignment at iteration 3, where TriCNES's GET preempts the operand-high fetch
(re-driving `$40`, no catch → Y=4) and RustyNES's GET is adjacent to the data
read (catch → Y=3). The byte-timer's 2-CPU-cycle granularity cannot express the
1-CPU-cycle separation, so the master-clock (1-CPU-cycle-resolution) model is
required — consistent with the φ2-open-bus refutation and the documented
18-rollback history.

Closing `$540[10,11]` requires the GET to land at the correct 1-CPU-cycle
