# v1.3.x left-edge video + stutter regression — root-cause audit

**Date:** 2026-05-24
**Scope:** two user-reported regressions in the RustyNES v2 frontend/PPU
**Status:** video bug ROOT-CAUSED + FIXED + proven; stutter root-caused to a
present-mode/pacer interaction with a targeted fix (not verifiable headless).

---

## 1. Symptoms (as reported)

1. **Video (primary):** a green tint / garbage in the leftmost ~1-2 columns
   (8-16 px) of the screen in many games. Super Mario Bros. shows greenish
   sprites while scrolling; Castlevania shows green-shaded content on the
   title/menu and garbage while scrolling. Affects many games.
2. **Performance (secondary):** fractional-second pauses / visible jitter
   during gameplay, across many games.

---

## 2. Video bug — ROOT CAUSE

### 2.1 The introducing commit

`086ce4d` ("fix(ppu): BG shift-register cycle-9 reload + post-emit shift",
session-8) converted the **BG pattern** pipeline to the Mesen2-faithful model:

- reload at phase 0 of each 8-cycle group (`reload_bg_shift_regs` at PPU dots
  1, 9, 17, …),
- shift AFTER `emit_pixel` (`DrawPixel` then `ShiftTileRegisters` per Mesen2
  `NesPpu.cpp::ProcessScanlineImpl` lines 881-884),
- an explicit `<<= 8` at the pre-fetch boundary (dots 328 / 336) to substitute
  for the per-cycle shifts that do not run during the dots 321-336 pre-fetch
  region (Mesen2 `NesPpu.cpp` lines 941-944).

That commit applied all three changes to the **16-bit BG pattern** shift
registers (`bg_shift_lo`/`bg_shift_hi`) **but left the attribute (palette)
pipeline on the OLD 8-bit `at_shift_lo`/`at_shift_hi` (u8) + 1-bit
`at_feed_lo`/`at_feed_hi` model.**

The two pipelines then advanced at *different rates* across the dots 321-336
pre-fetch boundary:

- The 16-bit pattern register gets the explicit `<<= 8` at dots 328/336, which
  moves the pre-fetched leftmost tile's data from bits 0-7 up to bits 8-15 so
  it is positioned at the read point (bit `15 - fine_x`) when the next
  scanline begins at dot 1.
- The 8-bit attribute register has **no equivalent pre-fetch advance** — during
  the pre-fetch region `shift_bg` does not run, so the attribute register is
  frozen at its dot-256 state. At the next scanline's dot 1 the attribute read
  samples a stale value for the leftmost tile.

Net effect: the **palette (attribute) bits drift one tile out of phase with
the pattern bits in the leftmost tile column (screen columns 0-7)**, producing
the wrong palette there. "Greenish" is the literal symptom because the
mis-sampled attribute frequently selects palette 0 → a green/teal NES color
instead of the correct grey/skin/brown.

### 2.2 Hardware reference defining correct behavior

- nesdev wiki "PPU rendering": the attribute shift registers are 8-bit but
  fed in lockstep with the pattern shifters; the canonical model used by
  accurate emulators keeps them aligned through the pre-fetch.
- Mesen2 `NesPpu.cpp::GetPixelColor` reads the BG color from the pattern
  shifters at `(_lowBitShift << offset) & 0x8000`, i.e. bit `15 - fine_x` of a
  16-bit register, and derives the attribute palette from
  `_previousTilePalette`/`_currentTilePalette` selected by
  `offset + ((_cycle - 1) & 7) < 8` — exactly the per-tile boundary our 16-bit
  AT shifters reproduce structurally. The decisive invariant is that the
  attribute MUST be sampled at the same per-pixel tile boundary as the pattern.

### 2.3 Empirical proof (headless capture, before/after)

Tooling: `crates/nes-test-harness/src/bin/capture_left_edge.rs` (diagnostic
bin; reuses the gitignored external ROM dumps; dumps full-frame + 8×-zoom
leftmost-32-columns PNG). A worktree pinned at the introducing-commit tree was
captured for the same frames and diffed pixel-for-pixel.

| ROM | frame | differing pixels | location | buggy → fixed (sample) |
|---|---|---|---|---|
| Castlevania (UxROM) | 500 | 256 | **cols 0-7, all rows** | green `(48,200,60)` → grey `(185,188,185)` |
| Super Mario Bros (NROM) | 900 | 47 | **cols 0-3** | green `(0,143,0)` → brown `(139,90,0)` |
| Balloon Fight (NROM) | 600 | 2 | **cols 0-7** | green `(48,200,60)` → white `(255,255,255)` |

In every case the difference is confined to the **leftmost tile column
(columns 0-7)** and is purely a palette swap — exactly the reported "green
left edge." The buggy left-edge crop visibly renders Castlevania's grey border
dashes as bright green; the fixed crop renders them grey.

### 2.4 The fix

Make the attribute shifters 16-bit and advance them in **exact lockstep** with
the pattern shifters (`crates/nes-ppu/src/ppu.rs`):

- `at_shift_lo`/`at_shift_hi` widened `u8 → u16`; `at_feed_lo`/`at_feed_hi`
  removed.
- `reload_bg_shift_regs` expands the 2-bit attribute to a full `0x00`/`0xFF`
  byte into bits 0-7 (the slot the pattern bytes occupy) — `(shift & 0xFF00) |
  expanded`.
- `shift_bg` shifts all four registers by 1.
- the pre-fetch `<<= 8` (dots 328/336) is applied to all four registers via a
  new `prefetch_shift_bg_regs()` helper (also keeps `tick` under the
  `too_many_lines` lint).
- `emit_pixel` reads the attribute at the same `0x8000 >> fine_x` bit position
  as the pattern (was `0x80 >> fine_x` on the 8-bit register).

Save-state: PPU snapshot schema bumped **v1 → v2** (`snapshot.rs`); v1 blobs
are upconverted on read (low byte preserved, transient feed bits dropped —
regenerated within one scanline of resumed rendering, lossless in practice).
`state_trace.rs` AT fields widened to u16 to match; `ppu_trace_diff` display
width widened to `{:04X}`.

> NOTE: the AT-register fix was already present uncommitted in the working tree
> from a prior session (a partial, unverified diagnosis). This audit *verified*
> it (paired before/after game-frame diff + new hermetic regression tests),
> completed the save-state migration tests, and brought the change to a
> clean fmt/clippy/doc/test state.

### 2.5 Regression guards added

`crates/nes-ppu/src/ppu.rs` (tests):
- `bg_attribute_boundary_no_scroll` + `bg_attribute_boundary_tracks_pattern_under_scroll`
  — render a solid pattern with attribute boundaries and assert the palette
  boundary stays glued to the pattern at every fine-X/coarse-X scroll.
- `bg_attribute_register_lockstep_through_prefetch` — the bulletproof guard:
  drives `reload_bg_shift_regs` + `prefetch_shift_bg_regs` + `shift_bg` and
  asserts the AT registers stay bit-identical to the pattern registers through
  the pre-fetch boundary. This fails to even compile under the old u8 model.

`crates/nes-ppu/src/snapshot.rs` (tests):
- `snapshot_round_trips_16bit_attribute_shifters` — full 16-bit AT round trip.
- `snapshot_reads_v1_attribute_shifters_as_low_byte` — v1→v2 upconversion.

## 3. Performance / stutter — root cause + fix

### 3.1 What was ruled out (measured, headless, release build)

`capture_left_edge` records per-frame `run_frame()` wall-time; a
`RUSTYNES_REWIND=1` mode was added to exercise the frontend's real
rewind-enabled hot path.

- **Core throughput is not the cause.** SMB release: median **3.13 ms/frame**,
  p99 3.58 ms, max 3.83 ms — ~5× headroom under the 16.67 ms NTSC budget.
- **Rewind capture is not the cause.** With per-frame rewind enabled
  (keyframe every 60 frames), median rose only ~0.1 ms (3.13 → 3.20 ms) and
  the slowest frames were **not** keyframe-aligned (440, 327, 232…). The LZ4
  XOR-delta path is cheap; the keyframe spike is not the stutter source.

### 3.2 Root cause: wall-clock pacer vs `Fifo` vsync double-pacing

`crates/nes-frontend/src/app.rs::pace_frames` is the authoritative timing
source — it paces frames on a wall clock at NTSC **60.098 Hz** and schedules
`ControlFlow::WaitUntil`. But `crates/nes-frontend/src/gfx.rs` **hardcoded**
`present_mode: wgpu::PresentMode::Fifo` (vsync), **ignoring the user's
configured `graphics.present_mode`** (the config field existed and supported
`"Mailbox"`, but `gfx.rs` never read it).

With `Fifo`, the surface ALSO gates presentation on the display refresh. On a
60.000 Hz panel the 60.098 Hz pacer and the 60.000 Hz vsync beat against each
other and drop/double one frame roughly every ~10 s — the classic
emulator-on-vsync "fractional-second pause" stutter, present across all games.

### 3.3 The fix

- `gfx.rs`: added `select_present_mode(pref, supported)` and wired the config
  `graphics.present_mode` through `Gfx::new(window, present_mode_pref)`. It
  maps `"mailbox"`/`"immediate"`/`"fifo"` (case-insensitive), validates against
  the surface's advertised modes, and **falls back to `Fifo`** when the
  requested mode is unavailable (`Fifo` is guaranteed on every backend).
- `config.rs`: default `present_mode` changed `"Fifo" → "Mailbox"` so the
  wall-clock pacer is the single timing authority and the vsync beat is
  removed for users without an explicit config. `Mailbox` presents the latest
  frame without a vsync gate and without tearing.
- `app.rs`: both native and wasm `Gfx::new` call sites pass the config value.

### 3.4 Honesty note / what could not be verified here

The agent ran headless (no display), so the stutter could not be *observed*
before/after. The fix is the standard, well-understood remedy for the
double-pacing beat and is low-risk (transparent `Fifo` fallback). **Please
verify on a real display.** If stutter persists after this change, the next
suspects (in order) are: (a) audio-queue underrun stalling the cpal callback
(`MAX_QUEUED_SAMPLES = 16384`, ~0.37 s), (b) per-frame `snapshot()` allocating
a fresh thumbnail every frame when rewind is enabled (`nes.rs::snapshot` calls
`thumbnail()` which does a full-framebuffer `vec![0u8; …]` + rescale every
frame — wasteful for rewind, which does not need the thumbnail), (c)
`desired_maximum_frame_latency` interacting with the pacer.

## 4. Baselines changed (with justification)

### 4.1 Commercial-ROM oracle insta snapshots — UPDATED (17 of 60)

`crates/nes-test-harness/tests/snapshots/external_real_games__*.snap` for:
batman_return_of_the_joker, mr_gimmick, final_fantasy, legend_of_zelda,
mega_man_2, kirbys_adventure, fire_emblem, uchuu_keibitai_sdf, famista_91,
balloon_fight, super_mario_bros, castlevania, ganbare_goemon, twinbee_3,
crisis_force, ganbare_goemon_2, ganbare_goemon_gaiden.

**Justification (per the no-blind-rebaseline guardrail):** each changed
snapshot differs **only in `fb_fnv1a64`** — `rom_sha256`, `cycles`,
`audio_samples`, and `audio_fnv1a64` are byte-identical (verified
programmatically across all 17). The framebuffer change is the *correct*
left-edge palette fix, proven by the paired before/after game-frame pixel diff
(§2.3) and visual inspection of the corrected frames (clean left edge; sacred
trio SMB/Excitebike/Kid Icarus legible). Updating these baselines records the
correct output, not a regression mask.

### 4.2 `screenshots/` PNG corpus — NOT regenerated (left for reviewer)

The 89-PNG `screenshots/` corpus is a **non-asserted visual reference** (only
written when `RUSTYNES_DUMP_DIR` is set; no test reads it). It is now stale for
the ~17 affected ROMs (still shows the buggy green left edge). Per the
guardrail it was deliberately **not** mass-regenerated. Recommend regenerating
it as a follow-up once this fix is reviewed:
`RUSTYNES_DUMP_FRAMES=1 RUSTYNES_DUMP_DIR="$PWD/screenshots" cargo test -p nes-test-harness --features test-roms,commercial-roms`.
