## The exact, unified target (from option-(a) ROM disassembly)

`AccuracyCoin.asm`: `TEST_DMCDMAPlusOAMDMA` (12963), `TEST_ExplicitDMAAbort` (13086), and
`TEST_ImplicitDMAAbort` (13157) **each** open with `JSR CheckDMATiming; CPY #4; BNE FAIL` — a NORMAL
(un-aborted) looping-`$4E` DMC DMA, measured by `CalculateDMADuration`, must yield **Y == 4** (where
Y = iterations of a loop reading `LDA $4000` + stalling 560/iter, the read walking 1 cyc earlier per

`TEST_Fail` stores `(ErrorCode<<2)|2` and `ErrorCode` defaults to 1 ⟹ a gate failure stores **0x06**.

- **DEFAULT: Y = 4 (pass). R1: Y = 3 (fail).** So three of the four fails collapse to ONE condition:
  **make R1's `CheckDMATiming` Y = 4.** (The 4th, `DMA + $4015` = `0x0A` = ErrorCode 2, fails one step
  later — re-check separately after Y==4.)
- **LIVE METRIC:** `scan_dma_abort` prints `CheckDMATiming Y` (read from `$50` at the `$0477`-set frame).
  Target: **Y 3 → 4.** Then the three tests' sweeps run; then wire the abort truncation; then `$4015`.

## Why no shortcut exists (proven in (b); do NOT re-try)

R1's GET-conflict is **1 cycle late** vs default's (Y=3 vs 4). Every localized 1-cycle shift fails:
- **form A — fire the GET 1 cyc earlier (off get-parity):** WEDGES the whole battery (the get-parity GET
  is load-bearing for the DMASync `$4000`-conflict / un-wedge / 6/10).
- **form B — shift the arm parity (`arm-align`/`arm-delay1`) or legacy inline-service:** breaks the 6/10.
- **fixed-offset GET (`phase-walk`):** inert (the span-start/arm is itself parity-locked, so a fixed
  offset from it stays locked) — Y unchanged.
- **idle-halt on/off, get-parity flip, rdy-heldaddr, dummy-read, abort-scheduling cuo:** all inert/refuted
  (see the kickoff + `project_v2_from_main_pivot` memory).

**Root (resolved):** R1's GET is 1 cyc late because the SERVICE starts 1 cyc late — the arm
(`dmc_dma_pending → true`) is detected in `tick_dmc` at **end_cycle N** (after that cycle's bus access),
and serviced in the **next** `read1`/`idle_tick` `while dmc_dma_pending()` loop (cycle N+1). Default's
burst detects the buffer-empty DURING the triggering `cpu_read` and services it INLINE in the SAME cycle
(0 latency), with a fixed-offset GET (`noop_cycles = short?2:3` then unconditional get) → Y==4 AND the
un-wedge together.

## The rewrite (the only remaining path)

**Reproduce default's inline-at-buffer-empty DMC service on the R1 master-clock substrate, WITHOUT the
legacy `tick_one_cpu_cycle`** (which broke lockstep timing because it bypasses R1's `start_cycle` /
`run_ppu_to` / `end_cycle` access shape). Concretely: the DMC DMA must be detected + serviced in the SAME
CPU cycle the buffer empties (0 arm→service latency), so the GET lands at default's cycle (Y==4), while
keeping (i) the `5ceab83` fire-phase that un-wedges DMASync, (ii) the 6/10, (iii) C1, (iv) no wedge.

### The central design tension (the thing to solve)

The arm/buffer-empty is currently known only at **end_cycle** (`tick_dmc` advances the DMC timer on
`!put_cycle`, the `5ceab83` fix). Inline service needs it known **before/at the access** of the same
cycle. So the rewrite must move the buffer-empty DETECTION earlier WITHOUT moving the timer's fire-PHASE
(which un-wedges + holds C1). Candidate approaches (R1i-1 picks one via the live Y metric):
- **(α) pre-detect:** at `start_cycle` (before `bus.read`), check whether the DMC timer WILL roll over
  this cycle (timer == 0 at the current phase) and, if so, service the DMA inline before the read — while
  still advancing the timer at end_cycle for the fire-phase. (Detection moved early; fire-phase unchanged.)
- **(β) split timer:** advance the timer at start_cycle for the buffer-empty detection but keep the
  IRQ-line / frame-counter sample at end_cycle (the C1 phase). Mirrors the F-2 decouple at the cycle level.
- **(γ) service-at-arm:** service the DMA in `end_cycle` immediately when `tick_dmc` arms it (same cycle as
  the buffer-empty), inserting the halt cycles there, so the next CPU read is the post-DMA read (matching
  default's "DMA inserted before the triggering read completes"). Requires care: end_cycle already did the
  access, so this models the DMA as occurring at the boundary — verify against default's exact cycle.

Each candidate keeps the GET at default's FIXED offset (`noop_cycles = dmc_dma_short()?2:3` then
unconditional get) once the service start is inline. The live Y metric (`scan_dma_abort`) tells you
immediately which candidate yields Y==4.

## R1i-0 + R1i-1 RESULTS (2026-06-04) — Y is invariant to DMA length; only the GET PARITY moves it, and that wedges

**R1i-0 (per-cycle oracle diff) DONE.** Dumped a CheckDMATiming DMA span default vs R1 (`ck_{def,r1}.csv`):
```
DEFAULT: arm@60 (R $FF4D, dmcP=1) | idle@61 (I) | halt@62-63 (r $FF4E) | GET@64 (r $FFC0=$00)   arm->GET=4
R1:      arm@63 (R $FF4D, dmcP=1) |  --no idle-- | halt@64-65 (r $FF4D) | GET@66 (r $FFC0=$00)   arm->GET=3
```
Default has an intervening IDLE before the DMA halts on the NEXT read (`$FF4E`); R1 halts immediately,
re-reading the arm-read's addr (`$FF4D`). So arm→GET = 4 (default) vs 3 (R1).

**R1i-1 (insert 1 alignment cycle, `mc-r1-dmc-inline-service`) ATTEMPTED → REFUTED.** Made R1's arm→GET = 4
by adding one alignment halt at span start. Result: **`CheckDMATiming Y` STAYED 3, AND `$4016` regressed
0x09→0x06 (a 6/10 break).** Reverted (byte-identical). **KEY NEGATIVE RESULT: Y is INVARIANT to DMA span
length** — the alignment cycle halts the CPU one extra cycle, shifting BOTH the GET and the walking
`CalculateDMADuration` read equally, so their relative phase (Y) is unchanged; meanwhile the extra cycle
moved `$4016`'s conflict landing (6/10 break). So the arm→GET=4-vs-3 from R1i-0 is NOT the Y lever.

**The Y lever is ONLY the GET's PARITY relative to the walking read** (form A confirmed: shifting it
wedges; R1i-1 confirmed: shifting DMA length is inert for Y + breaks 6/10). And default achieves Y=4 with
its burst GET on a DIFFERENT parity than R1's interleaved get-parity — default's burst DMASync works on
that parity, but R1's interleaved DMASync is parity-locked to the get-cycle (shifting it wedges R1's, even
though default's works on its own). **Conclusion (re-confirmed at the finest grain): the only fix is to
reproduce default's BURST service model — fixed-offset GET at default's parity, with default's DMASync-
conflict behavior — on the R1 substrate. This is NOT a cycle added to the interleaved model (inert/breaks
6/10) and NOT a GET-parity shift within it (wedges); it is the burst-model port done correctly (the legacy
`tick_one_cpu_cycle` port broke lockstep timing — the rewrite must use R1's start/run_ppu_to/end access
shape). R1i-1's alignment shortcut is refuted; the rewrite is candidate (β)/(γ): change WHEN the
buffer-empty is detected + service so the GET lands at default's parity WITHOUT shifting the read loop.**
This is the genuine multi-week core; R1i-0/R1i-1 narrowed it further (Y is parity-only, length-invariant).
