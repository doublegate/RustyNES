## What landed

### Trunk build validated end-to-end (the Sprint 1.2 deferred item)

Sprint 1.2 couldn't run `trunk build` because `cargo install
--locked trunk` failed on a libdeflate C-compiler issue. This sprint
worked around it by downloading the **pre-built trunk 0.21.14
binary** from GitHub releases (`trunk-x86_64-unknown-linux-gnu.tar.gz`),
which doesn't need to compile libdeflate-sys from source.

Two issues surfaced + fixed during the first real `trunk build`:

1. **Two target artifacts** (`nes_frontend` lib + `rustynes-v2` bin)
   — trunk couldn't pick. Fixed by adding `data-target-name="nes_frontend"`
   to the `<link data-trunk rel="rust">` in `index.html`.
2. **wasm-bindgen CLI/lib version mismatch** — `Trunk.toml` pinned
   the CLI to 0.2.100 but `Cargo.lock` resolved the library to
   0.2.121 (the caret `"0.2.100"` allows up to 0.2.x). The CLI and
   library version-check each other; fixed by aligning `Trunk.toml`
   to `wasm_bindgen = "0.2.121"`.

`trunk build --release` now succeeds: downloads `wasm-opt
version_123`, produces `dist/` with `index.html` + `.wasm` + `.js`
glue.

### Canvas-2D browser emulator (`src/wasm.rs`)

The pragmatic MVP path — a working browser NES emulator NOW, without
waiting for the full winit/wgpu/cpal/egui wasm32 port:

- **ROM loading** via `<input type="file">` + `FileReader` (raw
  web-sys, no crate wrapper). On selection, reads the bytes and
  instantiates `Nes::from_rom`.
- **Video** via the canvas 2D `ImageData` path. The PPU framebuffer
  is already RGBA8 256x240 — byte-identical to `ImageData`'s layout
  — so `ctx.put_image_data` is a direct blit, no conversion.
- **Keyboard input** via document keydown/keyup listeners mapping to
  player 1 (arrows = D-pad, Z = A, X = B, Enter = Start, Right Shift
  = Select — mirrors the native default keymap).
- **Run loop** via `requestAnimationFrame` using the classic
  wasm-bindgen `Rc<RefCell<Option<Closure>>>` self-reschedule
  pattern. Each tick: push buttons → `run_frame` → blit framebuffer.

The shared `Nes` + button state lives in a `thread_local!`
`Rc<RefCell<Emu>>` since web-sys event closures are `'static`.

### Size: 280 KB release wasm (0.09 MB gzipped)

Far under the 5 MB compressed budget. Why so small? The canvas-2D
`start()` never references the native `app`/`gfx` modules, so
dead-code elimination strips winit + wgpu + egui + cpal + gilrs
entirely from the final artifact. (Those crates still COMPILE as lib
deps, but the linker drops them.) When Sprint 1.4 unifies with
winit/wgpu, the size will grow to the multi-MB range — still within
budget per the Sprint 1.1 research.

### CI rustdoc failure fixed

The Sprint 1.2 lib restructure (binary-only → `[lib] + [[bin]]`)
exposed `nes-frontend`'s internals to `cargo doc --no-deps`, which
then failed under `-D warnings` on broken intra-doc links:

1. `debugger/mod.rs` linked `[`memory_panel`]` / `[`mapper_panel`]`
   / `[`input_rebind_panel`]` — private modules. Fixed by converting
   all 7 panel-module references in the overview doc-comment to plain
   code spans (they're implementation detail, not public API).
2. `lib.rs` linked `[`wasm::start`]` — the `wasm` module is gated
   `#[cfg(target_arch = "wasm32")]` so it's absent from native
   rustdoc. Fixed by making it a code span.

This was the `CI / rustdoc (push) Failing after 51s` failure the
maintainer flagged. Now green.

## NOT validated (environment limitation)

I can't open a browser in this environment, so the emulator was NOT
interactively tested (ROM load → visible frames → keyboard response).
The build is fully validated (trunk produces a valid artifact; the
code compiles + lints clean) but the runtime behaviour in a browser
is unverified. The first action item for whoever has browser access:

```bash
cd crates/nes-frontend/web/
/path/to/trunk serve   # or `trunk build` + serve dist/ statically
# Open http://127.0.0.1:8080
# 1. Check devtools console for the boot log lines.
# 2. Load a .nes ROM via the file picker.
# 3. Verify frames render + keyboard controls respond.
```

If frames don't render, likely suspects: the `ImageData` color
order (the PPU framebuffer is RGBA — confirm it's not BGRA), or the
rAF loop not firing (check the console for panics).

## Architecture note: canvas-2D vs winit/wgpu

The v2.0.0 plan specified winit + wgpu on wasm32. This sprint
deliberately took the canvas-2D shortcut to get a WORKING browser
emulator sooner, accepting that it's a SECOND rendering path
distinct from the native wgpu path. The trade-off:

- **Pro**: working browser emulator in one session; 280 KB artifact;
  no winit/wgpu/cpal wasm32 API wrestling.
- **Con**: the egui debugger overlay + NTSC filter don't work on
  the canvas path; it's architectural debt until Sprint 1.4
  unifies the paths.

The canvas path is retained as a lightweight "embed mode" even after
unification (a 280 KB emulator with no debugger is a legitimate
shipping target for a blog-embed or a minimal demo). The decision is
documented here for the eventual v1.3.0 ADR.
