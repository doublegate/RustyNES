## The pinned residual (recap)

The reload byte-timer is APU-rate (ticks every other CPU cycle, on `apu_phase`),
so its wrap — and thus the reload-DMA arm — is locked to `apu_phase` parity. The
`$540` X=10/11 reload halt therefore lands one CPU cycle later than TriCNES/Mesen:
the GET ends up adjacent to the `LDA $4000` DATA read (sees the GET's `$00` ->
Y=3) instead of preempting the operand-high fetch (re-drives `$40` -> Y=4). The
`REENABLE_BUMP` byte-timer-phase sweep is ODD-quantized (Y in steps of 2:
3,5,7,...) because each byte-timer unit = 2 CPU cycles, so the EVEN Y=4 is
fundamentally inexpressible by moving the byte-timer. The residual is a
1-CPU-cycle (sub-APU-cycle) master-clock sub-position effect.

## The model implemented

A **boundary-local 1-CPU-cycle DELAY of the reload arm**, at per-CPU-cycle
(sub-APU-cycle) resolution — the half-step the byte-timer phase cannot express.

1. **Boundary scope.** The `mc-r1-dmc-reenable-phase` `cannot_run_dmc_dma == 2`
   exclusion (TriCNES `CannotRunDMCDMARightNow`) precisely marks the X=10/11
   post-LOAD-GET reload boundaries (and only those). Per-cycle instrumentation
   (`SUBPOS_DIAG`, gated on `mc-r1-dmc-abort-probe`) showed two block patterns:
   * Pattern A (`$540[10,11]`, the WRONG ones): the reload buffer is already
     EMPTY at the block (`needs_dma()`), `cannot_run == 2`, on `!apu_phase`.
   * Pattern B (`$540[12,13]`, already correct): `buffer_full` here (the LOAD
     just refilled). Pre-arming pattern B moved `$540[12,13]` 04 -> 03 (wrong
     direction + wrong indices) — proving pattern A is the target.
2. **Direction.** Empirically, an EARLIER arm DECREASES Y while an earlier WRAP
   (the `REENABLE_BUMP` axis) INCREASES it — opposite axes. `$540[10,11]` needs
   Y UP, i.e. a LATER recognition.
3. **The lever.** At the pattern-A boundary, schedule the reload arm
   `SUBPOS_DELAY` CPU cycles out (default **1**) via a per-CPU-cycle countdown
   that suppresses the natural arm meanwhile. The `RUSTYNES_SUBPOS_DELAY` sweep
   (scan_dma_abort) is decisive:

   | `SUBPOS_DELAY` | `$540[10..15]` |
   |---:|:---|
   | 0 | `03 03 04 04 04 04` (baseline) |
   | **1** | **`04 04 04 04 04 04`** = Key3 |
   | 2 | `03 03 04 04 04 04` |
   | 3 | `04 04 04 04 04 04` |
   | 4 | `03 03 04 04 04 04` |
   | 5 | `04 04 04 04 04 04` |

   ODD delays close it, even delays don't — the inverse of the byte-timer's
   even-only quantization. The 1-CPU-cycle (odd) recognition delay supplies
   exactly the missing half-step. Default = 1.

## Why it is C1-safe (does NOT regress SH*/the cluster)

The fix moves NEITHER `apu_phase` (the C1 IRQ φ2 sample) NOR the byte-timer
phase globally. It is a per-CPU-cycle arm delay scoped by the `cannot_run`
exclusion to the X=10/11 post-LOAD-GET boundary ONLY (fires exactly 2x, on the
two `$540` pattern-A entries; pattern B and all steady-state reloads untouched).
This is the surgical opposite of the global `mc-r1-dmc-phase-minus1` (which
shifts every DMC GET and shatters SH* -> 83.45%/23).

## Reproduction

```
# Full battery (Implicit PASSES; 94.24%/8):
cargo test -p nes-test-harness --release --features test-roms,mc-r1-full-cpu \
  --test accuracycoin -- --nocapture

# $540 sweep (default delay = 1 -> Key3):
cargo build -p nes-test-harness --release \
  --features mc-r1-full-cpu,mc-r1-dmc-abort-probe --bin scan_dma_abort
for d in 0 1 2 3 4 5; do RUSTYNES_SUBPOS_DELAY=$d \
  ./target/release/scan_dma_abort tests/roms/accuracycoin/AccuracyCoin.nes 1700 \
  | grep 'IMPLICIT $540'; done
```
