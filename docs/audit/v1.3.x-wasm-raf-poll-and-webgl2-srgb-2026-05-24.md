## Issue 1 — wasm/GitHub-Pages: severe stutter + periodic brief total freezes

### Root cause

v1.3.2 moved the wasm frame loop into `RedrawRequested`
(`App::pace_and_produce_wasm`), which produces the due frames by
`web_time::Instant` delta and re-arms the next `requestAnimationFrame` via
`request_redraw()`. But the wasm idle/return paths were left on
`ControlFlow::Poll`:

- `pace_frames` (called from `about_to_wait`) set `Poll` on wasm;
- `start_nes` set `Poll` on wasm after the first frame kick;
- the bootstrap `run_wasm` set `Poll` and the pre-ROM `on_gfx_ready` branch
  never changed it.

On winit 0.30's web backend, `ControlFlow::Poll` reschedules the event loop
*immediately* (via `Scheduler.postTask` / `setTimeout(0)`). That is a
busy-loop, and it ran **in parallel** with the rAF-driven `RedrawRequested`
loop. Two schedulers competing for the single wasm main thread starve the
heavy `Nes::run_frame()` work → severe stutter. This is strictly worse than
the pre-v1.3.2 `WaitUntil` (which at least let the loop idle between wakes).

A second, independent freeze bug compounded it: `pace_and_produce_wasm`
early-returned on the `self.nes.is_none()` path **without** re-arming
`request_redraw()`. The rAF self-reschedule is the *sole* heartbeat of the
wasm frame loop; any tick that fails to re-arm permanently kills the loop and
freezes the canvas. (The known-good `wasm-canvas` embed path in `wasm.rs`
re-requests rAF unconditionally at the end of every tick — that is why the
user reports it as SMOOTH. It is the reference pattern.)

### Fix

Drive the wasm loop purely from rAF; idle on `Wait` everywhere else:

- `pace_frames` (wasm): `set_control_flow(ControlFlow::Wait)` instead of `Poll`.
- `start_nes` (wasm): `set_control_flow(ControlFlow::Wait)` instead of `Poll`.
- `on_gfx_ready` (wasm, pre-ROM branch): `set_control_flow(ControlFlow::Wait)`
  + first `request_redraw()` (so the rAF heartbeat is alive before a ROM
  loads, superseding the bootstrap `Poll`).
- `pace_and_produce_wasm`: the `request_redraw()` re-arm now runs
  UNCONDITIONALLY — production is gated behind `if self.nes.is_some()` but the
  re-arm is always reached, including the pre-ROM path. Verified there is
  exactly ONE `request_redraw` per rAF tick (no double-pump).

Native pacing (the `cfg(not(wasm32))` sleep-then-spin arm) is untouched.

### What could / could not be verified

- Verified headless: both wasm32 builds + both wasm32 clippy flavours green;
  single `request_redraw` per tick by inspection; native unchanged.
- Cannot verify headless: actual smoothness on Pages. **User must confirm in
  a browser.** The reasoning is grounded in winit 0.30's documented web
  `ControlFlow` servicing (`Poll` → immediate `postTask`/`setTimeout(0)`
  reschedule) and the known-good canvas-2D reference pattern.

## Issue 2 — wasm/WebGL2 color palette wrong (washed out / off)

None of the recent changes touch color, so this is a pre-existing WebGL2
render-path issue, newly noticed. Native (Vulkan/Metal/DX12) is correct.

### The native round-trip (correct, identity)

- PPU framebuffer bytes (already in the display/sRGB domain — the NES palette
  is authored as sRGB-ish display values) are uploaded into an
  `Rgba8UnormSrgb` texture (`gfx.rs` ~line 231).
- The surface format is the first `is_srgb` format (`gfx.rs` ~line 195).
- Blit: `textureSample(nes_tex)` → automatic sRGB→linear decode; the pipeline
  writes to the `*UnormSrgb` surface → automatic linear→sRGB encode.
- Net = identity. The bytes reach the screen untouched. Correct, and this is
  what the 60-ROM oracle + sacred trio are validated against.

### The WebGL2 divergence (root cause)

wgpu 22.1.0 with the `webgl` feature falls back to `Backend::Gl` on Pages when
WebGPU is absent. wgpu-hal's GL surface cannot present to a *real* sRGB default
framebuffer (WebGL2 has no `GL_FRAMEBUFFER_SRGB` toggle for the canvas
drawing-buffer). Instead, when the surface format `is_srgb()`, it:

1. renders the app pass into an intermediate `SRGB8_ALPHA8` texture
   (`wgpu-hal-22.0.0/src/gles/web.rs::configure`, lines ~346-375 — the
   intermediate is created with `describe_texture_format(config.format).internal`);
2. at `present()` (`web.rs` lines 222-278), takes the
   `swapchain.format.is_srgb()` branch and runs a dedicated program over the
   intermediate, `gles/shaders/srgb_present.frag`:

   ```glsl
   vec4 linear_to_srgb(vec4 linear) { ... 1.055 * pow(c, 0.41666) - 0.055 ... }
   void main() { frag = linear_to_srgb(texture(present_texture, uv)); }
   ```

   i.e. an EXTRA explicit `linear_to_srgb` encode on top of GL's automatic
   sRGB-framebuffer conversions on the intermediate. The encode/decode count
   no longer matches the native identity round-trip, so the palette comes out
   wrong (washed out / too dark / tinted).

The canvas-2D embed path (`wasm.rs`) has correct colors precisely because it
does ZERO conversion: `put_image_data` of the raw RGBA8 bytes, which the
browser composites as sRGB display data — exactly what the PPU bytes already
are. That is the ground truth: **the PPU framebuffer bytes must reach the
canvas untouched.**

### Fix (option (a)/(b) hybrid — keep everything UNORM on GL)

Detect the GL backend and keep the entire pipeline in the UNORM domain so
there is zero color conversion anywhere — the bytes pass straight through,
matching the canvas-2D path:

- `gfx.rs`: `let is_gl_backend = adapter.get_info().backend == wgpu::Backend::Gl;`
- surface format: on GL pick the first NON-sRGB (`!f.is_srgb()`) format; off GL
  keep the existing first-`is_srgb` choice;
- NES texture format: `Rgba8Unorm` when the surface is non-sRGB, `Rgba8UnormSrgb`
  when sRGB. The texture's sRGB-ness MUST match the surface's, or a single
  uncompensated decode/encode would tint the whole image.

The blit shader is a plain pass-through (`return textureSample(...)`, no manual
color math), so UNORM-everywhere = no decode + no encode + no extra present
pass (the `is_srgb()` present branch is not taken for a non-sRGB surface) =
bytes untouched. Native never hits the GL branch, so it stays on sRGB and is
pixel-identical.

### What could / could not be verified

- Verified headless: native clippy/build/test green; native format-selection
  branch unchanged (still first-`is_srgb`), so the oracle is byte-identical;
  both wasm builds green.
- Cannot verify headless: actual on-screen colors on Pages WebGL2 (no GPU /
  browser in this environment). **User must confirm in a browser.** Reasoning
  cites `wgpu-hal-22.0.0/src/gles/web.rs` (`present` + `configure`) and
  `gles/shaders/srgb_present.frag`.

Note on the NTSC filter: with the filter ON, the cosmetic look differs slightly
between the sRGB (native) and UNORM (WebGL2) domains because the blur math runs
on differently-encoded values. The filter is an optional effect and the base
(no-filter) path — what the oracle validates — is unaffected on native.

## Issue 3 — native local build: slight residual stutter

### Root cause (refinement, not a hard bug)

The v1.3.2 sleep-then-spin pacer (`App::block_until_native`, `SPIN_MARGIN =
1500 µs`) sleeps until ~1.5 ms before the target, then busy-spins to the exact
instant. The residual: a single `thread::sleep(remaining - SPIN_MARGIN)` can
overshoot its requested duration by several ms on a loaded/un-tuned host; that
overshoot can blow straight past `target`, so the precise spin never engages
and the frame is produced late — jitter.

### Headless measurement

A standalone Rust harness simulated the NTSC cadence with ~3.5 ms synthetic
per-frame work on this tuned CachyOS host:

| pacer | inter-frame interval (ms) |
|---|---|
| OLD (1.5 ms single-shot sleep + spin) | p50 16.639, p90 16.639, p99 16.640, rare ~20 ms preemption outliers |
| NEW (2 ms margin, 2 ms-capped chunked sleep + spin) | p50 16.639, p90 16.640, p99 ~16.644, same rare outliers |

On a *tuned* host the two are equivalent (the rare ~20 ms outliers are OS
preemption of the bench process itself and no margin strategy removes them).
The chunked approach's value is on a *loaded/un-tuned* host, where it bounds
how far a single oversleep can overshoot before the loop re-measures.

### Fix

- `SPIN_MARGIN` 1.5 ms → 2 ms.
- New `SLEEP_CHUNK = 2 ms`; `block_until_native` sleeps in
  `(remaining - SPIN_MARGIN).min(SLEEP_CHUNK)` naps and re-measures between
  them instead of one big sleep.

Cost: ~12% of one core spinning (was ~9%); ≈7 short naps per frame
(negligible). No good-case regression (p99 unchanged).

### What could / could not be verified

- Verified headless: cadence equivalence on this host; native build/clippy/test
  green.
- Cannot verify fully: subjective smoothness on the user's display under their
  load. **Final smoothness is the user's call.**
