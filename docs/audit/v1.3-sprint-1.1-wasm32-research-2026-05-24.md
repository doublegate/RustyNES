## Goal

Land enough wasm32 plumbing that
`cargo build --target wasm32-unknown-unknown -p nes-frontend`
compiles cleanly, so subsequent sprints (1.2 winit/wgpu/egui wiring,
1.3 ROM loading + audio + IndexedDB, 2 Pages deploy + CI) have a
foundation to build on. **Not in scope:** a working browser app —
that's Sprint 1.2+.

The chip stack (`nes-cpu`, `nes-ppu`, `nes-apu`, `nes-mappers`,
`nes-core`) is already `no_std + alloc`-compatible, verified by the
existing `thumbv7em-none-eabihf` CI cross-compile gate. Sprint 1.1
only touches the frontend crate's dep stack + 4 type-mismatch sites
in `app.rs`.

## Ecosystem state research (2026-05)

Pre-sprint research surfaced one significant deviation from the
v2.0.0 release plan's v1.3.0 design and several version-pin
recommendations:

### 1. **cpal stays** (vs. v2.0.0 plan's "replace cpal with web-audio direct")

The v2.0.0 release plan (May 2026) was authored before cpal's wasm32
backend was actively revived. **cpal 0.16+ ships two wasm32 backends
actively maintained as of 2026-03**:

- **Web Audio API** backend (default for `wasm32-unknown-unknown`,
  stable Rust 1.82+)
- **AudioWorklet** backend (lower latency, requires nightly +
  `-Zbuild-std` for atomics + COOP/COEP headers for SharedArrayBuffer)

**Recommendation:** ship the Web Audio backend in Sprint 1.x —
zero new abstraction, existing `cpal::Stream` glue works. Defer
AudioWorklet to v1.4+ if latency is audible.

### 2. **WebGL2 fallback still load-bearing**

WebGPU is Baseline since Jan 2026 (Chrome/Edge 113+, Firefox 141+
Windows, Safari 26+, Chrome Android 121+). But Linux Firefox + pre-A12
iPhones (~5–10% of traffic) still need WebGL2. wgpu auto-selects at
adapter request — keep the fallback.

### 3. **`--cfg=web_sys_unstable_apis` still required**

The WebGPU JS API is Baseline-stable but the Rust web-sys bindings
are still in the `unstable_apis` tier. wgpu's WebGPU backend on
wasm32 fails at link time without this rustflag. winit alone does
NOT need it anymore (was only needed for clipboard etc.).

### 4. **Version pins (research recommendations)**

| Crate | Current workspace | Recommended | Notes |
|---|---|---|---|
| `winit` | 0.30 | **0.30.13** (same line) | OK as-is |
| `wgpu` | 22 | 29.0.3 | Drift; v1.3.0 Sprint 1.2 may need bump |
| `cpal` | 0.15 | 0.17.3 | wasm32 backend lives in 0.16+; bump pending |
| `egui` | 0.29 | 0.34.x | Drift; pairs with wgpu 29 |
| `eframe` | (not used) | 0.34.2 | If switching to eframe |

**For Sprint 1.1 scaffolding** the existing pins compile cleanly to
wasm32 (the deps gracefully skip wasm32-incompatible code paths via
their own `cfg`-gates). Version bumps are deferred to Sprint 1.2
where real browser rendering needs them.

### 5. **Build system: trunk (vs wasm-pack)**

`trunk 0.21.x` is recommended for a desktop+web hybrid the size of
RustyNES: owns `wasm-bindgen` + `wasm-opt` versions, dev-server with
hot reload, single web entry. `wasm-pack --target web` is for
libraries consumed by JS; we're shipping a Rust app. Sprint 1.2
lands the `trunk.toml` and `index.html` host.

### 6. **Size budget**

Realistic numbers for the winit + wgpu + egui + cpal + idb stack:
**~8–12 MB uncompressed `.wasm`, ~2.5–4.5 MB Brotli**. The 5 MB
compressed budget from the v2.0.0 plan is tight but achievable with:
- `wasm-opt -Oz` (trunk runs by default in release)
- `panic = "abort"`, `lto = "fat"`, `codegen-units = 1`, `strip = true`
- The chip stack's `no_std + alloc` posture already excludes
  `HashMap`/RNG/threading overhead

## Sprint 1.1 changes

### Workspace `Cargo.toml`

Added 6 wasm-host workspace deps:

```toml
wasm-bindgen = "0.2.100"
wasm-bindgen-futures = "0.4.50"
console_error_panic_hook = "0.1.7"
js-sys = "0.3.77"
web-sys = "0.3.77"
web-time = "1.1"  # transparent native/wasm `Instant` shim
```

### `crates/nes-frontend/Cargo.toml`

Added a `[target.'cfg(target_arch = "wasm32")'.dependencies]` stanza
that pulls in all 6 wasm-host deps with `web-sys` enabled for the
minimal feature set needed by Sprint 1.2/1.3 (Document, Window,
HtmlCanvasElement, HtmlInputElement, FileReader, etc.).

`web-time.workspace = true` added unconditionally (it's a no-op on
native — re-exports `std::time` directly).

### `.cargo/config.toml` (NEW)

```toml
[target.wasm32-unknown-unknown]
rustflags = ["--cfg=web_sys_unstable_apis"]
```

Required by wgpu's WebGPU backend on wasm32.

### `crates/nes-frontend/src/app.rs` — 4 minimal-fix sites

1. **`use web_time::Instant` instead of `std::time::Instant`** (line
   37). On native, `web_time::Instant` IS `std::time::Instant`
   (re-exported). On wasm32 it routes to `Performance.now()`. winit's
   `ControlFlow::WaitUntil(Instant)` expects `web_time::Instant` on
   wasm32, so this consistency fixes the 3 type-mismatch sites.

2. **`open_rom_dialog` gated `#[cfg(not(target_arch = "wasm32"))]`**
   (line 158). `rfd::FileDialog` is native-only; the wasm32 build
   uses browser-native `<input type="file">` (wired in Sprint 1.3).

3. **`SysAction::OpenRom` match arm** gated to skip on wasm32 (line 537).
