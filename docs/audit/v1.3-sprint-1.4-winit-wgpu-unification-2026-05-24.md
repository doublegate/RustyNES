## Groundwork landed (1.4b)

### `gfx.rs` — `Gfx::new` is now `async`

The native path used `pollster::block_on(instance.request_adapter(...))`
+ `block_on(adapter.request_device(...))`. On wasm32 blocking the
browser event loop is impossible. Refactored:

- `pub async fn new(window) -> Result<Self, GfxError>` — the
  `request_*` calls are now `.await`ed.
- Native caller (`app.rs::resumed`) wraps it:
  `pollster::block_on(Gfx::new(window))` — behaviourally identical
  to before.
- wasm32 backends: `BROWSER_WEBGPU | GL` (WebGPU primary, WebGL2
  fallback). Native keeps `InstanceDescriptor::default()`.
- wasm32 limits: `downlevel_webgl2_defaults()` so requests don't
  exceed what WebGL2 exposes. Native keeps `downlevel_defaults()`.

### `Cargo.toml` — wgpu `webgl` feature

`wgpu = { version = "22", features = ["webgl"] }` — enables the
WebGL2 fallback adapter on wasm32 (for Linux Firefox + pre-A12 iOS
which lack WebGPU). No effect on native.

### Verified

- Native build + clippy + fmt: clean
- wasm32 build: clean (`gfx.rs` compiles for the wasm target)
- Native tests: 599 + 6 ignored (the async change is behaviourally
  transparent on native via `block_on`)

## Remaining: the App-lifecycle restructure (1.4b cont.)

The research (`a8a3151b` agent session) established the precise
known-working pattern for winit 0.30 + wgpu 22 + egui 0.29 on wasm32.
The remaining work:

### 1. User-event type + typed `ApplicationHandler`

```rust
enum AppEvent {
    GfxReady(Box<Gfx>),   // async wgpu init resolved
    RomLoaded(Vec<u8>),   // browser file-picker delivered ROM bytes
}
impl ApplicationHandler<AppEvent> for App { ... }
```

Native uses the same typed event loop
(`EventLoop::<AppEvent>::with_user_event()`); it simply never sends
events (its `resumed()` creates Gfx synchronously).

### 2. `App` gains a wasm-only proxy

```rust
#[cfg(target_arch = "wasm32")]
proxy: Option<EventLoopProxy<AppEvent>>,
```

### 3. `resumed()` cfg-split

- Native: create window → `block_on(Gfx::new)` → `on_gfx_ready(gfx)`
  (unchanged behaviour).
- wasm32: create window via
  `WindowAttributesExtWebSys::with_canvas(Some(canvas))` →
  `spawn_local(async move { let gfx = Gfx::new(window).await?;
  proxy.send_event(AppEvent::GfxReady(Box::new(gfx))) })`.

### 4. `user_event()` (new)

- `GfxReady(gfx)` → `on_gfx_ready(*gfx)` (install gfx + debugger;
  on wasm, NO NES yet — `rom_bytes` is empty at startup).
- `RomLoaded(bytes)` → `start_nes_from_bytes(bytes)`.

### 5. Extract shared helpers

- `on_gfx_ready(&mut self, gfx: Gfx)` — NTSC filter + egui debugger
  overlay + (native only: audio + NES-from-`rom_bytes` + rewind +
  frame timing).
- `start_nes_from_bytes(&mut self, bytes: Vec<u8>)` — the
  ROM-load path (used by native `load_rom_from_path` AND the wasm
  `RomLoaded` event).

### 6. cfg-gate native-only subsystems

- `gilrs` (gamepad): `#[cfg(not(target_arch = "wasm32"))]` on the
  field + the poll site.
- `cpal` `AudioOutput`: native-only. The wasm unified path either
  reuses the Sprint 1.4a web-sys audio (move the ring + AudioContext
  out of `wasm.rs` into a shared `audio` wasm submodule) OR defers
  audio to a 1.4c follow-up (video + input + debugger first).
- `Config` file I/O + `directories`: wasm uses `Config::default()`.
- Save-state-to-disk: wasm uses the Sprint 1.4a localStorage path.

### 7. `run()` cfg-split

```rust
let event_loop = EventLoop::<AppEvent>::with_user_event().build()?;
#[cfg(not(target_arch = "wasm32"))] event_loop.run_app(&mut app)?;
#[cfg(target_arch = "wasm32")] {
    use winit::platform::web::EventLoopExtWebSys;
    event_loop.spawn_app(app);   // by value, 'static, returns ()
}
```

### 8. Surface-configure timing (the #1 wasm bringup crash)

Guard against the 0×0 canvas at `resumed()`: clamp width/height to
`.max(1)`, do the first real `surface.configure` in the
`WindowEvent::Resized` handler, and early-return the render path if
either dimension is 0.

### 9. egui-wgpu 0.29 on wasm

Pass the surface's ACTUAL format (not hardcoded sRGB — WebGL2 often
only offers non-sRGB; the existing `Gfx::new` already does
`find(is_srgb).unwrap_or(formats[0])`, good), `msaa_samples: 1`,
`dithering: false`, and drive egui-winit with the canvas
`devicePixelRatio` scale factor.

### 10. `wasm.rs` feature-split

Gate the canvas-2D `start()` vs the unified winit `start()` behind
cargo features (`wasm-canvas` for the lightweight embed mode,
`wasm-winit` default for the full experience). Both `#[wasm_bindgen(start)]`
can't coexist, so exactly one is compiled per build.

## Honest assessment / why the restructure is staged separately

The App restructure is ~300-400 lines of intricate cfg-gated change
across `app.rs` (the core event loop), touching every native code
path via the `ApplicationHandler` signature change. Critically, it
has several **runtime-only** failure modes that don't show at compile
time and that I can't verify without a browser:

- 0×0 surface configure crash (the most common wasm-bringup failure)
- surface format selection panic on WebGL2
- the deferred-NES flow (App starts ROM-less on wasm)
- egui scale-factor / canvas sizing
- the proxy → spawn_local → user_event threading

The maintainer has browser access and verified the canvas path
interactively. The highest-value way to land the restructure is
with that browser in the loop for the inevitable iteration, rather
than a large blind implementation. The async-`Gfx` groundwork here
is the safe, native-verified foundation that the restructure builds
on; landing it separately de-risks the restructure (the foundational
async change is proven independently).

The canvas-2D path (with audio + save state) remains a complete,
working, verified browser emulator + the documented "embed mode" —
so v1.3.0 has a shippable web frontend regardless of when the
unified full-experience path lands.
