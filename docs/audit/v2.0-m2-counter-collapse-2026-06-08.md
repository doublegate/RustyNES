## TL;DR

The M-2 counter-collapse — moving the get/put `put_cycle` flip to END of the CPU
cycle (after the bus access), alongside the byte-timer-END move, while keeping
`apu_phase` (the APU IRQ line / C1 phi2 sample source) flipping at cycle-START so
C1 stays invariant — was implemented behind a new flag. It is the FIRST lever in 18
months to move **CheckDMATiming Y from 3 to 4 via the put_cycle parity** and to fix
the **DMC+OAM front-half parity** (floor `03,04,...` -> `04,03,...` = KEY) — idx0-6
+ idx8-15 now match the hardware KEY exactly, with ONLY idx[7] off.

But it is a **global parity shift**, not the per-context exact counter. The same shift
that fixes the DMC+OAM overlap (whose span is hardcoded `short?2:3`) **over-grows the
`dmc_dma_step` reload span by one cycle** in the Explicit/Implicit abort contexts
(Explicit `04`->`05`; Implicit baseline `00`->`02`), so those two REGRESS. C1 is
preserved, SH* 6/6, nestest 0-diff, and the floor (flag-off) is byte-identical.
No regression was forced into the floor (ADR-0002).

Net flag-on AccuracyCoin = **94.24% / 8** (floor 94.96% / 7): adds **Implicit DMA
Abort [error 2]** and rotates **Implied Dummy Reads [error 5 -> 31]**; DMC+OAM +
Explicit stay error 2 (the near-match does not pass — the result-byte gate needs the
FULL sweep == KEY).

## What was implemented (`mc-r1-counter-collapse`, default-off)

* `crates/nes-apu/src/apu.rs`:
  - `tick_with_external`: the `put_cycle = !apu_phase` flip is SUPPRESSED at
    cycle-start under the flag (it stays at start only under bytetimer-end/floor).
  - `dmc_tick_end` (end-of-cycle, after the bus access): (a) the DMC byte-timer is
    clocked on `timer_phase = !put_cycle` (NOT raw `apu_phase`) so the byte-timer
    FIRE stays locked to the now-shifted GET parity — ONE counter driving both the
    byte-timer and the get/put decision; (b) the `cannot_run` decrement uses the
    same `timer_phase`; (c) `put_cycle = !apu_phase` flips HERE at the very end,
    after the byte-timer/reload-arm.
  - `dmc_step_reload_arm`: the halt-subpos pre-wrap detector reads `put_cycle`
    (the shifted pre-wrap parity) instead of `!apu_phase`, so the landed Implicit
    `$540` pre-arm stays calibrated to the shifted wrap.
* Cargo: `mc-r1-counter-collapse` in nes-apu (raw gate) + nes-core (composes
  `mc-r1-dmc-bytetimer-end` + forwards the apu gate) + nes-test-harness.

`apu_phase` itself is UNCHANGED (still flips at cycle-start, line ~927); only the
get/put `put_cycle` and the DMC byte-timer/reload-arm move to end-of-cycle. The φ2
IRQ sample (`cpu.rs end_cycle::handle_interrupts`) stays on `master_clock`, so it is
invariant to DMA length — the reason C1 is preserved.

## Why it does not fully close (the precise per-index coupling — the deliverable)

1. **The DMC+OAM overlap span is hardcoded** (`dmc_overlap_begin` returns
   `short?2:3`; lockstep `service_dmc_dma_during_oam` likewise). So the byte-timer
   shift does NOT change the overlap span — it only re-times the looping-reload
   ARM, which moves WHICH OAM slot the GET steals -> the front-half parity flips
   from wrong (`03,04`) to right (`04,03`). idx0-6 + idx8-15 then match KEY exactly.
   The residual idx[7] (`03` vs `01`) is the overlap-REGIME-TRANSITION realign
   (the iteration where OAM first overlaps the DMC): a structural `dmc_overlap_*`
   accounting detail at the `04,03`->`02,01` boundary, NOT a parity issue.

2. **The `dmc_dma_step` (Explicit/Implicit) span is parity-EMERGENT**, so the SAME
   byte-timer shift that the overlap needs ADDS one cycle to the looping-reload
   self-cadence there (Explicit `04`->`05`; Implicit baseline `00`->`02`). The GET
   parity itself is right (Y=4 proves it), but the inter-GET reload period grows by
   one under the shifted wrap. This is the global-shift trade-off: a single parity
   flip cannot simultaneously be right for the hardcoded-span overlap AND the
   emergent-span dmc_dma_step.

3. **The wedge boundary is real**: with `put_cycle` end-flipped but the byte-timer
   LEFT on `apu_phase` (GET on `!a_N`, arm on `a_N` — one apart), the DMASync
   `$4000`-spin never catches `$00` -> the battery WEDGES (every DMA test `(unset)`
   even at 40000 frames). So the byte-timer MUST follow the shift (`!put_cycle`) to
   un-wedge — which is exactly what over-grows dmc_dma_step. The two are inseparable
   under a global counter.

## The minimal closure (what the FULL M-2 requires — confirmed by this measurement)

This session's lever PROVES the direction is right (Y 3->4 + DMC+OAM front-parity
fixed) but confirms the plan's reservation: the genuine closure is the **full M-2
counter-collapse** — collapse `cpu.cycles` + `cpu.cycles_emitted` + `apu.cpu_cycle`
+ `master_clock` into ONE canonical CPU master-cycle counter, make EVERY cycle a
real bus access (promote `mc-r1-internal-cycle` + the Stage-C indexed dummies),
DELETE `mc-r1-dma-cycle-budget`, so the span parity is exact in EVERY instruction
context. Only then does the overlap regime AND the dmc_dma_step reload chain each
land on its own correct absolute parity SIMULTANEOUSLY (the per-context exactness a
single global flip cannot give). The next surgical sub-target is the DMC+OAM idx[7]
overlap-regime-transition realign (isolated to `dmc_overlap_*`, no Explicit/Implicit
coupling) — closeable independently of the counter rewrite.
