## Step 1 — the umbrella fold (`crates/nes-core/Cargo.toml`)

`mc-r1-full-cpu` gains four members:

- `mc-r1-branch-poll-points` (W1) — discrete taken-branch IRQ poll points;
  closes `Interrupt flag latency` [error 10].
- `mc-ppu-2007-render-buffer` (W2) — the rendering-time `$2007` PPUDATA state
  machine; closes `$2007 Stress Test` (170/170 stable dots).
- `mc-r1-dmc-delayed-4015` (W3 Stages 1–3 + the `$540` grid close) — the
  TriCNES `APU_DelayedDMC4015` delayed status application on the unified
  single-driver DMA engine at the breakthrough parity. Pulls the whole engine
  chain automatically (`mc-r1-dma-unified-collapse` → `mc-r1-dma-unified` +
  `mc-r1-counter-collapse` → `mc-r1-dmc-bytetimer-end` →
  `mc-r1-dmc-oam-overlap`) plus the `mc-r1-brk-padding-read` CPU ride-along.
  Closes DMC+OAM, Explicit + Implicit DMA Abort, Delta-Mod, Implied-Dummy.
- `mc-r1-oam-dma-reg-window` — the active-window OAM-DMA register mirror
  decode; closes `APU Register Activation`.

**Cycle break:** the fold would create cargo feature cycles through the two
historical back-edges `mc-r1-dma-unified = ["mc-r1-full-cpu", …]` and
`mc-r1-dmc-oam-overlap = ["mc-r1-full-cpu", …]` (cargo rejects cyclic
features), so both back-edges are REMOVED — those flags are now umbrella
members and documented as "use via the umbrella". No other Cargo.toml needed
changes (the chain forwards each sub-crate flag individually).

Post-fold measurements: `--features test-roms,mc-r1-full-cpu` alone =
**100.00%** (127 pass + 12 pass-with-code of 139; battery completes, 7200
frames); the default battery re-measured **94.24%** with the byte-identical
documented fail set + error codes (IFlag latency [11], NMI Overlap BRK [2] /
IRQ [1], Delta-Mod [21], APU-RegAct [6], BG Serial In [2], `$2007` Stress [2],
Implied-Dummy [3]).

## Step 3 — snapshot/determinism (the serialization closure)

With the stack inside the umbrella, previously-transient gated state became
load-bearing across instruction boundaries. Serialized (every tail written
UNCONDITIONALLY — zeros when the feature is off — so the blob layout is
identical across feature builds; the default build's emulated behavior is
untouched):

| Section | Change | Fields |
|---|---|---|
| CPU | v1 → v2 (ADR-0003; v1 upconverts: `master_clock = cycles*12`, latches quiescent) | `master_clock` + `mc_need_nmi`/`mc_prev_need_nmi`/`mc_run_irq`/`mc_prev_run_irq`/`mc_prev_nmi_line` |
| APU | Stage-4 trailing tail (the module's append convention; pre-Stage-4 blobs upconvert — applied status synthesized from channel-active — and report the missing tail via `Apu::snapshot_restored_parity` so the bus only re-seeds the boot alignment for OLD blobs) | `put_cycle` + `parity_seed`, `cannot_run_dmc_dma`, `dmc_reenable_period_block`, `subpos_arm_countdown`, `dmc_need_halt`/`dmc_need_dummy_read`, `pending_dmc_dma_next`, the delayed-`$4015` sextet (`dmc_delayed_4015` countdown + `dmc_delayed_status` + `dmc_status_applied` + `dmc_set_implicit_abort` + `dmc_implicit_abort` + `dmc_edge_arm_suppress`) |
| BUS | trailing-default tail (replaces the Stage-1 clear-on-restore) | `dmc_halt`, `uni_oam_{active,halt,aligned,addr}`, **`ppu_clock` + `dma_mc_consumed`** |
| PPU | v2 → v3 (v1/v2 upconvert at inactive defaults) | `render_data_bus`, `ppudata_sm_countdown`, `ppudata_v_inc_pending`, `spr_fetch_{lo,hi}_raw`, `ppudata_spr0_nt_addr` (W2) + `bg_reload_render`, `mask_write_delay` (`mc-ppu-subpos`) |

**The empirical catch (found by the new continuation test):** restoring
`Cpu::master_clock` WITHOUT the bus-side `ppu_clock`/`dma_mc_consumed` pair
desynchronizes `run_ppu_to` — the restored emulator spun the PPU ~11 frames
inside the FIRST post-restore instruction (+4 bus cycles, PPU frame 12 → 23)
before the pair re-cohered. The pre-Stage-4 code was only self-consistent by
accident (neither side serialized). The pair now travels together.

New tests:

- `nes-apu` units: Stage-4 tail round-trip + pre-Stage-4-blob upconvert.
- Harness `save_state`: a synthetic NROM that keeps the DMC reload chain
  (fastest-rate looping sample), an OAM DMA per loop, and a `$4015`
  disable/re-enable race permanently hot;
  (a) `save_state_continuation_byte_identical_with_dmc_and_oam_dma_hot` —
  byte-identical EMULATION continuation (cumulative cycles + framebuffer FNV,
  12 frames) of a restored instance against the undisturbed original;
  (b) `save_state_re_encode_stable_every_frame_with_dmc_machinery_hot` —
  encode → decode → re-encode byte stability at every frame. Both green on
  BOTH builds. The audio STREAM is deliberately not bit-compared across the
  restore boundary: the blip pending-sample queue + band-limited accumulator
  are documented as intentionally not preserved, so a restored run has a
  short by-design audio transient even though the emulated state is exact.

Scope note: `Nes::run_frame` runs whole `Cpu::step`s, so snapshots land at
instruction boundaries — a DMA burst can never straddle a snapshot. "Mid-DMA"
therefore means the maximal API-reachable hot state: pending latches, the
delayed-`$4015` countdowns/status, the byte-timer phase, `cannot_run`, the
parity, and an in-flight looping sample — all exercised by the synthetic ROM.
