## The mechanism (TriCNES per-cycle rows, X=8 enable1 cyc 49326708)

1. At the disable's grid boundary TriCNES goes **silent immediately**
   (`BytesRemaining == 0` at the consume edge; the stale buffer byte is NOT
   consumed). At the X-swept re-enable: `StartDMCSample` (bytes-remaining +
   address ONLY — **the byte-timer is NOT reset**, the grid free-runs) +
   `APU_Silent` -> `DMCDMADelay = 2` -> the load GET lands at enable1+5.
2. **X=8**: the load GET cycle (49326713) COINCIDES with the consume edge —
   `DMCDMA_Get` runs in `_6502` (sets `CannotRunDMCDMARightNow = 2`), then
   `_EmulateAPU` the SAME cycle fires the byte-timer: bits 1 -> 0 -> 8, the
   consume edge sees `BytesRemaining > 0` but the arm gate
   `CannotRunDMCDMARightNow != 2` (Emulator.cs:1165) BLOCKS it — and the
   `:1186` decrement runs AFTER the edge block in the same end-tick, so
   `== 2` is observable ONLY at this same-cycle edge. **TriCNES holds no
   level request**: the skipped arm simply waits for the NEXT consume edge,
   576 cycles later (GET 49327293 = enable1+585).
3. **X=10**: the enable lands 2 cycles more before the boundary, so the edge
   falls 2 cycles AFTER the load GET — `CannotRun` has already decremented
   to 0 -> the edge arms normally -> the second GET at enable1+11 (both
   emulators identically).
4. **Ours (pre-fix)**: same first GET (+5; our level reload-arm at the
   write's end-tick compensates the silence-state difference — we consume
   the stale buffer byte at the post-disable edge, so `silence()` is false
   at the enable and the `load_arm` path never fires; the GETs land at +5
   regardless). At the X=8 coincident edge our `clock_output` consumed the
   just-delivered byte into the shifter -> `needs_dma()` (buffer-empty
   level) PERSISTED past the `cannot_run == 2` block -> re-armed at cooldown
   expiry (+4) -> the second GET at +13 — one boundary early. The
   `dmc_dma_cooldown`/`cannot_run` gates only DELAY a level-held need; the
   hardware (and TriCNES) arm is **consume-edge-quantized**.

## The fix (1 quantity, engine-ordering only — inside `mc-r1-dmc-delayed-4015`)

`crates/nes-apu/src/apu.rs`:

- New cfg-gated latch `dmc_edge_arm_suppress`. In `dmc_tick_end`, at a
  consume edge (`timer_phase && bits 1->8`): if the latch is set, RELEASE it
  (immediately before `dmc_step_reload_arm`, so the deferred arm fires
  exactly on-grid, like TriCNES's `BytesRemaining > 0` edge arm); else if
  `cannot_run_dmc_dma == 2 && needs_dma() && !pending` (the GET-coincident
  blocked edge), LATCH it.
- `dmc_step_reload_arm`: the level arm gains `&& !edge_suppressed`.
- Probe counter `EDGE_SUPPRESS_SET` (under `mc-r1-dmc-abort-probe`), printed
  by `scan_dma_abort`: fires **exactly 2x** over the whole battery — the
  Loop3 X=8 and X=9 restart races, nothing else (steady-state 576-chains arm
  AT their edges and never hit a blocked edge; the non-looping contexts have
  `bytes_remaining == 0` after the GET).

Post-fix X=8 re-trace: store data `00`; enable1->enable2 = 598; byte-timer at
enable2 = 27; single restart GET (+5); chain GETs rel-enable2 = -13, +563 —
**TriCNES-exact in every pinned column**.

### Relation to the Stage-3 DO-NOT-RETRY #1

Stage 3 refuted "consume-edge re-arm quantization (any scoping)" as a
quantization OF THE PERSISTENT RE-ARM — globally or `loop_flag`-scoped it
starves every looping chain (the prompt re-arm is load-bearing in the
calibrated contexts). This fix quantizes NOTHING in the steady state: the
level arm stays prompt everywhere; only an arm that the TriCNES `cannot_run`
gate has ALREADY blocked at a GET-coincident edge loses its level persistence
(measured: 2 occurrences in the whole battery). It is the "hardware's
free-running-grid quantization expressed some other way" that the
DO-NOT-RETRY entry itself predicted would be the closing shape.
