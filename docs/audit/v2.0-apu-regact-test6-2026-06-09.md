## What Test 6 actually checks (decoded from the upstream asm)

Suite `APU Tests :: APU Register Activation`, result `$45C`
(`AccuracyCoin.asm:8104+`; source preserved at `/tmp/RustyNES_v2/` and
quoted here):

* **Test 4** (error 4ŌåÆ5): `LDA #$40; STA $4014` from PRG ROM ŌĆö the 6502 bus
  is parked OUTSIDE `$4000-$401F`, so an OAM DMA sourcing page `$40` reads
  ONLY open bus (`$4015` is NOT read; the frame-IRQ flag survives). This is
  the inactive gate that already existed in `raw_oam_dma_read`.
* **Test 5** (error 5ŌåÆ6): the wacky setup. `JSR $3FFE`: opcode `$8D` (STA
  abs) comes from the `$2006` mirror (= PPU data bus, prepped to `$8D`),
  operand low `$14` from the `$2007` mirror (= PPU read buffer, prepped to
  `$14`), and a cycle-exact DMC DMA overwrites the bus with the sample byte
  `$40` so the operand-high read of `$4000` (open bus) yields `$40` ŌåÆ
  `STA $4014` executes with A = `$50` ŌåÆ OAM DMA from page **`$50`**
  (unmapped/floating) with the 6502 bus halted on the next opcode fetch at
  **`$4001`** ŌĆö INSIDE the window, so the APU/controller chip select is
  asserted for the whole DMA. The registers mirror every `$20` bytes
  (`$4000 | (src & $1F)`), producing the documented OAM image (the table at
  asm:8253-8275): `$50`-chain rows, `$44` at offset `$15` (`$4015` =
  frame-IRQ `$40` + triangle `$04`, read CLEARS the flag), `$41`/`$40` at
  `$16`/`$17` (controller bit | bus `& $E0`), then per-`$20` `$04`/`$01`/`$00`
  after the `$4015` value decays bit 6 off the bus. The chain ends at `$00`,
  so the resumed `$4001` fetch executes **BRK** ŌåÆ the prepared IRQ routine ŌåÆ
  `LDA #$A9` ŌåÆ `RTS` ŌåÆ `CMP #$A9` passes.
* **Test 6** (error 6ŌåÆ7) ŌĆö *the residual this stage closes*: copies OAM to
  `$500-$5FF` via `LDA $2004 / STA $2004` and compares every byte against
  the Test-5 expected table (rows 0-1 pattern `50 50 40 50` with the OAM
  attribute-byte `& $E3` drop, `$215 == $44`, `$216 & $FB == $41`,
  `$217-$234 == $40`, then `$04`/`$01`/`$00` per `$20` mirror period).
  **[error 6] = Test 5's DMA filled OAM with the wrong values** ŌĆö before
  this stage our reads returned the untouched `$50` latch everywhere, so the
  first mismatch was `$215` (`$50` != `$44`).
* **Test 7** (error 7ŌåÆ8): the bus-conflict variant ŌĆö same parked-window DMA
  but sourcing page `$02` (RAM: `$FF` half + `$00` half). Expected: `$x5` =
  `$24`/`$04` (`$4015` status with **bit 5 from the internal-bus latch** =
  the previous page-2 fetch), `$x6`/`$x7` = the externally-driven byte (the
  controllers are clocked but INVISIBLE ŌĆö the driven pins win the conflict;
  the eval also accepts `$E0`/`$E1` for consoles where the conflict pulls
  bits down).
* **Finale**: reads `$4016` twice ŌĆö EITHER outcome (controllers clocked
  during the conflict DMA or not) is a success (codes 9 / 5). TriCNES-style
  clocking ŌåÆ code **9** ("OAM DMA Bus Conflict Clocks"), which is what we
  now produce (`0x09`).

## The wedge mechanism, root-caused

Stage 3's refuted item 3 reported: ANY `$4016/$4017` VALUE change in the
mirror decode "wedges the AccuracyCoin framework bootstrap" (battery stalls;
the sub-test ROM never completes). The named hypothesis was that the
framework itself runs OAM DMAs parked on the input registers and hardware is
"protected" by something we lack. **Both wrong.** The cross-source analysis:

1. **No framework DMA ever parks in the window.** `dma_halt_addr` latches
   the preempted CPU read = the next opcode fetch after `STA $4014`. The
   framework executes from PRG ROM, so its DMAs park at `$8xxx`. The ONLY
   in-window parks in the whole battery are the two `JSR $3FFE` DMAs of
   this suite (asm:8314 + 8472, verified by grep).
2. **Hardware does NOT protect the framework from the side effects** ŌĆö the
   controllers really are clocked 8 times per conflict DMA (the suite
   finale exists precisely to report which behavior the console has, and
   both are accepted).
3. **The real wedge: the VALUE chain is load-bearing for the test's own
   control flow.** The byte on the data bus at the END of the Test-5 DMA is
   the opcode the halted CPU resumes with at `$4001` (open bus). On
   hardware the chain decays to `$00` ŌåÆ BRK ŌåÆ the recovery routine. The
   Stage-3 attempt (Mesen `ProcessDmaRead` obMask merge, but WITHOUT the
   put-half data-bus drive) left a non-`$00` byte on the bus ŌåÆ the resumed
   opcode was garbage ŌåÆ runaway execution through open bus ŌåÆ the test never
   writes its result ŌåÆ the battery (which runs tests sequentially) stalls.
   The `$4015`-arm-alone bisect being "harmless but flips nothing" is
   consistent: without the put-drive the status value never reaches the
   open-bus chain at all.

## The TriCNES mechanism, as ported (the citations)

`crates/nes-test-harness/golden/tricnes/tricnes-full-src/Emulator.cs`:

* **The window condition** (`Fetch`, 9252): the register overlay applies
  whenever `addressBus` (the 6502's own bus ŌĆö parked during DMA) is in
  `$4000-$401F`; the overlay decodes `Reg = Address & 0x1F` from the FETCH
  address. One block serves both normal CPU register reads and the
  DMA-active-window reads.
* **`Reg == $15`** (9255-9279): builds the status on `internalBus`
  (keeping internalBus bit 5), sets the delayed frame-IRQ clear, and
  `return internalBus` ŌĆö "reading from `$4015` can not affect the databus".
  `dataBus` keeps the previous fetch's value (Test 7's `$24` = bit 5 of the
  preceding page-2 `$FF` fetch).
* **`Reg == $16/$17`** (9281-9305): `bit | (dataBus & 0xE0)`; the shift is
  clocked via the 2-cycle `ShiftCounter` (double-read protection ŌĆö inert
  here, mirror reads are 64 cycles apart); then the conflict branch
  `if (DoOAMDMA && dataPinsAreNotFloating) return dataBus;` ŌĆö the
  externally-driven byte wins, the controller value is discarded (but the
  clock already happened).
* **The put drives the bus** (`OAMDMA_Put` 4111 ŌåÆ `Store(OAM_InternalBus,
  0x2004)` 9419): every `Store` puts the value on `dataBus`. This is the
  link that carries `$44` onto the bus after the offset-`$15` PUT, feeds
  `$41 = $01 | ($44 & $E0)` at `$16`, and decays the chain `$04 ŌåÆ $01 ŌåÆ
  $00` after the second `$4015` mirror read returns the cleared-flag `$04`.

Cross-checked against Mesen2 `NesCpu::ProcessDmaRead` (the
`enableInternalRegReads = (addr & 0xFFE0) == 0x4000` arm on the HALT
address): same physics; the obMask merge is Mesen's expression of the same
`& $E0` + conflict split. The TriCNES shape composes with our single
`open_bus` latch.

## Implementation map (all under `mc-r1-oam-dma-reg-window`)

`crates/nes-core/src/bus.rs`:

* `raw_oam_dma_read`: new active-window arm ŌĆö when
  `(dma_halt_addr & 0xFFE0) == 0x4000`, route to the decode below. The
  existing inactive gate (halt OUTSIDE + src INSIDE ŌåÆ silent open-bus
  return) is unchanged.
* New `oam_dma_read_reg_active(src)`: the normal external fetch first
  (`raw_cpu_read`, side effects included; skipped for src inside
  `$4000-$401F` itself so the readable registers don't double-fire),
  tracking whether the source region drives the pins (RAM/PPU/cart mapped =
  driven; `$4000-$401F` + `cpu_read_unmapped` cart space = floating). Then
  the overlay on `src & $1F`:
  - `$15`: `(apu.read_status() & 0xDF) | (internal_data_bus & 0x20)` ŌĆö
    internal-bus-only, `open_bus` untouched (mirrors the normal-CPU `$4015`
    composition; `read_status` carries the delayed frame-IRQ clear and the
    Stage-3 delayed-DMC bit 4).
  - `$16`/`$17`: `read_port(port)` (clocks the shift register; Four Score
    aware); driven ŌåÆ return the external byte (conflict, controller bits
    invisible); floating ŌåÆ `(open_bus & 0xE0) | bit`, stored to `open_bus`.
  - else: the external fetch value.
* New `oam_dma_put()`: the OAM PUT half ŌĆö `ppu.oam_dma_write(dma_byte)`
  plus, when parked in-window under the flag, `open_bus = dma_byte` (the
  TriCNES `Store` dataBus drive). All 5 put sites route through it
  (lockstep `clock_oam_dma_cycle`, both unified-engine parities,
  `oam_dma_step`, `oam_dma_overlap_cycle`); flag-off the helper is
  behavior-identical to the previous inline call.

Scope note: on silicon EVERY OAM put drives the bus; the model deliberately
scopes the drive (and the decode) to the parked-window case so all other OAM
DMAs ŌĆö including every DMA the DMC+OAM / Explicit / Implicit sweeps and the
framework bootstrap run ŌĆö are byte-identical to the floor. Widening the
put-drive to all DMAs is a separate open-bus-model audit, not needed by any
current test.

`crates/nes-core/Cargo.toml` + `crates/nes-test-harness/Cargo.toml`: the new
flag (composes `mc-r1-dmc-delayed-4015`) + forward.
