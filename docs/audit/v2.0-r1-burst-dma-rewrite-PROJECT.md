## Why from-scratch (not incremental) — the proven entanglement

## Architecture (the parallel burst-DMA model)

**Core idea:** when a DMC (or OAM) DMA fires, run default's BURST inline at the triggering `cpu_read`,
with each burst cycle going through an R1-COHERENT cycle (master_clock advance + `run_ppu_to` + APU/DMC
tick + φ2 IRQ sample) that reproduces default `tick_one_cpu_cycle`'s per-cycle PPU/APU/DMC effect — the
thing the legacy `tick_one_cpu_cycle` port got WRONG (it bypassed `run_ppu_to`/the master_clock split).

- **`burst_dma` module (`nes-core`):** `fn run_dmc_burst(&mut self, halted_addr) ` that emits
  `noop_cycles = short?2:3` halt re-reads + 1 GET, each via a `burst_cycle()` helper that does exactly
  what an R1 CPU cycle does (`start_cycle`-equivalent PPU pre-catch-up on `master_clock`, the held/sample
  read, `end_cycle`-equivalent: `master_clock += POST`, double catch-up, `cpu_clock_apu_dmc`,
  `handle_interrupts`). Mirror for OAM (`run_oam_burst`) + the DMC-during-OAM interleave (TriCNES
  `OAMDMA_Aligned`, GET=DMC/PUT=OAM priority).
- **Trigger point (the crux — match default):** detect the buffer-empty at the SAME cycle default does.
  Default's `tick_one_cpu_cycle` clocks the APU/DMC then `drain_dma` services in the SAME `cpu_read`. The
  burst model must run the DMC tick + buffer-empty check + burst service WITHIN the `cpu_read`/`read1`
  cycle that empties the buffer — NOT at the next read (the interleaved model's bug). Keep the IRQ-sample
  φ2 phase for C1 (the `handle_interrupts` at `end_cycle`); only the DMA SERVICE moves to inline.
- **Abort + truncation:** port Mesen `StopDmcTransfer` (cancel-if-`_needHalt` else `_abortDmcDma`) into
  the burst (the `mc-r1-dmc-phase-coherence` cuo fix bites once Y==4). The abort sweep then runs.
- **CPU integration:** `Cpu::read1`/`write1`/`idle_tick` call `bus.dma_burst_if_pending(addr)` at the
  point default's `drain_dma` is called (after the cycle's APU tick, inline), behind
  `mc-r1-dmc-burst-rewrite`; the interleaved `dmc_dma_step` while-loop is DISABLED under the feature.

## P2 PREMISE-CHECK (2026-06-04) — CRITICAL: the DMC halt placement is IDENTICAL; verify the divergence source BEFORE building the burst

Starting P2, read the actual default vs R1 DMA flow and found a premise problem that MUST be resolved
before implementing the burst engine:

- **Default `cpu_read` (bus.rs:2401):** `drain_dma(Some(addr))` is called BEFORE the read (leading), and
  the DMA is armed by the cycle's APU/DMC tick AFTER the read (default `read1`: read → `on_cpu_cycle`).
- **R1 `read1`:** leading `while dmc_dma_pending` drain BEFORE the read; armed at `end_cycle`'s
  `cpu_clock_apu_dmc` (tick_dmc) AFTER the read.
- ⟹ **BOTH drain-before-read and arm-after-read at the SAME cycle. The DMC halt PLACEMENT is identical.**
  So the burst model's "place the halt at default's read" premise is WRONG — the placement already matches.

**This forces a verification BEFORE P2 implementation:** the R1c-1 cumulative-cycle divergence (Y=3 vs 4)
is NOT (or not only) the DMC halt placement. Candidate REAL sources, in priority order:
1. **R1's C1/PPU timing** — R1 services interrupts at different cycles than default (the C1 fix) and the
   `master_clock`/`run_ppu_to` PPU phase differs; these accumulate cycle-count divergence in NON-DMA code.
   EVIDENCE: at frame ~306 (pre-DMA-tests) R1's instruction count already differs from default's (3056498
   vs 3058728) and C=92 (small/even); the ODD offset develops in frames 800→1280 (the PPU/CPU/APU test
   cluster). If the odd offset is from interrupt/PPU timing, the burst-DMA rewrite is MISDIRECTED and the
   offset is INHERENTLY entangled with C1 (no DMA model fixes it).
2. **OAM DMA 513/514 alignment** — a per-frame odd-cycle event in `drain_dma` (`extra = cycle&1==0?514:513`).
3. **The DMC DMA span/cycle-count** (the burst premise) — least likely now (placement identical).

**REVISED P2 (verify-then-build):** BEFORE the burst engine, run the instruction-level trace diff focused
on CLASSIFYING the cumulative-cycle-delta divergences: are the +N local jumps (R1c-1's `instr_cycle_diff`)
at (a) interrupt vectors / NMI-PPU-sync code [⟹ C1-entangled, burst misdirected, likely UNCLOSABLE under
R1], (b) OAM-DMA frames [⟹ fix the OAM alignment, NOT a DMC burst], or (c) DMC-DMA reads [⟹ the burst is
correct]. Gate: a written verdict on the divergence source. ONLY if (c) does the burst engine (P2-build)
proceed; if (a), the honest conclusion is the tail is incompatible with R1's C1 substrate (ship the C1 win);
if (b), the project pivots to an OAM-alignment fix (much smaller).

**This is the most important open question in the whole effort** and was surfaced by reading the code at
P2-start: do NOT build the burst engine until the divergence source is classified — it may be misdirected.
The scaffold (harness + skeleton + feature) stands; the burst IMPLEMENTATION is gated on this verdict.

## P2 RESULT (2026-06-04) — VERDICT (a): PPU `$2002`-vblank-phase. The burst-DMA rewrite is MISDIRECTED.

The classification is decisive and is NOT (b) OAM and NOT (c) DMC. Method + evidence:

1. **Aligned-code cost diff (`scripts/instr_cost_localize.py`, frame-306 window):** in a cleanly-aligned
   6184-instruction span, R1 and default have **identical per-instruction cycle cost — zero divergence**.
   The offset (C≈92) is a constant carried in from discrete events OUTSIDE the span ⟹ there is NO continuous
   DMA-driven divergence. This already eliminates (c)'s "burst accumulates per-fetch" premise.
2. **Cold-boot first-diff (`scripts/instr_boot_firstdiff.py`, BOOT-mode tracer):** both streams are
   byte-identical from the reset vector until idx 7829, where they FIRST diverge inside a tight 2-instruction
   loop. The branch at `$8048` costs **2 in R1 (falls through, exits) vs 3 in default (taken, loops once
   more)** — R1 exits the loop ONE iteration earlier.
3. **Disassembly of the loop** (`AccuracyCoin.nes`, NROM, $8000-mapped):
   ```
   8042: AD 02 20   LDA $2002      ; clear the vblank latch
   8045: AD 02 20   LDA $2002      ; poll PPUSTATUS
   8048: 10 FB      BPL $8045      ; spin until vblank (bit7) sets
   ```
   The canonical PPU vblank warm-up wait. R1's master-clock CPU/PPU phase makes the vblank flag observe-as-set
   one poll-iteration earlier than default.
4. **Full boot census (`scripts/instr_boot_census.py`, 20 frames / 211 598 instructions):** the ONLY
   divergence event in the entire window is this `$2002` poll loop (`8045`/`8048`/`804A`). **Zero** cost
   mismatches or stream forks at `$4014` (OAM-DMA trigger), at DMC registers (`$4010-$4013`), or at any
   interrupt vector. The divergence is 100% PPU-`$2002`-vblank-phase.

**Conclusion.** Y = 3-vs-4 is a DOWNSTREAM consequence of R1's corrected `$2002`/PPU-vblank phase (the same
phase that closes `cpu_interrupts_v2` / the C1 win and that R1 promoted to default as `$2002-flag-timing`).
That phase shifts where EVERY `$2002`-poll loop in the ROM exits, which cascades into the cumulative
CPU-vs-DMA phase that `CheckDMATiming` measures. **A DMA model — burst or interleaved — cannot change this**;
the DMC halt placement is already identical to default (premise-check) and the DMA path injects zero
divergence (step 1). The from-scratch burst-DMA PROJECT is therefore **misdirected and is HALTED.** The
scaffold (BOOT tracer + 3 census scripts + `burst_dma.rs` skeleton + the feature flag) is retained as the
diagnostic substrate but the burst engine is NOT built.

**What this means for the 4-test DMA tail.** It is one axis with the C1 win: under R1's (more hardware-faithful)
`$2002` phase, C1 passes and the DMA tail's `CheckDMATiming` lands Y=3; under default's `$2002` phase, the DMA
tail passes (Y=4) and C1 fails. Real hardware satisfies BOTH, so a phase that closes both exists — but it is a
PPU/`$2002`-vblank-sub-cycle refinement on the C1 master-clock surface, NOT a DMA rewrite. Pursuing it means
re-opening the `$2002`-flag-timing / master-clock vblank phase (the C1 axis), with the C1 trio as the
regression floor. Options for the user: (i) ship the C1 win behind the flag and accept default's 94.24% as the
shipping baseline with the 4 DMA tests `#[ignore]`'d under R1; (ii) re-open the `$2002`-vblank-phase on the C1
surface to find the both-pass phase (deep, same 17-rollback surface); (iii) leave R1 entirely flag-gated.
