## Root cause (confirmed, multiply)

RustyNES does **not** maintain one cycle clock. It has FIVE parallel integer counters
(`Cpu::master_clock`, `Cpu::cycles`, `LockstepBus::cycle`, `LockstepBus::ppu_clock`, `Apu::cpu_cycle`) +
TWO sub-cycle parity bits (`Apu::apu_phase`, `Apu::put_cycle`) + the DMC byte-timer, kept in lockstep only
by hand-written `+= 1` statements sitting next to each other â€” never derived from a single source of truth.
The per-cycle invariant that *should* hold (`master_clock == 12*bus.cycle + boot_offset`,
`cpu.cycles == bus.cycle âˆ’ oam_dma_cycles`, `apu.cpu_cycle == bus.cycle`, `put_cycle == (count+seed)&1`) is
maintained by convention, and several parts already do not hold.

**The reference invariant (Mesen2/ares/TriCNES, all three):** the DMC byte-timer expiry, the get/put
parity, AND the CPU cycle are all derived from ONE monotonic counter advanced exactly once per CPU cycle.
The DMC timer does not free-run; its expiry cycle is a pure function of that counter (Mesen
`_state.CycleCount++` in `StartCpuCycle` drives `ProcessCpuClock`â†’`NesApu::Exec`â†’`_currentCycle++` 1:1, and
the DMA gate is `getCycle = (CycleCount & 1)`; ares locks CPU+APU threads at `rate()==cpuDivider`; TriCNES
toggles `APU_PutCycle` and ticks the DMC timer in the same `_EmulateAPU` call). So the reload-fire-cycle is
deterministic relative to the CPU instruction stream, and `CheckDMATiming`'s fixed-clockslide walk yields a
stable Y.

**Why RustyNES gets Y=3:** `put_cycle` (which decides the DMC GET) is phased to `bus.cycle`, but
`bus.cycle` counts OAM-DMA cycles while `cpu.cycles` (the CPU instruction stream `CheckDMATiming` counts
against) does not. The OAM 513-vs-514 alignment (`bus.rs:1585`, keyed on `bus.cycle & 1`) and the deferred
`dma_mc_consumed` lump-fold inject a 1-cycle parity error between `put_cycle` and `cpu.cycles` that
accumulates over the run. By the time `CheckDMATiming` runs, the whole (locally-coherent) parity pair is 1
cycle off the CPU stream â†’ the reload GET lands one CPU-cycle off â†’ Y=3. (Confirmed: in a local trace
`apu_phase`/`put_cycle` are perfectly complementary â€” RA-1 locked them â€” so it is a CUMULATIVE drift, not a
local one, exactly the R-0 verdict: *"the get-half is correct globally ONLY if self.cycles parity tracks
CycleCount parity at EVERY fetch, which requires every DMA span (OAM+DMC) to match Mesen's exact per-DMA
value â€” a GLOBAL cumulative property, not a local gate."*)

## The rewrite: collapse to ONE integer clock

**Target:** a single monotonic `master_clock: u64` (master-clock units) owned by the CPU, incremented once
per CPU cycle (incl. every DMA cycle), with EVERYTHING phase-related derived by fixed arithmetic â€” no second
flip-flop, no independent seed. `cpu.cycles`, `bus.cycle`, `apu.cpu_cycle`, `apu_phase`, `put_cycle`, the
DMC-timer phase, the OAM 513/514 alignment, and the reload arm all read `master_clock`.
