# Status Report: DMA Tail Closure Effort (2026-06-06)

## Branch Context
- **Worktree:** `RustyNES_v2-from-main`
- **Branch:** `fix/v2.0-dma-tail-resolution` (active development)
- **Goal:** Resolve the 4-test DMA tail residual (CheckDMATiming Y=3 -> 4) without breaking the AccuracyCoin DMASync loop (the "wedge").

## Technical Progress & Analysis

### 1. The Y=3 Failure Root Cause
In the Master-Clock (R1) interleaved model, the DMC DMA span is emergent. Because reloads can be armed on either PUT or GET cycles, they sometimes start on a GET cycle, resulting in a 3-cycle duration (Halt-Dummy-Get). Real hardware (and the default RustyNES build) schedules reloads to start on a PUT cycle, resulting in a 4-cycle duration (Halt-Dummy-Align-Get).

### 2. The Architectural "Wedge"
Previous attempts to force a 4-cycle duration by delaying the arming phase (to ensure a PUT start) successfully locked Y=4 but caused the emulator to wedge (hang) on the AccuracyCoin battery.
- **Finding:** The wedge occurs because the test ROM uses a hyper-precise loop to synchronize the CPU to the DMC timer (`LDA $4015` polling).
- **Landmark Drift:** Deep tracing revealed that at the critical synchronization landmark, the DMC timer value in R1 differs from the default build (e.g., 26 vs 18). Even a 1-cycle shift in the absolute GET landing relative to the CPU instruction breaks the sync loop.

### 3. Sources of Cycle Drift
My investigation identified two primary sources of timing divergence between the Default and R1 implementations:
- **`mc-r1-dma-cycle-budget` Interaction:** When enabled, this feature allows DMA halt cycles to "eat" the idle cycles of the current instruction (e.g., the last cycle of a NOP). This causes the CPU to effectively execute the instruction faster than the default build, leading to total cycle drift.
- **OAM-DMA Parity Inversion:** In the default build, `self.cycle` (parity) is updated *after* OAM DMA. In R1, the OAM DMA is still serviced in a burst loop, but it updates `self.cycle` *internally* via `tick_one_cpu_cycle`. This results in the OAM DMA starting one parity phase off from Default, affecting the total cycle count and timer alignment.

### 4. Current State
I have successfully:
- Reproduced the Y=3 failure and the Y=4 wedge.
- Isolated the cause of the wedge to total cycle drift and initial timer phase mismatch.
- Verified that `mc-r1-dmc-phase-minus1` (clocking the timer on PUT cycles) is a valid lever to decouple span duration from GET position, but it requires the substrate to be perfectly cycle-aligned to the default build to resolve the sync loop.

## Next Strategic Steps
The bridge to full AccuracyCoin closure requires a "Phase 2" R1 substrate alignment:
1.  **Parity Alignment:** Harmonize the OAM-DMA entry parity in `bus.rs` between the two paths.
2.  **Budget Correction:** Refine the CPU loop (`read1`/`write1`/`idle_tick`) so that DMA halts do not borrow cycles from instructions that would desync the total cycle count relative to the default build.
3.  **Unified Phase:** Once the absolute cycle count and timer phase are locked to the default implementation's values at the test landmarks, the 4-cycle reload fix (via `phase-minus1` or trailing stall) can be engaged without wedging the test.
