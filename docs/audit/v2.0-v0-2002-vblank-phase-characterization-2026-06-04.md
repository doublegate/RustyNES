# V-0: the $2002-vblank phase is already hardware-correct under R1; the cold-boot alignment is INERT for Y

**Date:** 2026-06-04
**Branch:** `feat/v2.0-from-main`
**Axis:** V (close the 4-test DMA tail via the $2002-vblank phase — the user-chosen direction after P2 proved the burst-DMA rewrite misdirected)
**Floor:** `5ceab83` + `a8e482e` (R1 92.09%, C1 closed, DMA 6/10); default 94.24% byte-identical.

## Question

P2 proved the tail (`CheckDMATiming` Y=3 vs default's 4) is driven by R1 observing the boot
`$2002`-vblank poll exit one iteration earlier than default, not by any DMA model. Real hardware
satisfies BOTH C1 (on-time NMI/flag) and the DMA tail (Y=4), so a both-pass `$2002`-vblank
sub-cycle phase should exist. V-0: characterize it.

## Findings

### 1. The VBL-flag SET dot is identical (241, dot 1) in R1 and default

`crates/nes-ppu/src/ppu.rs:1272-1276` sets `PpuStatus::VBLANK` at `scanline == vblank_start_line()
&& dot == 1`, ungated. Only the /NMI-line *assertion* differs (R1 dot 1, default dot 3 lag-comp).
So the poll-exit difference is NOT the flag-set dot.

### 2. R1's poll-exit is HARDWARE-CORRECT (nesdev reference alignment)

The poll-exit difference is the read-SAMPLE phase. R1's `start_cycle` advances the PPU to
`master_clock - PPU_OFFSET` BEFORE the read (Mesen `_cpuDivider=12`/`_ppuOffset=1`/read-split,
`crates/nes-cpu/src/cpu.rs:50-66`), so a `$2002` read samples PPU state at read-start. Default's
lockstep runs the PPU 3 dots AFTER the read (`on_cpu_cycle`), so its read samples a PPU lagging ~1-3
dots and catches VBL-set one iteration LATE.

nesdev `PPU_frame_timing` (the `ppu_vbl_nmi_timing` spec) defines the reference alignment as the one
where "a read will see a change to a flag **if and only if it starts at or after the PPU tick where
the flag changes**." R1 samples at read-start = `master_clock - ppuOffset` = exactly this. **R1 is
the reference; default is the lag-comp artifact.** So R1's earlier poll-exit is correct and must NOT
be "fixed" toward default.

### 3. The cold-boot CPU/PPU alignment is COMPLETELY INERT for Y (24-value sweep)

R1's reset adds `master_clock += CPU_DIVIDER_NTSC` (12), but the cited Mesen reset is
`_masterClock += cpuDivider + cpuOffset` (13) — R1 drops the `+PPU_OFFSET`, leaving it one
master-clock behind the reference power-up alignment. Hypothesis: this shifts every poll-exit -> Y=3.

Tested behind `mc-r1-coldboot-ppuoffset` + a runtime `COLDBOOT_EXTRA_MC` atomic sweep
(`crates/nes-test-harness/src/bin/sweep_coldboot.rs`), extra = 0..23 (the full CPU/PPU sub-cycle
alignment space, 2 CPU cycles / 8 PPU dots and beyond):

```
extra_mc   Y   DMC+OAM  Explicit  Implicit  $4015
  0..23     3   0x06     0x06      0x06      0x0A   (IDENTICAL for every value)
```

**Every cold-boot alignment gives Y=3 and the same 4 failures.** The boot CPU/PPU phase washes out:
the game re-synchronizes to vblank every frame (via NMI / `$2002` poll) and `CheckDMATiming`
re-pins the CPU to the DMC/APU via DMASync, so the power-up offset has no residual effect at the
test. (`+PPU_OFFSET` does shift the test START frames ~13 earlier — execution changes — but Y does
not.)

## Conclusion

The user-chosen "$2002-vblank phase" axis does **not** expose a new localized both-pass degree of
freedom:

- R1's `$2002` read-sample timing is already hardware-correct (must not change toward default).
- The only legitimate pure CPU/PPU-phase knob (cold-boot power-up alignment) is **provably inert for
  Y** across all 24 sub-cycle values — it washes out at the per-frame vblank re-sync + DMASync re-pin.

With the PPU timing correct, the residual Y=3 is the **APU/DMC-vs-CPU cumulative phase**: R1 has
Mesen's CPU/PPU access but default's (lockstep-era) APU phase, and with R1's now-correct (earlier)
poll-exit the default-tuned APU phase lands Y=3 instead of the hardware Y=4. That APU phase is locked
to CPU-cycle parity (`put_cycle == cyc&1`), and every adjustment to it breaks the 6/10 DMASync
(exhaustively refuted upstream: 10+ levers — dummy-read, abort-cancel, fixed-span, arm-align,
timer-jitter, span-constancy, get-gating, arm-delay1, inline-service, dmc-oam; see
`v2.0-f2-recoupling-effort-2026-06-03.md` + memory `project_v2_from_main_pivot`).

**The both-pass phase, if reachable, requires matching Mesen's FULL APU+DMA cumulative phase under
R1's master-clock access shape — the deep multi-week re-architecture (DMC-timer + DMA-span +
CPU-cycle on one integer clock natively under `master_clock`/`run_ppu_to`), NOT a localized
`$2002`/alignment edit.** This re-converges with P2's and the F-2 effort's standing conclusion: the
4-test DMA tail and the C1 win are one axis, and no surgical change separates them.
