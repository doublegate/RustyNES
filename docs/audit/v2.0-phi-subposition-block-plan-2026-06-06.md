## The target (precisely diagnosed this session — supersedes "span 3→4")

The 4-test DMA tail (`CheckDMATiming Y` 3→4 → Implicit/Explicit Abort + DMC+OAM) is
NOT a DMA span bug: per-cycle trace (`9f062d5`) + TriCNES cross-ref prove our reload
DMA is **already span 4** and structurally matches TriCNES. `Y=3` is the **GET
fire-phase / φ-sub-position**: when the DMC GET drives the sample (`$00`) onto the
open bus relative to the CPU's `LDA $4000` sample point. A 1-sub-cycle (φ1/φ2)
offset → `CalculateDMADuration` catches `$00` one 1-cycle slide off → Y=3 vs 4. H2
(a CPU-cycle-count error in the measurement loop) is ruled out by nestest 0-diff.
The GET is DMASync-coupled (DMASync detects the same GET-`$00`), so a *uniform*
shift is absorbed and a *cadence* shift starves — the inert/wedge dichotomy behind
all 20+ prior levers. The fix must place the GET's open-bus drive + the CPU read at
the correct φ sub-positions **intrinsically**, matching TriCNES, on the C1-safe
Stage-D base (OAM unify, `7ae11d2`).

The same φ-sub-position axis is what BG Serial In + $2007 Stress need (PPU sub-dot);
`mc-ppu-subpos` is ported (`866f770`) as the PPU-side component (inert alone on R1 —
it needs this master-clock sub-position too).
