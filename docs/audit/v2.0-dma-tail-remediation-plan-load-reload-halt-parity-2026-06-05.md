## Two questions, answered

### (1) Why does the register-conflict path ($4015/$4016/$4017) never fire? ‚ÄĒ It SHOULDN'T (correct).
`CheckDMATiming` plays a 1-byte sample at `$FFC0` (`$4012=$FF`, `$4013=$00`). The DMC fetch address is
`$FFC0`; `$FFC0 & 0x1F = 0`, so the 2A03 register-activation address (6502-core bits 15-5 = `$4000`-range,
DMA bits 4-0 = 0) is `conflict_addr = $4000` ‚ÄĒ a no-op write-only register. The register-conflict path
(`$4015`/`$4016`/`$4017`) is *designed* to fire only when the DMC fetch addr's low 5 bits select one of
those registers (DMC Bus Conflicts `$046B`, `$4016` tests). For `CheckDMATiming` it correctly does
nothing. **Not a bug.** `CheckDMATiming` Y instead uses the DMC GET driving its fetched **sample byte**
(`$00`) onto the open bus (`raw_cpu_read($FFC0)` sets `open_bus=$00`, `bus.rs:2355`; `LDA $4000` reads it,
`bus.rs:2329`; ROM `AccuracyCoin.asm:18661`).

### (2) Why is the GET span 3 in RustyNES vs 4 on hardware? ‚ÄĒ The reload halt lands on the wrong parity.
nesdev `DMA.xhtml:113-115` (definitive): *"Load DMAs... are scheduled to halt the CPU on a **get** cycle
during the 2nd APU cycle after the write... Reload DMAs... are scheduled to halt the CPU on a **put**
cycle... load DMAs take 3 cycles and reload DMAs take 4."* The cadence:
- **LOAD** halt=get ‚Üí dummy=put ‚Üí get-read = **3** (halt, dummy, get).
- **RELOAD** halt=put ‚Üí dummy=get ‚Üí **align=put** ‚Üí get-read = **4** (halt, dummy, align, get).

`CheckDMATiming` uses a **looping** sample (`$4010=$4E`/`$4F`, loop bit set), so after the first (load)
byte it measures **reload** DMAs ‚Üí must be **4**. RustyNES:
- **LOAD arm** (`apu.rs:294-...` `write_status`): HAS the parity-chosen delay (`dmc_dma_delay = put_cycle ?
  3 : 4`), so the load halt lands on a GET cycle ‚Üí span 3. **Correct.**
- **RELOAD arm** (`apu.rs:737-755` `dmc_step_reload_arm`): just sets `pending_dmc_dma = true` at
  buffer-empty with **NO parity alignment**. The halt then starts on the next `read1` cycle at WHATEVER
  parity (`cpu.rs:774`), landing on a GET cycle ‚Üí span 3 (like a load). **This is the bug** ‚ÄĒ RustyNES
  does not schedule the reload halt on a PUT cycle.

## How the reference emulators do it (port target)

- **ares** (cleanest, `fc/apu/dmc.cpp` + `fc/cpu/timing.cpp:58-104`): RELOAD fires `dmcDMAPending()`
  immediately at the bit-counter wrap; LOAD fires after `dmaDelayCounter = periodCounter&1 ? 2 : 3`. The
  `dma()` loop branches on `io.oddCycle` (the APU get/put phase): halt-on-get ‚Üí dummy(put) ‚Üí get-read = 3;
  halt-on-put ‚Üí dummy(get) ‚Üí align(put) ‚Üí get-read = 4. The 3-vs-4 falls out of the phase, no special case.
- **Mesen2 / tetanes** (CPU-parity proxy): RELOAD `StartDmcTransfer` immediately; LOAD `_transferStartDelay
  = (CycleCount&1)?3:2`. The `ProcessPendingDma`/`handle_dma` loop's `getCycle=(CycleCount&1)==0` produces
  the alignment cycle for the reload case.
- **TriCNES** (most faithful, `Emulator.cs:4216-4341`): holds `DMCDMA_Halt` until a GET cycle clears it, on
  a true `APU_PutCycle` phase. Models the abort parity-inversion too.

**RustyNES already has** the get/put phase bit (`put_cycle`, `apu.rs:138`), the load delay
(`dmc_dma_delay`), and the loop with halt/dummy/align/get (`dmc_dma_step_impl`, `bus.rs:2179-2252`). **The
only missing piece is scheduling the RELOAD halt to land on a PUT cycle** (and confirming the load lands
on GET). The reload halt currently lands on whatever parity the buffer-empty timing gives.

### DT-P1 ‚ÄĒ schedule the RELOAD halt on a PUT cycle (MAKE-OR-BREAK)
Behind `mc-r1-dmc-reload-halt-parity` (default-off). Port the ares/nesdev model: the RELOAD DMA must halt
on a PUT cycle, the LOAD on a GET cycle. Implementation options (DT-P0's trace picks the cleanest):
- **(a) Gate the halt-start on the scheduled cycle type** in `dmc_dma_step_impl`/`read1`: the first
  (halt) cycle starts the DMA only when the current cycle matches the scheduled type (reload‚Üíput,
  load‚Üíget); if it's the wrong type, the CPU performs its read NORMALLY (no halt this cycle) and the halt
  starts on the next correct-type cycle. The CPU is NOT halted during the "wait" (it's a failed-halt
  attempt, not a stall ‚ÄĒ nesdev `DMA.xhtml:31` "tries again next cycle"), so this does NOT add a halt
  cycle; it shifts the reload halt-start by 0-1 cycle to the put phase, after which the loop's existing
  align cycle yields span 4.
- **(b) Align the reload pending-fire phase** so the resulting halt lands on put (adjust when
  `dmc_step_reload_arm` sets pending relative to `put_cycle`, ares-style: fire at the bit-counter wrap on
  the phase that puts the halt on put).
**MAKE-OR-BREAK GATE:** reload span 3‚Üí4 + `CheckDMATiming` Y 3‚Üí4 + the 4 tail tests pass + **6/10 hold +
C1 trio holds + no DMASync wedge + default byte-identical**. Critically, DMASync self-calibrates (it spins
until the `$00`), and the 6/10 are duration-TOLERANT, so a hardware-correct reload=4 should NOT break them
(this is WHY it differs from the refuted global arm-delay/arm-align, which forced the LOAD parity too and
broke the load DMAs). If the 6/10 break or it wedges, REPORT ‚ÄĒ the load/reload distinction may interact
with the abort/OAM cases; do not thrash.

## Why this is genuinely new (not a refuted lever)
The 10+ refuted levers all shifted parity/timing GLOBALLY (get-parity-flip, apu_phase-flip, cold-boot,
arm-delay/arm-align, trailing-stall, span-collapse) ‚ÄĒ they moved the LOAD and RELOAD together, breaking the
load DMAs (span 3) the 6/10 depend on, or shifted the GET off the `$4000` coincidence (wedge). **This fix
is LOAD/RELOAD-SPECIFIC: it leaves the load halt on GET (span 3, already correct) and moves only the RELOAD
halt to PUT (span 4) ‚ÄĒ the exact hardware behavior all four reference emulators implement and RustyNES
omits.** It is the first lever that targets the actual root cause (the missing reload-halt-parity), not a
global phase shift.
