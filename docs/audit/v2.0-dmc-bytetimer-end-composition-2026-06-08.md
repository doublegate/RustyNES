# Program M — DMC byte-timer FIRE-PHASE move (cycle-START -> cycle-END): composition tested, REFUTED as a standalone closure (2026-06-08)

## TL;DR

The session's pinned root cause (the DMC byte-timer is clocked at cycle-START
under `mc-r1-apu-unified-clock`; the references clock it at cycle-END, after the
CPU's bus access) was implemented behind a new flag composing the landed
M-1 + M-2 prerequisites. **Both promote orderings were measured.** Neither closes
Explicit DMA Abort or DMC DMA + OAM DMA; both REGRESS the DMA tail. **C1 is
preserved, nestest is 0-diff, the floor (flag-off) is byte-identical.** No
regression was forced into the floor (ADR-0002).

The composition the task hypothesized would "stop being absorbed" is **inert on
DMC+OAM** (structural reason below) and **mis-shifts Explicit/Implicit** (the
abort-arm is calibrated to the START position). The isolated byte-timer-position
move on the `cpu_clock` path is provably insufficient: the genuine closure
requires the **M-2 counter-collapse** — moving the get/put `put_cycle` flip to
end-of-cycle alongside the byte-timer AND applying the reorder inside the
OAM-overlap helper — while keeping the φ2 IRQ-sample dot independent of DMA
length to preserve C1.

## Why it does not close (the precise per-cycle coupling — the deliverable)

1. **DMC+OAM is inert because the overlap helper owns the byte-timer clock.**
   The DMC DMA + OAM DMA test measures spans DURING an OAM DMA, where
   `LockstepBus::service_dmc_dma_during_oam` (bus.rs:2067) drives every span
   cycle with `tick_one_cpu_cycle()` — NOT the `start_cycle`/`end_cycle` pair.
   `cpu_clock_apu_dmc` (where `dmc_tick_end` lives) is therefore NEVER called in
   the overlap, so moving the byte-timer to end-of-cycle has no effect on the
   measured `$50-$6F` sweep (byte-identical to floor). Closing DMC+OAM requires
   the byte-timer-position change INSIDE the overlap helper (M-2), not on the
   `cpu_clock` path.

2. **The within-cycle order is only HALF-moved.** This flag moves the byte-timer
   to end-of-cycle, but the get/put parity flip (`put_cycle = !apu_phase`,
   apu.rs ~1036) STILL happens at cycle-START (in `tick_with_external`, BEFORE
   the CPU's `bus.read`). TriCNES flips `APU_PutCycle` AFTER `_EmulateAPU`
   (after the access). So the DMA get/put decision still consumes the
   START-flipped parity while the byte-timer now ticks at END — the two are
   split across the cycle, the opposite of the reference's single
   "access -> tick -> flip" order. The reference invariant is unreachable
   without also moving `put_cycle` (and the bus's `get = !put_cycle` DMA
   decision) to end-of-cycle.

3. **Explicit/Implicit regress because the abort-arm is START-calibrated.** The
   `mc-r1-dmc-halt-subpos` reload pre-arm and the plain reload-arm both live in
   `dmc_step_reload_arm`; moving them to end-of-cycle shifts the abort-arm
   recognition by the within-cycle delta. The Implicit `mc-r1-dmc-halt-subpos`
   fix (the landed Y 3->4 lever) is calibrated to the START position, so it
   breaks (Implicit PASS -> FAIL); Explicit's idx[7] rotates the wrong way.

## The minimal closure (what M-2 actually requires)

Moving `put_cycle` to end-of-cycle = the RW-2 cumulative-OAM fold, which the
project history records as C1-regressing (92->89%). The plan's **M-2
counter-collapse** is the principled form: one canonical CPU master-cycle counter
from which BOTH `apu_phase` (byte-timer clock) AND `put_cycle` (DMA get/put
parity) derive and flip at the SAME end-of-cycle point (after the CPU access),
with the φ2 IRQ-sample dot kept on the DMA-length-independent `master_clock`
(handle_interrupts, cpu.rs:616-625) so C1 stays invariant. The byte-timer reorder
must then be applied identically inside `service_dmc_dma_during_oam`. The
isolated `cpu_clock` byte-timer-position move (this flag) is provably
insufficient and is retained default-off as the validated diagnostic that pins
this conclusion.

## DO-NOT-RETRY (measured this session)

* byte-timer-at-END, promote-BEFORE (over-shifts every DMA; broad regression).
* byte-timer-at-END, promote-AFTER, on the `cpu_clock` path only (inert on
  DMC+OAM; regresses Implicit; mis-shifts Explicit idx[7]).
* Do NOT move `put_cycle` alone (= RW-2, C1-regressing per project memory) —
  it must come as the full M-2 counter-collapse with the C1-invariant φ2 sample.
