**Date:** 2026-05-24

**Status:** **LANDED** (structural). Live trunk build verification

## Goal

Convert `nes-frontend` from binary-only to `[lib] + [[bin]]` so the
same module tree powers both the native desktop binary and the
wasm32 `cdylib` artifact that `trunk` consumes. Add a minimal
wasm-bindgen entry point + a placeholder `index.html` + `Trunk.toml`.

**Sprint 1.2's bound**: `cargo build --target wasm32-unknown-unknown
-p nes-frontend` continues to succeed (preserved from Sprint 1.1);
the new `lib.rs` + `wasm.rs` paths compile cleanly; `index.html` +
`Trunk.toml` are in place. Actually loading and running the wasm
artifact in a browser is Sprint 1.3 scope.

## What landed

### `crates/nes-frontend/Cargo.toml` — `[lib]` target added

```toml
[lib]
name = "nes_frontend"
path = "src/lib.rs"
crate-type = ["cdylib", "rlib"]

[[bin]]
name = "rustynes-v2"
path = "src/main.rs"
```

The `cdylib` is what wasm-bindgen + trunk consume to produce the
`.wasm` artifact. `rlib` preserves the regular Rust crate-graph
integration so the `[[bin]]` can import from the lib without
duplicating sources.

### `crates/nes-frontend/src/lib.rs` (NEW)

Declares all the existing modules as `pub mod` so both the binary
and the wasm `cdylib` can use them:

```rust
pub mod app;
pub mod audio;
pub mod config;
pub mod debugger;
pub mod gfx;
pub mod input;
pub mod ntsc;
pub mod save_state;

#[cfg(target_arch = "wasm32")]
pub mod wasm;
```

### `crates/nes-frontend/src/main.rs` — converted to thin shim

The binary no longer declares modules directly (those moved to
`lib.rs`); it just `use nes_frontend::app;` and delegates to
`app::run(rom_path)`. All native behaviour preserved.

### `crates/nes-frontend/src/wasm.rs` (NEW)

Minimal wasm-bindgen entry point gated `#[cfg(target_arch =
"wasm32")]`:

```rust
#[wasm_bindgen(start)]
pub fn start() -> Result<(), JsValue> {
    console_error_panic_hook::set_once();
    web_sys::console::log_1(&JsValue::from_str(
        "RustyNES v2 wasm32 — v1.3.0 Sprint 1.2 placeholder boot",
    ));
    Ok(())
}
```

This is intentionally a "does the wasm boot" placeholder. The
actual winit `ApplicationHandler::spawn_app` wiring is Sprint 1.3
work — it requires:
- `App::new_empty()` (or equivalent) so the wasm path can start
  WITHOUT a ROM path (the file picker lands later)
- a wasm-specific run loop that uses `EventLoopExtWebSys::spawn_app`
  instead of `run_app` (the latter blocks indefinitely and is
  incompatible with the JS event loop)

### `crates/nes-frontend/web/index.html` (NEW)

Minimal browser host. The `<link data-trunk rel="rust" href="../Cargo.toml">`
directs trunk to build the workspace crate; trunk generates the JS
glue that auto-invokes `start()` via `__wbindgen_start`. Placeholder
text + console-log instruction in the body. Sprint 1.3 will add the
actual canvas + ROM picker UI.

### `crates/nes-frontend/web/Trunk.toml` (NEW)

```toml
[build]
target = "index.html"
public_url = "/"
release = false

[serve]
address = "127.0.0.1"
port = 8080

[tools]
wasm_bindgen = "0.2.100"
```

Pinned `wasm_bindgen` version to match the workspace dep. Sprint 2's
CI workflow will set `public_url = "/RustyNES_v2/"` for the GitHub
Pages mount point.

### `crates/nes-frontend/src/config.rs` — minor clippy fix

Pre-existing `clippy::too_long_first_doc_paragraph` warning on
`PadBindings`'s doc comment surfaced when the lib restructure
exposed the type at module-path-1. Fix: split the first paragraph
across two paragraphs. Drive-by; not Sprint-1.2-specific behaviour.

## Tooling block — live trunk build verification deferred

`cargo install --locked trunk` failed mid-session because of a
libdeflate C compilation issue on the build host (unrelated to the
project — it's a system-level C-compiler / glibc symbol mismatch in
a transitive dep). The Sprint 1.2 deliverable is structurally
complete in the source tree, but I couldn't run an end-to-end
`trunk build` to produce the `dist/` artifact and verify the
wasm-bindgen glue generates correctly.

**Sprint 1.3 work item**: get trunk installed (either via the
host fix, or via a Docker-based trunk runner, or by using
`wasm-bindgen-cli` + `wasm-pack` manually as a fallback). The
first concrete validation Sprint 1.3 owes is:

```bash
cd crates/nes-frontend/web/
trunk build
# Verify dist/ contains index.html + .wasm + .js glue
# Open dist/index.html in a browser; check devtools console
# for the "RustyNES v2 wasm32" boot log from `wasm.rs::start`.
```

If the boot log appears, Sprint 1.2 is fully validated. If not, the
diagnostic from the browser devtools console points at the next
issue (likely a wasm-bindgen version mismatch or a missing web-sys
feature).
