## The TriCNES mechanism, as ported (the citations)

* **`APU_DelayedDMC4015`** (Emulator.cs:973 decl; 9512 set; 1214-1224 apply):
  every `$4015` write latches the DMC bit into `APU_Status_DelayedDMC`
  (Emulator.cs:9506) and sets the countdown to `APU_PutCycle ? 3 : 4`
  (9512 — "it will be decremented later this cycle, so it's really like
  2 : 3"). The countdown decrements EVERY CPU cycle in `_EmulateAPU` after the
  get/put branch work; at 0 the status APPLIES:
  `APU_Status_DMC = APU_Status_DelayedDMC; if (!APU_Status_DMC)
  APU_DMC_BytesRemaining = 0;` (1219-1223). A second write during the window
  resets the countdown with the new target — last write wins.
* **What stays IMMEDIATE at the write**: the enable-side `StartDMCSample`
  (9514-9517, only when `BytesRemaining == 0`), the load-delay arm
  `DMCDMADelay = 2` gated on `APU_Silent` (9519-9522), and the DMC IRQ-flag
  clear (9529).
* **The explicit don't-abort edge** (9533-9537): a DISABLE coinciding with
  "the APU cycle that fires a DMC DMA" —
  `(timer == 2 && get) || (timer == rate && put)` in TriCNES CPU-rate units —
  extends the countdown to `put ? 5 : 6` so the just-armed reload completes.
  Uniform unit mapping to our APU-rate byte-timer: `ours = (tricnes - 2) / 2`
  → `(timer == 0 && get) || (timer == timer_period && put)`.
* **The service gate** (`_6502` line 4218):
  `DoDMCDMA && (APU_Status_DMC || APU_ImplicitAbortDMC4015) && CPU_Read` — a
  pending/halted DMC DMA whose APPLIED status dropped is simply not serviced;
  the CPU resumes mid-DMA (the abort is EMERGENT). The stale `DoDMCDMA` /
  `DMCDMA_Halt` persist and resume if the status re-applies.
* **`$4015` reads** (9127/9260 + the 9268 footnote): bit 4 =
  `BytesRemaining != 0 && APU_Status_DelayedDMC` — a read right after a
  disable write must see bit 4 clear even though `bytes_remaining` is not yet
  zeroed.
* **Natural end** (`DMCDMA_Get`, 4148-4160): a non-looping sample's end
  clears `APU_Status_DMC` immediately.
* **The implicit-abort trio**: `APU_SetImplicitAbortDMC4015` latched at an
  ENABLE in the pre-fire window `(timer == 10 && get) || (timer == 8 && put)`
  (9540-9545) = ours `(4, get)/(3, put)`; transferred at the shifter-consume
  edge where it arms the DMA regardless of the buffer (1163-1175); the
  override dies at the END of the first pending cycle (9000-9003) — one
  serviced halt cycle on a read, none on a write ("it won't run at all",
  Loop2's JSR-push kill).

Cross-checked against Mesen2 `DeltaModulationChannel::SetEnabled` /
`NesCpu::StopDmcTransfer` (the `_needHalt` cancel) — Mesen models the same
physics with a transfer-cancel instead of a gate; TriCNES's gate is the shape
that composes with our unified engine.

## Refuted this session (DO-NOT-RETRY)

1. **Consume-edge quantization of the persistent `needs_dma` reload re-arm**
   (both global and `loop_flag`-scoped): starves the looping reload chains in
   EVERY context — EXPLICIT collapses to `01`s, `$500/$520` wedge at `2A/2B`.
   The prompt re-arm is load-bearing for the calibrated contexts; only the
   hardware's free-running-grid quantization expressed some other way can
   close `$540[8,9]`.
2. **Write-visible silence sampled at the unshifted (pre-collapse) byte-timer
   position** (a 2-CPU-cycle-lagged copy for the enable-side `APU_Silent`
   gate): INERT — the silence transition is not the `$540` X=8,9
   discriminator.
3. **The OAM-DMA active-window mirror decode** (Mesen `ProcessDmaRead`
   `enableInternalRegReads` arm, incl. the faithful `obMask` bus-conflict
   AND-merge + the `prevReadAddress` double-read skip): the `$4016/$4017`
   VALUE path (the merged byte / open-bus store) wedges the AccuracyCoin
   framework bootstrap — the battery stops at the APU-Reg-Activation test
   (the sub-test ROM `apu-reg-activation.nes` goes 0x1A-in-32-frames →
   never-completes). Bisected: the `$4015` internal-read arm alone is
   harmless but flips nothing (still 0x1A); ANY `$4016/$4017` value change
   wedges. Reverted fully per the no-speculative-additions discipline
   (the Sprint-2.4 rollback, now narrowed to the input-register value path).
   `REENABLE_BUMP` ±1: inert for `$540`.

## The residual map (the 2 open tests)

* **Implicit DMA Abort [error 4]** = KeyLoop3, exactly `$540[8,9]` (`04` vs
  KEY `00`). Loop3 (AccuracyCoin.asm:13367+) keeps the loop bit set
  (`$4010=$4E`) so no implicit abort occurs — it brackets pure reload-chain
  phase: the X-swept re-enable restarts the looping sample against the
  free-running byte-timer grid; the measured window (after `Clockslide_500` +
  the compensating slides) sees the chain's reload iff the restart's load
  completed before the grid boundary (the cliff). Hardware's cliff is at
  X=10; ours at X=8 — our restart chain locks one grid boundary earlier for
  those two X. All other chain contexts (DMC+OAM, Explicit, `$500`, `$520`)
  are KEY-exact, so this is a Loop3-context-specific 2-cycle lead in the
  load-to-first-consume coupling. The named diagnostic: tricnes_xdiff /
  trace_dma_4015 per-cycle at the X=8 vs X=10 Loop3 enables, landmark-locked
  on the `$4010=$4E` + `$4015=$10` write pair.
* **APU Register Activation [error 6]** = Test 6 (asm:8213+, the OAM-content
  verification after Test 5's JSR `$3FFE` + BRK trick parks the 6502 inside
  `$4000-$401F`). Needs the active-window mirror decode, but the
  `$4016/$4017` value path cannot land until the framework-bootstrap wedge is
  cross-diffed per-cycle (the framework itself runs OAM DMAs parked on the
  input registers; our merged-value/open-bus mutation diverges from hardware
  somewhere in that choreography). The `$4015` internal-bus arm alone is
  verified harmless-but-insufficient.
