## R1c-0 RESULT (2026-06-04) — NOT the boot-seed; the offset is the DMC-clock parity (entangled with the un-wedge)

Two findings pin the source + decide separability:

**(1) The offset is CONSTANT, and it is NOT the boot-seed.** `entry_r1.csv` shows R1's DMASync-sync phase is
a constant 74 across the whole readout+abort window (not drift). The `put_cycle`/DMC-phase boot-seed is
`seed_apu_alignment(0)` hardcoded at 3 R1 setup sites (`bus.rs:399/412/429`). **Flipping it to `(1)` left Y
= 3 and the 6/10 byte-identical — only the absolute boot timing shifted (frames 1597→1591).** `(alignment &
1)` has only 2 states (0/2 → `put_cycle=true`; 1/3 → false), both Y-equivalent. So the boot-seed is
DEFINITIVELY not the lever — a bounded "pick the right seed" fix is ruled out.

**(2) The offset is the DMC-clock-vs-CPU-cycle PARITY (`!put_cycle`), entangled with the un-wedge.** The
seed flip shifting the absolute timing but NOT Y proves Y is the DMC-clock-vs-CPU-cycle RELATIONSHIP, which
is parity-LOCKED (`put_cycle ↔ cpu_cycle` parity), set by `5ceab83`'s "clock the DMC on `!put_cycle`" (the
DMASync un-wedge). That parity is right for the un-wedge but 1 cycle OFF Y=4. Default clocks the DMC on
`apu_phase` (lockstep), which gives BOTH the un-wedge AND Y=4. R1's master_clock substrate (the C1 fix)
alters the `apu_phase`-vs-CPU-cycle relationship, so `5ceab83` had to recover the DMC phase via `!put_cycle`
— right for the un-wedge, 1 off for the Y measurement.

**SEPARABILITY VERDICT: ENTANGLED (but with the UN-WEDGE, not C1 directly).** form-A proved shifting the
DMC-clock parity to the Y=4 side WEDGES the DMA battery (the un-wedge needs `!put_cycle`). C1
(`cpu_interrupts_v2`) is a separate test and may survive a DMC-phase change, but the DMA BATTERY un-wedge
does not. So the DMC-vs-CPU phase is entangled with the DMASync un-wedge via R1's master_clock's altered
`apu_phase`-vs-CPU relationship. **The fix (R1c-1, entangled path): restore default's `apu_phase`-vs-CPU-
cycle relationship under R1's master_clock so the DMC clocks like default — giving BOTH the un-wedge AND
Y=4 — while C1's IRQ-sample accounting holds.** This is the master_clock co-design: the C1 fix shifted the
CPU-vs-APU phase by ~1 cycle; the rewrite must deliver C1's IRQ-sample phase WITHOUT shifting the CPU-vs-
APU(DMC) phase the DMASync/Y depend on. Default proves both are simultaneously achievable (lockstep does
it); R1's `master_clock`/`run_ppu_to`/`end_cycle` split currently trades one for the other.

**R1c-1 target (sharpened):** the divergence is in HOW R1's `master_clock` maps CPU cycles to the
APU/`apu_phase` clock (the `start_cycle`/`run_ppu_to`/`end_cycle` accounting), shifting `apu_phase`-vs-CPU
by 1 vs default. Pin that 1-cycle map difference (compare R1's `apu_phase` at a given CPU cycle vs
default's) and correct it so the DMC can clock on `apu_phase` (like default) without `!put_cycle`, closing
both. Gate: Y=4 + un-wedge + 6/10 + C1 (`cpu_interrupts_v2`) + no-wedge. NOT the boot-seed (ruled out).

## R1c-1 trace (2026-06-04) — the apu_phase map is IDENTICAL; the offset is a cumulative CPU-cycle-COUNT parity, not the DMC clock

Compared `(cpu_cycle&1, apu_phase, put_cycle)` R1 vs default (`entry_{def,r1}.csv`):
```
DEFAULT: cyc&1=0 -> apu_phase=1, put=0 | cyc&1=1 -> apu_phase=0, put=0   (put_cycle always 0 on default)
R1:      cyc&1=0 -> apu_phase=1, put=0 | cyc&1=1 -> apu_phase=0, put=1
```
**The `apu_phase`-vs-CPU-cycle map is IDENTICAL** (apu_phase=1 ↔ cyc even, both). And in R1
`!put_cycle == apu_phase`, so R1's DMC (clocked on `!put_cycle`) fires on **the SAME phase as default's
apu_phase clocking**. **The DMC clock is NOT shifted.** ⟹ R1c-0's "apu_phase map shifted by 1" hypothesis
is REFUTED — R1's APU clock + DMC fire-phase align correctly with default.

**So the Y=3-vs-4 is NOT the DMC clock phase (identical), NOT the seed, NOT the DMC service, NOT GET→read
visibility — all ruled out.** It is a **1-cycle offset in where the CPU instruction stream (the
`CalculateDMADuration` `LDA $4000` read loop) lands relative to the DMC.** Since the DMC fires on the same
phase, R1 must REACH the read loop at a cumulative CPU-cycle count 1 off default's (mod the DMC period),
from an **odd-cycle-count divergence** in R1's accounting vs default — a single +1/−1 event (candidate: an
OAM DMA alignment 513-vs-514, a DMC span 3-vs-4 at one fetch, or an instruction-cycle accounting
difference under the master_clock). `CalculateDMADuration` walks the read 1 cyc/iter, so Y = the initial
read-loop-start-to-GET cycle distance; a 1-cycle cumulative offset = Y 3 vs 4.

**R1c-1 NEXT (the pin):** find the ODD-cycle-count divergence. Compare R1's vs default's cumulative
`cpu_cycle` at a COMMON instruction landmark (e.g. the first `$4010=$4E` of CheckDMATiming, or each test's
entry) — the running R1-vs-default difference mod 2 (and mod the DMC period). Where it flips parity is the
source. Candidates to inspect: OAM-DMA 513/514 alignment (`drain_dma` `extra = cycle&1==0 ? 514 : 513`),
the cold-boot `master_clock += 12` (even — not it), any DMC fetch where R1's span ≠ default's by an odd
count. Once pinned, R1c-1 corrects that one event's cycle count to match default → the read loop lands at
default's offset → Y 3→4. **The hopeful angle:** the apu_phase/DMC-clock alignment is already correct
(R1c-1 trace), so this is NOT a full clock rewrite — it is a single odd-cycle accounting correction in the
CPU instruction stream's absolute position vs the DMC. **The risk (the R1c-1 gate, unverified):**
correcting an odd-cycle event shifts ALL of R1's subsequent timing by 1 cycle — which could break C1 (the
IRQ-sample timing depends on the cycle count) or a 6/10. So C1-safety is NOT assumed; the make-or-break
gate is Y=4 AND C1 trio holds AND 6/10 hold AND no wedge. If the odd-cycle fix flips Y=4 but breaks C1,
that re-confirms the deep entanglement (then the co-design is required); if it holds C1, the tail closes
with a bounded fix. R1c-1's pin + gated test settles it — narrower than the original "multi-week clock
rewrite" framing, but the C1-vs-DMC tension is still the thing being tested.
