## The structural truth (the deliverable)

* The DMA span is parity-EMERGENT from `dmc_dma_step_impl`'s `dmc_halt`-clearing
  loop (bus.rs:2519-2567): `in_dmc_dma`/`dmc_halt` set on cycle 1, halt clears on
  the next get cycle, GET fires on the get cycle after that. So span = 4 if the arm
  lands before a get, 5 if before a put.
* The counter-collapse end-flips `put_cycle`, shifting the arm one parity -> the
  hardcoded-span overlap arm re-times CORRECTLY (front-parity fix), but the
  emergent reload arm lands one parity late -> +1 align cycle -> over-grow.
* **The byte-timer (which sets the arm) and the get/put decision (which sets the
  GET) MUST share one phase** (else the DMASync spin wedges — proven). So the
  arm-shift and the GET-shift are locked together. There is NO within-`dmc_dma_step`
  knob that separates the abort-context reload from the steady-state reload (they
  are the same code path, same byte-timer).
* `mc-r1-internal-cycle` is inert -> the absolute count is already right ->
  collapsing counters cannot change the relative phase that drives the over-grow.

## What the FULL closure actually requires (revised, evidence-based)

The plan's "collapse 4 counters + every-cycle-bus-access -> per-context exact" path
is **refuted as written** (internal_cycle inert; counters coherent). The over-grow
is intrinsic to the **shared byte-timer phase**, not counter drift. The two
remaining viable directions, in priority order:

1. **DMC+OAM idx[7] alone (the named sub-target, NO Explicit/Implicit coupling).**
   The counter-collapse already gives DMC+OAM `04,03,...,02,01` == KEY except
   idx[7] (`03` vs `01`) — the overlap-REGIME-TRANSITION realign at the
   `04,03`->`02,01` boundary. This is isolated to `dmc_overlap_*` (bus.rs overlap
   path), independent of the emergent-span over-grow. Closing idx[7] would make
   DMC+OAM PASS under counter-collapse (the front parity is already KEY). This is
   the highest-probability single win and does NOT touch the wedge-prone
   `dmc_dma_step`. Attack: trace the overlap iteration where OAM first overlaps the
   DMC (`dmc_overlap_begin` short?2:3 regime boundary) and fix the slot accounting.

2. **Restructure `dmc_dma_step` to an EXPLICIT span counter that anchors the GET to
   the byte-timer get cycle (no wedge) but makes the pre-GET cycle COUNT structural
   (always halt+dummy+align-to-get = 4 for reload), context-free.** The challenge
   measured: the align-to-get is what varies; making it constant requires the arm
   to land deterministically relative to the get phase in EVERY context. The only
   way to decouple the abort-context reload from steady-state is if the byte-timer
   itself is NOT globally shifted — which re-wedges. This is a genuine multi-week
   rewrite of the DMC-DMA-on-master-clock (the same axis the project memory pins as
   "ONE deferred axis = cycle-exact DMC-DMA on the master clock").
