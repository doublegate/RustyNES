## The target architecture (TriCNES, `Emulator.cs`)

1. **Per-cycle interleaved DMA, NOT a burst.** `_6502()` runs once per CPU cycle; at the top, if a DMA
   is active and it's a read cycle, it runs **ONE** DMA cycle (halt/get/put) and the CPU instruction
   does not advance. (RustyNES drains the WHOLE span in a burst at `read1` entry — divergence C.)
2. **ONE global `APU_PutCycle` flip-flop**, toggled exactly once per CPU cycle (after the APU tick),
   **seeded at power-on/reset together with the DMC timer** from `APUAlignment` (e.g. `case 0: DMC=1022,
   PutCycle=true`). Get/put parity AND the DMC timer phase share ONE seed + ONE per-cycle counter, so
   they can never drift. The variable 3/4 span consumes cycles, each toggling PutCycle → shifts the next
   DMC arm phase = the integrated feedback (Mesen's `CycleCount&1` as a single boolean). (Divergence A.)
3. **Stateful OAM alignment `OAMDMA_Aligned`** (set by OAM get, cleared by DMC get) for the
   DMC-during-OAM overlap (adds one alignment cycle per DMC-get-during-OAM). (Divergence B.)
4. **Priority + halt-clear:** GET cycle (`!PutCycle`, read) → DMC priority; PUT cycle (`PutCycle`,
   write) → OAM priority; halts clear after a get; a halt cycle re-reads the CURRENT CPU address bus;
   DMA only runs on a read cycle.

## Why interleaved resolves the feedback-vs-cascade wall

In the **burst**, the whole span runs inside one `drain_dma`, so the DMC either advances for the
entire span at once (cascade → re-wedge) or not at all (no feedback). In the **interleaved** model,
each DMA cycle is a discrete CPU step that goes through `start_cycle`/`end_cycle` → `tick_dmc` ONCE →
the DMC byte-timer advances exactly once per DMA cycle (**feedback restored**), while the DMC *arm* is
naturally gated by the "DMA in progress" state machine (it cannot re-arm a new DMA mid-span) — **no
cascade**. The shared `put_cycle` flip-flop keeps the get/put parity and the DMC fire-phase locked
together. This is precisely the integrated feedback the burst cannot express.

## Design — CPU-driven interleaved DMA under R1

**Principle:** the R1 CPU loop already advances ONE cycle at a time via `start_cycle`/`end_cycle`
(each does `run_ppu_to` + `cpu_clock`/`tick_dmc` + `handle_interrupts`). Make a pending DMA consume
those cycles ONE AT A TIME instead of bursting.

- **New CPU state:** a `dma` sub-state machine (`Cpu` or a small struct): which DMA(s) active (DMC/OAM),
  the OAM byte index, `dma_halt` flags, `oam_aligned`. Mirrors TriCNES's fields.
- **`Cpu::read1` entry (under `mc-r1-substrate`):** before the real CPU read, while a DMA is pending
  and this is a read cycle, run DMA cycles via the normal `start_cycle`/(dma access)/`end_cycle` path —
  **one cycle per loop iteration** — until the DMA completes, THEN do the CPU's own read. Each DMA cycle
  is a full R1 cycle (PPU caught up, `tick_dmc` advances the DMC once, interrupts sampled). This
  replaces the bus-side burst `drain_dma`/`service_dmc_dma` for the DMC/OAM path under the feature.
- **The `put_cycle` flip-flop:** add to the integrated per-cycle counter (toggle once per `cpu_clock`,
  i.e. once per CPU cycle including DMA cycles). Seed it at power-on/reset WITH the DMC `dmc_ext_phase`
  from a shared `APUAlignment`-style seed (the determinism PRNG already seeds CPU/PPU phase — extend it
  to seed put_cycle + DMC together). The get/put decision uses `put_cycle`, NOT `self.cycle & 1`.
- **Get/put + priority:** port TriCNES's get-cycle=DMC-priority / put-cycle=OAM-priority / halt-clear-
  after-get / halt-re-reads-CPU-address rules. The DMC get delivers the sample (`complete_dmc_dma`);
  the OAM get/put does the `$2004` store with `oam_aligned`.
- **DMC arm gating:** keep `tick_dmc` advancing the timer every cycle (incl. DMA cycles), but suppress
  the delay/reload ARM while `dma_in_progress` (so no mid-span re-arm — the un-wedging requirement, now
  expressed correctly per-cycle rather than by freezing the timer).
- **Keep main's span semantics as the validation oracle**, but the SOURCE of the span is now emergent
  from the per-cycle interleave + put_cycle, not the `noop_cycles = short?2:3` burst constant.

## Phase B implementation spec (derived from TriCNES `_6502` + `DMCDMA_*`, 2026-06-03)

DMC-only per-cycle logic (TriCNES `Emulator.cs:4216-4340`, primitives `DMCDMA_Halted`/`Put`/`Get`):
- `DMCDMA_Halted()`/`DMCDMA_Put()` = `Fetch(addressBus)` → RustyNES `replay_dma_noop_read(halted_addr)`.
- `DMCDMA_Get()` = `Fetch(DMC_addr)` + deliver → RustyNES `dmc_dma_read(addr) + apu.complete_dmc_dma`.
- Halt set when the DMC arms; **cleared after a GET cycle**; the GET always lands on a get cycle
  (`!put_cycle`). Trace: arm-before-get → halt(get)+clear, put, **get** = 3-cycle span; arm-before-put
  → halt(put), halt(get)+clear, put, **get** = 4-cycle span. The 3-vs-4 span is thus EMERGENT from
  `put_cycle` parity at arm time — divergence-A's self-consistency, not main's fixed `short?2:3`.

Per-cycle step (`bus.dmc_dma_step(halted_addr)`, called once per R1 cycle while `apu.dmc_dma_pending()`):
```
if !in_dmc_dma { in_dmc_dma=true; dmc_halt=true; capture_deferred_dma_replay(); }
let get = !apu.put_cycle();
if get && !dmc_halt { byte=dmc_dma_read(addr); apu.complete_dmc_dma(byte); in_dmc_dma=false; } // GET
else { replay_dma_noop_read(halted_addr); if get { dmc_halt=false; } }                          // Halt/Put
```
Integration: (1) `Cpu::read1` (feature) runs `while bus.dmc_dma_pending() { cycle++; start_cycle;
bus.dmc_dma_step(addr); end_cycle; }` BEFORE the real read (DMA halts only on read cycles — TriCNES
`CPU_Read`); (2) `cpu_clock_apu_dmc` uses `tick_dmc_timer_only` (timer advances, NO arm → feedback
without cascade) while `in_dmc_dma`, else `tick_dmc`; (3) `drain_dma`/`cpu_clock` skip the DMC burst
under the feature (OAM stays burst — Phase C makes it interleaved), so the DMC is CPU-driven only.

- **DMC DMA + OAM DMA** — the overlap. Needs the OAM DMA ALSO interleaved (TriCNES OAMDMA_Get/Put/

These 4 are the genuine cycle-exact tail (a focused fresh effort): full OAM interleave + the

### DMA + $4015 Read (`0x0A`) — TRACE-PROVEN root cause: the test-positioned DMC DMA does not land at `BIT $4015` under R1

User-chosen investigation (2026-06-03, session 2). `0x0A` decodes as **Test 1 PASS / Test 2 FAIL**
(`AccuracyCoin.asm` 8467-8497): Test 2 uses DMASync's 432-cyc loop to position a DMC DMA so it halts
the CPU **right before** `BIT $4015`, delaying the read so the frame-IRQ flag resolves to CLEAR by
read time (V must end CLEAR).

**A correlated default-vs-R1 trace (`trace_dma_4015` bin + the R1 `trace_end_cycle` hook) REFUTED the
earlier "frame-counter-vs-`$4015`-replay ORDER" hypothesis.** The captured Test 2 window:

```
DEFAULT (PASS) — a 4-cycle DMC DMA fires AT $BF44/$BF45, halting the CPU before the BIT:
  47411130  R $BF44 $15  dmc_pending=1
  47411131-134  r ...    in_dmc  (4 DMC DMA cycles, halt at $BF45 + GET $FFC0)
  47411136  R $4015 $10  flag=CLR  -> PASS
R1 (FAIL) — NO DMC DMA fires; the CPU runs straight through:
  48334812  R $BF44 $15  dmc_pending=0
  48334813  R $BF45 $40
  48334814  R $4015 $50  flag=SET  -> FAIL
```

Default window = **4** `dmc_dma_cycle` rows; R1 window (+/-36 cyc around the BIT) = **0**. The DMC DMA
the test positions to halt the BIT **lands at a different cycle under R1** (the cumulative DMC-span /
DMASync `$4000`-conflict phase shift), so the BIT runs un-delayed and reads the flag still SET.

So `$4015` is NOT a frame-counter-vs-replay ordering bug, and NOT a surface `$4015`-handling bug. It
is the **same DMC-DMA-landing-cycle divergence** as the rest of the cluster — `$4016`/`$2007`/Open-Bus
pass because they observe the DMA's *direct effect* on the conflict register (landing-cycle-tolerant),
whereas `$4015` Test 2 needs the DMA to *delay a later instruction by an exact count* against a
concurrent frame-counter event (acutely landing-cycle-sensitive). Confirms the "one root" conclusion:
all 4 remaining fails are the cumulative DMC-span/phase reconciliation, not four separate bugs.

Tooling shipped (diagnostic-only, no behavior change): `crates/nes-test-harness/src/bin/trace_dma_4015.rs`
+ the `Bus::trace_end_cycle` hook (default no-op; LockstepBus pushes a `CycleRecord` per R1 cycle only
under `irq-timing-trace`) + DMC-DMA-step bus-access tagging. Default build byte-identical; banked 6/10
at `5ceab83` untouched.

### Explicit / Implicit DMA Abort (`0x06`) — TRACE-QUANTIFIED root cause: R1 DMC span is uniformly 1 cycle SHORT

`trace_dma_4015` pointed at the abort windows (`$0479` / `$0478`), default-vs-R1, with DMC-DMA-span
length histograms (contiguous `in_dmc` runs):

```
Explicit DMA Abort   DEFAULT (pass 0x01): {1:3,  3:32, 4:418, 5:15}   <- normal span = 4 cyc
                     R1      (fail 0x06): {2:17, 3:187}               <- normal span = 3; NO 4s/5s/1s
Implicit DMA Abort   DEFAULT (pass 0x09): {1:5,  3:99, 4:480}         <- normal span = 4 cyc
                     R1      (fail 0x06): {2:7,  3:122}               <- normal span = 3
```

Both abort tests show the SAME signature: the **R1 interleaved DMC DMA span is uniformly 1 cycle
SHORTER than the default burst** (normal 4->3, alignment 3->2, and the default 1-cycle DMAs vanish /
collapse to 2). The abort tests measure DMA *duration* via `CalculateDMADuration`; a systematically
1-short span yields wrong durations against the answer key
(`04 04 04 04 04 04 03 04 01 01 00 00 00 00 00 00`) -> `0x06`.

The breakthrough (`5ceab83`) fixed the DMC **fire-phase** (which DMASync catches -> un-wedged, 6/10),
but NOT the **span length**. The 6/10 passing DMA tests are duration-TOLERANT (they observe the DMA's
direct effect); the 4 fails are duration-SENSITIVE, so the 1-short span only bites them. This also
explains DMA + $4015: the 1-short span (+ phase) shifts the test-positioned DMA off the `BIT $4015`
cycle.

**Where the cycle is lost:** default `service_dmc_dma` runs `noop_cycles = short ? 2 : 3` preamble
cycles + 1 get = 3 or 4 (the reload/loop DMAs here are non-short -> 4). R1 `dmc_dma_step_impl` models
a SINGLE `dmc_halt` flag (halt -> align-to-get -> get = 3), collapsing Mesen's separate `_needHalt`
+ `_needDummyRead` two-cycle preamble into one. **Fix direction (NOT yet attempted — risks the 6/10):
give `dmc_dma_step_impl` the second preamble (dummy-read) cycle so the normal span is 4, then re-gate
the get-cycle parity so the 6/10 duration-tolerant tests hold.** This is the cumulative-span
reconciliation, now pinned to a concrete 1-cycle deficit in one function.

#### Dummy-read preamble fix ATTEMPTED — INERT (reverted; DO-NOT-RETRY this form)

Implemented behind a sub-feature `mc-r1-dmc-dummy-read`: `dmc_dma_step_impl` consumes the APU's armed
`dmc_need_halt` THEN `dmc_need_dummy_read` flags one-per-cycle (Mesen `processCycle` order), GET only
when both clear AND on a get cycle (`!put_cycle`). The new model WAS compiled+active (proven by the
`dmc_halt`-never-read dead-code warning). **Result: byte-identical to the 6/10 baseline** — same
result codes, same `set_frame`s, and the `trace_dma_4015` span histogram was UNCHANGED (`{2:17,
3:187}`, NOT the targeted `{3,4}`). So adding the second preamble flag does **not** lengthen the R1
span.

**Why it's inert:** R1's span is governed by the **`!put_cycle` get-parity gating of the GET**, not by
the preamble flag count. Reverted to `66d0c86`; banked 6/10 untouched.

### CORRECTION + UNIFIED ROOT CAUSE (parity-level per-span trace, 2026-06-03) — DMC-DMA PERIOD JITTER

The "R1 span is 1 cycle short" claim above was an **ARTIFACT** of counting spans by the `in_dmc_dma`
flag: R1 clears `in_dmc_dma` BEFORE the GET cycle's trace record, while the default burst clears it
AFTER — so the `in_dmc` count dropped R1's GET cycle but kept default's. Re-counting by consecutive
**`DmaRead` runs** (GET-inclusive, consistent both paths):

```
Explicit DEFAULT: {1:368, 2:8, 3:55, 4:414, 5:1}   <- normal span = 4, PLUS truncated 1s/3s (the abort)
Explicit R1:      {3:17, 4:187}                     <- normal span = 4 TOO; almost NO truncated runs
```

So **R1's DMC DMA span is NOT short — it is 4, same as default.** The dummy-read fix was inert because
there was no missing cycle. The real differences, from the abort-correlated parity trace:

1. **Default truncates DMAs on abort; R1 does not.** Default has a spread of 1/3/4-cycle DMA runs (the
   `STA $4015` abort cutting the DMA to durations 4/3/1/0 — what `CalculateDMADuration` measures); R1
   has almost only full-4 runs.
2. **...because the DMC DMA never LANDS at the abort under R1.** Nearest-DMA-to-abort offset: DEFAULT
   = `-6,-6,-5,-4,-3,-2` (DMA right there, sweeping 1/iter — the test design); R1 = `-34,-36,-77,-75,
   -33,...` (DMA 30-80 cyc away — the timed abort misses it entirely).
3. **...because R1's DMC DMA PERIOD JITTERS.** Inter-DMA-start interval: DEFAULT = clean discrete
   periods (432×137, 450×54, 576×131); R1 = 432 but JITTERED across 429..435. Default's per-fetch DMC
   recurrence is EXACT; R1's drifts ±~3 cyc per fetch.

**Unified root cause for ALL 4 remaining fails:** R1's emergent put-cycle-parity DMC span makes the
per-fetch DMC-DMA **period jitter** (±~3 cyc) where the default burst is exact. The jitter accumulates
across the landing-sensitive tests' long clockslides (multiple `Clockslide_500`s between DMASync and
the observed event), drifting the DMA off the test-positioned cycle — the `BIT $4015` (DMA+$4015), the
`STA $4015` abort (Explicit/Implicit), and the OAM overlap (DMC+OAM). The 6/10 pass because they
observe the DMA immediately after DMASync (no accumulated drift). The `5ceab83` breakthrough fixed the
DMC fire-phase (enough to un-wedge + pass the landing-tolerant 6/10) but the emergent span is **not
period-exact**, which the 4 landing-sensitive tests detect. **The real lever is making the per-fetch
DMC-DMA period deterministic/exact (no jitter) while keeping the fire-phase** — the genuine
cumulative-per-fetch DMC-span reconciliation, now quantified as a ±3-cycle period jitter. Tooling:
`trace_dma_4015` gained `put_cycle` + abort columns (`CycleRecord::put_cycle_post`).

#### Fixed-span fix ATTEMPTED — INSUFFICIENT (reverted; sharper diagnosis below)

Behind sub-feature `mc-r1-dmc-fixed-span`: `dmc_dma_step_impl` drives a FIXED `short ? 3 : 4`-cycle
span via a `dmc_span_idx` counter (the default burst's `noop_cycles + get`) instead of the
`!put_cycle` parity, leaving the DMC fire-phase (`tick_dmc` on `!put_cycle`) untouched. It DID change
execution (DMA-test frames shifted 1608->1597; spans went `{3:17,4:187}` -> `{3:7,4:191}`), did NOT
break the 6/10, but did NOT fix the 4 — **the period STILL jitters** (`{429:4,430:5,431:10,432:134,
433:5,434:6,435:5}` vs default's clean `{432:137,450:54}`) and the aborts STILL don't land
(`0/11 <= 8 cyc`).

**Sharper diagnosis (why fixing the span COUNT is insufficient):** the jitter is NOT the span length —
it is the **span-PARITY ↔ DMC-timer-phase coupling**. The DMC timer is clocked on `!put_cycle`
(F-2 decoupling), and `put_cycle` toggles once per CPU cycle. A span of EVEN length (4) preserves the
`put_cycle` parity across the DMA; an ODD-length span (3, the `short`/load DMAs) FLIPS it — shifting
the decoupled timer phase for the next fetch by one APU clock, which moves the next buffer-empty (and
thus the next DMA start) → period jitter. The fixed-span model still emits odd (3-cycle) short spans,
AND the timer advances *during* the span (`tick_dmc_timer_only`), so the period stays jittery. **The
true lever is the DMC-timer phase being invariant to the DMA span** — i.e. either (a) re-couple the
DMC timer to the CPU integer clock (undo F-2; risks the fire-phase + 6/10 + re-wedge — that's the deep
core the pivot was built to avoid), or (b) make EVERY span even-length AND credit the timer the exact
ticks consumed during the span so the post-span timer phase is span-invariant. Neither is a localized
tweak. Reverted to `e6f0271`; banked 6/10 untouched. DO-NOT-RETRY: fixed-span-count alone.

#### Option (b) ATTEMPTED in full — REFUTED (both halves; reverted)

Pointing the per-fetch parity trace (`put_cycle` column) at the jitter showed the decisive fact:
**`put_cycle` is locked to CPU-cycle parity** (`putcyc=1 ↔ cyc&1=1` for EVERY fetch), so it never
desyncs — and even the FIXED-span trace (constant span=4) STILL oscillates ±3, with the period
correlating with the **DMA-ARM parity** (`putcyc@start=1` → 432; `putcyc@start=0` → 429..431), a
net-zero ±3 limit cycle (a 435 fetch paired with a 429). So the jitter is NOT the span (even-span is
already what fixed-span tested) — it is the **arm-parity variance**, upstream of the span.

- **Even-span half** = the `mc-r1-dmc-fixed-span` result above: constant span=4 still jitters →
  REFUTED.
- **Tick-credit half** (`mc-r1-dmc-arm-align`, nes-apu): gate the DMC reload arm in `tick_dmc` to a
  fixed `put_cycle` parity (defer 1 cyc otherwise) so every fetch arms at the same phase. Result:
  **REGRESSION** — it deferred ALL DMA arms by a cycle, shifting the DMA timings the 6/10 depend on:
  `DMA + Open Bus` 0x01→0x0A, `$2007 R/W` →0x0A/0x06, `$4016` 0x09→0x06, `DMC Bus Conflicts`
  0xE1→0x06 (and frames slid 1608→2094). Reverted.

**Conclusion: option (b) is refuted.** Any arm-phase / span manipulation that cancels the jitter also
shifts the DMA timing the 6/10 rely on — because the 6/10 observe the DMA RIGHT AFTER DMASync, so they
are sensitive to the exact arm cycle the jitter-cancel changes. **The arm-parity variance is intrinsic
to the F-2 decoupling** (the DMC timer runs on `!put_cycle`, a phase that the interleaved DMA's
variable stall lets drift relative to the CPU instruction stream). The ONLY remaining lever is option
(a): **re-couple the DMC byte-timer to the CPU integer clock (undo F-2)** so the timer phase, the DMA
span, and the CPU cycle share one clock (what the default lockstep DMA does) — the deep master-clock
core the whole pivot was built to avoid touching. That is genuinely multi-week and risks the
fire-phase + 6/10 + re-wedge; it is the standing C1-vs-DMA tension. Four localized levers
(dummy-read, abort-cancel, fixed-span, arm-align) are now refuted; the tail does not yield to a
surgical edit. Banked 6/10 (`5ceab83`) untouched; default 94.24% byte-identical.
