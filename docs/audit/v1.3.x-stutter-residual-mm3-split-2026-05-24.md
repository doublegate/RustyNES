# v1.3.x residual-stutter (post-Mailbox) + Mega Man 3 MMC3 split — audit

**Date:** 2026-05-24
**Scope:** two follow-up reports after commit `8729526`
1. stutter / uneven framerate STILL present in SMB and other titles after the
   `Fifo → Mailbox` present-mode default;
2. "screen tearing / glitchy pixels" in a horizontal band on the Mega Man 3
   stage-select screen (above SHADOW MAN / below TOP MAN).
**Status:** (1) ROOT-CAUSED + FIXED (not fully verifiable headless — needs
real-display confirmation); (2) ROOT-CAUSED + determined **pre-existing, not a
regression**, on the v2.0-deferred C1 IRQ-sample-point axis — **no code fix
applied** (surgical change is unsafe; 17+ prior C1 rollbacks).

## 1. Stutter residual — root cause

### 1.1 The "boring cause" was ruled out, but with a twist

Step 1 of the brief: confirm the `Mailbox` default is actually in effect (a
user `config.toml` with `present_mode = "Fifo"`, or a surface lacking Mailbox,
would silently re-select Fifo). A probe test parsing the **actual** user config
at `~/.config/rustynes/config.toml` found:

- The on-disk config uses an **old, incompatible schema** (`[video] vsync`,
  `[emulation]`, `[input.keyboard_p1]`, …) that does **not** match the current
  `Config` struct. `toml::from_str` fails at line 25 (`[input.keyboard_p1]` —
  `InputConfig` requires `player1`/`player2`/`system`, which are absent).
- `Config::load_or_default` therefore catches the parse error, logs a warning,
  and **falls back to `Config::default()` → `present_mode = "Mailbox"`**.

So `Mailbox` IS the selected present mode. The present-mode beat was genuinely
removed by `8729526`. The residual stutter has a **different** root cause, which
is why it persists. (Separately: the stale on-disk config is a pre-existing UX
wart — the v1.x schema diverged from the user's old file — but it is not the
stutter cause and is out of scope here.)

### 1.2 Root cause: `ControlFlow::WaitUntil` delivery-cadence jitter

The prior investigation only measured `run_frame()` **emulation** cost (~3-4
ms/frame, ~5× headroom) and the rewind snapshot path — both ruled out. Neither
is where the stutter lives. The stutter is in frame **delivery cadence**.

`App::pace_frames` schedules the next frame with
`event_loop.set_control_flow(ControlFlow::WaitUntil(target))` and produces the
frame when the event loop wakes. On Linux (X11/Wayland) winit services that
wake through the `calloop` poll dispatch (`winit-0.30.13`
`platform_impl/linux/wayland/event_loop/mod.rs::poll_events_with_timeout` →
`loop_dispatch(timeout)` → `event_loop.dispatch(timeout, …)`, a `poll()`/
`epoll_wait()` with a timeout). The wake is therefore:

- a **lower bound**, not a precise instant — winit's own `ControlFlow` doc says
  so, and explicitly recommends *"Applications which want to render at the
  display's native refresh rate should instead use `Poll` and the VSync
  functionality of a graphics API"* — i.e. the current `WaitUntil` + `Mailbox`
  (no-vsync) combination is the documented **anti-pattern** for smooth output;
- **coalesced behind other event-loop fds** (the Wayland display fd, the proxy
  wake pipe, input) — when the compositor delivers an event near the deadline,
  dispatch services it first, delaying the timer wake by an unpredictable
  amount.

Because frame **production** happens at this jittery wake instant, and
`Mailbox` shows the latest produced frame at GPU flip time with no vsync gate,
the uneven production cadence is displayed directly as uneven motion — even
though the *average* FPS is exactly correct and emulation has huge headroom.
This is the classic "smooth average, jerky motion" emulator-pacing failure.

### 1.3 Measured delivery jitter (the missing measurement)

Headless caveat up front: a real `WaitUntil` wake can only be observed inside a
live winit event loop with a display, which this environment does not have. So
the substrate was measured directly (the OS timer/`poll` granularity that
`WaitUntil` rides on) and the two pacer recurrences were simulated at the NTSC
cadence. On this tuned CachyOS host (excellent kernel timer):

| pacer | inter-frame interval (ms) | production-instant deviation from ideal grid (ms) |
|---|---|---|
| bare `sleep(frame)` loop | p50 16.638, p90 16.693, p99 16.762, **max 16.826** | — |
| **current** (sleep/WaitUntil to target, ideal-grid anchor) | p50 16.639, p90 16.695, p99 16.762, **max 17.054** | p50 0.109, p99 0.194, **max 0.488** |
| **hybrid sleep+spin** (this fix) | p50 16.639, p90 16.639, p99 **16.640**, max 16.952 | p50 **0.000**, p99 **0.000**, max 0.313 |

The hybrid pacer collapses the production-instant deviation to ~0 at p99 and
tightens the interval p99 from 16.762 → 16.640 ms. On this host the bare-sleep
jitter is already small (sub-0.5 ms) because the kernel timer is tuned; on a
stock kernel — and, more importantly, when the wake is the **`WaitUntil`/
`calloop`-poll** path rather than bare `sleep` — the jitter is materially worse
and visible. The fix removes the jitter regardless of the host timer quality by
not depending on wake accuracy at all.

### 1.4 The fix (native only)

`crates/nes-frontend/src/app.rs`:

- New `App::block_until_native(target)` (cfg `not(wasm32)`): hybrid wait that
  `thread::sleep`s until `SPIN_MARGIN` (1.5 ms) before `target`, then
  `std::hint::spin_loop()`s to the exact instant.
- `pace_frames` now, on native, **blocks to the precise target** before
  producing the frame (so production lands on an even cadence), then keeps the
  event loop in `ControlFlow::Poll` so `about_to_wait` re-enters immediately
  and the next precise wait owns the cadence. `WaitUntil` is no longer used on
  native.
- **wasm32 is untouched**: it keeps `ControlFlow::WaitUntil` and never calls
  `block_until_native` (no usable `thread::sleep`/spin in the browser;
  `web_time::Instant` routes to `Performance.now()`; the browser's rAF cadence
  drives delivery). All native-only code is `#[cfg(not(target_arch =
  "wasm32"))]`-gated.

Spinning ~1.5 ms of every 16.6 ms frame costs well under 10% of one core and
only on the active emulator thread. Determinism is unaffected — the pacing
lives entirely in the frontend; the core (`nes-core` + chip crates) is not
touched.

### 1.5 Honesty note

This cannot be *observed* smooth headlessly (no display). The fix is the
standard, well-understood remedy for `WaitUntil`/no-vsync delivery jitter and
matches winit's own guidance. **Please confirm on the real display.** If
stutter persists after this, the next suspects are: (a) audio-queue underrun
stalling the cpal callback; (b) the per-frame `snapshot()` thumbnail
allocation when rewind is enabled; (c) compositor-forced vsync overriding
`Mailbox` (try `present_mode = "Immediate"` in a *correctly-schema'd* config,
or vsync-locked presentation with audio resampled to the display rate — a
larger change).

### 1.6 Audio path — assessed, left as a documented follow-up

`crates/nes-frontend/src/audio.rs` uses a `Mutex<VecDeque>` that drops-oldest
on overrun (cap 16 384 samples ≈ 0.37 s) and injects silence on underrun. With
precise pacing the producer cadence is now even, which is the dominant
stabiliser, so no audio change is made here (minimal-fix discipline). The
residual clock-domain mismatch (emulator wall-clock 60.0988 Hz vs the audio
device crystal driving cpal consumption) is bounded by the queue cap; if
audible drift remains after the pacing fix is confirmed on a display, the
follow-up is a small rate-matching feedback term nudging `frame_duration` from
the queue fill level. Left explicitly as a follow-up rather than a speculative
change.

## 2. Mega Man 3 stage-select band — root cause + determination

### 2.1 Reproduction tooling

`crates/nes-test-harness/src/bin/capture_left_edge.rs` extended with a
scripted-input mode: `RUSTYNES_PRESS_START="<frame>,…"` presses Start
(player 1) for `RUSTYNES_PRESS_HOLD` frames (default 4). Pure scripted input;
the core stays deterministic. Driving `Mega Man 3.nes` with a Start press at
frame 700 reaches the 3×3 robot-master stage-select by frame ~850. The
artifact band the user describes sits between the middle cell row (TOP MAN,
mid-right) and the bottom row (SHADOW MAN, bottom-right).

### 2.2 Regression-vs-pre-existing — DECISIVE: not a regression

A git worktree pinned at the parent commit `63d0a9b` (the commit *before* the
`8729526` AT-shifter widening) was built with the same capture bin and the same
scripted input, and the stage-select was captured at the identical frames.
Pixel-for-pixel diff, current HEAD `8729526` vs parent `63d0a9b`:

| frame | differing pixels |
|---|---|
| 850 (stage-select just settled) | **0** of 61 440 |
| 1100 | **0** of 61 440 |
| 1101 | **0** of 61 440 |

**The MM3 stage-select is byte-identical at HEAD and parent.** The AT-shifter
widening in `8729526` did **not** touch MM3 (expected: that fix only affects the
leftmost tile column under scrolling; the MM3 stage-select is static). The band
is therefore **pre-existing**, not collateral from `8729526`. The re-baselined
oracle/screenshots for MMC3-class games (e.g. Kirby's Adventure) did not encode
a new glitch — they changed only the leftmost-column palette, proven by the
`8729526` audit's per-ROM `fb_fnv1a64`-only diff.

### 2.3 The artifact and its mechanism

Frame-to-frame diffing isolates the band to a **single scanline, framebuffer
y≈153, on the right portion (x≥152)** — exactly "above SHADOW MAN". The
discontinuity is the **MMC3 scanline-IRQ raster split**: MM3's stage-select
animates the tiled "MEGA MAN III" background watermark and uses a mid-screen
MMC3 IRQ to change the horizontal scroll for the split, holding the menu cells
fixed above while the watermark scrolls below. The single split scanline shows
mixed (old-scroll / new-scroll) content because the IRQ handler's scroll write
lands partway across that scanline.

Rendering is deterministic (a re-run produces byte-identical frame 1101), so the
y=153 frame-to-frame variation is the legitimate watermark scroll, not random
flicker.

### 2.4 This is the C1 IRQ-sample-point axis (v2.0-deferred)

The split-scanline imprecision is governed by **when the CPU samples the IRQ
line relative to the A12 rise / PPU dot** — the canonical 6502 one-cycle IRQ
delay. Mesen2 models this with `NesCpu.cpp`'s `_prevRunIrq` (sample at end of
cycle, act at start of next); RustyNES uses `poll_irq_at_phase(M2Phase::High)`
in `Cpu::idle_tick`. The exact sample dot decides how many pixels of the split
scanline render with the pre- vs post-split scroll.

This is precisely the **Track C1 coordinated CPU/Bus/PPU IRQ-sample-timing
rework** documented in `docs/adr/0002-irq-timing-coordination.md` and the
CLAUDE.md version-policy section. It is the same architectural surface as the
open `cpu_interrupts_v2/{2,3,5}` residuals and `mmc3_test_2/4` sub-test #3, and
it has **17+ independent fix attempts rolled back**. It is explicitly gated for
v2.0's master-clock-precise scheduling refactor.

nesdev (`nesdev_wiki/MMC3.xhtml`) further documents that a **1-scanline shake**
at the MMC3 split is *hardware* behavior: "the 2C02's pre-render scanline will
decrement the counter twice every other vertical redraw, so the IRQ will shake
one scanline," and "the exact time and number of times the counter is clocked
will depend on the specific set of sprites present on every scanline." A
partial torn scanline at a mid-frame scroll write is also normal on real
hardware when a game writes scroll in the IRQ handler without waiting for
hblank — many MMC3 split games (MM3 among them) accept it.

### 2.5 Determination: no code fix

Per the brief's guardrail ("be surgical … 17+ historical rollbacks") and the
hard invariants (B4, `mmc3_test_2/*`, sacred trio), the correct engineering
decision is **not to touch the C1 axis** for this cosmetic, hardware-plausible,
pre-existing 1-scanline artifact. The IRQ *counter* logic (Phase B4 Sharp/NEC
reload-pending discriminator) is correct and validated; the residual is the
sub-instruction sample point, which the validated test suite already brackets
as `#[ignore]`'d (mmc3_test_2/4 #3) and which v2.0 will absorb. Closing it
surgically would risk the entire green suite for a ≤1-scanline cosmetic gain.

The MMC3 IRQ-counter logic, B4 invariant, and `mmc3_test_2` suite were all
re-confirmed green during this session (see §3).
