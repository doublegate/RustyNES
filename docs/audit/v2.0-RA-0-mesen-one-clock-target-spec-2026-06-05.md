## The Mesen model (TARGET)

**One master clock, one cycle counter, APU clocked at cycle START — including every DMA cycle.**

```
StartCpuCycle(forRead):                 EndCpuCycle(forRead):
  _masterClock += forRead?5:7             _masterClock += forRead?7:5     (total 12/cycle)
  _state.CycleCount++                      PPU.Run(_masterClock - _ppuOffset)
  PPU.Run(_masterClock - _ppuOffset)       NMI/IRQ φ2 edge detect  (NO apu clock here)
  _console->ProcessCpuClock()  ───────►  ProcessCpuClock → NesApu::Exec:
                                             _currentCycle++          (APU cycle == CPU cycle, 1:1)
                                             DMC byte-timer + frame counter advance / Run()
```

Load-bearing facts:

1. **APU/DMC is clocked at cycle START** (`StartCpuCycle` → `ProcessCpuClock`, `NesCpu.cpp:574`),
   BEFORE the access. The NMI/IRQ φ2 edge detector runs at cycle END (`EndCpuCycle`). So within one
   cycle: APU sets its IRQ flag at START, the CPU samples it at END of the SAME cycle.
2. **`_currentCycle` (APU) increments 1:1 with `CycleCount` (CPU)** via `ProcessCpuClock`, once per
   CPU cycle. There is NO separate APU phase seed — the APU phase IS the CPU cycle count.
3. **DMA cycles go through `StartCpuCycle`/`EndCpuCycle` too** (`ProcessPendingDma`: the halt at
   `:608`, every `processCycle()` at `:664`). Therefore **the DMC byte-timer and frame counter ADVANCE
   THROUGH the DMA halt/dummy/get cycles** — the DMC is NOT frozen during its own DMA span.
4. **The DMC DMA GET is gated on `(_state.CycleCount & 1) == 0`** (`:668` `getCycle`). Span = the
   initial halt cycle + dummy reads until the counter aligns to a get (even) cycle + the get. The 3-vs-4
   span is an EMERGENT consequence of the start parity on the unified counter, not a fixed constant.
5. `ProcessPendingDma` runs at the START of a read access (before the read), gated on `_needHalt`
   (`:577-606`). `_ppuOffset` is randomized at boot (`:362`) or 1; `_startClockCount=_endClockCount=6`;
   read cycle = Start(+5)/End(+7), write = Start(+7)/End(+5).

## The RustyNES R1 delta (what diverges — and why incremental parity tweaks failed)

RustyNES R1 (`crates/nes-cpu/src/cpu.rs` `start_cycle` :578, `end_cycle` :590;
`crates/nes-core/src/bus.rs` `cpu_clock` / `cpu_clock_apu_dmc` :~2688):

| Aspect | Mesen (target) | RustyNES R1 (current) |
|---|---|---|
| APU/DMC clock placement | cycle START (`ProcessCpuClock` in `StartCpuCycle`) | cycle **END** (`cpu_clock_apu_dmc` in `end_cycle`, after the access, before φ2) |
| APU phase source | unified `CycleCount` (no seed) | separate `apu_phase` / `put_cycle` seed |
| DMC during its own DMA span | **advances** every cycle | **frozen** (`tick_dmc_timer_only`/freeze — the F-2 un-wedge requirement) |
| GET parity | `CycleCount & 1 == 0` | `!put_cycle` (== `cyc&1==1`) |

The three structural differences (APU clocked at END not START; DMC frozen-not-advancing during the
span; separate seed not unified counter) are why the 10+ upstream parity levers all failed: they
patched the PARITY of a decoupled `apu_phase`, but the cumulative CPU-vs-APU phase is wrong because
the APU is clocked at the wrong HALF of the cycle and is frozen during DMA — a phase the reference
NEVER occupies. No parity flip of a decoupled, end-clocked, span-frozen DMC can reproduce a unified,
start-clocked, span-advancing one. The fix must be STRUCTURAL (match the model), not a parity guess.

## RA-1 (next): clock the APU/DMC at cycle START, unified, advancing through the span

Concrete change, behind a new sub-feature (default-off; gate every step on C1 trio + 6/10 + no-wedge
+ default byte-identical; floor `5ceab83`+`a8e482e`):

1. Move the APU/DMC clock from `end_cycle` to `start_cycle` (Mesen `ProcessCpuClock` at
   `StartCpuCycle`), so the APU sets its IRQ flag BEFORE the same cycle's φ2 sample in `handle_interrupts`
   (preserve the within-cycle "APU-sets-then-CPU-samples" order that C1 #4 `irq_and_dma` needs).
2. Drive the DMC byte-timer from the unified `self.cycles` (CycleCount) parity — drop the separate
   `apu_phase`/`put_cycle` seed; GET on `(self.cycles & 1) == 0` (Mesen `:668`).
3. Advance the DMC timer EVERY cycle INCLUDING its own DMA span (remove the freeze). The freeze was a
   workaround for the wedge under the decoupled model; under the unified start-clocked model the wedge
   should not recur because the get-parity, frame counter, and timer are all consistent on one counter
   (the exact reason Mesen does not wedge).

**RA-1 risk + gate:** the wedge (DMASync `LDA $4000;BNE` infinite spin) is the make-or-break — prior
work showed advancing the DMC during the span re-wedges UNDER THE DECOUPLED MODEL. RA-1 must verify
the unified start-clocked model does NOT wedge (Mesen's does not). If it still wedges, the residual is
a remaining decoupling (find it vs the Mesen `_currentCycle`/`CycleCount` 1:1 invariant). RA-2 then
makes the DMC span/landing the emergent Mesen function (`processCycle` get-alignment) → Y=4.
