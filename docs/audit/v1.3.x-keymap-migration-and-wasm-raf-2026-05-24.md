# v1.3.x frontend fixes — keymap migration regression + wasm rAF pacing

**Date:** 2026-05-24
**Scope:** `crates/nes-frontend/` only. Emulator core, determinism, and all
v1.3.0 accuracy invariants untouched.
**Author:** debug-agent session (root-cause + minimal fix).

This audit captures *why* two frontend issues happened, the empirical
evidence gathered, the alternatives considered, and the minimal fixes
applied. Both fixes are uncommitted at the time of writing (left for diff
review).

---

## Issue 1 — keyboard input dead after the v1.3.1 config migration

### Symptom

After upgrading to v1.3.1, all keyboard input was dead. Terminal showed:

```
unknown keycode "Up" / "Down" / "Left" / "Right" / "X" / "Z" / "RShift" / "Return"
/ "W" / "S" / "A" / "D" / "G" / "F" / "Q" / "E" in input config
```

### Root cause (regression we shipped in v1.3.1)

v1.3.1 added `Config::migrate_legacy` (`crates/nes-frontend/src/config.rs`)
to upgrade pre-v1.3.1 `config.toml` schemas. It renamed the legacy
`[input.keyboard_p1]` / `[keyboard_p2]` sections to `[input.player1]` /
`[player2]` **but carried the keycode string values over verbatim**.

The legacy values are old **winit-0.29 `VirtualKeyCode`** Debug names
(`Up`, `Down`, `Z`, `X`, `Return`, `RShift`, bare single letters, …). The
current schema and `input::parse_keycode` expect **winit-0.30 `KeyCode`**
Debug names (`ArrowUp`, `KeyZ`, `Enter`, `ShiftRight`, …) — the rename
landed in the winit 0.29 keyboard-API migration (`VirtualKeyCode` →
`KeyCode`). So after migration every binding hit the
`bind_button`/`try_bind` "unknown keycode" path
(`input.rs`, `eprintln!("… unknown keycode …")`) and was dropped.

Critically: **before** the migration, an old config either failed the
strict parse (→ silent fallback to *valid* defaults) or had its unknown
`[input.keyboard_pN]` sections ignored (→ *valid* defaults). Input worked.
The migration replaced working defaults with **unparseable** legacy
values — strictly a regression introduced by v1.3.1.

### Why the user's on-disk config can't self-heal via re-migration

`migrate_legacy` detects legacy input by the presence of a `[video]`
section or `[input.keyboard_pN]` sub-tables. The user's file was already
rewritten by v1.3.1 to the **current structure** (`[input.player1]` /
`[player2]`) — it just carries legacy *values*. So `migrate_legacy`
returns `None` on it (correctly — it is current-schema), the strict parse
succeeds (the values are just strings), and the breakage surfaces only at
`parse_keycode` time. The fix therefore has to live in `parse_keycode`,
not in the migration detector.

### Fix

`input::parse_keycode` now, on a miss against the current `KeyCode` name
table, retries the name through a new `legacy_keycode_alias` table that
maps winit-0.29 `VirtualKeyCode` Debug names to their winit-0.30
equivalents:

- `Up`/`Down`/`Left`/`Right` → `ArrowUp`/`ArrowDown`/`ArrowLeft`/`ArrowRight`
- bare single letters `A`..`Z` → `KeyA`..`KeyZ`
- bare digits `0`..`9` **and** winit-0.29 `Key0`..`Key9` → `Digit0`..`Digit9`
- `Return` → `Enter`; `LShift`/`RShift` → `ShiftLeft`/`ShiftRight`;
  `LControl`/`RControl` → `ControlLeft`/`ControlRight`;
  `LAlt`/`RAlt` → `AltLeft`/`AltRight`; `LWin`/`RWin` → `SuperLeft`/`SuperRight`;
  `Back` → `Backspace`; `Capital` → `CapsLock`; `Grave` → `Backquote`;
  `Apostrophe` → `Quote`
- `Space`, `Escape`, `Tab`, `F1`..`F12` are spelled identically in both
  APIs → resolve directly, never via the alias table.

The recursion is one level deep and provably terminating:
`legacy_keycode_alias` only ever yields a **current** `KeyCode` name, so
the re-entrant `parse_keycode` call hits a direct arm and cannot re-alias.
Current names are matched first, so an alias can never shadow a valid
current name (`KeyZ` and `ArrowUp` still resolve directly).

One supporting addition: `KeyCode::CapsLock` was not in the current
`parse_keycode` table (only the alias target needed it), so it was added
as a direct arm — without it the `Capital` alias would dead-end.

**Polish (also applied):** `migrate_legacy` now canonicalizes each legacy
keycode value to its current spelling as it is carried into `playerN`,
via the new public `input::canonical_keycode_name`. With aliases in
`parse_keycode` this is not required for correctness, but it means
freshly-migrated files are written with clean current names (`ArrowUp`,
`KeyZ`, …) rather than legacy spellings. `canonical_keycode_name` leaves
already-current and unrecognized names unchanged (the migration must not
silently drop an unknown value).

### Load-path proof (user's already-migrated config resolves with no action)

1. `Config::load_or_default` reads the on-disk `config.toml` (current
   structure, legacy values like `up = "Up"`, `a = "Z"`).
2. `migrate_legacy` returns `None` (no `[video]`, no `[input.keyboard_pN]`).
3. Strict `toml::from_str::<Config>` succeeds (values are strings).
4. `KeyBindings::from_config` → `pad_to_map` → `bind_button` →
   `parse_keycode("Up")` now returns `Some(KeyCode::ArrowUp)` (and `"Z"` →
   `KeyZ`, `"Return"` → `Enter`, `"RShift"` → `ShiftRight`, …).
5. Bindings populate; input is live again. **No manual edit, no
   re-migration, no file deletion required.**

### Tests added

- `input.rs`: `parse_keycode_accepts_legacy_{arrow,letter,named,digit}_names`,
  `parse_keycode_current_names_take_precedence_over_legacy`,
  `legacy_default_layout_resolves_end_to_end` (the exact 16 legacy values a
  pre-v1.3.1 default config carried), `canonical_keycode_name_rewrites_legacy_to_current`.
- `config.rs`: `legacy_keycode_values_are_canonicalized_and_resolve` — a
  legacy-default-layout migration whose every output keycode is asserted to
  `parse_keycode`-resolve.

## Issue 2 — wasm / GitHub Pages stutter (default `wasm-winit` frontend)

### Symptom

https://doublegate.github.io/RustyNES_v2/ stutters (uneven motion). The
v1.3.1 native sleep-then-spin pacer is `#[cfg(not(target_arch = "wasm32"))]`
-gated, so the deployed default `wasm-winit` (unified winit/wgpu/egui)
build does not get it.

### Root-cause investigation (winit 0.30.13 web backend source)

The task framed the fix as "winit 0.30 `Poll` schedules via
`requestAnimationFrame`, `WaitUntil` via `setTimeout`; switch to `Poll`."
Reading the actual winit-0.30.13 source under
`~/.cargo/registry/.../winit-0.30.13/` **falsified that premise** — the
mapping is different, and getting it right matters:

- `ControlFlow::Poll` → `PollStrategy` (default `Scheduler`) →
  `scheduler.postTask` (fallback `setTimeout` via `MessageChannel`).
  **Not** `requestAnimationFrame`.
  (`src/platform/web.rs` `PollStrategy`,
  `src/platform_impl/web/web_sys/schedule.rs` `Schedule::new`.)
- `ControlFlow::WaitUntil` → `WaitUntilStrategy` (default `Scheduler`) →
  `scheduler.postTask` with a delay (fallback `setTimeout`).
  (`schedule.rs` `Schedule::new_with_duration`.)
- `requestAnimationFrame` is wired **only** to redraw delivery:
  `Window::request_redraw()` → `canvas.request_animation_frame()` →
  `AnimationFrameHandler::request()` → `window.request_animation_frame(...)`,
  which fires the `RedrawRequested` event.
  (`src/platform_impl/web/window.rs:132-133`,
  `src/platform_impl/web/web_sys/animation_frame.rs`.)

So on the deployed build, frame **production** ran in `about_to_wait` →
`pace_frames`, which set `ControlFlow::WaitUntil(next)` — a `postTask` /
`setTimeout` wake that is **not** synced to the browser's vsync. The
emulator therefore produced frames on a clock independent of the display
refresh and the present (`RedrawRequested`, which *is* rAF-synced) showed
the jitter. Hence the stutter.

### Fix

Drive the wasm32 frame loop from the rAF-synced `RedrawRequested` instead
of from `ControlFlow`:

- New wasm-only `App::pace_and_produce_wasm`, called at the top of the
  `RedrawRequested` handler. It latches input, produces the frames due by
  `web_time::Instant` delta (catch-up still capped at `MAX_CATCHUP_FRAMES`,
  so a backgrounded tab doesn't burst on resume and wall-clock NTSC speed
  stays correct on non-60 Hz panels), updates the fps readout, then re-arms
  the next rAF via `gfx.window.request_redraw()`. That self-reschedule is
  the heartbeat of the loop — one production step per browser refresh.
- The shared catch-up body was factored out into `App::produce_due_frames`
  (used by both the native pacer and the wasm rAF loop) so the two paths
  can't drift.
- On wasm, `pace_frames` (still called from `about_to_wait`) is now a
  no-op that only keeps `ControlFlow::Poll` as an idle keep-alive between
  rAF callbacks; it no longer produces frames or sets `WaitUntil`.
- `start_nes` issues the first `request_redraw()` (the first rAF) and, on
  wasm, sets `ControlFlow::Poll` instead of `WaitUntil(target)`.

### What is deliberately NOT changed

- **Native** keeps the v1.3.1 sleep-then-spin pacer
  (`block_until_native`), the `Mailbox` present-mode default, and the
  `Poll`-driven precise cadence. All native production logic is inside the
  `#[cfg(not(target_arch = "wasm32"))]` arm of `pace_frames`.
- **`wasm-canvas`** embed frontend (`wasm.rs`, its own rAF loop) is
  untouched — it is a separate cargo feature and module, and the change is
  confined to `app.rs` wasm `cfg` branches that `wasm-canvas` does not
  compile differently.

### Why the task's premise was not followed literally

Switching wasm to `ControlFlow::Poll` *alone* (the literal task
instruction) would **not** have fixed the stutter in winit 0.30.13,
because `Poll` there is `scheduler.postTask`/`setTimeout`, not rAF. It
would also risk a busy-loop (postTask fires as fast as the scheduler
allows). The correct rAF-synced lever in this winit version is
`request_redraw()`, so production was moved onto `RedrawRequested` and
`Poll` is retained only as a benign keep-alive. This matches the task's
own escape hatch: *"Investigate empirically … if `Poll` busy-loops the
browser, consider … a rAF-driven approach instead."*

### Verification limits

Smoothness on the browser **cannot be verified headless** (no browser in
this environment). The change is reasoned directly from the winit 0.30.13
web-backend source and is confirmed buildable on both wasm feature
configs. Final smoothness confirmation requires the user to load the Pages
build.
