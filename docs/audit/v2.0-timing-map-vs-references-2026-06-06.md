# RustyNES v2 ŌĆö Cycle/Clock Timing Map vs Mesen2 + TriCNES (2026-06-06)

Built from three parallel deep source maps (Mesen2 `NesCpu.cpp`, TriCNES `Emulator.cs`,
RustyNES `cpu.rs`/`bus.rs`/`apu.rs` under `mc-r1-full-cpu`). Purpose: understand the

## 1. The fundamental cycle model (where RustyNES differs structurally)

| | Mesen2 | TriCNES | RustyNES (`mc-r1-full-cpu`) |
|---|---|---|---|
| Cycle counter | ONE: `_state.CycleCount` (++ in `StartCpuCycle`) | ONE: `totalCycles` / `CPUClock` (12 master/cyc) | FOUR: `cpu.cycles`, `cpu.cycles_emitted`, `apu.cpu_cycle`, `master_clock` |
| Is EVERY cycle a bus access? | **YES** ŌĆö read/write/`DummyRead`; no busless cycle | **YES** ŌĆö every `operationCycle` is a `Fetch`/`Store`; internal = dummy read | **NO** ŌĆö `idle_tick` is a no-access cycle; instruction declares `*cycles=N`, a **burn-loop** fills idle |
| DMA halt cycles | EXTRA, all in the one counter | EXTRA, freeze `operationCycle` | EXTRA, but accounted against a per-instruction **budget** (`cycles_emitted`) |

**The structural divergence:** Mesen and TriCNES model the 6502 as *every cycle is a memory
access* (a real read, a write, or a dummy read of the held address). RustyNES instead has
`dispatch()` **declare** an instruction's length (`*cycles = 2` for a NOP) and a **burn-loop**
(`cpu.rs:490 while cycles_emitted < cycles { idle_tick }`) fill the remainder with `idle_tick`
cycles that do **no bus access**. This is the `~40-50%-built` rewrite: the per-cycle access
shape exists for real reads/writes, but the *internal* cycles are still burn-filled, and a
`dma-cycle-budget` hack manages the accounting collision when a DMA halts mid-instruction.

This matters because **a DMC DMA halts the CPU on a bus access**. In the references every
cycle is a bus access, so the halt timing is exact. In RustyNES the busless `idle_tick`
cycles + the budget accounting make the halt insertion point and count approximate.

## 2. The DMC reload span ŌĆö the C1-vs-DMA core (3 vs 4)

All three maps reduce the CheckDMATiming Y=3-vs-4 to a single fact: **a looping DMC RELOAD
stalls the CPU 4 cycles in the references, 3 in RustyNES.** The mechanism is identical in
Mesen and TriCNES and is the precise thing RustyNES gets wrong:

### References: a reload's first serviced DMA cycle is a PUT ŌåÆ span 4

TriCNES per-CPU-cycle order (`_EmulatorCore` 868-927): **`_6502()` runs, THEN `_EmulateAPU()`
runs, THEN `APU_PutCycle` flips.** The reload arm lives in `_EmulateAPU`'s GET branch
(`bitsŌåÆ0 ŌåÆ DoDMCDMA=DMCDMA_Halt=true`, lines 1158-1169). So:

```
cycle N (GET):  _6502(N) runs ŌĆö does NOT see DoDMCDMA (arm hasn't happened yet)
                _EmulateAPU(N) ARMS the reload
                APU_PutCycle flips GETŌåÆPUT
cycle N+1 (PUT): _6502(N+1) is the FIRST cycle to see DoDMCDMA  ŌåÉ first service = PUT
```

The arm is **invisible to its own cycle's CPU**, so the first serviced cycle is the *next*
cycle (a PUT). On a PUT, `DMCDMA_Halt` does NOT clear (clearing is gated to the GET branch,
TriCNES line 4337). So the put-first sequence is one cycle longer:

```
PUT : DMCDMA_Halted   (halt stays)              stall 1
GET : DMCDMA_Halted   (halt clears, 4337)       stall 2
PUT : DMCDMA_Put      (dummy)                    stall 3
GET : DMCDMA_Get      (real fetch, DMA ends)     stall 4   => span 4
```

Mesen is the same fact in different clothes: `getCycle = (CycleCount & 1)==0`; the GET fetch
only fires on an even count; when the halt lands on an even count an **extra align cycle**
(`NesCpu.cpp:808-814`) is inserted to re-reach an even GET ŌåÆ 4. A `$4015`-enable **LOAD**
instead arms on a put (or via the parity-tuned `_transferStartDelay = even?2:3`) so its first
service is a GET ŌåÆ 3.

### RustyNES: the reload's first serviced cycle is a GET ŌåÆ span 3

Under `mc-r1-apu-unified-clock` the DMC byte-timer is clocked at **cycle START** (in
`tick_with_external` via `cpu_clock`), and `dmc_step_reload_arm` sets `pending_dmc_dma` on the
same `apu_phase==true` (= GET-parity) cycle. The `read1`/`idle_tick` DMA loop
(`while bus.dmc_dma_pending()`) then services it on that **same GET cycle** ŌĆö the arm is
**visible to its own cycle** (the opposite of TriCNES's `_EmulateAPU`-after-`_6502` ordering).
The legacy `dmc_dma_step_impl` (bus.rs:2441-2479) entered on a GET runs:

```
GET : in_dmc=true, dmc_halt=true; then getŌćÆdmc_halt=false   stall 1
PUT : replay                                                stall 2
GET : get && !dmc_halt ŌåÆ GET (real fetch)                   stall 3   => span 3
```

The legacy state machine is correct *given* the entry parity (GET-firstŌåÆ3, PUT-firstŌåÆ4,
verified both branches). The bug is the **entry parity**: RustyNES enters on GET where the
references enter on PUT, because RustyNES makes the reload arm **visible to its own cycle**.

## 3. Why every lever has failed (this session + 18 months)

- **Shift the byte-timer / arm parity** (`phase-minus1`, parity_seed): the GET, the byte-timer,
  and DMASync all ride the same `apu.cpu_cycle` seed, so a uniform shift moves DMASync's lock
  with the GET ŌåÆ **absorbed** (Y unchanged) or **desyncs** DMASync ŌåÆ **wedge**.
- **Anticipatory leading halt** (`mc-r1-dmc-anticipatory-halt`, this session): adds a stall
  *before* the GET ŌåÆ moves the GET +1 ŌåÆ DMASync re-syncs ŌåÆ **absorbed** (Y=3).
- **Trailing stall after the GET** (`mc-r1-dmc-trailing-stall`): holds the CPU past the GET ŌåÆ
  the looping reload can't complete ŌåÆ **wedge** ((unset)).
- **Budget toggle**: shifts the NOP's idle-cycle accounting, moves Y the wrong way (3ŌåÆ2).

The common thread: the span is **emergent from the first-service parity**, which is coupled to
the byte-timer/GET/DMASync via the shared seed. No surgical shift can change the entry parity
without moving the GET that DMASync locks onto.

## 4. What we are fundamentally trying to do vs what is occurring

**Trying:** make a looping DMC reload stall the CPU 4 cycles (so CheckDMATiming's setup window
= 1588 ŌåÆ Y=4), exactly as the references do ŌĆö by entering the DMA on a PUT cycle.

**Occurring:** RustyNES enters the reload DMA on a GET cycle (span 3) because the
`apu-unified-clock` arm is visible to its own cycle's DMA loop, whereas the references' arm
(`_EmulateAPU` after `_6502`, or the parity-gated `getCycle`) is invisible to its own cycle,
forcing first-service onto the next (PUT) cycle.

**The structurally-correct fix (all three maps agree):** make the reload arm **invisible to
the cycle in which it arms**, so the `read1`/`idle_tick` DMA loop first services it on the
*next* cycle (PUT parity) ŌåÆ the legacy state machine then emits span 4 naturally. Concretely:
a **1-cycle pending-visibility delay** on the reload arm only (a `pending_dmc_dma_next` latch
promoted to `pending_dmc_dma` at the next cycle boundary), matching TriCNES's
`_EmulateAPU`-after-`_6502` ordering ŌĆö NOT a parity shift of the byte-timer (which is
correct), and NOT a stall added to the existing span (which moves the GET / wedges). This is
the C1-safe move: it changes *when the reload becomes visible*, not the GET fetch cycle or the
DMASync cadence. Loads (which correctly enter on GET ŌåÆ span 3) must be excluded.

Verification target: CheckDMATiming setup window 1587 ŌåÆ 1588, Y 3 ŌåÆ 4; then the abort cluster
(gated on `CPY #4`) + DMC+OAM unblock; C1 trio + nestest 0-diff preserved; default
byte-identical (gated default-off).

(rendering_enabled && is_render_scanline) and dumped $500 at frame 6000. RESULT: $500 is

v/t reload masks (0x041F/0x7BE0 include the NN bits ŌĆö verified), NOT the per-dot buffer.
