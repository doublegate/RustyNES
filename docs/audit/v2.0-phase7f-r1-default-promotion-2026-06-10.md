**Date:** 2026-06-10
**Branch:** `refactor/v2.0-master-clock`

This is the program that makes the R1 master-clock path (AccuracyCoin 100% /
139, nestest 0-diff, `cpu_interrupts_v2` 5/5 strict) the **default** scheduler
for every shipping RustyNES v2 build, replacing the v1.7.0 integer-lockstep
default (94.24% / 8). It is the work item the Stage-4 audit deferred as "the
later Phase-7/F program".

## PF-1 — region-parametrize the R1 CPU master-clock loop (PAL 3.2:1)

The R1 CPU loop hardcoded `CPU_DIVIDER_NTSC = 12`, so it advanced
`master_clock` by 12 per CPU cycle regardless of region. The bus already ran
the PPU at the regioned `ppu_divider` (`run_ppu_to`), so PAL got 12/5 = 2.4
dots/cycle instead of the hardware-true 16/5 = **3.2:1**.

- **nes-cpu:** new `Bus::cpu_divider()` (default 12; `LockstepBus` overrides
  from `region_dividers().0` → NTSC 12 / PAL 16 / Dendy 15). `start_cycle` /
  `end_cycle` / `reset` derive the read/write master-clock split via
  `read_split(div)` / `write_split(div)` instead of the NTSC `const`s. For
  `div = 12` the splits are exactly `(5, 7)` / `(7, 5)`, so **NTSC is
  byte-identical** to the prior constants.
- **nes-core:** `LockstepBus::cpu_divider()`; the bus-side DMA-coherence fixup
  folds the region's true `cpu_divider` span into `dma_mc_consumed` (PAL's
  1-mc/cycle deficit vs the 3 physically-ticked dots is caught up by the next
  `run_ppu_to` → correct 3.2 average), while `ppu_clock` keeps `3 * ppu_div`.
  NTSC/Dendy (12 = 12 / 15 = 15) byte-identical.
- **region_timing:** the test encoded the legacy 3:1-for-PAL approximation
  (35,464 cyc/frame). R1 now measures the hardware-true **33,247** (PAL CPU
  1.662 MHz / 50 Hz / 312 lines). The PAL expectation switches on
  `mc-r1-full-cpu`; Dendy stays 3:1 (35,464); the legacy path keeps 35,464.

Canary: AccuracyCoin 100%, nestest 0-diff, `cpu_interrupts_v2` 5/5,
`ppu_vbl_nmi` 10/10, `save_state` 7/7, `visual_regression` 7/7, `region_timing`
4/4 both builds; fmt + clippy `-D warnings` clean.

## PF-2 — flip the umbrella to default-on (Option X)

`mc-r1-full-cpu` is now enabled by the **consumers'** defaults, not nes-core's:

- `nes-frontend` default `+= mc-r1-full-cpu` (and `wasm-canvas` forwards it),
  so `cargo run -p nes-frontend` + both wasm flavours ship R1.
- `nes-test-harness` default `= [mc-r1-full-cpu]`, so `cargo test` is the 100%
  path; `cargo test --workspace` unifies to R1.
- `nes-core`'s own default stays legacy+pivots.

**Why not nes-core's default?** Placing it there makes R1 un-strippable
through a consumer's `--no-default-features` (the dependency default leaks it —
measured: `--no-default-features --features test-roms` still gave 100%). With
Option X, `cargo test -p nes-test-harness --no-default-features --features
test-roms` cleanly drops to the v1.7.0 integer-lockstep baseline (**94.24% /
8**, verified) for A/B comparison. Embedders of `nes-core` opt into the v2.0
accuracy explicitly.

Verified: workspace builds + `cargo test --workspace` green (R1-unified); full
harness suite green; frontend builds R1; fmt/clippy/doc clean both builds.

## PF-3 — re-baseline goldens under R1 (visual proof first)

Under R1 the 60-ROM commercial oracle shifts in `cycles` + `audio_samples` +
`audio_fnv1a64` (DMC fetch cadence + the region-exact ratios); **6 of 60**
ROMs additionally shift a framebuffer checkpoint hash. Per the
`feedback_paired_pipeline_oracle_masking` lesson (re-baselining to green can
mask a real regression), every framebuffer change was **visually verified**
(R1 dump vs the committed v1.7.0 PNG) before any golden was re-baselined:

| ROM (mapper) | f600 verdict |
|---|---|
| Donkey Kong (NROM) | correct 25m screen; timing-dependent bonus-timer/sprite delta |
| Legend of Zelda (MMC1) | identical title screen; waterfall animation phase only |
| Kirby's Adventure (MMC3) | correct title screen |
| Castlevania III retrans (VRC6) | pixel-identical castle intro |
| Famista '91 (Namco 163) | correct title + field |
| Mr Gimmick (FME-7, f3661/f3901) | correct START/CONTINUE menu (identical) |

The framebuffer deltas are timing-dependent animation-phase shifts (the
captured frame lands on a slightly different game tick because R1's cycle
timing differs), **not visual regressions**. The other 54 ROMs — including the
SMB / Kid Icarus sacred trio — are framebuffer **byte-identical** under R1
(Kid Icarus's cycle count drops 21.24M → 19.92M = exactly ×30/32, the PAL
3.2:1 ratio correctly applying). Re-baselined: 57 oracle `insta` snapshots
(3 were already fully byte-identical) + the 7 changed screenshot PNGs.

**`audio_tests` stays legacy-only.** The 19-ROM bbbradsmith corpus asserts
audio-hash + cycle on **uniform-palette (black) frames** — there is nothing to
visually verify, so re-baselining them under R1 would be a trust-the-hash move
the verify-first discipline forbids. R1's effect on them is pure timing (the
expansion-audio *synthesis* code is untouched), so the legacy-baselined corpus
remains a valid scheduler-independent synthesis regression net, run by the
legacy A/B CI job. The **default (R1) build's** audio coverage is the
*visually/functionally verified* set: AccuracyCoin APU suite 100%, `apu_test`
8/8, `dmc_dma` 5/5, and the 60-ROM oracle's `audio_fnv1a64` (real-game VRC6 /
N163 / 5B / MMC5 expansion audio + DMC), now R1-baselined.
