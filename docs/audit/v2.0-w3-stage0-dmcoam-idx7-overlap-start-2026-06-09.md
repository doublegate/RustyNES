## The fix (overlap-accounting only; `dmc_dma_step_impl`/byte-timer/arm untouched)

1. **`nes-cpu` `Bus::oam_dma_overlap_ready()`** (new trait method, default =
   `oam_dma_in_flight()`): may a pending DMC join an OAM DMA as an overlap event?
2. **`read1` standalone DMC loop** gains `&& !bus.oam_dma_overlap_ready()` — do
   not drain the DMC unshared when an overlap is possible. Floor-inert: without
   the overlap feature the query is the trait-default `false` chain.
3. **The combined-loop entry** `dmc_ready && bus.oam_dma_in_flight()` becomes
   `dmc_ready && bus.oam_dma_overlap_ready()`.
4. **`LockstepBus::oam_dma_overlap_ready`** (gated `mc-r1-dmc-oam-overlap`):
   under `mc-r1-counter-collapse` = `owed > 0 || dma_pending.is_some()`; without
   the collapse flag it stays `owed > 0` (the audited Program-M state preserved).
5. **`dmc_overlap_begin(halted_addr)`** (signature gains the preempted CPU read
   address): when the event starts a pending `$4014`, `dma_halt_addr` =
   `halted_addr` (what `oam_dma_step` would have latched).
6. **`dmc_overlap_noop_cycle`** (collapse-gated): latch the pending `$4014` page
   on the FIRST shared halt cycle — inside the cycle's `start_cycle`, so the
   514/513 `self.cycle & 1` parity choice is identical to the no-DMC
   counterfactual — then the existing `oam_dma_overlap_cycle` consumes OAM's
   halt/alignment/transfer slots exactly like lockstep.

Post-fix trace: 19 events, with the two recovered boundary events latching inside
the first noop (`owed_at_get` = 511/510 — exactly the lockstep oracle's idx[6]/
idx[7] values).
