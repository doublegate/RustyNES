# Stage-E: every-cycle-a-bus-access refactor (the DMA-tail fix, multi-session)

Branch `refactor/v2.0-stage-e-bus-cycle-model`. Goal: close the CheckDMATiming Y=3→4 DMA
tail (and the abort cluster gated on it) by aligning RustyNES's cycle model to
Mesen/TriCNES — **every CPU cycle is a real bus access** — so DMC-DMA halts insert exactly
and the CheckDMATiming setup window matches (1587→1588). Per
`docs/audit/v2.0-timing-map-vs-references-2026-06-06.md`.

## Why (proven this session)

The reload is ALREADY span-4 (measured put-start). The 1-cycle residual is the
CheckDMATiming setup window (1587 vs 1588). Every per-cycle ACCOUNTING lever (budget
ON/OFF, read1-extra, anticipatory/trailing/visibility-delay) gives Y∈{2,3} or wedges —
because they only change accounting, while keeping `idle_tick` BUSLESS. The references have
NO busless cycle: an internal cycle is a dummy read, so a DMA halting during it inserts at
the exact bus-access point. The bolt-on flags cannot capture this; only making internal
cycles real bus accesses can. This is the one untested axis.

## Canary gate (EVERY step)

- `nestest` 0-diff (CPU state unchanged — dummy reads don't alter registers/flags).
- AccuracyCoin floor (`mc-r1-full-cpu`) pass-count NON-DECREASING; C1 trio

- Default build byte-identical (all Stage-E behind the umbrella flag).
- `cargo fmt`/`clippy`/`doc` clean.

## Stages (each gated, each a commit)

- **E.1 — `internal_cycle` foundation (THIS step).** Add `mc-r1-internal-cycle`: `idle_tick`
  performs a dummy read of the held address (`self.pc` for now) between `start_cycle` and
  `end_cycle`, so every internal cycle is a real bus access (Mesen `DummyRead`). Verify
  nestest 0-diff + measure Y (the crux: does an exact-bus-access internal cycle move Y?).
- **E.2 — held-address correctness.** Internal cycles must dummy-read the HARDWARE-correct
  address per instruction (PC for implied; the unfixed indexed address for page-cross; the
  stack pointer for stack ops). Compose with `cpu-implied-dummy-reads` /
  `cpu-stack-dummy-reads` (already in the floor). Cell-trace cross-diff vs Mesen.
- **E.3 — DMA halts always extra.** With internal cycles real, the DMA-halt loops in
  `read1`/`internal_cycle` must NOT count against `cycles_emitted` (Mesen: halts extra).
  Retire the `mc-r1-dma-cycle-budget` hack (the accounting collision it papered over is
  gone). Re-measure Y + the abort sweep.
- **E.4 — APU tick ordering.** If E.1-E.3 don't fully close it: move the DMC byte-timer +
  reload arm to AFTER the cycle's bus access (TriCNES `_EmulateAPU` after `_6502`), so the
  arm is invisible to its own cycle by construction (not via a latch — which absorbed).
- **E.5 — counter unification + OAM.** Collapse `cpu.cycles`/`cycles_emitted`/`apu.cpu_cycle`
  toward `master_clock` authority; OAM already unified (`7ae11d2`). Delete the burn-loop +
  `idle_tick` busless path once `internal_cycle` is the sole internal advancer.
- **E.6 — promote + final gauntlet.** Promote to default; serialize `master_clock`; 139/139.

## E.1 + E.3 RESULTS (2026-06-06) — PIVOTAL: the span-3 premise is REFUTED; Stage-E doesn't move Y

- **E.1** (internal_cycle = dummy-read PC; committed): nestest 0-diff. Y=3 (inert — the DMA
  halts BEFORE the dummy read, same as Mesen; making the internal cycle a bus access does
  not change which cycle the DMA enters on).
- **E.3** (E.1 + DMA-halts-extra via budget): nestest 0-diff. Y=**2** (same wrong direction
  as budget alone). The "exact" accounting (DMA extra, every cycle a bus access) gives Y=2,
  FURTHER from 4 than the floor's Y=3.
- **E.4 ≈ F-2** (DMC byte-timer/arm at END of cycle) was already tested earlier = Y=3
  (start-vs-end of the arm doesn't change the service cycle: the DMA loop services at N+1
  regardless, so the entry parity / span is unchanged).

**Conclusion:** the reload is ALREADY span-4 (measured put-start, HANG_PUT=4); the
timing-map agents' "RustyNES span-3" was a code-structure misread of the entry parity. The
Y=3 residual is NOT the reload span and NOT the cycle-model — it is the CheckDMATiming
SETUP WINDOW (1587 vs 1588), a 1-cycle CUMULATIVE difference somewhere in the setup
($4010 STA + clockslides + the reloads during them), which every accounting lever moves the
WRONG way (budget -> Y=2) or leaves at 3. The Stage-E every-cycle-bus-access refactor —
correct and worth keeping for the architecture (E.1 clean, nestest 0-diff) — does NOT close
the DMA tail.

**The genuine remaining diagnostic (E.2-redirected): a DIRECT cross-emulator cycle trace of
the CheckDMATiming setup window** (RustyNES trace_dma_4015 RUSTYNES_FULL_RANGE vs the
TriCNES /tmp/tricnes-harness, both anchored on $4010=$4E) to find the EXACT instruction
where the 1 cycle diverges — instead of guessing levers. This is the de-risk Step A/C that
was never fully executed and is now the only un-tried path to LOCATE (not guess) the cycle.
