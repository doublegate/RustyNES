## Implicit DMA Abort — characterized (default-as-oracle)

The test sweeps a `$4015` disable across 16 cycle-alignments and measures each aborted DMA's duration
(`CalculateDMADuration` Y) into `$500,X`, comparing against `Key1 = 00×10, 01, 01, 00×4`
(`AccuracyCoin.asm:13408`). Loop2→`$520`/Key2, Loop3→`$540`/Key3 (`04×6`).

Measured `$500` (`STA $500,X` writes, trace):
- **DEFAULT (passes):** `00 00 00 00 00 00 00 00 00 00 01 01 00 00 00 00` = Key1 ✓; Loop2 = Key2 ✓.
- **R1 (fails e2):** `01 01 01 01 01 01 01 01 01 01 01 01 01 01 01 01` — **Y=1 for ALL iterations.**

The implicit-abort rule (`asm:13164`, `DMA.xhtml:148`): the reload DMA on a 1-byte non-looping done
sample becomes a **1-cycle DMA**, AND **"will NOT occur if it would land on a write cycle"** → Y=0.
So the correct sweep is mostly Y=0 (abort lands on a write/put cycle) with Y=1 only where it lands on
a read/get cycle. R1 produces Y=1 uniformly = it never applies the write-cycle cancel.

## Root cause (located)

The abort machinery exists (`apu.rs`: `pending_dmc_abort`/`dmc_abort_delay`, armed in
`tick_with_external:644`; `bus.rs`: `service_dmc_abort:1940`, `drain_dma:1567` consumption). But it is
the **legacy/burst** model, NOT wired into R1's interleaved `dmc_dma_step_impl`:

- Legacy `drain_dma(read_addr)`: `if dmc_abort_pending { if read → service_dmc_abort (1-cyc halt, Y=1)
  else (write, read_addr=None) → complete_dmc_abort (cancel, Y=0) }`. Correct get/put split.
- Under R1, the done 1-byte sample fires a **normal reload** via `dmc_step_reload_arm`
  (`pending_dmc_dma=true`), serviced by `read1`'s interleaved loop → a normal 1-cycle DMA (**Y=1
  always**). The separate `pending_dmc_abort` flag is cancelled every cycle by `drain_dma(None)` in
  `cpu_clock:2817` (read_addr always None on the R1 path) → moot. `service_dmc_abort` calls the legacy
  `tick_one_cpu_cycle()`, unusable under R1.

So R1 routes the done-sample through the *normal reload* (Y=1) instead of the *abort* (get→1 / put→0).

## CORRECTION (deeper trace) — the abort IS armed under R1; the truncation doesn't engage

Two assumptions above were wrong; the accurate mechanism (traced `r1_abort.csv`, `abort_pend`/
`abort_dly` columns):
- `dmc.needs_dma()` is FALSE for a done sample (`bytes_remaining==0`, `dmc.rs:177`) — so it is NOT a
  "normal reload". The abort IS armed under R1: the load GET's `complete_dmc_dma` (`apu.rs:500`)
  computes `cycles_until_output` and sets `dmc_abort_delay` for the 1-byte-non-looping-done condition.
  `tick_with_external:644` matures it to `pending_dmc_abort`.
- At the `$4015` disable write the trace shows `abort_pend=1` (e.g. cyc 48680457, put=1), but then a
  **FULL DMA fires anyway** the next cycles (halt/dummy/GET `$FFC0`, span 3) — the abort does NOT
  truncate it under R1. So the abort matures but the interleaved `dmc_dma_step_impl` runs a complete
  DMA regardless (→ a GET happens → the measured behaviour is "DMA occurred" everywhere → Y=1 uniform,
  vs the answer key's mostly-Y=0). The legacy `drain_dma`/`service_dmc_abort` truncation path is simply
  not on the R1 interleaved code path.

This is **deeper than a flag-gated reroute**: the abort and the interleaved `dmc_dma_step_impl` (+ the
get/put parity + the `$4015`-write-cycle timing) are entangled, and each trace step reveals more (the
abort interacts with concurrent normal DMAs). It needs a focused investigation of exactly which DMA the
`CalculateDMADuration $4000` spin measures per sweep iteration (not the setup DMAs), then the Mesen
`StopDmcTransfer` (`_needHalt`→cancel / else 1-halt-cycle `_abortDmcDma`) ported INTO `dmc_dma_step_impl`
so the abort truncates the interleaved DMA — with the get/put-cycle Y=1/Y=0 split — without regressing
the 7/10 + Instruction Timing + the cycle-budget win. The drift-harness/default-as-oracle method
(`def_abort.csv` = ground truth) is the right tool to pin it per-iteration.

## COMPONENT-2 effort (2026-06-05) — the abort has THREE nested layers

Full iteration-0 DMA map (default vs R1) revealed the mechanism precisely:
- **DEFAULT** (3 DMAs): implicit-stop load `$DBA2`(span3), looping load `$DBBB`(span3), measured `$4000`.
  `$DBBB→$4000` = 558 cyc (one period) — the spin catches its FIRST reload → Y=0.
- **R1** (4 DMAs): adds an EXTRA reload `$DA0A` between `$DBBB` and `$4000`. The measured `$4000` lands
  one period later → CalculateDMADuration runs one extra iteration → Y+1.

**The `$DA0A` extra DMA is a SYMPTOM, not the cause:** R1's first looping reload halts on `$DA0A` (a
code addr) where default's halts on `$4000` (the spin) — so default's spin catches its reload (Y=0)
while R1's spin misses it and catches the NEXT reload (Y+1). A +1-cycle drift shifts the CPU's position
when the DMA halts → which address it halts on → which reload the spin catches.

**Layer 1 (load-span-3, `mc-r1-dmc-load-halt-late`, committed `a37ffb2`):** the drift harness CONFIRMS
this removes the `$DBA2` +1 — with it ON the abort region is cycle-identical (no delta change) until a
SEQ break at the clockslide (k=66). A real, validated fix. But INERT on the abort battery because:
**Layer 2 (clockslide computed-jump divergence):** at k=66 default does `JSR→$FF22` while R1 does
`JSR→$FF18` — a ~10-cycle difference in WHICH offset of the `$FFxx` C9-NOP calibration sled they enter.
This is NOT a propagated +1 (the stream is cycle-identical up to it); it's a structural divergence in
the clockslide's runtime-computed jump (some DMC-state-dependent value the calibration reads differs
between R1 and default). Source not yet pinned.
**Layer 3 (the abort write-cancel):** even with layers 1+2 fixed, the alignment-dependent Y=0/1/4
truncation (Mesen `StopDmcTransfer`) still has to be ported.

**Refuted this effort:** `mc-r1-dmc-loop-reload-defer` (simple `defer_dmc_reload_once` for the 1-byte
looping load under R1) — INERT, did NOT remove `$DA0A` (it's a period-later reload, not the immediate
one the defer targets; the real cause is the layer-2 clockslide drift). Reverted.

**Assessment:** the abort is a genuinely 3-layer-deep core. Layer 1 is done+validated. Layer 2 (the
clockslide computed-jump source) is the next pin — needs tracing what runtime value the `$FBxx`
clockslide uses to compute its `$FFxx`-sled entry, and why it differs by ~10 under R1. This is deeper
than the CheckDMATiming fix and may be multi-session in itself.

## LAYER-2 RESOLVED (was a phantom) + the real finding: abort Y is multi-factor

The "layer-2 clockslide JSR→$FF22 vs →$FF18" was a **windowing artifact** — the FULL_RANGE contiguous
trace shows BOTH builds do `JSR $FB65 → $FF18 → sled` identically. The windowed traces just emerged
from the unwindowed C9-sled at different points. No clockslide divergence exists.

Contiguous drift (load-halt-late ON, anchored `$4010=$4E`) shows the real residual: **+1 at `$DBBB`
(the LOOPING load, k=12)**. load-halt-late's all-loads +1 OVER-CORRECTED — it fixed `$DBA2` (non-looping
implicit-stop load, span 4→3) but broke `$DBBB` (looping load, was ALREADY span 3 → 4). Re-targeted the
+1 to **non-looping loads only** (`!self.dmc.loop_flag`) — `$DBBB` stays span 3, `$DBA2` gets the fix.
Validated: 93.53% held, nestest 0-diff, fmt+clippy clean. **STILL inert on the aborts.**

**The definitive finding (after ~6 turns on this core):** the abort Y is **genuinely multi-factor** —
every +1-cycle drift I pin and fix (`$DBA2` load span, `$DBBB` load span) is individually inert on the
measurement. Unlike CheckDMATiming (one clean +1 from one accounting bug), the abort sweep's Y is the
product of several compensating effects (the non-looping load span + the looping load span + the abort
write-cancel + the reload-suppress + the clockslide amplification), and fixing them one at a time does
not converge. This is the signature of needing the **full DMC-DMA-model rewrite** (replace the
get-parity-emergent span model with Mesen's explicit `StartDmcTransfer`/`StopDmcTransfer`/`_disableDelay`
model + the load/reload distinction, re-baselining the 7/10) — NOT incremental drift-patches. The
component fixes (load-halt-late, cycle-budget) are correct sub-pieces that the rewrite will subsume; they
are validated and banked default-off as the foundation. The rewrite is a dedicated multi-week effort.

## REWRITE START — the abort decomposes into EXACTLY TWO span-3 fixes (not "multi-factor")

Clean contiguous drift (FULL_RANGE whole iteration, anchored `$4010=$0E`) + TRUE-span measurement
(through the GET, NOT `in_dmc` runs) corrected the earlier pessimism. The implicit-abort iteration has
exactly TWO DMAs that are span-4 in R1 but span-3 in default — each a +1 the clockslides amplify:

1. **`$DBA2` — the implicit-stop LOAD** (`$4010=$0E` then `$4015=$10`, DMC inactive → load arm fires).
   R1 span 4 (halt-on-put), default span 3 (halt-on-get). **FIXED** by `mc-r1-dmc-load-halt-late`
   (+1 to the load `dmc_dma_delay` under R1, ALL loads — both are span-4). Drift confirms the `$DBA2`
   +1 is removed with it on.
2. **`$DBBB` — the 1-byte-looping RELOAD** (the step-9 `$4015=$10` hits an ALREADY-ACTIVE DMC, so the
   load arm does NOT fire; it's a reload from the looping output unit). R1 span 4 (halt-on-put),
   default span 3 (halt-on-get). **RESIDUAL** — `mc-r1-dmc-load-halt-late` doesn't touch reloads.

With fix-1 on, the contiguous drift is cycle-identical from `$4010=$0E` until `$DBBB` (+1), then the
`$DA0A` extra-reload + `$4000` SEQ-break are pure downstream symptoms of the `$DBBB` +1. So **closing
the implicit abort = the short-reload-span-3 fix on `$DBBB`.**

**Next (the reload-span-3 fix):** `dmc_step_reload_arm` (`apu.rs:799`) fires `pending_dmc_dma=true`
immediately with `dmc_dma_short=false` → the halt lands on the next read (put parity) → span 4. The
1-byte-looping reload must halt one cycle later (get parity, span 3), matching default. Options: a
parity-shift on the reload arm for the 1-byte-looping case (analogous to the load `+1` delay), or mark
it short and have `dmc_dma_step_impl` produce span-3 for short reloads. Validate on the BATTERY (the
7/10 + Instruction Timing + cycle-budget must hold) + the per-iteration `def_abort.csv` oracle. Then
the Explicit Abort + DMC+OAM (which share the abort path) follow. This is now a contained, well-pinned
fix — NOT the open-ended rewrite the earlier "multi-factor" read implied.

## RELOAD-SPAN-3 isolated to the deep core (the DMC-timer phase)

Tested `mc-r1-dmc-loop-reload-span3` (route the 1-byte-looping reload through a 1-cycle delay so it
arms next cycle → halt shifts to get → span 3). **REGRESSED HARD: 93.53% → 82.73%, DMA 1/10, SH\*
broke.** A 1-byte-looping reload fires EVERY period, so a whole-arm 1-cycle delay desyncs the broad DMA
behavior and MOVES the GET — breaking the readout tests (DMA+OpenBus/$2007) that observe the GET
landing. Reverted.

**Why it's the deep core:** the load fix (`+1 delay`) removes the LOAD's leading-halt cycle with the
GET preserved (the load has a delay countdown to retime). The reload has NO delay — it arms when the
DMC byte-timer empties the buffer, so its halt parity IS the **DMC-timer phase**. To make the abort
reload halt-on-get (span 3) WITHOUT moving its GET and WITHOUT breaking every other reload, the DMC
timer phase itself must be Mesen-correct everywhere — i.e. all reloads halt on the right parity
coherently. That is exactly the deferred **cumulative DMC-timer-phase / reload-arm-parity** core (the
`!put_cycle` fire-phase the memory log circled for many sessions). It cannot be a per-reload delay.

**Net of the sustained rewrite effort:** the implicit abort is now decomposed to ONE deep residual —
the 1-byte-looping reload's halt parity = the DMC-timer phase. The load half is solved
(`mc-r1-dmc-load-halt-late`, validated). The reload half is the genuine multi-week core: re-derive the
DMC byte-timer fire-phase so reloads halt-on-get where AccuracyCoin's reference (the passing default
build) does, coherently across the readout cluster (which must keep its current correct landings). This
is the `5ceab83`-breakthrough fire-phase work taken to exactness — a dedicated effort, NOT a flag.

## Fix shape (for implementation, behind a new flag e.g. `mc-r1-dmc-abort-interleave`)

1. When the abort is armed for the done-sample, **suppress the normal reload** (`dmc_step_reload_arm`
   must not set `pending_dmc_dma` when `dmc_abort_delay>0 || pending_dmc_abort`).
2. **Route the abort through the R1 per-cycle path** with the get/put distinction:
   - On a **get cycle** (a `read1` entry where `!put_cycle`): do the 1-cycle abort halt
     (`replay_dma_noop_read(halted)` + `complete_dmc_abort`, in_dmc for 1 cycle) → Y=1.
   - On a **put cycle** (CPU write, or a `put_cycle` read): `complete_dmc_abort` with NO halt → Y=0.
   The put-cycle cancel must be reachable on write cycles too (write1 / cpu_clock), since the CPU may
   be writing when the abort matures.
3. Reference: Mesen `StopDmcTransfer` (`NesCpu.cpp:911`: `_needHalt`→full cancel, else `_abortDmcDma`
   consumed after 1 halt cycle) + `DeltaModulationChannel::SetDmcReadBuffer` aborted-DMA glitch
   (`_bitsRemaining==1 && timer<2 → _disableDelay=3`). TriCNES `APU_ImplicitAbortDMC4015`.

Validate on the BATTERY: Implicit/Explicit Abort `$500/$520` match Key1/Key2 (or the Alt keys); the
7/10 + CheckDMATiming + Instruction Timing hold; nestest 0-diff. Default-as-oracle (`def_abort.csv`,
`r1_abort.csv`) is the per-iteration ground truth; the drift harness (`drift_harness.py`) aligns them.

## 2026-06-05 — root-cause pinned (artifact correction + the real gap)

Two findings this session, both evidence-backed (floor = R1 + `mc-r1-dma-cycle-budget`, 93.53%, 9 fail,
verified):

1. **The "reload must halt-on-get / span 3" framing is an `in_dmc`-undercount ARTIFACT.** The
   `in_dmc_dma` flag clears on the GET cycle BEFORE its trace row, so counting `in_dmc==1` runs
   undercounts the true span by 1 (the documented anti-artifact rule). Measuring TRUE span
   (`in_dmc-run + 1 GET`) on `floor_0478.csv`: histogram `{span4: 257, span3: 32, span2: 2}` —
   R1 reloads are ALREADY true span 4, i.e. the Phase-1 Mesen-model value (`reload DMAs = 4`). So
   re-deriving the byte-timer fire-phase "to make reloads span 3" chases the artifact. DO NOT pursue.
   (Cross-path span diff vs the DEFAULT build is ALSO unreliable — default uses the legacy burst-DMA
   path where `in_dmc` is traced differently; the `+1` heuristic doesn't transfer. Within-R1 only.)

2. **The real Implicit/Explicit Abort gap: the R1 read/write path NEVER consumes `dmc_abort_pending`.**
   `Cpu::read1` (cpu.rs:784) loops `while bus.dmc_dma_pending()` → `dmc_dma_step` → `dmc_dma_step_impl`,
   which has NO abort branch. `Cpu::write1` (cpu.rs:852) has no DMA/abort loop at all (R1 halts only on
   READ cycles, TriCNES `CPU_Read`). The abort machinery exists — `complete_dmc_dma` (apu.rs:503-514)
   arms `dmc_abort_delay` on the 1-byte-done LOAD, it matures to `pending_dmc_abort` (apu.rs:641-646) —
   but ONLY the legacy/OAM-interleave path services it (bus.rs:1567/1884/1941/2044, via `cancel_dmc_dma`
   /`complete_dmc_abort`). Under R1 the abort is armed and then dropped on the floor, so every iteration
   runs the reload as a normal DMA → **Y=1 uniformly** (`01×16`), missing Key1's `00×10,01,01,00×4`.
   The "01→00 on put cycles" Key1 needs IS the write-cycle cancel, NOT a fire-phase shift.

**Concrete fix (next):** wire the abort into the R1 per-cycle path with the get/put distinction
(audit "Implicit DMA Abort" steps 1-3 above):
- In `read1`'s pre-access DMA region, when `dmc_abort_pending()` matures on a GET/read cycle
  (`!put_cycle`): one halt cycle (`replay_dma_noop_read` + `complete_dmc_abort`, `in_dmc` for 1 cycle)
  → **Y=1**.
- When it matures on a PUT/write cycle (a `put_cycle` read, OR inside `write1`): `cancel_dmc_dma` /
  `complete_dmc_abort` with NO halt → **Y=0** ("does NOT occur on a write cycle").
- Gate behind a new default-off flag; validate Key1/Key2 on the battery + the 7/10 + CheckDMATiming +
  Instruction Timing hold + nestest 0-diff before promoting. This closes Implicit + Explicit Abort and
  supplies the abort half of DMC+OAM.

### Implemented `mc-r1-dmc-abort-cancel` — measured INERT (wrong layer; the counter matures too late)

Built the read1/write1 servicing exactly as scoped (flag `mc-r1-dmc-abort-cancel`, default-off):
- `Bus::dmc_abort_pending` / `dmc_abort_is_get_cycle` / `dmc_abort_halt_step` / `dmc_abort_cancel`
  (nes-cpu trait no-ops; LockstepBus delegates to `apu.dmc_abort_pending/put_cycle/cancel_dmc_dma`).
- `Cpu::read1` (before the `dmc_dma_pending` loop): abort pending + get cycle → 1 halt cycle then
  `cancel_dmc_dma` (Y=1); abort pending + put cycle → `cancel_dmc_dma`, no halt (Y=0).
- `Cpu::write1` (top): abort pending → `cancel_dmc_dma`, no halt (Y=0, can't RDY-halt a write).

**Result: AccuracyCoin 93.53% / 9 fail — BYTE-IDENTICAL to the floor** (Implicit/Explicit Abort still
`[error 2]`). fmt + clippy clean, default build untouched. The hook never bites.

**Root cause (trace-confirmed, `floor_0478.csv`):** the abort arms via `complete_dmc_dma` →
`dmc_abort_delay` (a 2-3 cycle countdown, `dmc_abort_delay_for` ∈ {2,3}) → `pending_dmc_abort`. That
counter matures **during/after the reload DMA it should cancel** — at cyc 48688847 `abort_pend` flips
to 1 on a cycle that is itself `in_dmc==1` (the DMA already running: `W $4015`[dly3] → `R $DB21`[dly2]
→ `I`[dly1] → `r $DB22`[in_dmc=1, abort matures]). By the time `dmc_abort_pending()` is true at the
next `read1`, the DMA has already run, so `cancel_dmc_dma()` clears already-clear flags → no-op. Also
note the abort arms only ~5× across the window, not the 16× the sweep needs — `dmc_abort_delay_for`'s
{2,3} guard is too narrow. This is the SAME "span completes before `dmc_abort_delay` matures → the flag
never bites" shape the pivot log already refuted (memory `project_v2_from_main_pivot`, Phase C s2).

**Verdict:** the read1/write1 servicing is the correct *cancel* logic but the wrong *layer/timing*. The
abort cannot ride the delayed `dmc_abort_delay` counter — it must be **decided synchronously at the
1-byte deliver point** (Mesen `SetDmcReadBuffer` → `StopDmcTransfer` sets `_abortDmcDma`, consumed on
the very next `processCycle`: `if(_needHalt)` full-cancel else 1 halt cycle), with the get/put
write-cycle suppression applied at that decision, BEFORE the reload runs. The flag is KEPT default-off
(byte-identical, gates green) as the validated servicing half; the remaining work is the synchronous
arming: replace the 2-3 cycle `dmc_abort_delay` with an immediate `_abortDmcDma`-style edge set at
`deliver_sample`/`complete_dmc_dma` and consumed on the next CPU cycle via the now-wired
`dmc_abort_pending` path. DO-NOT-RETRY: the delayed-counter `dmc_abort_delay` servicing (inert, proven
twice).

### Premature-clear bug FOUND + FIXED — the abort now bites (but not yet Key1)

The "inert" result above had a concrete cause: `drain_dma(None)` is called from `cpu_clock` EVERY R1
cycle (bus.rs:2817), and its legacy abort branch (bus.rs:1567, active under `not(dmc-get-put-scheduler)`)
ran `complete_dmc_abort()` with `read_addr=None` — wiping `pending_dmc_abort` one cycle after it matured,
BEFORE the read1 hook could see it. Fixed by gating that branch to `not(mc-r1-dmc-abort-cancel)` so the
