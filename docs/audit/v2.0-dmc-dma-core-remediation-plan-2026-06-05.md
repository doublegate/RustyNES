## The reframe (why this is not "lever #24")

Every prior attempt (23 levers — see the DO-NOT-RETRY table at the end) treated Y=3 as a *cycle-counting /
parity* problem and applied a localized shift. RW-1 collapsed the dual parity flip-flops to one counter +
seed (byte-identical, kept); **RW-2 collapsed the OAM cumulative counter and Y STILL did not move** —
refuted in all three forms (theory; OAM-parity probes; full inline interleave, which regressed C1).

So the residual is **NOT** the OAM cumulative drift, and **NOT** any single parity/span lever already
tried. The three research passes converge on two facts that reframe the work:

1. **The reference model is structurally specific, and we have never validated against it cycle-by-cycle.**
   Mesen runs the ENTIRE DMA as a prefix at the top of every `MemoryRead` (never on writes), on ONE master
   cycle counter, with `getCycle = (CycleCount & 1) == 0`, halt OUTSIDE the loop and dummy/align/get
   INSIDE, the halted CPU re-issuing its read AFTER the span, and a TWO-latch open-bus model where `$4015`
   reads update only the internal bus and every DMA read updates the external bus. The "span 3 vs 4" is
   EMERGENT from the get/put parity at the halt cycle; the byte timer only sets the 2-vs-3 *request delay*
   (`dmaDelayCounter = periodCounter & 1 ? 2 : 3`). (Agent-1: `NesCpu.cpp:502-509,581-759,761-829`;
   `DeltaModulationChannel.cpp:247-298`; `OpenBusHandler.h`.)

2. **`CheckDMATiming` Y is a DATA-BUS measurement, not a span-length measurement.** `CalculateDMADuration`
   (asm:12899) walks `LDA $4000` one cycle earlier per iteration and counts (`INY`) how many iterations
   read `$00`. The `$00` is the DMC sample driven onto the EXTERNAL open bus by the GET; it PERSISTS on
   open bus (write-only `$4000` returns the latch) until the next halt/dummy cycle re-reads the parked
   `$4000`→`$40`. So Y = how many of the walking-read windows the GET's `$00`-on-open-bus covers — a
   function of the GET's exact cycle AND the open-bus persistence model, not the structural span count.
   This is WHY `mc-r1-dmc-mesen-span` / `fixed-span` were inert (the span was already 4) and why the
   held-address-conflict lever was the wrong mechanism (CM-P0: Y uses the sample byte, not the register
   conflict). **We have never modeled the open-bus `$00` persistence + walking-read alignment against a
   cycle-aligned Mesen reference.**

The conclusion: stop guessing levers. **Build the Mesen per-cycle oracle cross-diff first** (repeatedly
cited as "needed/blocked" and never completed), pin the EXACT cycle where our GET / open-bus diverges, then
implement the DMC-DMA + data-bus path to match Mesen cycle-for-cycle. The validation substrate IS the
deliverable that breaks the 23-lever refutation cycle.

## Canonical TARGET model (the spec to match — Agent-1 + Agent-3)

- **One counter.** `CycleCount++` once per CPU cycle, INCLUDING every DMA cycle. `get = (CycleCount&1)==0`
  (even). No separate DMA-phase counter. (RustyNES already has this after RW-1: `apu_phase`/`put_cycle`
  derive from `cpu_cycle + parity_seed`; the get-cycle is `!put_cycle`. The remaining question is the
  absolute alignment + the open-bus model, NOT the derivation.)
- **DMA runs at the top of `MemoryRead`, never on writes.** Halt cycle OUTSIDE the loop; the loop decides
  per-cycle: get→DMC-fetch (priority) else OAM-read else DMC-dummy; put→OAM-write else align-dummy. Span
  3 (load, halt-on-get) vs 4 (reload, halt-on-put) is emergent.
- **Two open-bus latches.** `_externalOpenBus` (every read incl. DMA updates it) + `_internalOpenBus`
  (CPU-only; `$4015` read updates ONLY this). The halted CPU re-issues its read AFTER the span. The GET's
  sample byte lands on the external bus and persists until the next read.
- **The `$4000-$401F` register glitch** (`ProcessDmaRead`, `NesCpu.cpp:761-829`): when the parked 6502 addr
  is in `$4000-$401F`, a DMA read of `$40xx` activates the register + its `$20`-byte mirror
  (`0x4000 | (addr & 0x1F)`) with side-effects; `$4015` is read internally and does NOT drive the external
  bus. (RustyNES `dmc_dma_read` already does this for the DMC path; `raw_oam_dma_read` does NOT — the
  `APU Register Activation` Test 6 gap, but that is upstream-coupled to this same DMC setup.)
- **Abort:** `StopDmcTransfer` = cancel-if-`_needHalt`-else-`_abortDmcDma`; `_disableDelay` = 2/3 by parity;
  a `$4015` D4=0 during a DMA aborts it one APU cycle later, costing one halted cycle (or not occurring at
  all if the halt is delayed by a write cycle). ONLY Mesen models this fully; ares/tetanes do not.
- **DMC+OAM overlap:** DMC steals get cycles, OAM pauses + realigns; overlap costs 1/2/3 cycles
  (TriCNES `OAMDMA_Aligned` stateful flag). This is the `DMC DMA + OAM DMA` answer-key tail.
