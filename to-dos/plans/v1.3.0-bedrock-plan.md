## Context

RustyNES shipped **v1.0.0 "Cycle-Accurate"** → **v1.1.0 "Scriptable"** → **v1.2.0 "Curator"** (the last is
merged to `main`, awaiting the release cut). v1.2.0 added mapper tiering (87 families) + ROM ingest/DB/editor,
NTSC knobs + composable shader stack + HD-pack loader, the peripheral set (Family BASIC / SNES mouse /
Arkanoid / Genie DB), Lua depth (onNmi/onIrq/setInput), menu-bar UX (FontAwesome icons + contextual
enable/disable + remappable shortcuts), web touch + Power Pad wasm + experimental piccolo wasm Lua, a
turn-key netplay deploy bundle, and a PGO/BOLT CI job — all with **AccuracyCoin 100% (139/139)** held.

**v1.3.0 is a foundation + tooling + polish release**, NOT architectural closure. Theme: modernize the
toolchain, fix frame-pacing measurement/judder, build out the GeraNES-class **developer tooling**, widen
mapper breadth, and polish input/netplay UX — while the hard contract holds. Synthesized from three research
passes (GeraNES feature inventory, accuracy/compat backlog + browser-RA feasibility, and a frame-pacing
root-cause) plus the previously-captured toolchain-modernization plan.

**Hard contract (unchanged):** determinism (same seed+ROM+input ⇒ bit-identical fb+audio); **AccuracyCoin
100% (139/139)** + oracle byte-identical after every change; chip stack stays `#![no_std]`+alloc; new
features off-by-default → shipped/wasm/no_std byte-identical; core test harness never reads frontend
DB/overrides; pin-the-test-ROM-first for accuracy work.

### B — Frame pacing & present (the maintainer's "Presented judder" report)
Root-caused: **"Presented" is timestamped AFTER `surface.present()` returns** (`gfx.rs:1111` → `app.rs:5886`)
so it captures GPU-submit + vsync-gate + coalesced-`RedrawRequested` jitter, while **"Produced" is timestamped
immediately after emulation** (`emu.rs:612`) and stays clean. ~60-70% a MEASUREMENT artifact (wrong timestamp
location) + ~30-40% real judder (NTSC 60.0988 Hz producer beating a 60.000 Hz display → periodic catch-up,
worse on the Fifo fallback). Fixes (frontend-only, no core/determinism surface):
- **B1** move `record_presented()` to the START of the `RedrawRequested` branch (timestamp the display-refresh
  signal, not GPU-submit return) — removes the submit-queue jitter; relabel the panel series to clarify
  semantics ("at redraw") + a note.
- **B2** make the measurement **display-sync-aware**: when the display-sync pacing regime is active, the
  refresh IS the clock → timestamp at redraw entry; document that Wallclock/Mailbox regimes differ.
- **B3** mitigate the real beat: audit the present-mode selection (Mailbox vs Fifo fallback) + the
  sleep-then-spin pacer + the ~10 s Fifo-fallback hitch; consider a frame-pacing improvement so the
  producer cadence and display refresh don't accumulate a catch-up hitch.
- **B4 (defer to v1.4)** a real GPU present-timing query (`KHR_present_timing` / `DXGIGetFrameStatistics`) for
  true scan-out timing — platform-specific, high effort. Note only.

### C — Developer tooling (GeraNES-inspired; the headline theme)
- **C1 PPU Event Viewer** — enhance the existing `debugger/event_panel.rs` into GeraNES's full dot-grid
  (341×262) PPU register R/W trace (blue=read/red=write) with hover (addr/value/scanline/dot). Core capture
  already exists via the `debug-hooks` event log (output-only, byte-identical off). HIGH value (accuracy +
  ROM-hacking debugging).
- **C2 Symbol-file loading** — load `.sym`/Mesen-label/`.elf` symbol files; annotate the CPU-panel disasm +
  memory views with labels. New ELF/label parser (frontend; e.g. `object`/`goblin` for ELF, a small parser
  for Mesen `.mlb`). Frontend-only. HIGH value (TAS/ROM-hacking).
- **C3 Memory Compare / search** — baseline snapshot → filter changed regions (bitwise masks, range, diff
  list) for cheat-hunting/state inspection. Frontend-only. MEDIUM.
- **C4 Trace-log + symbol annotation** — extend the existing trace panel with symbol names + CSV/text export.
  Frontend-only, low effort. MEDIUM.

### I — Browser RetroAchievements (casual-mode-only)  *(maintainer-approved, was deferred)*
Ship browser RA for **casual achievements/leaderboards only**, with loud in-UI caveats that **hardcore
is unavailable in the browser**. Known blockers to engineer around (document them in `compatibility.md`
regardless): rcheevos is C → needs an **emscripten second build track**; browsers **forbid setting the
RA `User-Agent` header** (auth/identification path must adapt — RA allowlists the client by UA today, so
coordinate an alternative or accept casual-only identification); **hardcore integrity collapses** in a
browser (DevTools can patch the wasm) → casual-only is the only honest mode. Off the critical path; budget
for RA-API-drift maintenance. ~2-3 wk. Gate it behind a feature so the native build is unaffected.

### C1-RES — Hard-tier accuracy residuals (RE-BASELINED 2026-06-15, then close)
The maintainer's directive: *we already did the master-clock refactor, so re-test all of these against the
**current** master-clock state before assuming any work.* Done — the picture is far smaller than the old
"~3-4 wk fractional refactor" fear:
- **`cpu_interrupts_v2` — ALREADY CLOSED.** All 5 sub-tests (incl. 2/3/5) are now **plain strict `#[test]`s,
  5/5 pass, 0 ignored** on the default master-clock build. The old plan line listing `{2,3,5}` as residuals
  is **stale** — remove it.
- **`mmc3_test_2_4_scanline_timing` — OPEN** (1-CPU-cycle bracket residual after "C1 step B4"; ADR 0002).
- **`apu_reset_len_ctrs_enabled` — OPEN** (length counters not re-enabled / triangle altered across reset).
- **`apu_reset_4017_written` — OPEN** (last `$4017` value not re-applied on reset).
- **`mmc3_test_2_6_mmc3_alt` — NOT A BUG** (NEC rev B; project defaults to Sharp rev A — by-design `#[ignore]`).
- The `opcodes.rs`/`apu.rs`/`ppu.rs` unit `#[ignore]`s are **permanent-by-design pins** of superseded
  pre-master-clock granularity — cannot/should-not un-ignore; real coverage is the strict ROM tests + AC 100%.

**Diagnosis verdict (two debug agents, 2026-06-15) — the 3 residuals share ONE root cause, and it is a
deep refactor, not three small fixes:**
- **MMC3 scanline #3** (ADR-0002): the integer "3-PPU-dots-per-CPU-cycle" model cannot represent the M2
  sub-cycle phase the test brackets. Needs Mesen2's **fractional 12-master-clocks-per-CPU-cycle** model with
  a φ1/φ2 memory-access split so the CPU IRQ-poll samples at the correct phase. **15+ documented rollbacks**;
  Session-29 empirically falsified the localized fix; ADR-0002 Stop Condition #3 forbids another point-fix.
  1–3 wk + regression hunting, **HIGH risk to AccuracyCoin 139/139** (touches the calibration baked into
  ppu_vbl_nmi, sprite-hit, DMC-DMA, nestest-0-diff, the 60-ROM oracle). Genuinely v2.0-scale.
- **Both `apu_reset`**: same axis — `Nes::reset()` is a *function-call* reset that doesn't model the
  cycle-accurate CPU reset-vector delay (9–12 clocks) + frame-counter re-arm phase. The `4017_written`
  `last_4017` storage field is ~1hr + determinism-neutral, **but alone it neither passes the test nor avoids
  regressing the currently-passing `4017_timing`** (boundary value 12) — two prior attempts confirmed this.
  No clean partial win to ship. The APU register semantics (`length.rs`, `frame_counter.rs`) are already correct.

**MAINTAINER DECISION (2026-06-15): KEEP DEFERRING the 3 residuals; bank `cpu_interrupts_v2` as
newly-closed.** No master-clock refactor in v1.3.0 — the risk/rollback history (15+ rollbacks, ADR-0002
Stop Condition #3, HIGH risk to 139/139) does not justify it for zero production-ROM gain. **v1.3.0
C1-RES scope is therefore DOCUMENTATION-ONLY:** (1) update `docs/STATUS.md` + ADR-0002 to record that
`cpu_interrupts_v2` is now 5/5 strict (closed) and the remaining hard-tier set is exactly 3 (MMC3 scanline
#3 + 2 apu_reset), all sharing the single fractional-master-clock root cause and deferred as a future
v2.0-scale item; (2) keep the 3 tests `#[ignore]`'d. The `irq_trace` golden oracle + `cpu-instr-cycle-trace`
v2.0 scaffold remain in place as the gate if a future release ever takes on the refactor.
