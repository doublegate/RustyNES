## Context

RustyNES is a pure-Rust workspace: a `#![no_std]`+alloc cycle-accurate chip stack
(`rustynes-{cpu,ppu,apu,mappers,core}`) + a `std` desktop frontend `rustynes-frontend`
(winit + wgpu + cpal + egui), a WebAssembly build (winit/wgpu + a lightweight canvas embed,
with a deterministic Pointer-Events **touch overlay**), rollback netplay, RetroAchievements,
TAS, save-states/rewind/run-ahead, Lua scripting, and a composable shader stack. It ships at
**v1.5.0 "Lens"** with **v1.6.0 "Studio"** in development; this plan slots **after the
v1.6.0+v1.7.0 desktop maturity** as the first **platform** (not accuracy) release.

**v1.8.0 ports RustyNES to a complete, shippable Android app.** The pure-Rust core compiles to
AArch64 and produces **bit-identical** framebuffer+audio on ARM — so the entire emulator value
(accuracy, shaders, save-states, TAS) ports for free; only the *host shell* (lifecycle,
rendering surface, audio sink, touch UI, file access, packaging) is new. The accuracy bar is
held by the existing chip stack and host CI; nothing in this release touches it.

## Hard contract (unchanged; applies to everything below)

- **Determinism:** same seed+ROM+input ⇒ **bit-identical** framebuffer+audio. The chip stack +
  the emitted-frame/sample contract are **never touched**, so **AccuracyCoin 100% (139/139)** +
  the oracles stay green on **host** CI (Android CI only proves the build links + an instrumented
  smoke boot). The core is platform-independent → a state saved on desktop loads on Android and
  a `.rnm` TAS replays identically, so **desktop⇄Android netplay cross-play stays valid**.
- **All Android work lives in new host crates** — a shared `rustynes-mobile` bridge + a
  `rustynes-android` platform crate + a Gradle module — exactly as `rustynes-frontend` and the
  wasm path are separate hosts over the same core. The chip stack stays `#![no_std]`+alloc;
  `unsafe` is confined to the JNI/surface glue (each with a `// SAFETY:` note).
- **NEVER bundle commercial ROMs.** All ROM/SRAM/save-state I/O goes through the Storage Access
  Framework (document picker + **persistable URI grants**); the app collects nothing by default.
- New mobile pacing/throttle knobs are **frontend-only** — run-ahead's speculative frames are
  already rolled back and never reach save-states/TAS/RA, so auto-throttling cannot change the
  persistent timeline.

## Architecture (LOCKED: hybrid, with a winit spike)

**Ship target — Option (c) Hybrid.** Keep `wgpu` rendering the NES image (full shader stack
intact) onto an Android `SurfaceView`/`SurfaceHolder` whose `ANativeWindow` is handed to wgpu.
The core + render/audio loop run on a dedicated native thread (the Android analogue of the
desktop `emu-thread`); **Jetpack Compose** owns all chrome (top bar, settings, SAF ROM picker,
touch overlay, save-state manager). Touch/gamepad events publish into the existing lock-free
`SharedInput` and are read at the **same late-latch a keypress uses** — so TAS/netplay/rollback
see them identically and there is **no new determinism surface** (the exact pattern the wasm
touch overlay already proved).

**beta.1 spike — Option (a).** Compile the existing winit+wgpu+egui `App` to Android via
`android-activity` (the `wasm-winit` path is the structural precedent). This is the fastest proof
the core renders + sounds on real ARM, and winit 0.30's `resumed`/`suspended` already models the
surface-loss contract. The egui debugger panels can survive as an optional in-surface power-user
overlay, so the spike's reuse is not wasted; the shippable app then graduates (a)→(c) by lifting
the wgpu surface + core loop into `rustynes-android` and replacing the egui chrome with Compose.

**Rejected:** Option (b) pure-Compose-no-wgpu — discards the NTSC/CRT/Bisqwit shader stack +
the 8:7-PAR/overscan/letterbox blit + the wgpu present path (the v1.0–v1.5 rendering crown
jewels) for Material polish; an XL rewrite at lower fidelity.

### Shared `rustynes-mobile` bridge (factored here, reused by v1.9.0 iOS)

A new platform-agnostic crate that owns the **typed control surface** over the core — load ROM
(from a byte buffer, never a path), set input (the `Buttons` mask per port), tick/run-frame,
borrow the current framebuffer, save/load state, query status — generated via **UniFFI** so the
Kotlin (Android) and Swift (iOS) bindings come for free and type-checked. Platform crates
(`rustynes-android`, and v1.9.0's `rustynes-ios`) wrap it with only the thin, hot, hand-rolled
glue UniFFI can't express: handing the native surface handle (`ANativeWindow` / `CAMetalLayer`)
to wgpu, and the audio sink. This keeps the hand-rolled `unsafe` JNI/FFI surface tiny and shares
~all of the binding layer across both mobile targets.

## STANDING directive — continual research + opportunistic improvement

When implementing any workstream below, **re-mine the web + the Rust-on-Android ecosystem**
(winit/wgpu/cpal/android-activity/cargo-ndk/UniFFI release notes, Play policy updates) as needed
— the ecosystem moves fast. And make any **beneficial enhancement or measured performance/battery
optimization** encountered en route even if not explicitly listed here, provided it respects the
hard contract (determinism-neutral, core untouched, gated where it changes shipped behavior).

## Workstreams

Lettered, each independently shippable as a beta; cumulative on `main` via `feat/*` PRs.

### A — Foundation & build  → beta.1
Stand up `rustynes-mobile` (UniFFI control surface) + `rustynes-android` (`jni` 0.21 hot glue) +
a Gradle module. Toolchain: **NDK r27 LTS** (verify 16 KB page alignment — a Play requirement for
Android 15+ regardless of NDK), Rust targets `aarch64-linux-android` (ship) + `x86_64-linux-android`
(emulator/CI) + optional `armv7-linux-androideabi`, built with **`cargo-ndk`** invoked from Gradle
via **`cargo-ndk-android-gradle`** → `.so` in `jniLibs/` → **AAB** (NOT `cargo-apk`: deprecated +
APK-only, can't publish to Play). `minSdk 26` (AAudio floor), `targetSdk 35` (Play mandate since
2025-08-31). CI: GitHub Actions mirrors `release.yml` — build arm64+x86_64, `gradle bundleRelease`,
upload the AAB; the determinism gate stays on host CI. **Effort: L. Determinism-safe** (new host).

### B — Rendering + video filters  → beta.2  *(SHIPPED via AGSL; wgpu deferred)*
**Decision (2026-06-19):** ship the render path as the per-frame RGBA `Bitmap` blit (proven
performant — the NES is light) and deliver the **GPU shader experience via an AGSL `RuntimeShader`**
applied over the Compose render path (`Modifier.graphicsLayer { renderEffect = … }`): None /
Scanlines / CRT (soft-scanline + RGB aperture mask + vignette), an "FX:" cycle control + persisted
choice. `RuntimeShader` is API 33+, so it's feature-detected (`videoFiltersSupported`) and the plain
blit is the always-on fallback. Post-processing only — the emulated framebuffer/audio is untouched,
so determinism holds. **This is the pragmatic, Compose-native, low-risk path to the user-facing
shader goal; it avoids the native-surface rearchitecture.**

The **full wgpu-on-`SurfaceView` native renderer** (reusing the desktop WGSL NTSC/CRT/Bisqwit stack,
`ANativeWindow`→wgpu via `raw-window-handle`, the `surfaceDestroyed`/`surfaceCreated` surface-loss
lifecycle, Choreographer pacing) is the documented **deeper-architecture follow-up** — worth it for
shader-stack parity + max perf/battery, but not required to ship a polished v1.8.0. **Determinism-
safe** either way (presentation-only). **Effort: AGSL = S (done); native wgpu = L (deferred).**

### C — Audio  → beta.3  *(nearly free)*
**`cpal 0.18` (the current workspace pin) has a native AAudio backend at API 26**, so `audio.rs`
(the lock-free SPSC ring + allocation-free callback + 4-tap Hermite ±1% DRC resampler) works
**unchanged** — the single biggest de-risk. Add: AAudio `PerformanceMode::LowLatency` (2-burst
buffer, MMAP where supported), a mobile DRC latency-target default (mobile bursts are spikier —
bump headroom like the desktop >75 Hz rule), stream pause/resume on `onPause`/`Suspended`, and
audio-focus handling (`AudioManager`) forwarded to mute/pause. **Effort: S. Determinism-safe** —
the resampler is timing-only; the core's emitted samples never depend on wall-clock feedback.

### D — Input: touch + gamepad + haptics  → beta.3
**Reuse the deterministic wasm touch-overlay design** (`wasm_touch.rs`): D-pad/A/B/Start/Select +
selectable target port (P1–P4 Four Score) + Power Pad mat, drawn by Compose (Option c) or egui
(spike), funnelled into the `Buttons` mask → `SharedInput` → `EmuCore::latch`. Hardware gamepads:
Android `InputDevice` + `KeyEvent`/`MotionEvent` (`gilrs` does NOT cover Android) → the same mask,
via `GameActivity`'s `InputIterator` or the Compose `Activity`'s `dispatchKeyEvent`/
`onGenericMotionEvent` over JNI. Light haptics on touch-press via `Vibrator`/`VibratorManager`.
**Effort: M (touch + gamepad), S (haptics). Determinism-safe** — all input converges on the one
late-latched `Buttons` mask, identical to wasm touch.

### E — Storage / SAF + lifecycle  → beta.4  *(retires Risk #3)*
ROM import via the **Storage Access Framework** (`ACTION_OPEN_DOCUMENT`, optional
`ACTION_OPEN_DOCUMENT_TREE` for a library view) with **persistable URI grants**
(`takePersistableUriPermission`) so ROMs survive reboot; bytes read via `ContentResolver` → handed
to `Nes::from_rom` (the wasm byte-picker is the closest precedent — no path, raw bytes). Save-states
/SRAM/settings in app-private `Context.filesDir` (the Android analogue of the desktop `ProjectDirs`):
keep the existing **`.rns` format + `<rom-sha256>/slot-N.rns`** layout (byte-identical, just a
different root → cross-device portability). Save-on-background (`onPause` → SRAM + auto-state) +
the desktop `emu-thread`/`Arc<Mutex<EmuCore>>` model (native `pthread` host — a better fit than
single-threaded wasm). **Effort: M–L. Determinism-safe** (format unchanged).

### F — Compose shell + feature-port  → beta.5
Native Material 3 shell: top bar, settings (DataStore or `toml` in `filesDir`), save-state manager,
ROM library/recents on content URIs, the touch-overlay layout. Port the feature matrix —
**ports clean:** core, audio, video+PAR+overscan, NTSC/CRT/Bisqwit shaders + ShaderStack,
save-states/rewind/run-ahead, TAS `.rnm` (SAF import/export), per-game DB, custom palettes
(SAF), HD-pack (SAF tree, off-by-default). **Re-map:** clipboard screenshot → Android
share-sheet/MediaStore. **DROP:** `rustynes help`/ratatui (terminal-only). **DEFERRED (per the
locked MVP):** Lua (mlua *does* cross-compile with the NDK — feature-gate for a later increment),
RetroAchievements (rcheevos C cross-compiles — needs a Compose OAuth/login UI), netplay (UDP
compiles but mobile NAT/CGNAT needs the TURN story that's already a deferred desktop item).
**Effort: L. Determinism-safe.**

### G — Play Store / signing / legal  → rc.1  *(Risk #2 is partly non-engineering)*
Enroll in **Play App Signing** (Gradle `signingConfig` upload key); AAB build + 16 KB-alignment
check + targetSdk 35 in CI; IARC content rating (an empty emulator shell = Everyone/PEGI 3; the
*user's* ROMs aren't the app's content); data-safety = "no data collected" for the MVP. Emulators
**are allowed** on Play (RetroArch/Dolphin/PPSSPP ship there) given **no bundled ROMs / no
downloader** — compliant by design. Mitigate the Aug-2025 Play-Protect "ROM-based systems"
language + the 2026–27 developer-verification rollout by (1) preparing dev verification early and
(2) shipping a **guaranteed sideload/F-Droid + GitHub-Releases channel** so the project never
depends solely on Play (also the home for the egui-debugger power-user build). **Effort: M;
approval/calendar risk is outside our control.**

### M — Monetization / licensing (freemium $2.99 unlock)  → rc.1
A **single free Play listing** with a **one-time, non-consumable in-app purchase** ("Full
Unlock", product id `full_unlock`, **$2.99 USD**) via the **Play Billing Library** (`com.android.
billingclient:billing-ktx`, v7+). NOT a paid app and NOT a second "demo" listing — Play's modern
guidance is one free app + an unlock IAP. Pieces:

- **`Entitlement` / `BillingManager`** (new Kotlin): on start, connect `BillingClient`, query the
  `full_unlock` product + the user's purchases, and expose a `@Volatile isUnlocked` the shell
  reads. A purchase must be **acknowledged** (`acknowledgePurchase`) or Play auto-refunds in 3
  days. Entitlement is cached locally (DataStore) for offline launches but is **re-verified
  against Play** on each connect (the cache is a convenience, not the source of truth). No
  server: a non-consumable IAP is owned forever and restored via `queryPurchasesAsync` (handles
  reinstall / new device / "restore purchases").
- **Demo gating (the three locks + the timer), all enforced in the shell, core untouched:**
  1. **10-minute session timer** — a monotonic per-launch budget; on expiry, pause the emulator
     and show a non-dismissable "Demo time's up — Unlock full version for $2.99" sheet with the
     Buy + Restore buttons. Resets on next launch (per-session, not per-day).
  2. **Save-states disabled** — the Save/Load slot buttons are hidden/disabled in demo.
  3. **Save-on-background + auto-resume disabled** — `onPause` writes no `auto` state and a ROM
     never auto-resumes in demo.
  4. **Battery-backed SRAM not persisted** — the on-cart save RAM is never written to disk in
     demo (a battery save inside the snapshot is fine in-session, but nothing survives close).
  When `isUnlocked` flips true (purchase or restore), all four lift immediately, no restart.
- **UI:** an always-visible "Unlock ($2.99)" affordance in demo; a "Restore purchase" entry; a
  subtle "Demo — N:MM left" indicator. After unlock, these disappear.
- **Determinism-safe / accuracy-safe:** every gate is a *host* policy over persistence + pacing;
  the emulated framebuffer/audio for a given input is **byte-identical** in demo and paid (the
  demo is the same emulator, just without the persistence features + a session clock). The core,
  the `.rns` format, and AccuracyCoin are untouched.
- **Maintainer-manual (can't be CI-self-certified):** create the `full_unlock` managed product
  in the Play Console, the pricing ($2.99) + tax, a license-tested account, and the closed-track
  upload to exercise the real purchase flow (Play Billing can't be fully tested on a sideloaded
  build). A **`BuildConfig`/debug override** (`debug.force_unlocked` / `debug.force_demo`) lets the
  gating + timer + UI be verified on-device without Play. **Effort: M.**

### Perf / thermal / battery  *(cross-cutting, in B/C/F)*
Perf is **not** the gate (the NES is light: ≤2 ms/frame headless, ample on modern arm64) —
**thermals/battery** are. Default run-ahead **off or 1** on Android (keep the budget-throttle
hysteresis), a smaller default rewind window, Choreographer vsync pacing (no battery-draining
busy-spin), and respect `PowerManager.OnThermalStatusChangedListener` to back off
run-ahead/fast-forward. **Determinism-safe** — all frontend pacing/scheduling knobs.

The byte-identical core; the entire wgpu render + shader pipeline; the cpal/Hermite-DRC audio

## Maintainer decisions (researched defaults; flag to override)

1. **Bindings:** **UniFFI** for the typed surface + raw `jni` only for the surface/audio hot path
   *(default: yes — minimises hand-rolled `unsafe`)*.
2. **Activity backend:** `GameActivity` (better input/IME) vs a plain Compose `Activity` hosting a
   `SurfaceView` *(default: plain Compose `Activity` for cleaner Compose integration; revisit if
   input plumbing argues for GameActivity)*.
3. **ABIs:** arm64-only ship vs arm64+armeabi-v7a *(default: arm64-only — smallest AAB; x86_64 for
   emulator/CI regardless)*.
4. **NDK pin:** r27 LTS vs r28/r29 (16 KB-by-default) *(default: r27 LTS, Rust's supported LTS)*.
5. **Distribution posture:** Play-first + sideload backup, or F-Droid/sideload-first treating Play
   as best-effort *(default: Play-first with a guaranteed sideload/F-Droid channel)*.
6. **Lua on Android:** ship native mlua (cross-compiles) vs defer scripting from the mobile build
   *(default: defer per the locked MVP; feature-gate for a later increment)*. **RESOLVED in v1.8.6:**
   native `mlua` shipped — `rustynes-script`'s `ScriptEngine` was made `Send` (mlua `send` +
   `Arc`/atomic newtypes) so the UniFFI bridge can hold it; host/CI-verified, on-device pending.

## Out of scope / deferred (documented, not built in v1.8.0)

Netplay on mobile (NAT/CGNAT/TURN — already a deferred desktop item), RetroAchievements
(Compose login UI over the cross-compiled rcheevos), Lua scripting, and the egui debugger as a

(into the post-v1.8.6 tail).

## Optimised release order (rationale)

Sequenced by **user-facing impact × dependency order × risk**: fix the biggest gameplay pain first
(touch input is currently single-touch — a real bug), then deepen the shell (settings/state), then
the high-effort/high-risk native renderer, then the power-user feature port, then connectivity.
**The Google Play launch is held to v1.8.6 (feature-complete)** per the maintainer's updated timing;
v1.8.0–v1.8.5 distribute via **GitHub Releases / sideload** and ship **full-featured** (freemium
behind `PLAY_BUILD`, off until v1.8.6). Play-listing growth assets are still *prepared* across the
early releases so the listing is ready when the build is.

## v1.8.2 — Input & the virtual controller  *(the priority increment)*

Two coupled deliverables: **multi-touch input** and a **stylised virtual NES controller** that the
touch regions are derived from, both **resizing + remapping correctly across the Z Fold 7's two
displays**.

### A — Multi-touch (fixes the single-input bug)

**Root cause.** The current overlay gives each on-screen button its own
`pointerInput { detectTapGestures { onPress … } }`. `detectTapGestures` tracks a *single* gesture, so
pressing A while holding Right (or two D-pad directions for a diagonal) drops one input — only one
on-screen button registers at a time.

**Fix — one container-level multi-pointer handler.** Replace the per-button gesture detectors with a
**single `pointerInput` on the controller container** that runs an `awaitPointerEventScope` loop,
and on every `awaitPointerEvent()` recomputes the *entire* `Buttons` mask from **all currently-pressed
pointers**: for each `PointerInputChange` with `pressed == true`, hit-test its `position` against the
button regions and OR in every region it falls inside; then `controller.setButtons(0, mask)` and
`change.consume()`. This is the RetroArch-style overlay model and it gives, for free:

- **simultaneous presses** (A + B, A + a direction, Start + Select),
- **D-pad diagonals** (a finger on the cross's corner falls inside both the Up and Right regions —
  the regions are allowed to overlap at the diagonals so ↖↗↙↘ register as two bits),
- **slide / roll** off one button onto another without re-tapping (handled because the mask is
  recomputed from live positions every event, not latched per-down),
- correct **late-latch** semantics (the mask still flows through the existing `SharedInput`/late-latch
  path, so TAS/netplay/determinism are unchanged — this is purely a host input-collection change).

Reuse the deterministic wasm touch-overlay region model (`wasm_touch.rs`) for the hit-test math.

### B — The virtual NES controller (styled per `assets/RustyNES_Icon/rustynes.svg`)

Render an on-screen **NES-001 control pad** matching the controller drawn by the icon generator's
`nes_controller()` (`assets/RustyNES_Icon/make_icon.py`) — the same one in `rustynes.svg`:

- **Body:** light-grey rounded-rectangle shell (`#cfcdc6` face, `#a7a299` shadow side, `#6f6e69`
  edge key-line, corner radius ≈ 0.14·h), with a dominant **dark-charcoal face plate** (`#37373a`,
  `#1d1d1f` border, `#525256` inner bevel).
- **D-pad (left):** a recessed dark square well (`#27272a`) with the black **cross** (`#161618`
  base + a top-light gradient, `#26282d` centre hub + a `#4a4d53` highlight).
- **Select / Start (centre):** a light-grey pill housing (`#c6c4bf`, `#7d7c77` stroke) with two dark
  pills (`#2e3138` + a `#474b53` top catch-light) and **red** `SELECT` / `START` labels (`#e60012`).
- **A / B (right):** a light-grey housing with two dark ring-wells (`#26282c`) and **concave
  bright-red** buttons (`#9a1c1c` rim → a dished red face with a `#ff9a90` lower catch-light), **red**
  `B` / `A` labels below. (NES layout: **B left, A right**.)
- **Logo:** the red "racetrack" capsule outline (`#e60012`, **no wordmark** — matches the icon).
- Two medium-grey accent bars (`#9a9a98`) across the upper-centre of the plate.

**Implementation.** Add a **controller-only export** mode to `make_icon.py` (emit just
`nes_controller(...)` to a standalone SVG) and convert it to an **Android `VectorDrawable`** (the
faithful, low-effort path); render it behind the touch layer. Pressed buttons get a lit/inset
variant (brighten the red dish / darken the D-pad arm) driven by the live mask, so the art reacts to
the multi-touch state. (Alternative: draw it directly with a Compose `Canvas` from the same geometry
constants — keep both options open; the VectorDrawable is the default.)

### C — Responsive sizing + hit-region remap for BOTH Z Fold 7 displays  *(maintainer requirement)*

The controller art **and** its touch regions must **resize and remap together** for the active
display, so they can never desync:

- **Single source of truth.** Derive every button region from the **measured controller bounds**
  (`BoxWithConstraints` / `onGloballyPositioned`) as fractions of the rendered controller rect — the
  exact fractions `nes_controller()` uses (D-pad centre at `left+0.205w, cy+0.07h`; A/B at
  `left+0.805w ± 0.068w`; Select/Start at `cx ± 0.064w`; arm half-extents `0.245h × 0.088h`; button
  radius `0.118h`; etc.). Because the regions are computed from the *same* rect the art is drawn into,
  a resize moves both in lockstep.
- **Per-posture layouts, recomposed on fold/unfold.** The activity already absorbs the
  `screenSize`/`smallestScreenSize` config change without a restart, and the Compose layout is
  responsive; key off the active size class:
  - **Cover (folded, ~1080×2520 portrait):** a compact controller pinned low, sized for **one-handed
    / two-thumb** reach (button clusters toward the bottom corners), screen above.
  - **Inner (unfolded, ~1968×2184, near-square):** a **larger** controller with the D-pad and A/B
    clusters spread to the **outer edges** for a held-in-two-hands tablet grip, with more vertical
    room for the NES image. Optionally a side-by-side (controller flanking the screen) landscape
    variant.
  - A **size / opacity / horizontal-offset** setting (per posture) so players can tune reach; the
    hit regions follow the rendered size automatically.
- **Verification:** on-device, fold/unfold mid-game and confirm (a) the controller rescales without a
  restart, (b) a press in the *new* layout hits the right button (regions tracked the resize), and
  (c) multi-touch + diagonals work on both displays.

### D — Polish

Light **haptics** on press (`Vibrator`/`VibratorManager`, a short tick; user-toggleable);
**hardware-gamepad remapping** + the on-screen layout sharing the one `Buttons` mask; an optional
**turbo/autofire** toggle per face button (reuse the desktop turbo model). **Effort: M.
Determinism-safe** — all input converges on the one late-latched mask.

### E — The `PLAY_BUILD` flavor (interim sideload builds are full-featured)

Per the updated Play-launch timing, wire a **`play` product flavor / `BuildConfig.PLAY_BUILD`
flag** (default **false**) and gate all freemium logic on it: the 8-minute session timer, the three
persistence locks (save-states / save-on-background+resume / battery-SRAM), and the `LicenseManager`
Billing client only engage when `PLAY_BUILD == true`. The default sideload/GitHub/dev build →
`PLAY_BUILD=false` → **unlimited, full-featured**; the freemium code stays compiled but dormant. This
keeps every interim (v1.8.2–v1.8.5) sideload release fully playable while the Play AAB (flipped on at
v1.8.6) stays freemium. **Effort: S.**

`load_state` but still allows in-session play). The Android UI is a Compose login screen + a toast
unlock HUD + a `.rap` progress sidecar. **TLS cross-compiles cleanly with NO OpenSSL** —
`ureq` + `rustls` + `ring` (the RA HTTP path), which sidesteps the OpenSSL-on-NDK cross-compile pain.
**On-device login round-trip + unlock toasts + sidecar persistence are deferred** (no device this
cycle).

**Netplay — direct-IP / same-LAN only for v1.8.6; CGNAT/TURN DEFERRED.** The mature GGPO-style
rollback core (`rustynes-netplay`: `UdpTransport` in `connection.rs` + `RollbackSession<T>::advance`
in `session.rs`) has **no `!Send` barrier and cross-compiles to Android as-is**, so a host-and-join
*direct-IP / same-Wi-Fi-LAN* session is the tractable v1.8.6 scope. The Kotlin frame loop branches to
an `np_advance_frame` entry point — **netplay REPLACES `runFrame`**, because rollback re-runs frames
(speculative advance + rollback-and-resimulate); the bridge cannot just emit one frame per call in a
netplay session. Determinism + cross-play hold (a state saved on desktop loads on Android; a `.rnm`
replays identically), so **desktop ⇄ Android cross-play stays valid**.

**The CGNAT / cross-NAT TURN story is explicitly DEFERRED** (carryover — see the post-v1.8.6 roadmap).
The relay infrastructure **already exists in-repo** — `deploy/` is a turn-key Docker bundle (coturn
STUN/TURN `turnserver.conf` + a signaling-server image + a Caddy TLS proxy + `.env`-driven config),
and `rustynes-netplay` already has the `stun` (RFC 5389 binding-request) + `signaling` (N-peer mesh
offer/answer/ICE) + `mesh_net` machinery. What is **unrun** is the *ops* step: a **hosted, running
relay** with a domain + TLS + a live **cross-NAT** session over it. That is a **maintainer ops / cost
decision** (recurring TURN hosting + bandwidth + abuse surface), exactly as `docs/netplay-webrtc.md`
§5 records the browser path's relay as "deployment-ready, NOT verified — the maintainer's manual run."
So v1.8.6 ships LAN/direct-IP netplay and defers internet/CGNAT netplay to the relay being stood up.

After Lua + RA + LAN netplay land, Android is at near-desktop parity (the remaining gaps are the
deferred TURN traversal, the on-device sign-offs, and the egui debugger — which stays an optional
sideload power-user overlay, never a first-class mobile surface).

**This is the gate to the Google Play launch — but the launch is decoupled from the connectivity
work.** Per the maintainer's updated timing, Play publication happens **here**, when the Android build
is feature-complete — not before. The capstone steps (most prepared in v1.8.0/v1.8.1, exercised now
end-to-end; confirm the current `PLAY_BUILD` / freemium wiring in `android/app/build.gradle.kts` +
the `LicenseManager` Billing code first):

1. Flip the **`PLAY_BUILD`** flavor on for the Play AAB (freemium $2.99 Full Unlock + the 8-minute
   demo become active); sideload/GitHub releases stay `PLAY_BUILD=false` / full-featured.
2. Create the **`full_unlock`** managed product at **$2.99** in the Play Console; enroll in **Play
   App Signing** with the upload keystore; finalize IARC rating + data-safety (no data collected) +
   the store listing assets (from v1.8.3).
3. **Closed-track + license-tested** verification of the real purchase flow → promote to production.

**Effort: L** (the connectivity workstreams) **+ M** (Play launch, mostly non-engineering).

Determinism-neutral (movies drive the same input path). **Effort: M.**

**D — HD-Pack Builder (touch).** The **HD-Pack Builder** (ADR 0017) authoring flow on mobile, over the
**`rustynes-hdpack`** shared crate already compiled into the Android build (v1.8.5 extracted it and
taught it `Read + Seek` / raw-bytes loading for SAF). The new surface is the **touch authoring UI**
(capture the current tile set, edit/replace tiles, assign conditions, export the pack via SAF). Output-
only / off-by-default, so the emulation stays byte-identical. Bridge: builder control + export-bytes on
`rustynes-mobile`, forwarding to `rustynes-hdpack`. **Effort: L.**

**E — Audio depth.** Bring the desktop **audio-depth** workstream (ADR 0020) to Android: a **20-band
EQ**, **stereo panning / Schroeder-reverb / crossfeed**, **per-channel mixing** sliders, an **output-
device picker**, and a standalone **NSF / NSFe music player** (load a `.nsf` via SAF, a track list +
transport, no ROM needed — the desktop NSF player). All of it is **frontend DSP** over the emitted
samples (the core's synthesis is untouched), so it is **determinism-neutral** — exactly as on desktop.
Bridge: the audio-DSP config setters + the NSF-player surface on `rustynes-mobile`. **Effort: M.**

**F — Shader / filter breadth.** Extend the v1.8.4 wgpu `SurfaceView` renderer beyond the shipped
WGSL NTSC/CRT/Bisqwit set with the desktop **hqNx / xBRZ** scalers, the **CRT preset bank**, and a
**constrained `.slangp` / `.cgp` import** (the same constrained subset the desktop accepts). Reuses the
shared **`rustynes-gfx-shaders`** crate path (WGSL on Android via wgpu). Presentation-only →
determinism-safe. Bridge: shader-config selection (likely already partly present from v1.8.4).
**Effort: M.**

**G — Cheats + Game Genie.** A Compose **cheat manager**: **Game Genie** codes (the
`rustynes-core/src/genie.rs` runtime overlay, off by default, not in the save-state), **raw-RAM
(GameShark-style)** cheats (`Nes::poke_ram` applied after `run_frame`), and the **Game Genie code DB**,
persisted **per-ROM** (the desktop `[[raw]]` / Game-Genie TOML model, keyed by ROM SHA via the per-game
DB). Empty cheat list = byte-identical, so determinism holds. Bridge: apply-genie / poke-ram setters on
`rustynes-mobile` (genie already core; poke-ram is the documented caller-side overlay). **Effort: M.**

**H — Symbol / source-map debugging.** SAF loaders for **`.sym` / `.mlb` / `.nl`** symbol files and
**ca65/cc65 `.dbg`** source maps, feeding the debugger (A) so addresses show labels and the homebrew
dev sees source context — the desktop symbol/source-map loaders reused, fed raw bytes via SAF. Gated
with A (`debug-hooks`). Bridge: load-symbols-bytes on `rustynes-mobile`. **Effort: S.**

## The shared bridge is now connectivity-complete — iOS inherits Lua / RA / netplay

By the end of v1.8.6 the shared **`rustynes-mobile`** bridge exposes the **full** mobile surface —
core (ROM/input/run-frame/framebuffer/save-load-state/identity) **plus Lua, RetroAchievements, and
netplay** — over UniFFI. **v1.9.0 iOS (SwiftUI / wgpu→Metal) reuses all of it for free** via the
generated **Swift** bindings: it inherits Lua, RA, and netplay without re-deriving the FFI surface.
The hard `!Send` → `Send` work that those connectivity features required is **already done in shared
crates**: `rustynes-script`'s `ScriptEngine` is now `Send` (mlua `send` + `Arc`/atomic newtypes) and
`rustynes-ra`'s `RaClient` carries the audited `unsafe impl Send`. The std crates iOS consumes
alongside the core are **`rustynes-ra`** (RetroAchievements), **`rustynes-hdpack`** (HD-pack), and
**`rustynes-gfx-shaders`** (the WGSL NTSC/CRT/Bisqwit stack, WGSL → Metal via wgpu) — all
platform-agnostic, all shared. So the iOS plan's "Netplay / RA / Lua deferred" MVP cut can be revisited
upward cheaply: the Rust-side groundwork is no longer iOS-specific work, only the SwiftUI login /
overlay / netplay-UI chrome remains. The bridge having gone connectivity-complete on Android is the
single biggest de-risk for the iOS release.
