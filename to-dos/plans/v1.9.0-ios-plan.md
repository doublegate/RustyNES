# RustyNES — Plan: v1.9.0 "iOS / iPadOS App"

> **Maintainer decisions (locked 2026-06-17):** architecture = **Rust core + native
> SwiftUI UI + wgpu→Metal**, reusing the **shared `rustynes-mobile` bridge crate** with the
> v1.8.0 Android port (one core-binding layer for both mobile targets). MVP is **focused**:
> core + video/shaders + audio + touch/controller + save-states/rewind/TAS. **Netplay and
> RetroAchievements are DEFERRED** to a follow-up (mirrors the v1.8.0 Android MVP cut).

Full backing research: [`research/v1.9.0-research-ios.md`](research/v1.9.0-research-ios.md)
(Apple-specific) + the shared-bridge section of
[`research/v1.8.0-research-android.md`](research/v1.8.0-research-android.md).

## Context

RustyNES is a pure-Rust, cycle-accurate NES/Famicom emulator: a `#![no_std]`+alloc chip
stack (`rustynes-{cpu,ppu,apu,mappers,core}`) under a `std` desktop frontend
(`rustynes-frontend`, winit+wgpu+cpal+egui) plus a WebAssembly build with an on-screen
touch overlay. This plan ports it to a complete, shippable **iOS + iPadOS** application,
built on a mature **v1.6.0 "Studio" + v1.7.0** desktop and the **v1.8.0 Android** port
that first stands up the shared `rustynes-mobile` bridge. The pure-Rust core compiles to
Apple ARM and produces **byte-identical** output, so this release adds a platform shell —
it does not touch emulation.

**TL;DR feasibility (from the research):** *de-risked by precedent.* Apple's **April-2024
App Review Guideline 4.7** permits retro-console emulators worldwide, and **Delta** (an
NES-capable emulator) shipped and topped the App Store with user-provided ROMs. RustyNES
is a **pure interpreter** → it needs **no JIT**, the one capability iOS withholds — so the
single hardest iOS-emulator constraint simply does not apply.

## Hard contract (unchanged; applies to everything below)

- **The chip stack + the emitted-frame/sample contract are NEVER touched.** All iOS work
  lives in a `rustynes-ios` Swift/UniFFI layer over the shared `rustynes-mobile` bridge
  crate. The desktop / native / `no_std` / wasm builds stay **byte-identical**, and
  **AccuracyCoin 100% (139/139)** + the oracles stay green on the existing host CI.
- **Determinism holds on ARM** (same seed+ROM+input ⇒ bit-identical FB+audio), so
  desktop ⇄ Android ⇄ iOS netplay cross-play remains valid once netplay lands. The iOS app
  consumes the same `Nes` snapshot/output the oracle validates; the DRC resampler + run-ahead
  stay frontend-side (timing-only), never in core synthesis.
- **NEVER bundle commercial ROMs.** Ship zero copyrighted ROMs in the `.ipa`. ROM import is
  user-provided only — Files / `UIDocumentPicker` / share-sheet / iCloud Drive — copied into
  the app sandbox keyed by SHA-256 (the desktop save-identity scheme). Optionally bundle only
  CC0/public-domain homebrew demos.
- New surfaces are additive; nothing changes the deterministic core path.

> **STANDING directive — continual research + opportunistic improvement.** When implementing
> any workstream, re-mine the web + the Apple/Rust ecosystem (winit/wgpu→Metal, cpal/CoreAudio,
> UniFFI, GameController) and the in-repo `docs/` as needed; and make any beneficial
> enhancement or measured performance optimization encountered en route even if not listed
> here — provided it respects the hard contract above.

## Architecture

**Adopted — Option (b): a thin Rust core behind a native SwiftUI UI, with the NES
framebuffer rendered into a `CAMetalLayer`/`MTKView` via wgpu→Metal.** This is the
architecture every shipping Rust-core iOS app converges on (Mozilla/UniFFI, Ferrostar,
LiveKit). RustyNES's existing `rustynes-core` ⇄ `rustynes-frontend` split *is* the
"deterministic core / platform shell" boundary an FFI bridge wants, so the bridge is thin
and the core is reused byte-for-byte.

**Shared mobile foundation.** A common `rustynes-mobile` bridge crate (introduced by the
v1.8.0 Android plan) exposes the UniFFI surface over `rustynes-core` — `nes_new(rom_bytes)`,
`run_frame() -> framebuffer`, `drain_audio() -> samples`, `set_controller(port, buttons)`,
`set_pointer(...)`, `snapshot()/restore()`, `reset()/power_cycle()`, region/mapper/identity
queries — mirroring the `Nes` public API ~1:1. `rustynes-ios` is the Apple-side shim
(staticlib → XCFramework + the generated Swift package); `rustynes-android` is its Kotlin
counterpart. **One core-binding layer, two platform shims.**

**Rejected:**
- *(a) winit+wgpu+egui verbatim on iOS* — winit's mobile story is weak and an egui-only UI
  feels alien (no native navigation/share-sheet/document-picker/Dynamic-Type/Stage-Manager/
  VoiceOver); App Review and users penalize non-native chrome. Kept ONLY as a throwaway
  **M0 "does the core run on-device"** spike (fastest path to a pixel in the Simulator).
- *(c) hybrid egui-debugger-embedded-in-SwiftUI* — reasonable **later** (ship the egui
  debugger as an opt-in "Developer" tab once the native app is stable), not the v1.9.0 baseline.
- *No-shared-crate (bind the core independently per platform)* — duplicates the FFI layer
  across Android/iOS; rejected in favor of the shared `rustynes-mobile` bridge.

**JIT — confirmed NON-ISSUE.** iOS forbids third-party JIT / W^X. RustyNES needs none: the
CPU/PPU/APU core is a pure cycle-stepped **interpreter** (no dynamic codegen); `mlua` is
**interpreted** Lua 5.4 (not LuaJIT); the experimental `piccolo` backend is pure Rust;
wgpu→Metal shaders are compiled by Apple's toolchain at build/load (not app-generated code).
The headline iOS emulator restriction is irrelevant to an NES interpreter.

**App Store policy verdict — FEASIBLE & PRECEDENTED.** Guideline 4.7 (Apr 2024) permits
retro-console emulators; Delta is the proof. Compliance posture RustyNES already satisfies:
**no bundled copyrighted ROMs**, **no in-app download mechanism**, **no links to ROM sites**,
a clear **in-app ownership notice**, conservative **age rating**, and a clean **privacy
manifest** (no telemetry/accounts in the MVP — RA login, if ever shipped, gets disclosed).

## Workstreams

> Effort key: S ≈ days · M ≈ 1–2 wks · L ≈ 3–5 wks · XL ≈ 6+ wks (solo, part-time).

### A — Foundation: build + shared bridge + signing  (L)
Add `crates/rustynes-ios` (`staticlib`) over the shared `rustynes-mobile` UniFFI surface.
`rustup target add aarch64-apple-ios aarch64-apple-ios-sim` (+ `x86_64-apple-ios` only if
Intel CI/devs remain); UniFFI + `cargo swift`; `xcodebuild -create-xcframework` (lipo the
sim arches first); a checked-in SwiftUI Xcode/SwiftPM project consuming the XCFramework.
Apple Developer Program, App ID, distribution cert + provisioning profiles, **fastlane
match** (certs in a private git repo). GitHub Actions `macos-latest` + fastlane (gym/match/
pilot), **gated to tags + manual dispatch** (macOS minutes bill at 10×, mirror the PGO/release
gating). *Determinism-safe* (host shim only). *Risks:* UniFFI Rust-lib ↔ generated-Swift
version coupling (the iOS analog of the wasm-bindgen pin discipline); XCFramework arch/lipo
mistakes; macOS CI cost; provisioning churn.

### B — Metal rendering + surface  (M)
wgpu Metal surface from a `CAMetalLayer` (an `MTKView`'s layer) — *or* a ~30-line hand-written
Metal nearest-filter blit of the 256×240 RGBA texture (decide after the M0 spike; the hand
blit may be lighter for a single textured quad). Letterbox to the NES **8:7 PAR** reusing the
`gfx.rs` math. `CADisplayLink` pacing honoring **ProMotion 120 Hz** (emulate at the console
rate 60.0988 Hz, present 1:1, DRC absorbs the beat — reuse the existing display-sync logic).
Surface recreate on resize/scale/drawable-loss. *Determinism-safe* (output-only). *Risks:*
ProMotion vs 60.0988 Hz beat (mitigated by existing display-sync/DRC, needs on-device check);
drawable lifecycle on background; wgpu-into-external-`CAMetalLayer` ergonomics.

### C — CoreAudio  (M)
cpal CoreAudio host (iOS support incl. **AVAudioSession** via `objc2-avf-audio`); reuse the
lock-free SPSC ring + Hermite DRC **verbatim** (target-agnostic `resampler.rs`). AVAudioSession
category/activation; handle **interruptions** (calls/Siri), **route changes** (headphones),
the **silent switch**, and **background** policy; request a small `preferredIOBufferDuration`
for low latency. *Determinism note:* DRC is a frontend resampler stage — core samples
untouched, audio oracle + cross-play preserved. *Risks:* interruption edge-cases; buffer-size
negotiation vs underruns on older devices.

### D — Touch + controller input  (M)
SwiftUI translucent touch overlay reusing the **proven wasm overlay design** (D-pad / A / B /
Start / Select + Power Pad mat + target-port selector). **GameController** (`GCController`)
discovery in Swift (MFi + Xbox/PlayStation/Backbone auto-discovered; gilrs is unreliable on
iOS — push button state across the FFI). Hardware keyboard via `UIKeyCommand` /
`.keyboardShortcut`; trackpad/pointer drives Zapper/SNES-mouse/Arkanoid Vaus (the core already
models pointer devices). Optional **Core Haptics** rumble (off by default). *Determinism note:*
all inputs converge on the same `set_controller` late-latch bit-mask as desktop/wasm →
TAS/netplay-identical. *Risks:* controller hot-plug; gesture-vs-overlay conflicts; on-device
touch ergonomics (a manual, on-device verify carryover, Delta-style).

### E — Storage + lifecycle  (M)
ROM import via `UIDocumentPicker` / SwiftUI `.fileImporter` / share-sheet "Open in" / iCloud
Drive, with `.nes`/`.fds`/`.nsf`/`.zip` UTType registration; copy into the sandbox
(`Application Support/RustyNES/`), SHA-256-keyed. `.rns` save-states + battery SRAM in the
sandbox (the format is platform-independent → **cross-device save portability**). SwiftUI
`ScenePhase` pause/resume (invalidate `CADisplayLink`, stop audio, drop the drawable on
background; rebuild on foreground). `PrivacyInfo.xcprivacy` (no data collected in the MVP).
*Risks:* security-scoped URL handling for imported files; background-task limits. (iCloud save
sync = future.)

### F — Feature port (MVP-scoped)  (L)
Ports cleanly: save-states / rewind / run-ahead (S), cheats (Game Genie / raw RAM) (S), TAS
`.rnm` (S), shaders NTSC/CRT/palette (M — WGSL→Metal; ship plain blit + 1–2 shaders first,
measure on-device). **Deferred to v1.9.x:** Lua console (`mlua` arm64 static link, interpreted,
no JIT — gate behind a "Developer" toggle), native RetroAchievements (`rc_client` arm64 +
keychain token + login UI + privacy disclosure), and **netplay** (valid determinism/cross-play,
but mobile NAT/CGNAT + iOS background limits make the *transport* the blocker → local Wi-Fi /
GameKit / TURN later). *Risks:* shader perf on older devices (measure-first); Lua/RA scope creep
(hard "ship-later" flags).

### G — iPadOS UX  (M)
Designed in from day one: SwiftUI scenes resize, the Metal view letterboxes to 8:7 PAR in any
window (split-view / Stage Manager — pause non-foreground scenes); hardware keyboard (Magic
Keyboard) + trackpad pointer; larger-canvas layout. PiP / external-display = future note.
*Determinism-safe* (UI only).

### H — App Store release  (M)
TestFlight (App Store Connect API key in CI secrets) internal → external; privacy nutrition
labels + manifest; age rating questionnaire; **review-safe metadata** (no ROM references, no
download path, ownership notice); iPhone + iPad screenshots; submit + iterate. *Risks:* review
rejection over perceived ROM facilitation (managed — no bundled content/download path; Delta
precedent); first-app account/agreement latency.

## Risks (ranked)

1. **App Store review rejection** over perceived copyrighted-ROM facilitation — *managed/
   precedented* (no bundled ROMs, no download path, ownership notice; Delta). Highest-impact,
   now low-probability with discipline.
2. **macOS CI cost + signing/provisioning friction** (10× minutes, fastlane match, App Store
   Connect setup) — gate the iOS pipeline to tags + manual dispatch.
3. **Two UI codebases (SwiftUI iOS + egui desktop) divergence** — mitigated by keeping the FFI
   surface = the `Nes` API and never re-implementing core logic in Swift.
4. **Audio interruption / lifecycle bugs** (AVAudioSession, drawable loss, background) — needs
   on-device testing the headless CI can't do.
5. **ProMotion 120 Hz pacing beat** vs 60.0988 Hz — mostly mitigated by existing display-sync/
   DRC; on-device validation across refresh rates.
6. **UniFFI/XCFramework version coupling** — a CI gotcha mirroring the wasm-bindgen pin.
7. **On-device shader perf** on older devices — measure-first; ship the plain-blit baseline.

## Maintainer decisions

1. **Architecture (b) SwiftUI + UniFFI + Metal** — *locked* (see header). winit-reuse stays an
   M0 spike only.
2. **Shared `rustynes-mobile` bridge with v1.8.0 Android** — *locked* (one core-binding layer).
3. **MVP scope: ship without netplay + native RA + Lua console** — *locked* (defer to v1.9.x).
4. **UniFFI vs cbindgen** — recommended **UniFFI** (`cargo swift`); cbindgen only if the FFI
   surface stays tiny. *(Default: UniFFI.)*
5. **wgpu-into-`CAMetalLayer` vs a hand-written Metal blit** — decide after the **M0 spike**
   (hand blit may be simpler/lighter for one textured quad). *(Default: spike both, pick lighter.)*
6. **Apple Developer account ownership** (personal vs org), **bundle ID**, and whether RA/accounts
   ever ship (drives privacy labels) — maintainer-only (account + legal). *(Default: personal
   account, no accounts in MVP → clean privacy labels.)*
7. **CI budget** — accept macOS 10× minutes, gate the iOS pipeline to tags + manual dispatch
   (mirror PGO/release gating). *(Default: gated.)*
