## Strategy — five concurrent attack vectors

### Vector A — Multi-axis cutover-sweep extension

The existing `crates/nes-test-harness/src/bin/cutover_sweep.rs` already implements a battery-driven
knob-grid harness reading 10 `RUSTYNES_CUT_*` env vars via the `cutover` module in
`crates/nes-core/src/bus.rs:219-229`. Plus `crates/nes-cpu/src/r4_knobs.rs` has 6 more axes
(`GET_PARITY`, `CASCADE`, `HALT_PATH`, `OAM_WRITE_SEM`, `MID_LOOP_DMC_POLL`, `PPU_OFFSET`).

**Extend** with per-test gradient signals (not just `$0461/$0462/$0463` but ALL 29 failing result
bytes simultaneously — see `crates/nes-test-harness/src/accuracy_coin_catalog.rs` for the address
table). Add 3-4 NEW axes the prior sweeps didn't cover:
- **A1:** $2002-read PPU sample position (`master_clock - 1` vs `-2` vs `-0`) — re-test now that
  R-phase substrate is in place; PHASE_COMP sweep was on combo only (§115/§116 refutation may not
  apply to R-phase)
- **A2:** DMC abort countdown duration (1/2/3 CPU cycles) — the `§75` cluster
- **A3:** APU frame-counter IRQ assertion sub-cycle phase (φ1 vs φ2 of CPU cycle N)
- **A4:** SH* "had-DMA" detector (cycle-diff vs stall_count vs internal_data_bus value)

Use `scripts/r4_sweep_ac.sh` as the template — extend its 4-nested-loop to N axes; pipe per-knob
combo to a CSV of `(knob_tuple, per_test_status_27_bytes)` for analysis.

**Closure expectation:** identifying a single combo that flips 1-3 tests in CPU-Interrupts or
APU-Registers-and-DMA without regressing the sacred trio. ADR-0002 binds: STOP on any oracle break.

**Per-cluster reading priority:**

  `NstApu.cpp`

For each failing test, run our impl + ALL 5 references in a paired-instruction trace; identify
the smallest algorithm delta that's consensus-correct on the other 4 emulators.

### Vector C — Cross-branch port (`main` v1.7.0 + `debug/v2.0-master-clock-testbed`)

`main` (`ce312e6`, v1.7.0) reaches **126/139** AccuracyCoin pass; current `refactor/v2.0-master-clock`
reaches 110/144. **Some `main`-passing tests are R-phase regressions** — not real losses, just
substrate-relative behavior shifts. The legacy substrate's `cpu-c1-attempt-17-access-reorder` φ1/φ2
split landed several SH\* and CPU-Behavior-2 tests via specific dummy-read sub-cycle positions.

Approach: for each failing test on the current branch, check `main`'s status. If `main` passes
**without** the `cpu-c1-attempt-17-access-reorder` feature, port the behavior to R-phase. If
`main` passes only WITH the feature, the test's expectation is captured by the φ1/φ2 sub-cycle
sample point that R-phase replaced — investigate whether the equivalent R-phase mechanism is
addressable.

- §117 B4 PPU infra (already validated 60/60 oracle byte-identical) — see if BG serial-in / stale-
  shift residuals partially flip on testbed

  knobs DON'T close which tests)

`git diff main..refactor/v2.0-master-clock -- crates/nes-cpu/src/cpu.rs` will surface the SH\*

## Phased execution

### Phase E1 — Data harvest (1-2 days, READ-ONLY, no test changes)
1. Build per-test result-byte aggregator: extend `accuracycoin_catalog.rs` to emit a CSV row
   per battery run of `(timestamp, knob_tuple, $0461, $0462, ..., $0488)` for all 27 failing
   AccuracyCoin tests. Sweep harness reads this.
2. Run extended cutover sweep across A1+A2+A3+A4 axes (~200-500 grid points) — one battery per
   point, ~30s each = 2-4 hour total. Identify any knob combo flipping ≥1 test without breaking
   another or the gauntlet.
3. Capture all 5 reference emulator implementations of the failing-test algorithms (read-only
   audit). Extract algorithm-difference table per test cluster.
4. Run `git diff main..refactor/v2.0-master-clock` for cpu.rs / mmc3.rs / apu.rs / bus.rs;
   identify which file regions differ; note `main`'s behavior per failing test.

**Gate:** at end of E1, a written summary of (knob-combo wins, cross-emulator deltas,
branch deltas) — NO code changes yet.

### Phase E2 — New trace D1/D2/D3 builds (1-2 days)
Build the three new traces as feature-gated fixtures + Mesen2 Lua reference companions + Python
cross-diff. Validate each on a known-passing test first (e.g., D1 on cpu_dummy_writes_ppumem, D2
on a known-passing FC test, D3 on a known-passing PPU misc test) — confirm traces match Mesen
byte-for-byte before applying to the failing tests.

**Gate:** all three trace pipelines produce non-empty correctly-formatted CSV; cross-diff on
known-passing tests = 0 divergences.

### Phase E3 — Per-test targeted attempt (3-5 days, ADR-0002 STOP gate per attempt)
For each cluster in priority order (highest expected-flip-rate first):
1. APU Registers and DMA tests (10 tests) — likely R4-style port from Mesen2/ares with Vector D2
   trace steering. Expected flip: 3-7 tests.
2. SH\* family (5 tests) — Vector B cross-emulator audit + D1 trace. Expected flip: 2-4 tests.
3. CPU Behavior 2 (3 tests) — nestopia + fceux cpu.cpp algorithm porting under D1. Expected flip:
   1-2 tests.
4. IFlagLatency + NMI Overlap IRQ (2 tests) — A3 sweep + D2 trace. Expected flip: 0-2 tests.
5. mmc3 #3 — puNES `irqA12.c` algorithm + targeted A12 trace + Vector A sweep on A12-related
   knobs. Expected flip: 0-1 test.
6. PPU Misc (6 tests, bleeding edge) — D3 trace + sub-PPU-dot algorithm work. Expected flip: 0-2
   tests (most are 2-3 day each).
7. APU Tests (2 tests) — Mesen DMC trace cross-diff. Expected flip: 0-1 test.
8. INC $4014 (1 test) — downstream of APU Registers cluster (DMC bus interaction). Likely closes
   with #1 above.

**Per-attempt protocol:**
- Single-axis change at a time
- Run full gauntlet (60-ROM oracle + ppu_vbl_nmi + cpu_interrupts_v2 + nes-ppu + mmc3) after every
  attempt
- ADR-0002 binding: on ANY regression in the gauntlet, STOP, report with trace evidence, await
  user direction (revert / sub-flag / iterate)
- Commit at every green checkpoint (standing authorization for this branch)

**Expected total flip:** ~8-15 additional tests of the 29 (≈118-125/144 = 85-90%). Honest upper
bound — multiple tests share root causes so successful fixes may cascade; multiple tests are
bleeding-edge (`PPU Misc`) and won't flip without multi-day work.
