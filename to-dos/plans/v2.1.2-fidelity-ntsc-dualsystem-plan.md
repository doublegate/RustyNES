# v2.1.2 "Fathom" — NTSC-palette / composite-shader + Vs. `DualSystem` frontend plan

> **Status:** proposed (2026-07-10). Two Fathom-line fidelity carryovers taken to
> completion: the generated NTSC palette (F1.4) + the composite-shader ladder
> (F2.2), and the Vs. `DualSystem` second-screen frontend (F2.1). Version target
> **v2.1.2** is a proposal — these are additive, off-by-default, display/frontend
> changes on the byte-identical v2.0.0 "Timebase" core, so they slot as a patch
> in the accuracy/fidelity v2.1.x line (the mobile store launch owns v2.2.0). The
> maintainer confirms the final version at cut time.

## 0. Why this plan, and what already exists

The v2.1.0 "Fathom" plan
(`to-dos/plans/v2.1.0-fathom-accuracy-remediation-plan.md`) sequenced three
fidelity items that never landed in a shipped release:

- **F1.4** — a *generated* 512-color (64 × 8-emphasis) NTSC palette synthesized
  from the composite waveform, replacing/supplementing the hand-authored table.
- **F2.2** — an optional NTSC composite shader ladder (generated palette →
  separable `nes_ntsc` → Bisqwit per-dot true composite).
- **F2.1** — presenting the Vs. `DualSystem` second screen in the desktop
  frontend.

A code survey (2026-07-10) shows **most of the shader ladder already exists** and
**the dual-console core is complete but unwired in the frontend**. This plan is
therefore mostly *finish + verify + wire + document*, not greenfield.

### The two genuine gaps

- **Palette:** the base table is hand-authored; F1.4 wants a *deterministic
  generator* from the YIQ composite model (ares `fc/ppu/color.cpp`), fed into the
  existing `build_rgba_lut_from_base` → `set_custom_palette` path, default-off.
- **Dual present:** the frontend uses a bare single `Nes` on a single-texture
  present path; F2.1 needs a dual path (`Emu::Dual`) plus a second-framebuffer
  present, input routing, and sub-audio — without regressing the single path.

The composite shaders themselves (F2.2) are effectively built; F2.2 here is
*verify end-to-end, fill the separable middle rung if missing, expose/persist
cleanly in the UI, phase-correctness, and a deliberate visual re-bless*.

## 1. Goals / non-goals

**Goals**

1. A deterministic, `#![no_std]`-safe generated NTSC base palette, default-off,
   selectable alongside the hand table and any loaded `.pal`.
2. A coherent, UI-selectable, persisted NTSC/CRT shader ladder end-to-end, with
   the generated palette feeding its front, phase-locked dot-crawl verified.
3. Desktop presentation of the Vs. `DualSystem` second screen (side-by-side /
   stacked), P1–P4 input routing, coin/service, and sub-audio — with the
   single-console path **byte-identical / behavior-unchanged**.

**Non-goals (explicit, documented)**

- No change to the deterministic core golden vectors for the **single-console**
  path. The generated palette is the one framebuffer-affecting item and ships
  **default-off behind a flag** with a deliberate re-bless (v2.0.3 precedent).
- Composite/CRT shaders stay **display-only** — never fed back into
  framebuffer/audio golden vectors or save-state hashing (determinism boundary,
  `accuracy-ledger.md` Notes).
- Dual mode's advanced frontend features (run-ahead, rewind, netplay rollback,
  TAS piano-roll, save-state round-trip) are **scoped to main-console-only or
  disabled in dual mode for this release**, documented as a known limitation with
  follow-up tickets. Full dual parity is a later effort.
- PlayChoice-10 Z80 menu-screen stays out of scope.
- Real Vs.-cabinet end-to-end boot verification remains **fixture-limited**: the
  staged GVS dumps are the MAME `maincpu` half only (the 5 `#[ignore]`'d
  `vs_dualsystem` boots, `accuracy-ledger.md`). F2.1 is verified through the
  synth harness (`crates/rustynes-test-harness/tests/vs_dualsystem_synth.rs`) and
  any complete 64 KiB dual dump a maintainer supplies.

## 2. Feature A — Generated NTSC palette (F1.4) + composite-shader ladder (F2.2)

### A1 — Generated NTSC base-palette module (in-core, deterministic)

- **New:** `crates/rustynes-ppu/src/palette_gen.rs` (or extend `palette.rs`):
  `pub fn generate_base_palette(p: &NtscPaletteParams) -> [[u8; 3]; 64]`.
  Model the ares `fc/ppu/color.cpp` YIQ integration: per color, integrate the
  composite waveform over the subcarrier (12 phases), apply
  saturation/hue/gamma/brightness/contrast, and handle the `$xD`/`$xE`/`$xF`
  black/edge cases (and the darker `$0D`). Pure function of documented params;
  `const`-friendly; no `std`, no float non-determinism that varies by target
  (use a fixed integration; if `f32` is used, round deterministically).
- `NtscPaletteParams { saturation, hue, gamma, brightness, contrast }` with
  documented defaults matching the current hand table's intent.
- Feed the generated base straight into the existing
  `build_rgba_lut_from_base(&base)` → `set_custom_palette(Some(base))`. **No new
  emphasis path** — reuse the LUT builder.
- **Tests:** determinism (same params ⇒ byte-identical output on repeated calls);
  a spot-check of a few known entries (grays `$00/$10/$20/$30`, the `$0D` black)
  against ares/Mesen2 reference RGB within a documented tolerance; a golden
  512-entry snapshot committed under `tests/golden/`.

### A2 — Palette selection, config, UI, re-bless

- Config: add a palette-source selector to `config.rs` graphics section —
  `palette_source: enum { BuiltinHand, GeneratedNtsc, File(PathBuf) }` (or extend
  the existing `palette_file`/named-palette bank). **Default = `BuiltinHand`** so
  default builds stay byte-identical.
- UI: extend `debugger/settings_panel.rs:743-796` (the palette selector +
  "Load .pal…") with a "Generated NTSC" option and the `NtscPaletteParams` knobs
  (sliders), live-applied via `nes.set_custom_palette(...)`.
- **Visual re-bless (deliberate):** when `GeneratedNtsc` is selected it changes
  framebuffer output. Re-bless *only* the affected visual-corpus snapshots under
  the generated palette in a dedicated commit, exactly as v2.0.3 did — the
  default-build (`BuiltinHand`) snapshots MUST NOT move. Confirm
  `cargo test --workspace --features test-roms` visual suite is unchanged on
  default and intentionally-updated only under the flag.

### A3 — Composite-shader ladder completion + verification (F2.2)

The ladder already exists (`Ntsc` simplified blur → `Lmp88959` composite →
`CompositeRt` Bisqwit per-dot). Work here is **finish + verify**, not rebuild:

- Audit each `BuiltinPass` end-to-end: selectable in `settings_panel`, `id`
  round-trips through config persistence, `params()` knobs surface in the UI, and
  the `R16Uint` index path (`wants_index`, gfx.rs:1036) is correctly toggled for
  `CompositeRt`.
- Confirm the generated palette (A1) feeds the *front* of the ladder: RGBA
  passes (`Ntsc`/`Lmp88959`/`Crt`) consume the palette-mapped RGBA; the Bisqwit
  pass consumes the raw palette **index** framebuffer (so it must be independent
  of the RGBA palette choice — verify and document that interaction).
- Phase-correctness: verify `ntsc_phase()`/`video_phase` produces correct
  dot-crawl on `Lmp88959`/`CompositeRt` (a known artifact test ROM / 240pee).
- If the "separable `nes_ntsc`" middle rung is genuinely absent as a distinct
  option (the current `Ntsc` is a simplified blur, not the separable-kernel
  `nes_ntsc`), decide: either (a) document the current three rungs as the ladder
  and drop the separable rung as redundant with LMP88959, or (b) add a
  separable-kernel WGSL pass. **Recommend (a)** unless a visual gap is found —
  avoid adding a fourth near-duplicate composite.
- **Determinism guard:** add/confirm a test that the shader path never mutates
  the framebuffer/audio golden hash (display-only assertion).

### A4 — Docs (A)

- `docs/frontend.md` — palette-source selector + shader ladder section.
- `docs/accuracy-ledger.md` — move "NTSC composite / analog palette" and
  "generated palette" rows from **Deferred (F1.4/F2.2)** to **Remediated /
  Shipped**; keep the determinism-boundary note.
- `docs/ppu-2c02.md` — the generated-palette model + params.
- `CHANGELOG.md` `[Unreleased]` — "Added" entries (default-off palette + shaders).

## 3. Feature B — Vs. `DualSystem` second-screen frontend (F2.1)

The core is complete; the frontend must adopt it without regressing the single
path. Staged to keep blast radius bounded.

### B1 — Dual detection + `Emu` adoption at the load boundary

- At ROM load (`emu.rs` load path, ~1131-1199), branch on dual detection:
  `Emu::from_rom(bytes)` already returns `Dual` for NES2.0 type 5/6 or a `vs_db`
  match, else `Single`. Introduce an `EmuCore` representation that can hold either
  — **minimally invasive option:** keep `pub nes: Option<Nes>` as the primary
  (main) console (so all existing run-ahead/rewind/snapshot/debugger/netplay code
  keeps compiling and working on the main console unchanged), and **add**
  `pub dual: Option<Box<VsDualSystem>>` alongside. When `dual` is `Some`, the
  per-frame step drives the `VsDualSystem` (which owns both halves) and the
  `nes` handle mirrors/aliases its `main()` for the existing machinery.
  - Rationale (Central-State Ownership): avoid threading `Emu` through the entire
    frontend in one release; the additive `dual` field confines the new state and
    leaves the single-console owner path byte-identical.
  - Document the aliasing contract precisely (who steps whom; that the debugger /
    save-state operate on `main()` in dual mode this release).

### B2 — Second-framebuffer present path

- `Gfx` today has one `nes_texture` + `surface` + pipeline (gfx.rs:288-311). Add
  a **second RGBA source** for `sub_framebuffer()` and composite both into the
  present: default **side-by-side** (512 × 240 logical), with a stacked
  (256 × 480) option in config. Simplest robust approach: upload both into a
  single 512 × 240 (or 256 × 480) staging buffer the existing single-texture
  pipeline presents — **no second surface/window required**, so the shader ladder
  (A) applies to the combined image uniformly.
- Aspect/letterbox: extend the existing letterbox math for the 2:1 (or 1:2)
  combined aspect. Confirm CRT/NTSC passes tile correctly across the seam (or
  render per-half then compose — pick the one that looks correct; document).
- Config: `dual_screen_layout: enum { SideBySide, Stacked }` (default
  `SideBySide`).

### B3 — Input routing + coin/service

- The core `VsDualSystem::set_buttons` already maps P1/P2→main, P3/P4→sub and has
  coin/service routing (vs_dualsystem.rs:235/245/260). Wire the frontend's P1–P4
  input (existing multi-pad support) + a coin/service binding through it in dual
  mode. Single mode unchanged.

### B4 — Sub-audio drain

- `VsDualSystem::sub()` exposes the second console's audio. For this release,
  **mix main+sub to the single output** (or a config toggle main-only / mixed).
  Keep it simple; document. No change to single-console audio.

### B5 — Scope-bounding advanced features in dual mode (documented limits)

- Run-ahead, rewind, netplay rollback, and TAS piano-roll **assume a single
  `Nes` snapshot**. For v2.1.2, in dual mode: **disable run-ahead/netplay/TAS**
  (grey out with a tooltip) and either disable save-state or scope it to the main
  console. Debugger operates on `main()`. Each is a one-line guard + a UI
  disable, plus a `docs/frontend.md` "Vs. DualSystem limitations" subsection and
  follow-up tickets (`T-PS-dual-runahead`, `T-PS-dual-savestate`,
  `T-PS-dual-netplay`).
- **Why:** full dual parity for these is a large, separate effort; shipping
  presentation+input+audio now is high-value and low-risk, and honest scoping
  beats a half-working rollback.

### B6 — Verification path (fixture reality)

- End-to-end real-cabinet boot is fixture-blocked (maincpu-half dumps; the 5
  `#[ignore]`'d `vs_dualsystem` boots). Verify F2.1 via:
  - the synth harness `vs_dualsystem_synth.rs` (drives both halves) — extend it
    to assert both framebuffers are produced and the frontend compose path yields
    a 512 × 240 (or 256 × 480) image;
  - a manual smoke test with any complete 64 KiB dual dump a maintainer stages
    under `tests/roms/external/`;
  - a frontend unit test that, given two synthetic framebuffers, the compose
    function produces the correct combined dimensions + pixel placement.
- Do **not** un-`#[ignore]` the fixture-blocked boots — they remain
  fixture-limited (`accuracy-ledger.md` unchanged for those rows).

### B7 — Docs (B) + ADR

- **ADR 0032** (Michael Nygard) — "Vs. `DualSystem` desktop presentation: additive
  `dual` field + combined-framebuffer present + scoped-down advanced features."
  Context (core complete, frontend single-`Nes`), Decision (additive `dual`
  alongside `nes`, main-console aliasing, disabled advanced features), Consequences
  (present/input/audio now; run-ahead/netplay/TAS/dual-save-state deferred with
  tickets). Cross-link ADR 0002 / the DualSystem core.
- `docs/compatibility.md:132-133` — DualSystem now *presented* on desktop (was
  core-only); note the scoped limitations.
- `docs/frontend.md` — dual-screen layout + limitations subsection.
- `docs/accuracy-ledger.md` — flip the "Vs. `DualSystem` second screen" row from
  **Deferred (F2.1)** to **Shipped (frontend present; advanced features scoped)**.
- `docs/STATUS.md` — feature matrix + the unchanged `#[ignore]` fixture rows.
- `CHANGELOG.md` `[Unreleased]` — "Added" entry.

## 4. Testing, determinism & gates

- **Single-console byte-identity is the hard gate.** Before/after, the default
  build (hand palette, no shader, single console) must produce byte-identical
  framebuffer + audio golden vectors and identical `external_real_games` 60/60.
  A dedicated CI assertion + local check.
- Generated-palette snapshot re-bless is an **isolated, reviewed commit** touching
  only the flagged-on visual corpus (v2.0.3 pattern).
- New unit tests: palette-gen determinism + reference spot-checks; dual compose
  dimensions/placement; shader display-only (no golden mutation).
- All standard gates: `cargo fmt --all --check`, `cargo clippy --workspace
  --all-targets -- -D warnings` **plus every feature combo** (scripting, hd-pack,
  retroachievements — never `--all-features`), wasm32 clippy (both default and
  `wasm-canvas`), `RUSTDOCFLAGS="-D warnings" cargo doc`, `no_std`
  cross-compile (`thumbv7em-none-eabihf`), `pre-commit run markdownlint`.
- AccuracyCoin stays **141/141**, nestest 0-diff (neither feature touches the
  deterministic single-console core).

## 5. Sequencing, effort, risk

| Step | Effort | Risk | Golden-vector impact |
|---|---|---|---|
| A1 palette generator (in-core) | M | low | none (until selected) |
| A2 palette select + UI + **re-bless** | S–M | low | flagged-on corpus only |
| A3 shader ladder verify/finish | M | low | none (display-only) |
| A4 docs (A) | S | — | — |
| B1 dual detection + `dual` field | M | **med** (touches load/step path) | none (single unchanged) |
| B2 combined present path | M | med | none |
| B3 input routing | S | low | none |
| B4 sub-audio | S | low | none |
| B5 scope-guard advanced features | S | low | none |
| B6 verification (synth + unit) | M | low | — |
| B7 ADR 0032 + docs (B) | S | — | — |

**Order:** A first (self-contained, lower risk, immediate visual payoff), then B
(B1 → B2 → B3/B4 → B5 → B6 → B7). A and B are independent; A can ship even if B
slips. Land A behind its flag, re-bless, verify default byte-identity; then build
B additively so the single-console path never regresses.

**Highest risk:** B1's load/step-path change. Mitigation: the additive `dual`
field (single path compiles/behaves unchanged), a byte-identity gate on the
single path, and scoping the advanced features out of dual mode rather than
half-implementing them.

## 6. Definition of done

  reference-spot-checked; default build byte-identical.
- Shader ladder fully selectable/persisted/phase-correct; display-only guard
  green.
- Vs. `DualSystem` desktop present (side-by-side + stacked), P1–P4 + coin/service
  input, sub-audio; single-console path byte-identical; advanced features cleanly
  disabled-in-dual with follow-up tickets.
- ADR 0032 + all docs (`frontend`, `compatibility`, `accuracy-ledger`, `STATUS`,

- All gates green; AccuracyCoin 141/141; nestest 0-diff.
