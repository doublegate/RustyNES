# v2.1.0 "Fathom" — Accuracy-Gap Remediation Plan

> **Status:** proposal / planning doc (2026-07-09). This line ships **as
> `v2.1.0` and up, ahead of** the **v2.2.0** joint iOS + Android store launch
> (the mobile ports move to v2.2.0; the accuracy work lands first so the mobile
> apps launch on the improved core). Version numbers below (`v2.1.0` …
> `v2.1.x`) are proposed anchors, not blessed releases.
> Codename **"Fathom"** (nautical, continuing the "Harbor" theme — a fathom is
> a depth measurement; this line is about measured depth of accuracy).
>
> This plan closes or formally dispositions every item in the 2026-07-09
> accuracy-gap audit. It is grounded in four research sweeps recorded in this
> planning session: the ADR-0002 lessons-learned corpus, a hardware/reference
> sweep (NESdev + Mesen2 + ares under `ref-proj/`), a subsystem-scoping sweep,
> and a BestEffort-mapper-tier sweep. Read this **with** ADR 0002 and
> `docs/mappers.md` open.

---

## 1. Executive framing — what is actually remediable

The raw gap-list conflates five very different categories. The single most
important planning insight is that they are **not** one backlog:

| Category | Items | Disposition |
|---|---|---|
| **Genuine safe wins** (low-risk, well-tested, display-only or self-contained) | PPU palette backdrop-override, OAMADDR 257-320, open-bus refresh-map + per-bit decay, generated NTSC palette | **Do — high value, low risk (Phase F1)** |
| **Doc reconciliation** (already implemented, docs stale) | MMC5 split-screen + audio, Vs. `DualSystem` core | **Do — near-zero effort (Phase F0)** |
| **Frontend fidelity** (no core / determinism risk) | DualSystem second-screen presentation, optional NTSC composite shader, optional OAM decay | **Do — frontend-only (Phase F2)** |
| **Real engineering** (format / medium models) | NSF non-60 Hz + NSFe, FDS medium model | **Do — scoped, larger (Phase F4)** |
| **Honesty raise** (BestEffort → Curated) | 112 BestEffort mapper families | **Do — incremental, evidence-gated (Phase F3)** |
| **The deferred residual** (ADR 0002) | MMC3 R1/R2 scanline-IRQ timing | **Instrumentation-first; likely formal close (Phase F5)** |
| **By-design non-goals** (no oracle / out of scope) | APU mixer "bit-exactness", PC10 Z80, the 16 non-accuracy `#[ignore]`s | **Document, do not chase (§7)** |

**Two gaps are already closed** — MMC5 vertical split-screen (`$5200-$5202`) and
MMC5 audio (`$5000-$5015`) are implemented in `crates/rustynes-mappers/src/mmc5.rs`;
the Vs. `DualSystem` core is implemented in `crates/rustynes-core/src/vs_dualsystem.rs`
with `sub_framebuffer()` already exposed. `docs/mappers.md:115` and
`docs/compatibility.md:132-133` are stale and describe them as deferred.

**One gap is a non-goal** — the APU mixer "±4%" is enforced *inside* Blargg's
`apu_mixer` ROM, which is an analog-cancellation test with tolerance, not a
0-tolerance digital oracle. A correct non-linear lookup-table mixer (which we
have) passes it; there is **no stricter oracle to tighten against**. The ±4%
framing in `docs/apu-2a03.md:225` is *honest and should stay*. We will not
present this as a remediable gap.

## 2. Sacred gates (bind every phase)

No phase may regress any of these. They are the promote/rollback contract
inherited from the v2.0.0 "Timebase" work and ADR 0002:

- **AccuracyCoin 141/141 (100.00%, RAM-authoritative)** — never drops.
- **nestest 0-diff**, `cpu_interrupts_v2` 5/5 strict, all `ppu_vbl_nmi/*`,
  `ppu_sprites` 19/19, mmc3 A12-IRQ 18/18, the passing `mmc3` strict tests.
- **60-ROM commercial byte-identity oracle 60/60** — any re-bless is documented
  with the exact tile/scanline delta and a TriCNES-faithfulness justification
  (the v2.0.3 precedent).
- **Determinism contract** — same seed + ROM + input ⇒ bit-identical framebuffer
  + audio + save-state. Nothing display-only (NTSC filter, OAM decay) may feed
  the framebuffer/audio golden-vector or save-state hash. Rate/jitter stays in
  the frontend.
- **`#![no_std]` chip stack** — `rustynes-core` still cross-compiles to
  `thumbv7em-none-eabihf --no-default-features`.
- **Host gate green** — `cargo fmt --check`, `clippy --workspace --all-targets
  -D warnings` (every feature combo incl. `retroachievements`), both wasm32
  clippy legs, `RUSTDOCFLAGS="-D warnings" cargo doc`, markdownlint
  (`pre-commit run markdownlint --all-files`, pinned v0.39.0).
- **Feature-flag discipline** — anything risky lands additive + default-off, and
  is promoted to default only after the oracle proves it (the ADR 0030 /
  2-cycle-ALE precedent). Both paths tested.

**Stop conditions** (ADR-0002 discipline, generalized): any pre-existing strict
test regressing → rollback; a partial flip (some but not all targets of a
change) is designed-out, not shipped; reaching a prior attempt's diagnosis →
stop and record.

## 4. Phase F1 — PPU display-accuracy wins (low-risk, high-value)

These four are the heart of the value. All are either display-only or governed
by a well-understood Blargg refresh-map, all benefit directly from the one-clock
every-cycle-bus-access scheduler, and all follow ares's explicit-FSM modeling
over Mesen2's timestamp-delta approach where a silicon-faithful boundary matters.

### F1.1 — Palette backdrop-override ($3F00-$3FFF) — **do first**

Not "corruption" — a *display* artifact. When rendering is disabled and loopy
`v` points into the palette range, the palette's shared address line is driven
by `v`, so the pixel output is `palette_ram[v & 0x1F]` (with the
`$10/$14/$18/$1C → $00/…` universal-backdrop mirroring) instead of the backdrop.
No bytes mutate.

- **Touch-point:** the pixel-output stage in `crates/rustynes-ppu/src/ppu.rs`.
  When rendering is disabled, branch on `(v & 0x3F00) == 0x3F00` → output the
  mirrored palette entry. ~3-line change (per Mesen2 `NesPpu.cpp` render path
  + ares output stage).
- **Oracle:** no dedicated Blargg code; validate against Mesen2/ares golden
  framebuffers (Micro Machines split-screen, `full_palette` demo) added to the
  `tests/golden/` corpus. **Must not** mutate palette RAM (read-through only).
- **Effort:** S. **Risk:** low. Display-only ⇒ zero determinism/save-state risk.

### F1.2 — OAMADDR forcing (dots 257-320) + `$2004` attribute mask + `& 0xF8` copy

Three related, well-tested 2C02 behaviors:

- **Force `oam_addr = 0` across dots 257-320** of rendering scanlines (the
  window logic already exists for the data-bus model at `ppu.rs:1740/1750`;
  it just never zeroes `oam_addr`). Trivial in a per-dot scheduler.
- **`$E3` mask on `$2004` reads of attribute bytes** (`addr & 3 == 2`) — bits
  4-2 don't exist on the OAM attribute latch.
- **`OAMADDR & 0xF8` 8-byte OAM copy** at render-start when OAMADDR ≥ 8 (the
  sprite-eval hardware bug).
- **Oracle:** Blargg `sprite_ram` (8 codes incl. the `$E3` mask code 5),
  `oam_read`, `oam_stress`. Wire these into the test-harness if not already.
- **Effort:** S-M. **Risk:** low. The mid-render `$2004`-exposes-live-eval case
  is harder (needs the real sprite-eval FSM) but few titles depend on it — scope
  it as a stretch sub-ticket, gated on a discriminating ROM.

### F1.3 — Open-bus refresh-map exactness + per-bit decay

The current model (`ppu.rs:484-1562`) is per-*group* (masks `0x1F/0x20/0xC0`)
with a single ~600 ms reload (`OPEN_BUS_DECAY_RELOAD = 1_000_000`). Blargg's
`ppu_open_bus` does **not** stopwatch the decay interval — it verifies the
per-register **refresh map** (which bits each register drives vs passes through)
and that decay eventually reaches 0.

- **Get the refresh map exact** (the load-bearing part): `$2002` = `---D DDDD`
  (top 3 driven+refreshed, low 5 decay); `$2004` = all 8 driven; `$2007` =
  `DDDD DDDD` non-palette / `DD-- ----` palette (bits 5-0 driven on palette
  reads). Audit the existing `refresh_open_bus`/`touch_open_bus` calls against
  this table.
- **Go per-bit** (Mesen2 `SetOpenBus` scheme): `decay_stamp: [u64; 8]` in frames
  (or scanlines), each bit decays independently past a coarse threshold
  (~3-30 frames all pass). Deterministic, allocation-free.
- **Oracle:** `ppu_open_bus.nes` + a unit test asserting each bit clears within
  its window. **Do not** build real-time microsecond per-bit-group timers — no
  test rewards it and it threatens determinism.
- **Effort:** S-M. **Risk:** low.

### F1.4 — Generated 512-color NTSC palette in-core

Replace / supplement the current palette with a **generated** 512-entry palette
(64 colors × 8 emphasis) synthesized once at init from the composite waveform
(ares `fc/ppu/color.cpp` YIQ integration: 12 subcarrier phases per color,
saturation/hue/gamma, `$xD/$xE/$xF` edge cases). Deterministic, one-time cost,
`const`-friendly, `#![no_std]`-safe.

- **Touch-point:** palette-generation module (new or extending the existing
  palette path). Keep it a pure function of documented parameters.
- **Oracle:** `full_palette` / `240pee` visual corpus + Mesen2/ares screenshot
  baselines. This is display fidelity; **no** automated pass-count moves.
- **Effort:** M. **Risk:** low. Note: a generated palette *changes framebuffer
  output*, so this is the one F1 item that touches the golden-vector hash — land
  it behind a flag first, re-bless the visual corpus deliberately (v2.0.3
  precedent), and keep the current hand palette selectable.

**Phase F1 effort:** M total. **Value:** the largest genuine-accuracy gain in the
plan, at the lowest risk.

## 5. Phase F2 — Frontend fidelity (no core/determinism risk)

- **F2.1 — DualSystem second-screen presentation.** The core already produces
  `sub_framebuffer()` (`vs_dualsystem.rs:229`) and `sub()` audio; no frontend
  surfaces it (grep of `rustynes-frontend`/`rustynes-libretro` for
  `sub_framebuffer` = nothing). Add a frontend dual-output mode (side-by-side /
  stacked window) consuming `sub_framebuffer()`, plus an optional sub-audio
  drain. **Frontend-only, no core change.** Effort: M. Risk: low.
- **F2.2 — Optional NTSC composite shader.** Layer the accuracy ladder as
  Mesen2/ares do: generated palette (F1.4, in-core default) → an `nes_ntsc`-style
  separable-kernel WGSL stage → optionally a Bisqwit per-dot composite decode
  (worker/shader). **All display-only, in the frontend/shader**, never fed back
  into golden vectors or save-state hashing. Effort: M (separable) / L (per-dot).
  Risk: low, provided it stays outside the deterministic boundary.
- **F2.3 — Optional OAM decay (off-by-default).** Mesen2-style ~3000-CPU-cycle
  8-byte-row decay, **opt-in, default-off, PAL-disabled** (constant refresh
  masks it). Kept out of the deterministic default path to protect
  byte-identical save-state/TAS. Effort: S. Risk: low (gated off).

## 6. Phase F3 — Mapper honesty raise (BestEffort → Curated)

**Frame:** the tier is an *honesty marker*, not behavior — runtime, determinism,
and save-state round-trip are identical across tiers. The 112 BestEffort
families are register-decode + save-state-round-trip unit-tested; promotion to
Curated means **landing real accuracy evidence** (fixture/spec + committed oracle
snapshot). The honesty gate (`mapper_tier_honesty.rs`) enforces that no oracle
dir references a non-accuracy-gated mapper, so **promotion and evidence land
together** — you cannot wire an oracle for `mapper-NNN` until NNN is Curated.

**Composition (from the tier sweep):** 112 = **86 no-IRQ** (easy) + **26
IRQ-bearing** (hard) + **0 expansion-audio** (none exist in BestEffort).

**Mechanics per promotion:** move the id BestEffort→Curated in `tier.rs`
(`match` arm + `CURATED_IDS` + `BEST_EFFORT_IDS`, in lockstep or the disjoint-set
tests fail) **simultaneously with** landing the fixture/spec + snapshot. Promote
to **Curated, not Core** (Core is pinned at exactly 51).

**Fixture source (commercial dumps).** The maintainer's local No-Intro set
(~473 titles) lives at `~/Dropbox/ROMs/Nintendo Entertainment System - Famicom
(2020)/` (zipped `.nes`; the loader reads `.zip` directly). Use it to stage
per-mapper fixtures into the **gitignored** `tests/roms/external/`, run the
oracle with `--features commercial-roms,test-roms --test external_real_games`
(recapture via `INSTA_UPDATE=auto RUSTYNES_DUMP_FRAMES=1 … cargo insta accept`).
**Never commit a commercial ROM** — commit only the `.snap`/screenshot vectors;
the `mapper_tier_honesty.rs` commercial gate permits a staged dump only for a
*classified* mapper. Prefer a CC0/homebrew ROM (F3.2) whenever one exists so the
gate runs in CI without the local set.

**Sequencing (cheapest honest gain first):**

- **F3.1 — Synthetic boot-smoke sweep, all 112.** Extend the existing
  `v21_coverage_mappers.rs` `synth_rom(...)` pattern (needs no ROM) to run
  ~60 frames of every BestEffort family, asserting no panic. Catches real bugs,
  costs nothing, unlocks nothing (stays BestEffort) — pure safety net. Effort: S.
- **F3.2 — CC0/homebrew no-IRQ promotions.** Promote the families with freely
  distributable test ROMs so the honest Curated count rises without commercial
  dumps: **218** (Magic Floor, tepples), **30** (UNROM-512), **28** (Action 53),
  **111** (GTROM/Cheapocabra), **31** (2A03 Puritans), **29** (Sealie CUFROM).
  Commit the ROM (CC0) + a byte-identity `.snap`. Effort: M.
- **F3.3 — Shared MMC3-clone A12 timing oracle.** Build one A12/IRQ timing
  oracle on the `Mmc3CloneMapper` core (`sprint11.rs:332`) that promotes the
  clone cluster together (44/49/52/115/134/189/205/238/245/348/366/268/513/176).
  This is where the F5 MMC3 timing work (if it lands) pays a dividend. Effort: L.
- **F3.4 — Licensed no-IRQ Famicom games** (staged commercial dumps, gitignored;
  commit snapshots only): **76** (Digital Devil Story), **95** (Dragon Buster),
  **97** (Kaiketsu Yanchamaru), **77** (Napoleon Senki), **72/92** (Jaleco).
  Effort: M.
- **F3.5 — Hardest last:** JY ASIC (**35/90/209/211** — selectable IRQ source)
  and **253** (Waixing VRC4-clone, scaled CPU-cycle). Effort: L.

**Value:** raises the *honest* accuracy-gated count from 60 toward ~80+ with
real evidence, prioritizing IRQ-bearing boards (the real-bug surface) and
popular titles. Spread across several v2.1.x point releases.

## 7. Phase F4 — Expansion-format engineering (larger, scoped)

- **F4.1 — NSF non-60 Hz.** The header play-speed word is currently unparsed;
  playback is hardwired to the 60 Hz vblank NMI (`nsf.rs`). Parse the µs/tick
  speed field and decouple `play` from vblank (cycle-counted software timer or
  APU-frame-IRQ driver so `play` fires at the declared period). Effort: M.
- **F4.2 — NSFe.** Add an `NSFe` chunk parser (`INFO/DATA/bank/time/auth`)
  feeding the same player. Effort: M.
- **F4.3 — FDS medium model.** The hard one. Model the disk as a byte stream
  with real gap regions + per-block CRC-16 synthesized on write, and the head as
  a continuous linear position with velocity/seek-time (replacing the fixed
  `HEAD_RESEEK_CYCLES = 8_000`). Effort: L. **Constraint:** write-CRC
  verification needs a real FDS BIOS (copyright-gated, out of CI); land it behind
  the existing `RUSTYNES_FDS_BIOS` env path + keep the simplified model as the
  default. Likely a stretch item — revisit whether the fidelity justifies the
  medium-model rewrite.

## 8. Phase F5 — The MMC3 R1/R2 residual (ADR 0002 — instrumentation-first)

**This phase is different. Read ADR 0002 and the two 2026-07-02 audit docs
before touching anything.** The residual is a **≤1-CPU-cycle differential**
between the mapper-IRQ observation path and the `$2002` VBL-sync path, on four
`#[ignore]`'d MMC3 sub-tests (R1: `mmc3_test_2/4` #3 + `mmc3_test_v1/4` #3;
R2: `mmc3_test_v1/5` #2 + `mmc3_test_v1/6` #2). **Zero production-ROM impact.**

**Hard constraints from the lessons-learned corpus:**

- **21+ falsified levers** are on the DO-NOT-RETRY list. Do **not** re-propose
  any of: intra-cycle CPU phase split, constant IRQ-visibility pipeline (1 or 2
  M2 cycles), bus access-ordering swap, A12 filter-threshold tweaks, A12
  emission-dot shift, `run_ppu_to` fencepost change, or **either direction of
  the M2-phase-conditional rising-edge visibility deferral** (mechanism proven
  inert for this ROM).
- **The per-dot split-around-access scheduler already shipped** (beta.2/beta.4).
  Do **not** propose "implement a per-dot scheduler" — it exists.
- **The mechanism finding:** the ROM measures an interval between two
  observations on the same CPU timeline; any consistent re-phasing of the
  PPU-vs-CPU batch boundary shifts *both* together. The bracket is
  **structurally unreachable on any 3-dots-per-CPU-cycle batched catch-up
  model** — the residual is *differential*, not positional.
- **The re-open bar:** "re-open only with a genuinely new, falsifiable
  single-axis hypothesis, not a re-derivation of anything on the DO-NOT-RETRY
  list." Any partial-flip or repeat-diagnosis is a mandated stop.

**F5.0 — PREREQUISITE instrumentation study (pure, zero-risk — do this and
nothing else first).** Exactly one live axis survives: the MMC3 filter's
`gap >= 3` acceptance test currently operates at **integer CPU-cycle**
granularity (`mmc3.rs:606`), whereas real silicon's "three falling edges of M2"
is a *falling*-edge elapsed-low-time question at **M2 (half-cycle) resolution**
— the ares model uses an explicit `irq_delay = 6` M2 down-counter. But the
2026-07-02 campaign found that *no qualifying A12 rise this ROM produces ever
lands in the post-access half of a CPU cycle.* The open question is whether that
"always pre-access" property is **specific to this ROM's phase alignment** or a
**structural consequence of NTSC MMC3 A12 timing generally** (BG/sprite fetches
at dots 260/268/…/316 vs the CPU's 5/7 NTSC read split).

- Instrument the A12-rise-vs-CPU-access phase for **all** NTSC MMC3 fetches
  (not just this ROM) — pure logging, no chip-code change, **no regression risk
  to the sacred 100%**.
- **Two outcomes, both valuable:**
  - **If "always pre-access, structurally"** → the entire remaining
    phase-conditional search space is dead everywhere on NTSC MMC3. Output:
    **formally close ADR 0002**, keep the four tests `#[ignore]`'d by design,
    document the closure. This is the expected and *good* outcome.
  - **If a qualifying rise can land post-access** → axis B is genuinely live:
    prototype an M2-edge-precise (`master_clock`/`M2Phase`-resolution) *falling*-
    edge low-time filter (ares-style down-counter consuming the canonical
    `master_clock`, not integer CPU cycles), behind a **default-off flag**,
    under the full stop-condition protocol. Promote to default **only** if it
    flips all four targets **and** holds every sacred gate. Any partial flip is
    designed-out.

**F5.1 — Disposition either way.** The most likely and correct plan output is
"instrument, confirm structural, close the ADR by design." Do **not** risk a
22nd rollback of the 141/141 for the 1-cycle bracket — this is the one place
"aggressive" yields to "sacred" (v2.0.0 plan Risks #3).

## 9. By-design non-goals (document, do not chase)

- **APU mixer "bit-exactness"** — no 0-tolerance oracle exists; the correct
  non-linear LUT passes `apu_mixer`. Keep the honest ±4% framing. (The analog
  HPF/LPF chain is a fixed-coefficient approximation with no pass/fail ROM;
  optional future work is measured-RC coefficients + oversampling, validated by
  audio-capture regression only — not a correctness gap.)
- **PlayChoice-10 Z80 second-screen menu** — a full Z80 core + PC10 board. XL,
  out of scope.
- **The 16 non-accuracy `#[ignore]`s** — stay as-is: the 8 permanent historical
  pins (superseded pre-master-clock unit assertions; real-ROM coverage
  supersedes them), the 2 interactive CPU-reset `_full_protocol` tests
  (structural), the STUN/TURN live-network probes (`--ignored` by design), the
  HD-pack local-pack test, and `mmc3_test_2/6` (by-design NEC-rev-B vs default
  Sharp-rev-A). **One is addressable as a fixture task:** the 5 Vs. `DualSystem`
  GVS boot tests need combined 64 KiB dual dumps (the staged dumps are the MAME
  maincpu half only) — a *fixture-sourcing* task, not an emulator bug; fold into
  F0/F2 if a combined dump is obtained.

## 12. Maintainer decisions (resolved 2026-07-09)

1. **Sequencing — START THE SAFE WINS NOW.** F0 + F1 begin immediately as
   **v2.1.0**; they are core-accuracy work independent of the mobile ports. The
   larger phases (F3-F5) land as later v2.1.x releases, all **ahead of** the
   v2.2.0 joint mobile store launch (so the apps ship on the improved core).
   First slice underway: **F0 + F1.1 → F1.2 → F1.3** (the v2.1.0 bundle).
2. **F5 — INSTRUMENTATION FIRST, AXIS-B CODE ATTEMPT AUTHORIZED.** Run the
   zero-risk F5.0 logging study first. **If** it proves axis B live (a qualifying
   A12 rise can land post-access on NTSC MMC3 generally), the flagged
   M2-half-cycle *falling*-edge filter prototype (F5.1) is **pre-authorized**
   under the full stop-condition protocol — no further approval needed to
   *attempt* it, but it promotes to default ONLY on an all-four-target flip with
   every sacred gate held; any partial flip or repeat-diagnosis is a mandated
   stop. If F5.0 proves the property structural → close ADR 0002 by design.
3. **F1.4 — DEFAULT-OFF, PROMOTE LATER.** The generated NTSC palette ships behind
   a flag with the current hand palette as default; promotion to default (with a
   deliberate visual re-bless) is a later-release decision (ADR 0030 precedent).
4. **F4.3 FDS.** Deferred as a stretch (real-BIOS / copyright keeps write-CRC out
   of CI; the simplified model has zero known game impact).
5. **First slice:** **F0 + F1.1** — doc truth-up + the palette backdrop-override.
