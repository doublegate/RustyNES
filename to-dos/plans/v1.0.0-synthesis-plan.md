# RustyNES synthesis — harvest v2.8.0 into the public `RustyNES` repo, cut v1.0.0 (CURRENT PLAN)

## Context — why this change

Two sibling projects exist:
- **`RustyNES_v2`** (this repo, `Commercial_Private-Projects/RustyNES_v2`, **v2.8.0**) — a
  cycle-accurate NES emulator: AccuracyCoin 100% (139/139), 51 mappers, FDS, Vs/PC10,
  netplay (UDP + browser WebRTC), RetroAchievements, TAS movies, save-states, rewind,
  run-ahead, display-sync pacing, a dedicated emulation thread, an egui debugger, and a
  WASM/GitHub-Pages build. Crates: `nes-cpu/ppu/apu/mappers/core/frontend` +
  `nes-netplay/cheevos/test-harness`. Frontend = winit + wgpu + egui + cpal.
- **`RustyNES`** (the original PUBLIC repo `OSS_Public-Projects/RustyNES`,
  `github.com/doublegate/RustyNES`, **v0.8.6**) — an earlier, less-accurate emulator with
  a **polished eframe/egui/glow desktop frontend** whose unique value is UX polish, plus a
  large doc ecosystem + branding/images. Crates: `rustynes-cpu/ppu/apu/mappers/core/desktop`.

**Goal:** transplant v2.8.0's emulation technology into the public `RustyNES` repo, retain
the parent's advanced/missing UX features (wired onto the new stack), synthesize the docs +
README (no duplication, all images kept), and release the result as **v1.0.0** on the public
repo. The parent's emulation core is being *replaced*; its core API is a subset of v2's (the
investigation confirmed v2 provides every capability the parent frontend needs, renamed).

**Locked decisions (from the user):**
1. **Frontend base = v2.8.0's** winit/wgpu/egui/cpal frontend (keeps ALL of v2's features);
   the parent's eframe `rustynes-desktop` is RETIRED and its UX polish ported in.
2. **Crate naming = `rustynes-*`**: rename the transplanted v2 crates to
   `rustynes-cpu/ppu/apu/mappers/core/frontend` + `rustynes-netplay/cheevos/test-harness`;
   binary stays `rustynes`.
3. Target repo = the existing public `RustyNES` (preserve its git history); cut **v1.0.0**.
4. **NEVER commit commercial Nintendo ROMs** (hard rule, carried from v2). "Commercial ROMs
   available" = the oracle *harness* + the gitignored `tests/roms/external/` convention
   (user supplies dumps) + the committable CC0/public-domain test ROMs only.

## Investigation findings (two Plan agents, read-only)

- **The parent's "advanced features" are frontend UX, not emulation.** v2's core is a strict
  superset. Parent UX worth porting: a persistent **menu bar** (File/Emulation/Options/Debug/
  Help), **recent-ROMs MRU** (max 10, persisted), **first-run Welcome modal** + keyboard-
  shortcuts window, a **tabbed Settings window** (Video/Audio/Input/Advanced), **themes**
  (light/dark/system), **8:7 pixel-aspect** correction, a **status bar** (ROM name / state /
  fading messages / FPS), loading/pause/FPS overlays, and **ROM pre-warming** (evaluate — v2's
  run-ahead/display-sync may make it moot). v2 already has: drag-drop, save/load state, rewind,
  the CPU/PPU/APU/memory/OAM/mapper debugger panels (richer than the parent's), a settings panel,
  gamepad, 2-player, Four Score, config persistence. So the port is the *shell* (menu bar +
  recent ROMs + welcome + Settings *window* + status bar + themes + pixel-aspect), not the deep panels.
- **Core ↔ frontend API rename map** (parent `Console` → v2 `Nes`): `from_rom_bytes*`→`from_rom*`,
  `power_on`→`power_cycle`, `step_frame*`→`run_frame`, `audio_samples()+clear`→`drain_audio*`,
  `set_controller_1/2(u8)`→`Buttons`/`Controller` (need a u8→`Buttons` adapter; bit order
  A=0x01..Right=0x80 matches), `cpu()/bus().ppu/apu`→`*_snapshot()` debug VIEWS, `peek_memory(u16)`
  →`cpu_bus_peek(&mut,u16)`. (Mostly irrelevant since we ADOPT v2's frontend, not the parent's —
  but the map confirms v2's core covers everything.)
- **Parent's self-defined v1.0.0** = 300+ mappers + 100% TASVideos + every planned feature
  (netplay/TAS/Lua/achievements/WASM/CRT). Cutting v1.0.0 on this synthesis requires
  **redefining** that bar around "production-quality + hardware-accurate" (which v2 IS).
  Lua scripting is the one advertised-but-unbuilt feature in BOTH (note as post-1.0).
- **Docs**: parent has a deep `docs/` (api/cpu/ppu/apu/mappers/formats/features/...), a huge
  `to-dos/` tree, `ARCHITECTURE/OVERVIEW/ROADMAP/VERSION-PLAN/CHANGELOG(88K)`, AI configs
  (CLAUDE/GEMINI/WARP), `ref-docs/` (huge UI/UX design specs). README uses HTML `<img>` for 3 of
  6 images; the **gameplay screenshot** (`RustyNES-Screen_SMB_FirstLight.png`), 2nd blueprint,
  and logo-icon are orphaned (surface them). README claims **SDL2** but the stack is
  eframe/egui (fix). ROADMAP header says "v2.15.0" at v0.8.6 (fix). Both CHANGELOGs use
  Keep-a-Changelog — keep parent's v0.1.0→v0.8.6 history, prepend a v1.0.0 entry.
- **Infra**: parent `.github` = ci.yml (macOS disabled), docs.yml (rustdoc→Pages),
  release.yml (macOS commented), security.yml (cargo-audit/deny). **No WASM web demo** (Pages =
  rustdoc only). v2 has the just-optimized ci/release/web (+ composite action) + a live wasm
  Pages deploy. Dual MIT/Apache in both (compatible). Parent: edition 2024 / rust 1.88; v2:
  edition 2021 / rust 1.86 — reconcile to one toolchain (v2 code should build on 1.88).

## Strategy — phased synthesis

Work happens IN the public `RustyNES` repo (preserve history) on a `feat/v1.0.0-synthesis`
branch. Each phase ends green + committed; FF-merge + tag only at the end (confirm before push).
**The source `RustyNES_v2` project is treated as READ-ONLY and is never modified** — every
operation reads from a staging COPY placed inside `RustyNES` (Phase 0), so there is zero risk of
perturbing this project.

### Phase 0 — Stage everything into `RustyNES` + relocate carry-over content
- **0a — Complete staging copy.** `rsync` ALL of `RustyNES_v2/` into
  `RustyNES/temp/rustynes_v2-harvest/` (the parent already has a `temp/`), EXCLUDING only the
  non-content build/VCS internals: `target/`, `.git/`, `fuzz/target/`, and the `nesdev_wiki/` +
  `tests/roms/external/` + `perf-logs/` heavies are copied but flagged (see 0b). This is the
  single source the rest of the plan pulls from; nothing reaches back into `RustyNES_v2`. Remove
  `temp/rustynes_v2-harvest/` at the very end (Phase E) once everything is relocated.
- **0b — Relocate the "carry-over for future use" dirs** from the staging copy to their permanent
  `RustyNES/` homes (these are reference/tooling assets the user wants kept):
  - `fuzz/` → `RustyNES/fuzz/` — the cargo-fuzz harnesses only (exclude the 169M `target/`/corpus).
  - `tests/` → `RustyNES/tests/` — committed CC0/public-domain ROMs + golden + harness tests.
    **`tests/roms/external/` (commercial dumps, ~tens of MB) is carried but stays GITIGNORED —
    never committed to the public repo.**
  - `perf-logs/` → `RustyNES/perf-logs/` — the Perf-panel CSV logs (reference; **gitignored**).
  - `nesdev_wiki/` → `RustyNES/nesdev_wiki/` — the 208M local NESdev mirror (reference;
    **gitignored — do NOT commit 208M to the public repo**; add to `.gitignore`).
  - Plus (also future-use): `deploy/` (netplay Docker/compose bundle) → `RustyNES/deploy/`;
    `screenshots/` → `RustyNES/screenshots/`; `scripts/` → `RustyNES/scripts/`. (`ref-docs/`,
    `docs/`, `to-dos/` are synthesized in Phase C, not bulk-moved.)
  - Update `RustyNES/.gitignore` (merge v2's) so `target/`, `nesdev_wiki/`, `perf-logs/`,
    `tests/roms/external/`, `fuzz/target/`, `temp/` are all ignored before any commit.

### Phase A — Transplant v2's stack + rename to `rustynes-*` (the mechanical heavy lift)
- Copy v2's `crates/nes-{cpu,ppu,apu,mappers,core,frontend}` + `nes-{netplay,cheevos,test-harness}`
  into the parent's `crates/`, REPLACING the old `rustynes-{cpu,ppu,apu,mappers,core}` and
  retiring `rustynes-desktop`. Also bring v2's `Cargo.toml`/`Cargo.lock`, `rust-toolchain.toml`,
  `.cargo/`, `deny.toml`, workspace `[workspace.lints]`, `tests/`, `fuzz/`, benches.
- **Rename `nes-* → rustynes-*`** everywhere: crate `name`s, `[workspace.members]`, every inter-crate
  dep (path + version), and every `use nes_core::` → `use rustynes_core::` (underscore form),
  `nes_cpu`→`rustynes_cpu`, etc. Scripted (sed) + verified. Frontend crate `nes-frontend`→
  `rustynes-frontend`; binary `rustynes-v2`→`rustynes`. Reconcile edition/toolchain to the parent's
  (1.88 / 2024) or keep v2's (1.86 / 2021) — pick one; verify it builds.
- Gate: the full v2 gauntlet, now under `rustynes-*` — `cargo test --workspace --features test-roms`
  (AccuracyCoin 100%, oracles byte-identical), fmt/clippy/doc/no_std/wasm. The transplant must
  preserve byte-identical emulation (same code, renamed). Binary `rustynes <rom>` runs games.
- **Milestone: the new hardware-accurate core + v2 frontend is live in the RustyNES repo.**

### Phase B — Port the parent's UX polish into `rustynes-frontend`
Port (egui code is portable from eframe → v2's winit+egui context; v2 already runs egui):
- A persistent **menu bar** (egui `TopBottomPanel`: File/Emulation/Options/Debug/Help) driving
  v2's existing actions (open/recent/save-state/load-state/reset/power/pause/fullscreen/settings/
  debugger/about). Keep `~` for the deep debugger.
- **Recent-ROMs MRU** (config + menu), **first-run Welcome modal** + **Keyboard-Shortcuts window**,
  a **Settings window** (tabbed Video/Audio/Input/Advanced — promote/merge v2's settings panel),
  **themes** (light/dark/system), **8:7 pixel-aspect** toggle, a **status bar** (ROM/state/
  fading messages/FPS), loading/pause overlays. Evaluate ROM pre-warming (likely drop).
- Reference parent files (harvest the egui code, re-wire to v2's `App`/config):
  `crates/rustynes-desktop/src/{app.rs,gui/menu.rs,gui/settings.rs,gui/mod.rs,config.rs,input.rs}`.
- Gate: builds + runs; the new UX works; v2's existing features unbroken.

### Phase C — Synthesize the documentation (no duplication, all images kept)
- **README.md** — rewrite into ONE production-quality, user-facing README: keep the parent's
  structure + branding + ALL images (banner, both blueprints, emu-compare, the orphaned SMB
  screenshot) + add v2's screenshot **montage** + a few gameplay shots; replace the feature list
  with the ACCURATE v2.8.0 reality (100% AccuracyCoin, 51 mappers, FDS, Vs/PC10, netplay, RA, TAS,
  save-states, rewind, run-ahead, display-sync, WASM demo); fix SDL2→winit/wgpu; merge both
  Quick-Start/Build/Config/Controls/Acknowledgments/Citation. Arrange images logically (hero
  banner → highlights → screenshot showcase/montage → architecture blueprint → comparison).
- **CHANGELOG.md** — keep parent's v0.1.0→v0.8.6 Keep-a-Changelog history; prepend a rich
  **[1.0.0]** entry summarizing the synthesis (core replaced with v2.8.0 tech; enumerate the
  capabilities) + fold v2's CHANGELOG highlights.
- **ROADMAP.md + VERSION-PLAN.md** — redefine the v1.0.0 bar around the synthesized reality
  (mark accuracy/mappers/netplay/RA/TAS/save-states/rewind/WASM **done**; 300-mapper/100%-TASVideos/
  Lua become post-1.0 stretch); fix the "v2.15.0" header.
- **The synthesis rule (applies to EVERY markdown tree below):** for a file that exists in BOTH
  projects with the same/related name, **combine + synthesize into one** — retain all unique
  knowledge from each, drop duplication, prefer v2's accurate current state where they conflict on
  facts, keep the parent's structure/branding where it reads better. A file unique to one project
  is carried as-is. Nothing is silently dropped.
- **The re-versioning rule (cross-cutting — applies to ALL moved/synthesized markdown + TODOs):**
  the inbound docs are saturated with `v2.x.x` references (the engine's prior release line
  v2.0.0…v2.8.0). In the synthesized project the version line is `v0.1.0…v0.8.6 → v1.0.0` and the
  ENTIRE v2 engine lands at once as **v1.0.0**, so the project must NEVER present `v2.x.x` as one of
  its own releases. Reconcile every inbound version reference:
  - **Feature attribution** ("added in v2.2.0", "new in v2.7.0") → reframe as a **v1.0.0** capability
    (or describe the feature with no version tag). The "v2.8.0 NEXT/previous-release" framings in
    `CLAUDE.md`/`STATUS.md` collapse to a single v1.0.0 current-state summary.
  - **CHANGELOG**: keep RustyNES's own `v0.1.0…v0.8.6` history, then map the engine's `v1.x`+`v2.x`
    lineage onto an INTERMEDIATE **`v0.9.0…v0.9.7` series + a final `v1.0.0`** (the mapping table
    below) — DISTRIBUTING the v2 changelog content across those stages instead of one giant v1.0.0
    block. Each `0.9.x` entry summarizes its stage (Keep-a-Changelog form, dated the synthesis day),
    `v1.0.0` covers the production cut (parent UX + docs synthesis). The next line continues from `1.0.0`.
  - **Delta-only — do NOT re-state what `v0.8.6` (and prior) already shipped.** The `v0.9.x` entries
    (and the synthesized docs/README generally) document ONLY the NEW / CHANGED / ENHANCED content and
    the genuine DIFFERENCES the engine brings relative to the parent's existing `v0.8.6`-and-earlier
    state. Do NOT re-announce features, prose, or images that already existed in the parent and carry
    over unchanged/identical (the parent's own `v0.1.0…v0.8.6` CHANGELOG already covers those). Where
    the engine REPLACES a parent feature with a better implementation, document the **enhancement (the
    delta)** — e.g. "cycle-accurate master-clock scheduler → AccuracyCoin 100%", NOT "added CPU
    emulation". Before writing each `v0.9.x` entry, diff its stage's content against what `v0.8.6`
    already documented and keep only the real additions/changes. (Same principle applies when merging
    same-topic `docs/`/`ref-docs/`/`to-dos/` files — keep the differential knowledge, not duplicate text.)
  - **Release notes**: give each mapped stage a `docs/release-notes/v0.9.x.md` (condensed from the
    corresponding `v2.x.x` notes) + `v1.0.0.md` for the production cut. The original `v2.x.x` note
    files may be retained under `docs/release-notes/engine-lineage/` (labelled "engine development
    history") for the curious, but are NOT presented as RustyNES releases.
  - **STATUS/CLAUDE/ADR/audit/benchmarks/ROADMAP/to-dos**: replace "ships v2.8.0" / "v2.x.x current"
    framing with **v1.0.0**. Where deep historical narrative genuinely needs the old anchors (the
    accuracy-program history), keep the technical content but relabel the version markers as the
    **upstream engine lineage** ("developed across the RustyNES_v2 v2.0–v2.8 line; v1.0.0 here") so
    it reads as history, not as this project's release numbers. ADR/audit FILES keep their dates.
  - **Crate/workspace versions**: `2.8.0` → `1.0.0` (Phase A rename + Phase E bump).
  - **Sanity sweep** (end of Phase C): `grep -rni 'v2\.[0-9]\|2\.8\.0\|2\.7\.' docs/ ref-docs/
    to-dos/ *.md` should surface ONLY deliberately-retained "engine lineage" mentions — no stray
    release claims, no "current version 2.x", no v2.x in feature lists or badges.

a readable progression. `v1.0.0` is reserved for the genuinely-new synthesis work (the parent UX

# RustyNES v2 — CI/CD pipeline consolidation (post-v2.8.0)

## Context — why this change

v2.8.0 is shipped. During the release, a doc-only fix-push triggered the entire
CI matrix unnecessarily, and the `Deploy Web Demo` workflow runs **independently
of CI passing** (it can publish broken code to Pages). The three workflows
(`ci.yml`, `release.yml`, `web.yml`) also repeat the toolchain + apt-deps +
cache setup, and `ci.yml` has no path filtering or concurrency cancellation.
An earlier pass (commit `d504be4`) already: added a shared `rust-setup`
composite action, merged fmt+clippy+rustdoc into one `lint` job pinned to 1.86
(closing a local-vs-CI gap), dropped the redundant 1.86 test entry, added
workflow-level `paths-ignore` + PR concurrency cancel. **This plan goes further**:
turn the workflows into ONE smart, state-dependent CI/CD chain where (a) jobs
only run when relevant files changed, (b) nothing proceeds if a predecessor
fails (`needs:`), (c) Pages deploys **only** after CI is green, and (d) `main`
gets lightweight branch protection requiring a single aggregate check.

**Locked decisions (from the user):** keep the v2.8.0 tag as-is (the doc-only CI
red is fixed-forward on main); branch protection = **lightweight** (require the
`ci-success` check + block force-push/deletion + linear history, but STILL allow
direct `git push origin main` — no required PRs).

**Key research findings (June 2026):**
- Workflow-level `paths-ignore`/`paths` + a *required* status check = the
  "Expected — Waiting for status" trap: a skipped *workflow* leaves its required
  check pending forever and blocks PRs. But a job skipped by a job-level `if:`
  reports **Success**. → With branch protection, path-filtering must move to a
  job-level change-detector, not the workflow trigger.
- Canonical pattern: a `changes` job (`dorny/paths-filter@v3`) outputs booleans
  → downstream jobs gate on `if: needs.changes.outputs.<x> == 'true'` → a single
  **`ci-success`** aggregate job (`if: always()`, fail if any `needs.*.result`
  is `failure`/`cancelled`, pass otherwise) is the ONLY required check. Skipped
  (path-filtered) jobs pass through it cleanly.
- Composite actions are the right reuse unit *within one repo* (already in use);
  reusable workflows are for *cross-repo* sharing (not needed here). Keep the
  `rust-setup` composite; consolidate at the job level.
- `Swatinem/rust-cache` `save-if: ${{ github.event_name == 'push' }}` keeps PR
  runs restore-only (don't pollute/grow the cache); pushes save it.

Sources: GitHub Docs "About/Troubleshooting status checks"; community discussions
#26698/#54877/#13690; devopsdirective.com "Required Checks for Conditional Jobs"
(the `if: always()` aggregate-gate pattern); oneuptime.com "Monorepo Path Filters
in GitHub Actions"; `dorny/paths-filter` README; the composite-vs-reusable
comparison articles.

## The design — one state-dependent CI/CD pipeline

**Consolidate `ci.yml` + `web.yml` into a single `ci.yml` pipeline** (delete
`web.yml`); keep `release.yml` separate (tags are a distinct lifecycle) but make
it fail-safe. Triggers: `push`/`pull_request` to `main` + `workflow_dispatch`,
with **no** workflow-level path filter (so required checks never hang).

Jobs + their gating (all compile jobs `uses: ./.github/actions/rust-setup`):

1. `changes` — `dorny/paths-filter@v3`, outputs `code` + `web`:
   - `code`: `crates/**`, `Cargo.toml`, `Cargo.lock`, `.cargo/**`,
     `rust-toolchain.toml`, `scripts/**`, `tests/**`, `.github/**`.
   - `web`: the same minus `tests/**`/`scripts/**` + `scripts/wasm_size_budget.sh`.
   - A docs-only push matches neither → every downstream job skips → `ci-success`
     still passes.
2. `lint` (fmt+clippy+rustdoc, 1.86), `test` (ubuntu/macos/windows), `test-roms`
   (release), `no_std`, `wasm` (clippy ×2), `bench` — each
   `needs: changes`, `if: needs.changes.outputs.code == 'true' || github.event_name == 'workflow_dispatch'`.
3. `web-build` (trunk build + size-budget gate; upload the Pages artifact only on
   non-PR) — `needs: changes`, `if: web changed || dispatch`.
4. `web-deploy` (`actions/deploy-pages`) — `needs: [web-build, lint, test,
   test-roms, no_std, wasm, bench]`, `if: github.event_name != 'pull_request'
   && github.ref == 'refs/heads/main' && needs.web-build.result == 'success'
   && !contains(needs.*.result, 'failure')`. Job-level `permissions: {pages:
   write, id-token: write}` + job-level `concurrency: pages` + `environment:
   github-pages`. **This is the fix**: Pages publishes only when the whole CI
   chain is green.
5. `ci-success` — `needs: [lint, test, test-roms, no_std, wasm, bench,
   web-build]`, `if: always()`; shell loops `needs.*.result`, `exit 1` on any
   `failure`/`cancelled`, else pass. **The single required status check.**

Per-job `cache-key`s already isolate caches; add `save-if: push` to the
composite's cache step.

`release.yml` (tag-triggered) gains a lightweight `verify` job (`cargo test
--workspace`, debug) that the per-platform `build` matrix `needs:`, so a tag
whose code is broken ships no binaries ("don't proceed if a predecessor fails"
for the release chain too). Keep the anti-clobber note (no `body:`).

## Files to change

- `.github/workflows/web.yml` — **delete** (merged into `ci.yml`).

  `allow_force_pushes:false`, `allow_deletions:false`,
  `required_linear_history:true`, `restrictions:null`. Lightweight — direct
  pushes to `main` still work; the check gates PRs + force-push/deletion blocked.
