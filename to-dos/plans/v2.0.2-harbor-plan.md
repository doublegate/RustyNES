## The octal-latch campaign (oracle-first discipline)

The v2.0.1 close (ADR 0030) proved that **both** bounded fork attempts — Option 2
(Mesen2's persistent `_ppuBusAddress` model, PR #234) and Option 1 (the 2-cycle-ALE
fetch refactor, draft PR #236) — reached only 139/141 flag-on, and concluded the fix
needed a **dedicated Timebase-scale campaign** cross-diffed against a per-cycle bus
oracle. That campaign is this release's work
(`docs/audit/v2.0.2-octal-latch-campaign-2026-07-08.md`). Two empirical corrections
to the v2.0.1 plan:

1. **The assumed oracle was wrong.** The per-cycle bus cross-diff proved the vendored
   **Mesen2 build does NOT pass these two tests** — both result bytes read `0x0A`
   (corruption not reproduced). So ADR 0030's prior "Option 2 = proven-correct Mesen2
   recipe" premise was false. The correct definitive oracle is **TriCNES**
   (`ref-proj/TriCNES/Emulator.cs`, MIT, commit `9199870` — the **AccuracyCoin
   author's own emulator**), which models the multiplexed AD/A bus + octal latch at
   transistor level and does drive `$2F19` / `$0FFF`.
2. **A whole-dot port suffices.** The full 2-cycle-ALE fetch refactor was **not**
   required. Porting TriCNES's octal latch at RustyNES's whole-dot cadence, behind the
   pre-existing `mc-ppu-bus-addr-hybrid` flag, reaches 141/141 flag-on.

## The model (behind `mc-ppu-bus-addr-hybrid`, default-off)

The reworked model adds an 8-bit `octal_latch`, a 14-bit `address_bus`, and a
`copy_v` / `pattern_latch_stale` transient pair (all `#[cfg]`-gated). Every
background fetch resolves its effective VRAM address through `octal_effective` — the
TriCNES `FetchPPU` splice `(address_bus & 0x3F00) | octal_latch` — which is
**transparent** (returns the intended address, reloads the latch) except on the two
modeled corruption events:

- **ALE + Read:** the `$2007`-read PPUDATA state-machine countdown landing during
  render freezes `octal_latch` on the read's DATA byte; the next pattern fetch reads
  `{PAR high 6}:{stale $FF}` = `$0FFF`. Verified per-dot: `$2007`R @ sl3 dot223 →
  latch = `$FF` @ dot228 → pattern read `$0FFF` @ dot230.
- **Hybrid Addresses:** a `$2006` second write during render sets `copy_v` and
  captures the stale octal-latch low byte; the next nametable fetch splices
  `{new v high $2F00}:{stale $19}` = `$2F19`. Verified: W2006 @ sl4 dot182 → hybrid
  NT read `$2F19` @ dot186.

A12 / MMC3 notification stays on the **intended** (un-spliced) fetch address at every
call site, so fetch-address / A12 IRQ timing is unchanged. No snapshot-format bump
(`octal_latch` / `address_bus` self-heal on the next in-blanking fetch ALE, and
`copy_v` / `pattern_latch_stale` are one-shots consumed within a few dots), so
`PPU_SNAPSHOT_VERSION` stays at 4.

## Verified result (the honesty bar)

- **Flag-OFF (shipped default):** byte-identical to v2.0.1 — AccuracyCoin **139/141
  (98.58%)**, nestest 0-diff, mmc3 + `ppu_sprites` all pass. The two new PPU tests
  remain the honest measured gaps in the shipped build.
- **Flag-ON (`mc-ppu-bus-addr-hybrid`, experimental):** AccuracyCoin **141/141
  (100.00%)**, framebuffer 100%, **no regression** — nestest 0-diff, mmc3 (A12
  clocking + IRQ) all pass, `ppu_sprites` 19/19, `mmc1_a12` pass.

**This release does not claim the shipped build is 141/141, and does not claim the
flag has been promoted.** Both are the deliberate v2.0.3 step.

## Decision — refine-then-promote (maintainer, 2026-07-08)

The model ships **flag-off** in v2.0.2 for two reasons:

1. The **60-ROM commercial byte-identity oracle** — ADR 0030's mandated
   promote-safety gate — is maintainer-run and could not be exercised at
   implementation time (no local commercial dumps).
2. The Hybrid path carries a **`+1 coarse-X` latch-capture approximation** of
   RustyNES's whole-dot cadence (the corrupted NT fetch is one tile past the write's
   in-flight tile) — exact for the tested alignment, but not yet a first-principles
   model that generalizes across both test alignments.

Per the *Feature-Flag Additive Change* guardrails, the model bakes behind the
default-off flag in v2.0.2; the shipped default stays the honest **139/141**.

## Carried forward (into v2.0.3)

- **Rework the Hybrid path to a first-principles latch-carry model** — the
  `octal_latch` naturally holding the stale low byte across dots (per TriCNES's
  ALE-driven latch), replacing the `+1 coarse-X` reconstruction from `v` at the
  `$2006` write, so it is exact across BOTH test alignments and generalizes.
- **Run the 60-ROM commercial byte-identity oracle flag-on** + broader
  `$2007`/`$2006`-during-render title validation + a CI job asserting flag-on 141/141.
- **Promote `mc-ppu-bus-addr-hybrid` to default** (shipped AccuracyCoin 141/141),
  weighing the ADR 0028 save-state / byte-identity implication at that point. ADR 0030
  is updated (not superseded) when promotion lands.
- The v2.0.1 Harbor carryovers remain open: Android feature-parity finalization +
  foss/play on-device verify + monetization wiring (v2.0.4), host-localizable bridge
  warnings, and wgpu/naga 30 (blocked on an egui 0.36+ that targets wgpu 30).
