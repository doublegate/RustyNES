# v2.0.x Mobile Finalization & v2.1.0 Joint Store Launch — plan

> **Maintainer decision, 2026-06-23.** The Android (v1.8.x) and iOS (v1.9.0) apps
> will **not** ship to their app stores on their original, independent timelines.
> Both store launches are **held until after v2.0.0 "Timebase"** so the shipped
> apps include v2.0.0's master-clock rewrite, hard-tier accuracy-residual closure,
> and Vs. DualSystem — rather than launching pre-2.0.0 and immediately re-releasing
> on top of a breaking core change. v2.0.0 is the one release licensed to break
> byte-identity / save-state compatibility (ADR 0002), so the apps must re-port and
> re-verify onto it regardless; doing the store launch *after* that re-port is the
> single-launch path.

This plan supersedes the **release / launch** halves of
[`v1.8.0-android-plan.md`](v1.8.0-android-plan.md) and
[`v1.9.0-ios-plan.md`](v1.9.0-ios-plan.md). Those two documents remain the
authoritative **engineering** specs for the Android and iOS apps (architecture,
the shared `rustynes-mobile` UniFFI bridge, the wgpu→SurfaceView / wgpu→Metal
render paths, SAF / Files, store policy); what changes here is **when** each app
ships and the **phasing** of its final polish.

## The new release line at a glance

| Phase | Versions | What ships | Store action |
|---|---|---|---|
| Interim Android | **v1.8.x** (current) | GitHub-**sideload** APK/AAB builds; the v1.8.9 batch + the batched on-device pass | none — **no Play production** |
| Interim iOS | **v1.9.0 → v1.9.9** | iOS/iPadOS built across the **v1.9.x TestFlight train** (foundation → power-user → connectivity → store-readiness → creator tools; mirrors Android v1.8.0–v1.8.9); **TestFlight** | none — **no App Store** |
| Timebase | **v2.0.0** | The one-clock + every-cycle-bus-access master-clock rewrite (ADR 0002). The core the apps re-port onto. | none (desktop release) |
| Android final | **v2.0.1 → v2.0.4** | Final Android additions / modifications / enhancements / fixes, re-ported onto the v2.0.0 core | none (still pre-launch) |
| iOS final | **v2.0.5 → v2.0.8** | iOS finalization on the v2.0.0 core, building on the v1.9.0 foundation | none (still pre-launch) |
| Readiness | **v2.0.9** | True correctness checks + ready-for-release verification for **both** apps + last-minute fixes / testing | both stores brought to review-ready |
| **Joint launch** | **v2.1.0** | Set **all** release flags; **Google Play production + Apple App Store** go live **together** | **both apps released** |

## Why hold to post-v2.0.0

- **v2.0.0 is breaking.** The fractional-master-clock collapse changes save-state
  format and byte-identity (ADR 0002). An app shipped at v1.8.x / v1.9.0 would
  have to re-port + re-migrate its on-device save states the moment v2.0.0 lands —
  shipping after v2.0.0 avoids a store re-release and a save-state break in users'
  hands.
- **One launch, not two staggered ones.** Releasing Android and iOS *together* at
  v2.1.0 gives a single coordinated launch (store listings, marketing, support
  load) instead of two separate store-review cycles months apart.
- **The apps inherit v2.0.0's accuracy.** The hard-tier residual closure + Vs.
  DualSystem land in v2.0.0; launching after it means the mobile apps ship at the
  same accuracy bar as the desktop, not a release behind.

## Interim work (now → v2.0.0) — continues

Nothing freezes. Between now and v2.0.0:

- **Android (v1.8.x):** keep cutting **GitHub-sideload** releases. Finish the
  **v1.8.9** batch (the side-pane / foldable UX fix + the held docs) and run the
  **batched on-device verification pass** (all v1.8.6–v1.8.8 items, see
  [`../v1.8.x-on-device-verification.md`](../v1.8.x-on-device-verification.md)).
  The PGS / Play-Integrity / Chromecast features stay **default-off behind their
  build flags** (`PGS_ENABLED`, `PLAY_INTEGRITY_ENABLED`, `CHROMECAST_ENABLED`) —
  they are flipped on only at **v2.1.0**, not now.
- **iOS (v1.9.0):** build the **interim foundation** per
  [`v1.9.0-ios-plan.md`](v1.9.0-ios-plan.md) — the SwiftUI shell + wgpu→Metal
  render path + the shared `rustynes-mobile` bridge — distributed by **sideload /
  TestFlight** only. No App Store submission.

The Google Play **production** launch that was the old v1.8.x "Workstream P"
(maintainer task) is **deferred into this train** and becomes part of **v2.1.0**.

## Post-v2.0.0 finalization — proposed per-version scope

The four-release Android and four-release iOS allocations below are a **proposed**
breakdown of "bring each app to release quality on the v2.0.0 core." They are the
suggested phasing; the maintainer may re-slice within each four-release window.

### Android — v2.0.1 → v2.0.4

- **v2.0.1 — core re-port onto Timebase.** Regenerate the UniFFI Kotlin bindings
  against the v2.0.0 `rustynes-mobile`; migrate / re-bless on-device save-state +
  `.rnm` formats to the v2.0.0 core; re-verify determinism (AccuracyCoin + the SMB
  / Zelda on-device smoke) on real ARM. Establishes a green Android build on v2.0.0.
- **v2.0.2 — feature-parity finalization.** Fold in the v1.8.9 UX fixes; finalize
  the gated Play-services wiring (PGS cloud-saves / achievements, Play Integrity)
  *still default-off*; finish the box-art library + scrapers, capture / share
  (screenshots, MP4 clips, PiP, tile, shortcuts, widget), TV / Leanback, and the
  accessibility pass against the v2.0.0 behaviour.
- **v2.0.3 — polish + performance.** Regenerate Baseline Profiles against v2.0.0;
  re-tune R8 full-mode keeps for the new core; app-size + startup + Compose
  stability pass; adaptive / foldable final WYSIWYG; vitals instrumentation.
- **v2.0.4 — Android release candidate.** Play Console product / listing / signing
  assets staged; internal / closed testing track; crash-free-rate + ANR gate; close
  every open item from the batched on-device verification pass.

### Android — the `foss` / `play` flavor split (lands across v2.0.1–v2.0.4; ADR 0025)

A cross-cutting Android-finalization deliverable: split the build into a **`foss`**
(default) flavor and a **`play`** flavor so RustyNES can ship to **F-Droid + GitHub
sideload** (clean, Google-free, ad-free) *and* **Google Play** (the full proprietary +
ad-supported build) from one codebase. F-Droid **requires** a build with no
proprietary/Google deps and no ads/tracking, and a `BuildConfig` flag gates code but not
packaging — so a product flavor is the only way to get a genuinely clean channel. Full
rationale + mechanics in **ADR 0025**. Work items (sequence within v2.0.1–v2.0.4):

1. **Add the `distribution` flavor dimension** — `foss { isDefault = true }` + `play`;
   `PLAY_BUILD` set per-flavor. Add the **`installDebug` → `installFossDebug` alias** so
   the dev/CI command is unchanged; update `android.yml` to the flavored tasks.
2. **Move every proprietary SDK to `playImplementation`** — Billing, Cast, Play Games v2,
   Play Integrity, in-app update/review, **+ AppLovin MAX + RevenueCat**. The `foss`
   variant links none of them.
3. **Façade the glue** — an interface in `src/main`, a **no-op impl in `src/foss/`**, the
   real impl in **`src/play/`**, for `LicenseManager`/`Billing`, `CloudSave`/`PlayGames`,
   `Integrity`, `Cast`/`ChromecastSender`, update/review, and the monetization
   `AdGate`/`RewardedGate` + RevenueCat wrapper. `MainActivity` calls only the façade.
4. **Move the ad/billing manifest entries to `src/play/AndroidManifest.xml`** — the
   `AD_ID` permission, the AdMob `APPLICATION_ID`, AppLovin activities, the Play Games
   `app_id` meta-data — so they never merge into the `foss` manifest.
5. **Wire the monetization SDK glue** into `src/play/` (the `rustynes-monetization` crate
   `.so` + bindings are already in the build from the v1.8.9 build-out, dormant): the
   AppLovin/RevenueCat shells from `crates/rustynes-monetization/shells/`, the
   `feature_enabled` gating of SaveStates/SaveOnExitResume/BatterySaves/FastForward/
   Shaders/Cheats, the run-out paywall + countdown UI, `begin_session` at launch,
   `export/restore_progress` persistence, and the offline-grace path.
6. **Verify both flavors on-device** — `foss` (no ads, no Google, fully free) *and* `play`
   (freemium gating, ad cadence, RevenueCat entitlement flips) — part of the v2.0.9
   readiness pass.
7. **Submit at v2.1.0**: `foss` → **F-Droid** (+ GitHub Releases), `play` → **Google Play**.

> **Already done in v1.8.9 (the clean part):** the `rustynes-monetization` crate (our own
> `AdPolicy` Rust core — no Google, no ads) is wired into the Android build (the `.so` via
> cargo-ndk + UniFFI Kotlin bindings), dormant. Only the **proprietary glue** above is
> flavor-scoped here at v2.0.x.

### iOS — v2.0.5 → v2.0.8

> The iOS app is built across the **v1.9.0 → v1.9.9** TestFlight train (full plan:
> [`v1.9.x-ios-train-plan.md`](v1.9.x-ios-train-plan.md)) — foundation, the Metal
> renderer + shader stack, connectivity + scripting, platform excellence, and
> creator tools — then **finalized here in v2.0.5–v2.0.8 onto the v2.0.0 Timebase
> core**, exactly as Android (v1.8.x) is finalized in v2.0.1–v2.0.4.

- **v2.0.5 — core re-port + foundation finalize.** Bring the v1.9.x SwiftUI / Metal
  shell onto the v2.0.0 core; regenerate the UniFFI **Swift** bindings; migrate
  on-device save states; determinism re-verify on real Apple silicon.
- **v2.0.6 — feature parity.** Game Center, iCloud save sync, MFi / hardware
  controllers, share / capture, iPad multitasking + Stage Manager, VoiceOver /
  accessibility — matched to the Android feature set where the platform allows.
- **v2.0.7 — polish + performance.** Metal pipeline + ProMotion tuning; app
  thinning / on-demand resources; the App Store **privacy manifest** + required-
  reason API audit; energy / thermal pass.
- **v2.0.8 — iOS release candidate.** App Store Connect product / listing staged;
  TestFlight external beta; App Review readiness (guideline self-audit, emulator-
  policy posture, age rating).

### Readiness — v2.0.9

- **True correctness checks, both apps.** Run the **batched on-device verification
  pass** for Android *and* the equivalent iOS pass; cross-check feature parity;
  confirm AccuracyCoin / determinism on both platforms against the v2.0.0 core.
- **Verify *both* Android flavors on-device** (ADR 0025): the **`foss`** build (no ads,
  no Google SDKs, fully free — the F-Droid/sideload artifact) *and* the **`play`** build
  (freemium gating, ad cadence, RevenueCat entitlement flips).
- **Last-minute fixes + testing.** Triage anything the readiness passes surface;
  no new feature work — stabilization only.
- **Both stores review-ready.** Listings, screenshots, privacy disclosures, signing,
  and the closed-test promotion paths all staged so v2.1.0 is a flip, not a build.

### Joint launch — v2.1.0

- **Flip every gated flag to production** (`PGS_ENABLED`, `PLAY_INTEGRITY_ENABLED`,
  `CHROMECAST_ENABLED`, the iOS equivalents) and any `PLAY_BUILD` / `APPSTORE_BUILD`
  switches.
- **Submit + release both apps together** — Google Play production promotion and
  Apple App Store release, coordinated for the same launch window.
- **Ship the `foss` Android flavor to F-Droid + GitHub Releases** alongside the `play`
  flavor on Google Play (ADR 0025) — the clean, Google-free, ad-free channel.
- **This is the milestone the old "Workstream P" maintainer task collapses into.**

## Gates

- Each Android/iOS finalization release keeps the **desktop / core determinism
  contract**: AccuracyCoin holds at its v2.0.0 value, the chip stack stays
  `no_std`-clean, and host CI stays green. Mobile changes are presentation / host
  side.
- **No store submission before v2.1.0.** Sideload (Android) + TestFlight (iOS) are
  the only distribution channels until then.
- The **batched on-device verification pass is a hard gate** on the v2.1.0
  production launch (it already gated the deferred Play launch; it now gates the
  joint launch). See [`../v1.8.x-on-device-verification.md`](../v1.8.x-on-device-verification.md).
