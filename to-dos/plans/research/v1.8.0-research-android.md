# RustyNES v1.8.0 "Android app" — Research & Release Plan

**Status:** Research (READ-ONLY pass). No code modified. Date: 2026-06-17.
**Author of research:** Claude (Opus 4.8 1M).
**Mission:** Port the pure-Rust RustyNES NES emulator to a complete, shippable Android application.

> Scope note: this slots *after* the in-flight v1.6.0 "Studio" (TAStudio / movie-interop) train.
> It is a **platform** release, not an accuracy release. The accuracy bar is held by the
> existing chip stack; this release adds a new *frontend host* (Android) without touching the
> determinism contract. The v1.0.0 lineage facts (AccuracyCoin 100% / 139, determinism, the
> `rustynes-*` workspace shape) are the immovable substrate.

---

## 0. TL;DR for the maintainer

- **Architecture pick: Option (c) — Hybrid.** A thin Rust core lib behind a small JNI/UniFFI
  surface, **plus** the existing `wgpu` renderer drawing the NES image onto an Android
  `SurfaceView`, hosted in a native **Jetpack Compose** shell that owns the chrome (menus,
  touch overlay, settings, file pickers). This maximises reuse of the byte-identical core +
  the existing `wgpu`/`cpal`/Hermite-DRC stack while giving native lifecycle, native file
  access (SAF), and a native-feeling UI. Option (a) (winit `android-activity` reuse of the
  whole egui frontend) is the **fastest first-boot** and is recommended as the **beta.1
  spike** to prove the core renders + sounds on-device, then progressively replaced by the
  Compose shell. Option (b) (pure Compose, no wgpu) is rejected — it throws away the
  shader/NTSC/CRT pipeline and the proven present path.
- **Toolchain:** Android NDK **r27 LTS** (or r28; pin one), Rust targets
  `aarch64-linux-android` (ship), `x86_64-linux-android` (emulator), `armv7-linux-androideabi`
  (optional legacy), built with **`cargo-ndk`** invoked from **Gradle** via the
  `cargo-ndk-android-gradle` plugin → `.so` into `jniLibs/` → **AAB** for Play. `minSdk 26`
  (AAudio floor), `targetSdk 35` (Play mandate as of Aug 2025).
- **Audio is nearly free:** `cpal 0.18` (already the workspace pin) has a **native AAudio
  backend** at API 26 — the existing lock-free ring + Hermite DRC resampler is platform-
  agnostic and works unchanged. Determinism-safe (resampler is timing-only).
- **The 3 biggest risks:** (1) **wgpu surface-loss on rotate/background** — must drop the
  surface on `Suspended`/`surfaceDestroyed` and recreate on `Resumed`; the emu core keeps
  running headless. (2) **Google Play emulator/verification policy uncertainty** (Aug-2025
  Play Protect language + 2026-2027 developer-verification rollout) — ship-to-Play is a real,
  non-engineering risk; mitigate with sideload/F-Droid as a guaranteed channel. (3) **No
  bundled ROMs + SAF file access** — all ROM/SRAM/state I/O must go through the Storage
  Access Framework with persistable URI grants, which is new plumbing the desktop `rfd`/
  `directories` path doesn't cover.
- **Beta train:** beta.1 first-boot-on-device (winit spike) → beta.2 surface+lifecycle
  hardening → beta.3 audio+input+touch → beta.4 storage/SAF+save-states → beta.5 Compose shell
  + feature-port matrix → rc.1 Play-policy/legal/signing + AAB → **v1.8.0** release.

## 1. Architecture decision (the keystone)

Three candidate shapes were evaluated against four axes: **code reuse**, **native feel**,
**effort**, and the **determinism contract**. The determinism contract is *invariant under all
three options* — the chip stack is `#![no_std]` + `alloc`, already CI-proven to compile against
`core+alloc` (the `thumbv7em-none-eabihf` gate), and produces bit-identical framebuffer+audio
given the same seed+ROM+input on any ISA, including ARM/AArch64. So determinism is not a
differentiator; it is a *constraint we must not break* (see the per-option notes).

### Option (a): Reuse `rustynes-frontend` whole via `android-activity` + winit

Compile the existing winit+wgpu+egui `App` to Android. winit 0.30's `ApplicationHandler`
already models `resumed`/`suspended` (the exact hooks Android needs), and the Android backend
is driven by the `android-activity` crate. The egui shell becomes the touch UI.

- **Reuse:** Maximal — the entire `App`, `gfx.rs`, the shader stack, the audio engine, the
  debugger overlay, run-ahead, pacing all come along. The single biggest lever.
- **Native feel:** Poor-to-moderate. egui is an immediate-mode in-game UI, not Material; file
  pickers, sharing, and system back-gesture feel foreign. No native menus.
- **Effort:** **Lowest to first boot** (S–M), but it inherits *every* desktop assumption
  (`rfd` native dialogs, `directories` XDG paths, keyboard hotkeys) that does **not** exist on
  Android, so "complete app" effort balloons (M–L) chasing those gaps inside egui.
- **Determinism:** Safe — identical code path to native; the new surface is presentation-only.
- **Reference:** the project already has `wasm-winit` (full winit+wgpu+egui in the browser),
  which is structurally the closest precedent — Android `android-activity` is the same
  "winit on a foreign host" shape minus the wasm single-thread constraint.

### Option (b): Thin core lib behind UniFFI/JNI + pure Jetpack Compose UI (no wgpu)

`rustynes-core` exposed through UniFFI; Compose renders the framebuffer (e.g. as an
`ImageBitmap`/`Canvas` blit), Oboe/AAudio for sound, Compose for all chrome.

- **Reuse:** Core only. **Throws away** `gfx.rs`, the NTSC/CRT/Bisqwit shader stack, the
  letterbox/PAR/overscan blit, the wgpu present path, and the GPU-timing work — all the
  v1.0–v1.5 rendering investment. Re-implementing 8:7 PAR + overscan + (eventually) shaders
  in Compose/RenderEffect is a large, lower-fidelity rewrite.
- **Native feel:** Best — fully Material 3.
- **Effort:** **Highest** (XL) for parity; you rebuild the renderer.
- **Determinism:** Safe (core untouched) but you must re-prove the *audio* path's DRC servo
  on a non-cpal sink, re-introducing risk the existing engine already retired.
- **Verdict:** Rejected. The rendering pipeline is a crown jewel; discarding it for Material
  polish is a bad trade for an emulator whose value prop includes CRT/NTSC fidelity.

### Option (c): Hybrid — Rust core + wgpu game surface in a Compose `SurfaceView` shell **[RECOMMENDED]**

Keep `wgpu` rendering the NES image (full shader stack intact) onto an Android
`SurfaceView`/`SurfaceHolder` whose `ANativeWindow` is handed to wgpu. Wrap the core + the
render/audio loop in a thin native lib (a new `rustynes-android` crate, JNI via the `jni`
crate or UniFFI for the typed control surface). Compose owns everything *around* the game
surface: top bar, settings, ROM picker (SAF), touch-overlay controls, save-state manager,
RetroAchievements/netplay screens.

- **Reuse:** High — core + `wgpu`/`gfx.rs` + shaders + `cpal`(AAudio) + the audio engine +
  run-ahead + pacing all reused. Only the *chrome* and the *host glue* are new (and the
  chrome is genuinely better as native).
- **Native feel:** Strong — Material shell, native pickers, native back/lifecycle, native
  notifications/quick-resume.
- **Effort:** **Medium-High** (L). More than (a)'s first boot, far less than (b)'s rewrite,
  and the *complete-app* effort is the lowest of the three because the platform-native pieces
  (storage, lifecycle, pickers) are done the Android-idiomatic way instead of being faked
  inside egui.
- **Determinism:** Safe. The core runs on a dedicated thread (the Android analogue of
  `emu-thread`); wgpu surface + AAudio are presentation/output only. The existing lock-free
  `SharedInput` + late-latch model maps cleanly: touch/gamepad events are published into
  `SharedInput` exactly as keyboard events are on desktop, so TAS/netplay/rollback see them
  identically and add **no new determinism surface** (this is precisely how the wasm touch
  overlay was made deterministic — `docs/frontend.md` "Touch controls").

**Recommended path:** ship **(a) as the beta.1 spike** (fastest proof the core renders and
sounds correctly on real ARM hardware, reusing the `wasm-winit` precedent), then **graduate to
(c)** for the shippable app — lift the wgpu surface + core loop out of the egui `App` into the
`rustynes-android` lib and replace the egui chrome with the Compose shell incrementally. This
de-risks early (you see Mario on a phone in beta.1) while converging on the right long-term
architecture. The egui debugger panels can remain available as an optional in-surface overlay
for power users, so (a)'s reuse is not wasted.

can't express cleanly (handing the `Surface`'s `ANativeWindow` to wgpu, pushing the audio
sink). This split keeps the JNI hand-rolled surface tiny (high-risk code minimised) while the
broad API is generated and type-checked.

## 3. Subsystem mapping

### Rendering — wgpu on Android (Vulkan/GLES) + surface loss

- wgpu's Android backend is **Vulkan first, GLES fallback** (`webgl` feature is wasm-only; on
  Android wgpu uses the Vulkan or GLES backends natively). The workspace already enables
  `wgpu` with the `webgl` feature for wasm; Android needs the default Vulkan/GLES backends —
  a target-gated feature set, not a code change to `gfx.rs`'s WGSL (the shaders are WebGL2-safe
  already, per `docs/frontend.md`, which guarantees GLES3 compatibility).
- **The surface-loss problem (RISK #1).** On Android the `ANativeWindow` backing a wgpu
  `Surface` is **invalidated between `Suspended` and `Resumed`** (rotate, background, lock).
  The app **must drop the `Surface` on suspend and recreate it on resume**; creating a surface
  before `Resumed` panics. winit 0.30's `ApplicationHandler::{resumed, suspended}` is exactly
  this contract — the existing `App` already creates the surface in `resumed` on the wasm/native
  path, so Option (a) inherits the right shape. For Option (c), the Compose `SurfaceHolder`
  callbacks (`surfaceCreated`/`surfaceDestroyed`/`surfaceChanged`) drive the same
  create/drop/reconfigure on the Rust side via JNI.
  - **Determinism note:** surface recreation is **presentation-only**. The emu core (and its
    framebuffer/audio production) keeps running headless across a surface gap — no frame is
    re-emulated, no state perturbed. The present path simply has nowhere to draw until the
    surface returns. This is the same decoupling the desktop `emu-thread` already provides
    (the occlusion watchdog keeps emulation+audio running when the compositor throttles
    redraws — `docs/frontend.md` display-sync section).
- **Frame pacing:** present to the device refresh via Android **Choreographer** (the legacy
  doc already noted this). Map it onto the existing pacing matrix: a phone panel near 60 Hz
  → `display`-sync (1 emulated frame/refresh, audio DRC absorbs the 60.0988-vs-60 beat);
  90/120 Hz panels → wall-clock pacing (exactly as the desktop high-refresh path). The pacer
  is platform-agnostic; only the "refresh signal" source changes (Choreographer vsync callback
  instead of winit `RedrawRequested`).
- **Effort: L.** Surface lifecycle is the bulk of it; the renderer itself is reuse.

### Audio — cpal → AAudio (NEARLY FREE)

- **`cpal 0.18`** (the *current workspace pin*) ships a **native AAudio backend** (it migrated
  from the Oboe wrapper to `ndk::audio`), minimum **API 26**. So the existing
  `audio.rs` engine — the lock-free SPSC ring, the allocation-free callback, the 4-tap Hermite
  resampler with ±1% dynamic rate control — **works unchanged** on Android. This is the single
  biggest de-risk in the whole port.
- Request **low-latency mode** (AAudio `PerformanceMode::LowLatency`, buffer = 2 bursts) for
  the tightest latency; the DRC servo's latency target may need a mobile-specific default
  (mobile bursts are larger/spikier — bump the target like the desktop >75 Hz +20 ms headroom
  rule). MMAP path gives lowest latency where the device supports it.
- **Lifecycle:** pause/stop the cpal stream on `onPause`/`Suspended`, restart on resume. Audio
  focus (transient loss on calls/notifications) handled in the Compose/Activity layer via
  `AudioManager` and forwarded to mute/pause.
- **Determinism note:** the resampler is timing-only; the *core's emitted samples* (the audio
  oracle contract) never depend on wall-clock feedback. Unchanged on Android.
- **Effort: S.** Mostly lifecycle + a mobile DRC default; the engine is reuse.

### Input — touch overlay + hardware gamepad + haptics

- **Touch overlay:** the project **already designed and shipped a deterministic touch overlay
  for wasm** (`wasm_touch.rs` + the `web/index.html` Pointer-Events DOM overlay): D-pad / A /
  B / Start / Select, selectable target port (P1–P4 for Four Score), and a 12-button Power Pad
  mat, all funnelled into a `Buttons` mask read at the **same late-latch a keypress uses**.
  This is the reuse template. On Android the overlay is drawn by **Compose** (Option c) or
  egui (Option a); the bitmask → `SharedInput` → `EmuCore::latch` plumbing is identical, so
  touch input is recorded/replayed identically by TAS + netplay (no new determinism surface).
- **Hardware gamepads:** Android delivers controller input via `InputDevice` + `KeyEvent`
  (buttons, `AKEYCODE_BUTTON_*`) and `MotionEvent` (axes, `AXIS_HAT_X/Y` for d-pad, sticks).
  `gilrs` (the desktop gamepad lib) **does not** cover Android — gamepad input must come
  through the Android input pipeline. With `GameActivity`/`android-activity` the
  `InputIterator`/`KeyEvent`/`MotionEvent` are exposed to Rust; with Option (c) the Compose
  `Activity` receives `dispatchKeyEvent`/`onGenericMotionEvent` and forwards normalised button
  state over JNI. Map to the same `Buttons` mask. **Effort: M.**
- **Haptics:** light vibration on touch-button press via Android `Vibrator`/`VibratorManager`
  (Compose layer). Purely cosmetic; off the determinism path. **Effort: S.**
- **Determinism note:** all input converges on the single `Buttons` mask at the existing late
  latch — the contract is preserved exactly as it is for wasm touch.

### Lifecycle + the emu thread

- Android process/activity lifecycle (`onCreate/onStart/onResume/onPause/onStop/onDestroy` +
  `surfaceCreated/Destroyed`) maps onto: create core on first ROM load; **save-on-background**
  (`onPause` → write SRAM + an auto save-state); drop surface on `surfaceDestroyed`; pause the
  emu thread + audio stream; resume + recreate surface on return.
- The desktop **`emu-thread`** model (dedicated emulation thread + lock-free `SharedInput` +
  `Arc<Mutex<EmuCore>>`) ports directly — Android is a full `std` `pthread` host (unlike
  single-threaded wasm), so the *native* threading model is available, which is actually a
  better fit than the wasm path. **Effort: M.**

## 4. App lifecycle + storage (no bundled ROMs — RISK #3)

- **ROM import via SAF.** Commercial ROMs are **never bundled** (project policy + Play policy).
  The user picks a `.nes`/`.fds`/`.zip` via the **Storage Access Framework** document picker
  (`ACTION_OPEN_DOCUMENT`). The app **takes a persistable URI grant**
  (`ContentResolver.takePersistableUriPermission`) so the ROM (and an optional ROM *folder* via
  `ACTION_OPEN_DOCUMENT_TREE` for a library view) survives reboot. ROM bytes are read through
  the `ContentResolver` and handed to `Nes::from_rom`. This replaces the desktop
  `rfd`/drag-and-drop path; it is **new plumbing** with no desktop analogue beyond the wasm
  file-picker shape (which is the closest precedent — wasm already receives raw bytes from a
  browser picker rather than a path).
- **Save-states / SRAM / settings:** app-**private** internal storage
  (`Context.filesDir`) — the Android analogue of `directories::ProjectDirs` (which is
  desktop-only and must be replaced). Save-states keep the existing `.rns` format and the
  SHA-256-keyed slot layout (`<rom-sha256>/slot-N.rns`) — byte-identical format, just a
  different root dir. SRAM/NVRAM battery saves on `onPause`. Settings → an Android-idiomatic
  store (DataStore in Compose, or keep the existing `toml` written to `filesDir`).
- **Determinism note:** the save-state format and SHA-256 keying are unchanged, so a state
  saved on desktop loads on Android and vice-versa (the format is platform-independent).
- **Effort: M–L** (SAF + persistable grants + the library/recent list rebuilt on content URIs).

## 6. Perf / thermal / battery

- **Headroom is ample.** The core targets ≤2 ms/frame headless on a 2018 laptop; modern
  arm64 phone cores (and the AArch64 codegen) comfortably emulate the 2A03/2C02 in real time.
  The NES is a light target — perf is **not** the gating concern; *thermals/battery* are.
- **Run-ahead** costs N extra `run_frame`s + ~140 µs state churn per visible frame. On mobile
  this is a battery/thermal tax — keep the existing **budget-throttle hysteresis** (auto-drops
  run-ahead when produce-cost p95 rises) and default run-ahead **off or 1** on Android, exposed
  in settings. **Rewind** keyframe cache RAM is fine on modern phones but make the window
  smaller by default.
- **Frame pacing to device refresh** via Choreographer (above) — avoids busy-spin wall-clock
  pacing that drains battery; prefer the vsync-driven `display`-sync regime on ~60 Hz panels.
- **Thermal:** cap unthrottled fast-forward bursts; respect Android thermal-status callbacks
  (`PowerManager.OnThermalStatusChangedListener`) to back off run-ahead/fast-forward.
- **Determinism note:** all of the above are *frontend pacing/scheduling* knobs — the core's
  emitted frames/samples are unchanged. Auto-throttling run-ahead does **not** change the
  persistent timeline (run-ahead's speculative frames are already rolled back and never reach
  save-states/TAS/RA, per `docs/frontend.md`).
- **Effort: M** (mostly Choreographer + thermal callbacks + mobile defaults).

## 7. Play Store policy + legal (RISK #2)

- **Emulators are allowed** and present on Play today (RetroArch, Dolphin, PPSSPP all ship via
  Play in 2025). The standard compliance posture: **no ROM downloader / no bundled copyrighted
  ROMs** (RetroArch's Play build famously strips the core downloader for exactly this). RustyNES
  already mandates user-provided ROMs only — compliant by design.
- **2025 policy turbulence (the real risk).** Google's **Aug-2025 Play Protect** language
  added "ROM-based systems and software that replicate proprietary environments," and a
  **developer-verification** rollout (full legal identity; APKs must be signed by a verified
  developer to install on certified devices) is scheduled **early-access Mar 2026 → all devs
  Sep 2026 → enforcement 2027** in early regions then global. None of this *bans* emulators
  today, but it adds identity/verification overhead and ongoing policy uncertainty.
  - **Mitigation:** (1) prepare developer verification early (legal identity / DUNS if org).
    (2) Ship a **guaranteed alternate channel** — sideloadable signed APK + **F-Droid** /
    GitHub Releases — so the project is never solely dependent on Play approval. This is also
    the natural home for the egui-debugger/power-user build and any feature Play balks at.
- **Content rating:** complete the IARC questionnaire (an emulator shell with no bundled
  content is typically Everyone/PEGI 3; the rating of *user-supplied* games is not the app's).
- **Privacy:** the app collects **nothing** by default (offline emulator). Data-safety form =
  "no data collected/shared." RetroAchievements/netplay (if shipped) introduce account/network
  data → disclose then. No ads, no analytics in the MVP.
- **Signing:** enroll in **Play App Signing**; Gradle `signingConfig` with an upload key; the
  AAB is signed and uploaded.
- **Effort: M** (paperwork + signing + a release track), but **calendar/approval risk is
  non-engineering and outside our control** — flagged as risk #2.

## 8. Completion definition (first-boot → parity → Play)

"To completion" = **a signed AAB on a Play internal/closed track (and a sideload APK), that
boots, loads a user ROM via SAF, renders with the shader stack, sounds correct via AAudio,
accepts touch + hardware-gamepad input, survives rotate/background/resume without crashing,
saves/loads state + SRAM, and holds the determinism contract** (a state saved on desktop loads
on Android; a TAS replays identically). Stretch: native RetroAchievements login.

### Proposed beta train → shippable AAB

| Milestone | Goal | Architecture | Key risks retired |
|---|---|---|---|
| **beta.1 — First boot** (S–M) | `android-activity`+winit spike (Option a): the existing `App` boots on a real device + emulator, renders a test ROM, makes sound. | (a) | Proves core+wgpu+cpal/AAudio work on ARM. |
| **beta.2 — Surface & lifecycle** (L) | Rock-solid surface drop/recreate on rotate/background/lock/multi-window; emu keeps running headless across the gap; Choreographer pacing. | (a→c) | **Risk #1** (surface loss). |
| **beta.3 — Audio + input + touch** (M) | Low-latency AAudio mode + lifecycle pause/resume + audio focus; deterministic touch overlay (reuse wasm design); hardware gamepad via Android input; haptics. | (c) | Input determinism, audio lifecycle. |
| **beta.4 — Storage + SAF + states** (M–L) | SAF ROM picker + persistable grants + library/recent; save-states/SRAM to `filesDir`; save-on-background; settings store. | (c) | **Risk #3** (no-bundled-ROM/SAF). |
| **beta.5 — Compose shell + feature port** (L) | Native Material shell (top bar, settings, save-state manager, ROM library), feature-port matrix (shaders, TAS, palettes); decide RA/netplay defer. | (c) | Native-feel; parity. |
| **rc.1 — Store/legal/signing** (M) | Play App Signing, AAB build in CI, 16 KB alignment check, targetSdk 35, data-safety + content rating, dev verification prep, F-Droid/sideload artifact. | (c) | **Risk #2** (Play policy) — partially; the rest is approval-calendar risk. |
| **v1.8.0 release** | Signed AAB on a closed track + public sideload APK; docs (`docs/android.md`), CHANGELOG, a `rustynes-android` crate + Gradle project committed. | (c) | — |

Determinism-preserving invariant across **every** milestone: the chip stack and the
emitted-frame/sample contract are never touched; all Android work lives in a new host frontend
(a `rustynes-android` crate + Gradle module + JNI/UniFFI glue), exactly as `rustynes-frontend`
and the wasm path are separate hosts over the same byte-identical core. AccuracyCoin/golden/
nestest gates stay on host CI and remain 100% / 139.

## 9. Ranked risk list

1. **wgpu surface loss on rotate/background (HIGH, engineering).** Mishandling → crashes/black
   screen on the most common user action. Mitigated by the winit `resumed`/`suspended` contract
   (Option a) or `SurfaceHolder` callbacks (Option c) + keeping the emu headless across the gap.
   Retired in beta.2.
2. **Google Play emulator/verification policy (HIGH, non-engineering, outside control).**
   Aug-2025 Play Protect language + 2026-27 dev-verification rollout create approval/calendar
   uncertainty. Mitigated by full compliance (no bundled ROMs/downloader) **and** a guaranteed
   sideload/F-Droid channel so the project never depends solely on Play.
3. **SAF / no-bundled-ROM storage rework (MEDIUM-HIGH).** Entirely new I/O model vs desktop
   `rfd`/`directories`; persistable URI grants, content-URI library. Closest precedent is the
   wasm byte-picker. Retired in beta.4.
4. **Input breadth (MEDIUM).** `gilrs` doesn't cover Android; hardware gamepads + touch must be
   funnelled through the Android input pipeline into the single `Buttons` mask without breaking
   the late-latch determinism. Retired in beta.3.
5. **Architecture transition cost a→c (MEDIUM).** Lifting the wgpu surface + core loop out of
   the egui `App` into `rustynes-android` is real refactoring; mitigated by doing (a) first as a
   throwaway-or-keep spike and keeping egui as an optional overlay.
6. **Audio lifecycle / latency tuning (LOW-MEDIUM).** cpal AAudio is reuse, but low-latency
   mode + audio focus + a mobile DRC default need device tuning. Retired in beta.3.
7. **Battery/thermal from run-ahead/rewind (LOW).** Mitigated by mobile-default off/1 run-ahead
   + thermal-status backoff + vsync pacing. Not a blocker.
8. **16 KB page alignment + ABI/NDK pinning (LOW).** A build-config detail (NDK r28+ default,
   Play requirement); caught by a CI alignment check.

## 10. Top open decisions for the maintainer

1. **Architecture commitment:** approve **(c) hybrid** as the ship target with **(a) as the
   beta.1 spike**? Or go pure-(a) for v1.8.0 and defer the Compose shell to v1.9.0? (Affects
   scope by ~2 betas.)
2. **Bindings:** **UniFFI** for the typed control surface + raw `jni` only for the
   surface/audio hot path — or hand-roll all JNI (smaller dep tree, more `unsafe`)?
3. **Activity backend:** `GameActivity` (better input/IME) vs a plain Compose `Activity`
   hosting a `SurfaceView` (cleaner Compose integration)? Influences the input plumbing.
4. **Feature scope for MVP:** defer **netplay** (mobile NAT/TURN) and **RetroAchievements**
   (login UI) out of v1.8.0, shipping core+video+audio+touch+gamepad+states+TAS first? (Strong
   recommendation: yes — both are "needs work" and netplay is *already* a deferred item.)
5. **Distribution posture:** Play-first with sideload backup, or **F-Droid/sideload-first** and
   treat Play as best-effort given the 2025-27 policy uncertainty?
6. **ABIs to ship:** arm64-only (smallest AAB, simplest) vs arm64+armeabi-v7a (legacy reach)?
   x86_64 for emulator/CI regardless.
7. **NDK pin:** r27 LTS (Rust's supported LTS) vs r28/r29 (16 KB-by-default, newer Clang)?
8. **Lua on Android:** ship native mlua (it cross-compiles) or defer scripting from the mobile
   build entirely?

## Appendix — concrete crates / tools / versions

- **Build:** `cargo-ndk` (bbqsrc), `cargo-ndk-android-gradle` (willir) or Mozilla
  `rust-android-gradle`; Android NDK **r27 LTS** (r28/r29 ok); Gradle AGP current; AAB output.
- **Host glue:** `android-activity` (0.6+, `game-activity` feature) for Option (a); `jni`
  crate (0.21) + **UniFFI** (~0.28+) for Option (c).
- **Renderer:** `wgpu` 29 (workspace pin) — Vulkan/GLES backends on Android (the `webgl`
  feature stays wasm-only); `winit` 0.30 (Option a) with `android-native-activity`/
  `android-game-activity`.
- **Audio:** `cpal` 0.18 (workspace pin) — **native AAudio backend, API 26 floor** (no `oboe`
  crate needed). Reuses `audio.rs`/`resampler.rs` verbatim.
- **UI shell (Option c):** Jetpack Compose (Compose BOM ~2025.x, Kotlin 2.x), Material 3,
  Activity Result API for SAF, DataStore for settings, `Vibrator` for haptics, Choreographer
  for vsync.
- **SDK levels:** `minSdk 26`, `targetSdk 35` (Android 15, Play mandate since 2025-08-31).
- **Reuse precedents in-repo:** the `wasm-winit` path (`src/wasm*.rs`) is the structural
  template for Option (a); `wasm_touch.rs` is the template for the deterministic touch overlay;
  the legacy `docs/archive/legacy-v0.8-docs/platform/MOBILE_BUILD.md` has prior JNI sketches
  (now superseded by this plan's cpal-AAudio + hybrid findings).
