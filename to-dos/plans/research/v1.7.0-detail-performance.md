# RustyNES v1.7.0 Performance Optimization Opportunities — Measure-First Workstream

**Date:** 2026-06-17  
**Scope:** Novel, determinism-safe optimizations holding AccuracyCoin 139/139, byte-identical audio/framebuffer  
**Current state:** v1.0.0 ships at 2–4 ms/frame headless; v1.4.0 F1–F2 micro-opts + v1.5.0 H1–H8 frontend pass yielded rendering/pacing improvements.

---

## Executive Summary

RustyNES has aggressive constraints: `#![no_std]` + `alloc`, determinism-locked (same seed+ROM+input = bit-identical output), and already applies PGO + BOLT + mapper-caps-gating. The v1.4.0–v1.5.0 work addressed the low-hanging fruit (scanline-flag cache, MMC5 early-return, frontend pacing). The v1.7.0 opportunity lies in **second-order optimizations** that require **measurement before implementation**:

1. **PPU dot-loop batching** — idle scanlines & 8-dot fetch spans can skip per-dot register thrash
2. **APU/BLEP recompute amortization** — phase-row kernel lookup & sample-rate resampler
3. **Unified DMA fast-path** — common case short-circuit within the determinism envelope
4. **Save-state snapshot speed** — XOR-delta compression + serialization pipelining
5. **Cache locality & branch elimination** — struct layout + pixel-composition masks
6. **Frontend decoupling** — GPU/filter work on separate thread (if measurement shows win)
7. **Wasm-specific** — WASM Linear Memory layout & tighter buffer bounds

Each has **evidence requirements** (which benchmark shows the win), **determinism proofs** (how to prove byte-identical output), and **effort estimates**. The workstream is ranked by (win magnitude × feasibility × safety).

## The Determinism & Measurement Contract

**Byte-identity guarantee:** every optimization must prove it does NOT change:
- CPU cycle-exact execution (timing)
- PPU dot-by-dot pixel composition
- APU sample values (not timing; resampler is frontend-only)
- Mapper state transitions (all side effects)

**Measurement first:** each optimization MUST name the Criterion bench that will show the win:
- `nes_run_frame_nestest` (CPU-dominated, minimal PPU work)
- `nes_run_frame_flowing_palette` (rendering-heavy, full BG/sprite/palette churn)
- `ppu_tick_one_frame` (PPU dot loop in isolation)
- `blip_square_wave_0_1s_ntsc` (APU BLEP synthesis)
- `mapper_dispatch::cpu_read` (mapper hot-path)

**Proof strategy:** A/B baseline + implementation on same host; run full oracle (AccuracyCoin 139/139, `nestest` golden-log, visual_regression, apu_mixer/apu_test byte-exact); PGO-enabled determinism gate in CI.

## Tier 1 Optimizations (High Win Likelihood, Moderate Risk)

### T1.1: PPU Dot-Loop Batch Advancement (Idle & Fetch Spans)

**Title:** Skip per-dot register updates on known-idle PPU dots

**Source/Evidence:**
- `crates/rustynes-ppu/src/ppu.rs` lines 1–500 (the per-dot `tick()` scheduling loop)
- `docs/scheduler.md` (One tick = one PPU dot; 89,342 dots/frame)
- Performance baseline: `ppu_tick_one_frame` = 497 µs (33× realtime); full-frame PPU cost is ~6–8% of frame time per v1.4.0 F1

**What it is:**
The PPU's per-dot scheduler advances through 89,342 dots per frame. On idle scanlines (241–260 NTSC, no rendering), every dot still:
- Increments the dot counter
- Advances the master clock (R1 backend)
- Checks for frame-complete transition
- (Optionally) calls synthetic bus reads

On rendering scanlines, the 8-dot background tile fetch pipeline (CHR pattern fetch, nametable/attribute read) can batch-advance across identical dot phases without recomputing shifter state on **every single dot** when the fetch result is predictable (e.g., repeating tile patterns).

**Why v1.7.0-worthy:**
- The PPU `tick()` call dominates the per-dot work; it's called 89,342 times/frame
- Idle scanlines are 20 of 262 NTSC scanlines = ~7.6% of dots with zero visual work
- The 8-dot fetch pipeline has structurally identical steps; many games have long runs of static BG tiles
- Measurement ceiling: the `ppu_tick_one_frame` bench isolates PPU cost (currently 497 µs); a 5–10% win = 25–50 µs/frame

**Effort:** M–L (requires profiling which dot phases are hot, batching logic is ~200 LOC, validation is high)

**Expected-win-area:** `ppu_tick_one_frame` bench (+5–10%); full-frame `flowing_palette` (+2–3%, PPU is 30–40% of budget)

**Determinism/byte-identical risk: MEDIUM**
- Batching must preserve scanline-by-scanline framebuffer pixel ordering (can't skip a visible dot)
- Mid-scanline PPUCTRL/PPUMASK changes must still interrupt batch spans
- Proof: compare pixel-for-pixel framebuffer hash on each scanline boundary; run full oracle with batching on/off

**Measurement plan:**
1. **Baseline:** `cargo bench -p rustynes-ppu --bench ppu_throughput -- --save-baseline v1.7.0-pre` (idle-heavy test ROM if available; else synthetic)
2. **Implementation:** Flag for batch mode in `Ppu::tick()`, skip per-dot register updates when safe
3. **A/B:** `--baseline v1.7.0-pre` on both `nestest` and `flowing_palette` full-frame benches
4. **Oracle:** `cargo test --features test-roms` on AccuracyCoin, visual_regression, golden-log with/without batch flag

### T1.2: Unified DMA Cycle Fast-Path (Common Case Short-Circuit)

**Title:** Skip unified DMA dispatch on CPU-only cycles (no pending/in-flight DMA)

**Source/Evidence:**
- `crates/rustynes-core/src/bus.rs` lines 4250–4300 (`unified_dma_pending`, `unified_dma_cycle_impl`)
- `docs/performance.md` lines 71–74 ("the +6–8% lives on the bus side… `unified_dma_cycle`")
- Benchmark: `full_frame` A/B showed R1 scheduler +6–8% overhead vs. legacy; most is `unified_dma_dispatch` per-cycle

**What it is:**
The R1 master-clock scheduler runs `unified_dma_cycle()` every CPU cycle to check for pending/in-flight DMA. On a frame with no DMA activity (like `nestest` menu), this is still invoked ~29,780 times/frame. The function checks:
1. Is OAM DMA pending? → latch halt/address
2. Is OAM DMA in-flight? → increment read/write counters
3. Is DMC DMA serviceable? → perform DMA read

On most cycles (no DMA), all three checks fail; the function is a no-op. A branch-free predicate `unified_dma_pending()` already exists (line 4253) to gate the whole loop—**but the inline dispatch still runs even when pending is false** (a false-positive performance trap if the inlining breaks branch prediction).

**Why v1.7.0-worthy:**
- The v1.4.0 mapper-caps-gating (ADR 0001) proved that skipping no-op per-cycle calls yields measurable wins (−4% to −10%)
- The unified DMA cycle is the **only per-cycle dispatch that runs unconditionally** (others are already gated by caps flags)
- Non-DMA cartridges (NROM, many mappers) would benefit; DMC-using carts (many MMC games) have sparse DMA windows
- Potential win: 5–15% on DMA-free frames; average across mixed workload ≈ 3–5%

**Effort:** S–M (flag the early-exit path, add `#[cold]` or `#[inline(never)]` to the impl, bench to verify branch prediction stays intact)

**Expected-win-area:** `full_frame` benches (+3–5% on nestest, +1–2% on flowing_palette if its loader uses DMA)

**Determinism/byte-identical risk: LOW**
- The skip is purely an early return; no state change
- Proof: compare CPU cycle counts and PPU dot counts at frame-end; run oracle unchanged

**Measurement plan:**
1. **Baseline:** `cargo bench -p rustynes-core --bench full_frame -- --save-baseline v1.7.0-pre`
2. **Implementation:** Add inline early-return gate + branch hints at the dispatch point
3. **A/B:** Criterion baseline comparison on both workloads
4. **Oracle:** Full AccuracyCoin + cycle-count hash validation

### T1.3: APU/BLEP Phase-Row Index Amortization

**Title:** Cache per-phase kernel-row index across multiple BLEP taps

**Source/Evidence:**
- `crates/rustynes-apu/src/blip.rs` lines 150–250 (BlipBuf::add_sample, the delta scatter)
- `docs/performance.md` line 87: `blip_square_wave_0_1s_ntsc` = 611 µs (164× realtime; 3.4 ns/sample)
- v1.4.0 F2 dropped the "phase-row index cache" due to quantization reordering risk (see line 225)

**What it is:**
The BLEP decimator's `add_sample()` is called once per CPU cycle (29,780 times/frame). Each call:
1. Computes `phase = input_sample_frac % 1.0` (fractional output position)
2. Derives `phase_row = (phase * PHASES) as usize` (index into kernel table; PHASES=32)
3. Loads kernel[phase_row][0..TAPS] (32 kernel coefficients)
4. Scatters delta × kernel[phase_row][m] to the output ring

The phase-to-phase-row conversion is done inline per call. However, **the phase advances slowly** (by `step = sample_rate / cpu_rate` per sample; step ≈ 0.024 at 44.1 kHz ÷ 1.79 MHz). So many consecutive samples land in the **same phase_row**, recomputing the same index repeatedly.

**Why v1.7.0-worthy:**
- The v1.4.0 analysis correctly flagged a quantization risk (reordering f64→f32→i32 conversions breaks byte-identity), BUT
- The risk is **NOT inherent to caching the phase_row**; it's specific to changing **when** the quantization happens
- A simple cached `phase_row` local variable (set once per call or on phase boundary crossing) avoids the reordering entirely
- Potential win: 2–5% on `blip_square_wave_0_1s_ntsc` bench (fewer modulo/integer conversions per tap)

**Effort:** S (one cached local in `add_sample()`, conditional invalidation when phase wraps)

**Expected-win-area:** `blip_square_wave_0_1s_ntsc` bench (+2–5%); full-frame APU cost is ~5–10% so overall ~0.5–1%

**Determinism/byte-identical risk: LOW**
- The kernel lookup **result** is identical; only the timing changes
- Proof: compare sample-by-sample bitwise audio output with/without cache

**Measurement plan:**
1. **Baseline:** `cargo bench -p rustynes-apu --bench spectral::blip_square_wave_0_1s_ntsc -- --save-baseline v1.7.0-pre`
2. **Implementation:** Add `cached_phase_row: Option<usize>` to `BlipBuf` or use thread-local; set on entry to `add_sample()`, invalidate on phase wraparound
3. **A/B:** Criterion comparison
4. **Oracle:** APU audio oracle (`apu_mixer`, `apu_test` byte-exact comparison)

## Tier 2 Optimizations (Medium Win, Higher Complexity)

### T2.1: Save-State Compression Speed (XOR Delta + Streaming)

**Title:** Replace full-snapshot rewind ring with streaming XOR-delta compression

**Source/Evidence:**
- `crates/rustynes-core/src/rewind.rs` lines 1–100 (LZ4 keyframe + XOR delta strategy is already IN PLACE)
- `docs/performance.md` line 18: Save-state save/load latency ≤ 1 ms target
- Current: keyframe every 60 frames (1 second); every other frame is XOR delta (already compressed)

**What it is:**
The rewind ring already uses XOR-delta compression (line 72: `Body::Delta`). However, the **serialization path** in `save_state.rs` and **rewind apply path** may still be doing full-snapshot allocation + copy per restore. Mesen2 pipelines the decompression (decompress LZ4 in one thread while decoding PPU state in another).

The optimization is to:
1. Decompress LZ4 keyframe + in-flight deltas on a **separate thread** (not the emu thread)
2. Pre-fault the snapshot buffer pages before calling `pop_back()` to avoid OS page faults during restore
3. Stream XOR application byte-by-byte (cache-friendly) instead of buffering the whole delta

**Why v1.7.0-worthy:**
- Current rewind is feature-gated; users with F5-held rewind enabled see the cost every frame (~320 KiB snapshot)
- The v1.3.0 H3 already reuses the snapshot buffer allocation; the next tier is **CPU time** on compress/decompress
- Potential win: 10–30% on rewind capture latency (currently ~1–3 ms for 64 KB compressed snapshot)

**Effort:** M (threading + sync, or async I/O if wasm is a target; validation is high due to state machine)

**Expected-win-area:** New bench `snapshot_restore` (or already exists?); measure `rewind_capture_enabled=true` full-frame cost; expect +0.5–1% win if rewind is armed

**Determinism/byte-identical risk: LOW**
- Decompression is purely a side effect; the emu's state output is identical
- Proof: compare rewind-restored state byte-for-byte; frame output must match oracle

**Measurement plan:**
1. **Baseline:** `cargo bench -p rustynes-core --bench snapshot_restore` (if not exist, create one with F5 rewind loop)
2. **Implementation:** Thread-pool for decompress; `Arc<Mutex<Vec<u8>>>` for pre-allocated buffers
3. **A/B:** Benchmark with/without threading
4. **Oracle:** Determinism oracle with rewind_capture enabled; frame hash unchanged

### T2.2: PPU Cache Locality & Struct Layout

**Title:** Reorder PPU register fields to improve CPU cache hit rate on hot per-dot accesses

**Source/Evidence:**
- `crates/rustynes-ppu/src/ppu.rs` lines 100–500 (Ppu struct definition)
- Each `tick()` reads/writes: `scanline`, `dot`, `pixel`, `nmi_pending`, rendering-state flags (cached or not)
- CPU cache line = 64 bytes; typical NES PPU struct is much larger (~2 KB state)

**What it is:**
The PPU struct holds:
- Loopy registers (`v`, `t`, `w`, `x`, `fine_y`)
- Shift registers (BG + sprite, 2–4 u16/u32 per)
- Rendering state (current BG tile, sprite line data, etc.)
- Frame/scanline counters
- Flags (visible, rendering, pre_render, cached_visible, nmi_pending, etc.)

Hot per-dot access pattern:
1. Read `scanline`, `dot` (check if visible/pre-render)
2. If rendering: read shift registers, increment x
3. Read `nmi_pending`, conditionally write it
4. Update `pixel` buffer

A cache-unfriendly struct might spread these across multiple 64-byte lines, causing 3–5 cache misses per dot. Reordering to co-locate hot fields reduces this to 1–2 misses.

**Why v1.7.0-worthy:**
- The PPU's `tick()` is called 89,342 times/frame; even 1 extra cache miss per dot = ~90 K misses/frame
- On a 3–4 GHz CPU with ~4 ns/cycle, a cache miss (∼20 cycles) is 80–120 ns—visible at this scale
- Potential win: 2–5% on `ppu_tick_one_frame` bench if the struct is sufficiently unfriendly (measured via `perf cache-references`)

**Effort:** M (requires profiling to identify actual cache footprint, refactoring struct layout, validation that derived state computation doesn't break)

**Expected-win-area:** `ppu_tick_one_frame` bench (if measured via perf cache-miss ratio); full-frame +1–2%

**Determinism/byte-identical risk: LOW**
- Struct layout is internal; no external ABI change
- Proof: framebuffer hash per-scanline identical (pixel composition is deterministic by dot)

**Measurement plan:**
1. **Profile:** `perf stat -e cache-references,cache-misses ./target/release/bench ppu_tick_one_frame` (baseline)
2. **Identify hottest fields:** Use `perf record --call-graph dwarf` to find offset accesses
3. **Reorder:** Move `scanline`, `dot`, `rendering`, shift registers, `nmi_pending` to front of struct
4. **Re-profile:** Verify cache-miss ratio decreases
5. **Bench:** `ppu_throughput` A/B
6. **Oracle:** Visual_regression golden framebuffer hash

### T2.3: Pixel Composition Branch Elimination

**Title:** Replace conditional priority logic with arithmetic masks in sprite-to-BG rendering

**Source/Evidence:**
- `crates/rustynes-ppu/src/ppu.rs` lines 3600–3750 (pixel compositing + priority logic)
- Hot path: for every visible pixel (61,440 pixels/frame = 256×240), decide BG vs. sprite via priority rules
- Current: `if sprite_opaque && (!bg_opaque || sprite_priority) { emit sprite } else { emit bg }`

**What it is:**
The NES sprite-to-background priority rule is:
- Sprite priority=0 → sprite on top (unless transparent)
- Sprite priority=1 → BG on top (unless transparent)
- Both transparent → universal BG color
- Opaque sprite priority=1 + opaque BG → BG (not sprite)

This is typically implemented with nested ifs. A branch-free version computes:
```
selected_palette = mask ? sprite_palette : bg_palette
```
where `mask = sprite_opaque & (sprite_priority == 0 || !bg_opaque)` is computed without branches.

**Why v1.7.0-worthy:**
- 61,440 conditional branches per frame = one of the densest inner loops
- Modern CPUs (Skylake, Zen) have excellent branch prediction, but speculative penalty on mispredicts (~10–20 cycles)
- Games with fast-moving sprites or scrolling backgrounds cause frequent priority changes, increasing mispredict rate
- Potential win: 1–3% if mispredict rate is >5% (measured via `perf` branch-misses)

**Effort:** S–M (the logic is already tight; profile first to confirm the win exists, refactor pixel composition to use masks)

**Expected-win-area:** `flowing_palette` full-frame bench (heavy sprite usage); +1–2% if rendering time is 30% of budget

**Determinism/byte-identical risk: LOW**
- The pixel output must be identical (same priority rules applied)
- Proof: per-pixel framebuffer hash; run oracle unchanged

**Measurement plan:**
1. **Profile:** `perf stat -e branch-instructions,branch-misses ./target/release/bench nes_run_frame_flowing_palette`
2. **If mispredict rate >5%:** refactor pixel composition to eliminate branches
3. **Bench:** `flowing_palette` full-frame A/B
4. **Oracle:** Visual_regression golden framebuffer byte-exact comparison

## Tier 3 Optimizations (Speculative, Requires Investigation)

### T3.1: Frontend Decoupled GPU/Filter Thread

**Title:** Move NTSC composite filter + GPU upload to a separate thread (non-blocking present)

**Source/Evidence:**
- `crates/rustynes-frontend/src/gfx.rs` (Bisqwit NTSC filter, wgpu command-encoder submission)
- `docs/performance.md` lines 282–290 (H5 GPU-timing shows `gpu_ms` in perf panel)
- `docs/frontend.md` (present-blocking, GPU submit latency)

**What it is:**
Currently, the emu thread produces the framebuffer, the present thread copies it (after H1 decoupling), and the GPU upload + filter submission happens **synchronously** on the winit event loop. On high-refresh panels (144+ Hz), the present thread may stall waiting for the GPU to finish the previous frame's upload.

Mesen2 runs the Bisqwit NTSC filter on a **separate GPU submit thread**, decoupled from both the emu and present threads. The emu publishes to a ringbuffer; the filter thread consumes, applies composite RT, and submits; the present thread reads the blit result.

**Why v1.7.0-worthy:**
- The perf log (H7–H8) surfaces `gpu_ms`; on some systems this is 2–5 ms (15–30% of the 16.67 ms budget)
- The NTSC Bisqwit path (12-sample window per pixel) is CPU-bound on the GPU but memory-bound on the CPU (reading index framebuffer)
- Decoupling removes the blocking dependency; the emu + present can proceed while GPU work is in-flight

**Effort:** L (requires thread-safe ringbuffer + GPU submit queue; high validation complexity)

**Expected-win-area:** Perf log `gpu_ms` + `present_max_ms` metrics; measure on actual 144+ Hz panel (no headless path); expected ≈ 20–30% reduction in `gpu_ms` if contention is real

**Determinism/byte-identical risk: MEDIUM**
- The GPU output is **identical** (deterministic operations), but timing changes (fewer stalls)
- Frontend timing is not part of the determinism contract (only emulation output)
- Proof: compare presented framebuffer RGBA byte-for-byte with/without threading; frame hashes match

**Measurement plan:**
1. **Measure baseline:** Run on 144 Hz panel with perf logging enabled; capture 60 s of `gpu_ms`, `present_max_ms`, `produced_max`, `catchup_bursts`
2. **Implementation:** Thread + Arc<Mutex> ringbuffer for index_fb + ntsc_phase
3. **Validate:** No regression in emulation output (oracle runs; determinism holds)
4. **Remeasure:** Perf log on same hardware; compare `gpu_ms` + `present_max_ms` before/after
5. **Gate:** Only commit if `gpu_ms` p99 reduction > 20% AND no emulation regression

### T3.2: Run-Ahead Double-Simulation Batching

**Title:** Run ahead-frame + visible frame back-to-back to amortize launch overhead

**Source/Evidence:**
- `crates/rustynes-core/src/nes.rs` (rewind_capture_enabled = false for run-ahead frames)
- `crates/rustynes-frontend/src/app.rs` (run_ahead loop)
- Current: run hidden frame, capture rewind, run visible frame, capture rewind (4 rewind snapshots per 2 frames)

**What it is:**
Run-ahead (predictive display input) emulates one extra frame ahead (without capturing to rewind) to reduce input latency. Current flow:
1. Run hidden frame (run_ahead=true, rewind_capture_enabled=false)
2. Run visible frame (rewind_capture_enabled=true)
3. Repeat

Each frame triggers `run_frame()` overhead (scheduler setup, reset flags, etc.). If both frames are executed in one `run_frame()` call (e.g., `run_n_frames(2, [false, true])`), the setup is amortized.

**Why v1.7.0-worthy:**
- Run-ahead is enabled by default in the frontend; every frame does 2× the work
- If scheduler setup is 5–10% of frame cost, batching saves 2.5–5% per logical frame
- Potential win: 2–5% on full-frame benches if run-ahead is active

**Effort:** M (requires new `run_n_frames` API, rewind-capture vectorization, validation)

**Expected-win-area:** Full-frame benches with run_ahead=1; measure `produced_max` + frame timing in perf log

**Determinism/byte-identical risk: LOW**
- Each frame's output must be identical to running separately; proof by oracle

**Measurement plan:**
1. **Baseline:** Perf log with run_ahead=1; measure frame cost + jitter
2. **Implementation:** Extend emu API to accept frame count + rewind-capture array
3. **Bench:** Full-frame with run-ahead enabled; measure time per logical frame
4. **Oracle:** Output per frame must match the sequential baseline

## Tier 4 Optimizations (Wasm-Specific, Long-Tail)

### T4.1: WASM Linear Memory Optimization

**Title:** Align hot data structures to page boundaries in WASM linear memory

**Source/Evidence:**
- `crates/rustynes-frontend/src/wasm.rs` (WASM build)
- WASM linear memory is segmented by page (65,536 bytes); allocations across pages cause extra indirection

**What it is:**
In WASM, memory access crossing a 64 KiB boundary adds overhead (some engines check bounds twice). The PPU and APU hot data structures should be pre-allocated at fixed offsets that don't cross boundaries.

**Why v1.7.0-worthy:**
- WASM targets are increasingly common (browser NES emulators)
- The gain is marginal on native (2–3%), but 5–10% on WASM if crossing is frequent

**Effort:** S (one-time layout file, linker script or allocator hook)

**Expected-win-area:** WASM build only; full-frame bench on WASM target

**Determinism/byte-identical risk: LOW** — layout-only change

**Measurement plan:** Bench WASM binary before/after; measure full-frame time in browser
