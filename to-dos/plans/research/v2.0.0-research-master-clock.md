## 0. The single most important framing fact (read this first)

**The master-clock refactor that the task brief describes as "the big v2.0.0 prize" has
already been built and is already the shipping default core.** This radically changes what
a *RustyNES* v2.0.0 release actually is.

The history is the classic RustyNES "engine-lineage versioning trap":

- The internal engine line (the `nes-*` worktree under `refactor/v2.0-master-clock`)
  built the **R1 master clock** — a `master_clock: u64` on the CPU advancing 12 units per
  CPU cycle, with `start_cycle`/`end_cycle` φ1/φ2 split access, `run_ppu_to(exact mc)` PPU
  catch-up, and region-parametrized dividers (NTSC 12 / PAL 16 / Dendy 15 on a common
  master clock → **exact 3.2:1 PAL** as integer dividers, not a float).
- That stack was promoted through `mc-r1-substrate` → `mc-r1-full-cpu` umbrella →
  **default and only scheduler** (Phase-7/F, `docs/audit/v2.0-phase7f-r1-default-promotion-2026-06-10.md`).
- It reaches **AccuracyCoin 100.00% (139/139)**, `nestest` 0-diff, `cpu_interrupts_v2`
  **5/5 strict** (the 3 C1 IRQ-sample sub-ROMs that the brief lists as "unreachable" are
  **CLOSED**), `ppu_vbl_nmi` 10/10, `$2007` Stress closed, the NMI-suppression / `$2002`
  race closed, SH\* unstable stores 6/6.
- The engine line then continued **past** "v2.0.0" to internal markers v2.1.0 / v2.6.0 /
  v2.7.1 (mappers, Vs/PC10 RGB, N-peer netplay). All of this is folded into what the
  **RustyNES release line** ships as **v1.0.0 → … → v1.5.0 "Lens"** (current tagged
  release; `Cargo.toml` = 1.5.0).

So: **do not plan to "build the master clock from scratch." It exists, it ships, it is
100% on the oracle.** What the brief calls the residuals — `$2002`/NMI suppression, PAL
3.2:1, the C1 trio — were closed *by* that work.

### What is genuinely left (the real v2.0.0 scope)

Two things the brief is right about remain open, and they are the true architectural prize
for a RustyNES **v2.0.0** major bump:

1. **The R1 master clock rides on a FIVE-counter substrate that should be ONE clock.**
   `docs/audit/v2.0-cumulative-cycle-accounting-rewrite-plan-2026-06-05.md` documents it:
   the core keeps `Cpu::master_clock`, `Cpu::cycles`, `LockstepBus::cycle`,
   `LockstepBus::ppu_clock`, `Apu::cpu_cycle` **plus** two sub-cycle parity bits
   (`Apu::apu_phase`, `Apu::put_cycle`) **plus** the DMC byte-timer, kept in lockstep only
   by hand-written `+= 1` statements next to each other — never derived from one source of
   truth. Several invariants *already* do not hold. The reference invariant
   (Mesen2/ares/TriCNES, all three) is **one monotonic counter; everything phase-related is
   fixed arithmetic of it**. RustyNES is not there.

2. **The CPU is NOT "every cycle is a bus access."** `docs/audit/v2.0-timing-map-vs-references-2026-06-06.md`
   is explicit: Mesen and TriCNES model the 6502 as *every* cycle being a real read, a
   write, or a dummy read of the held address — no busless cycle. RustyNES instead has
   `dispatch()` *declare* an instruction's length and a **burn-loop** (`cpu.rs:490 while
   cycles_emitted < cycles { idle_tick }`) fill the remainder with `idle_tick` cycles that
   do **no bus access**, with a `dma-cycle-budget` hack to reconcile a DMA that halts
   mid-instruction. The audit calls this "the `~40-50%-built` rewrite."

These two are the same rewrite. Collapsing to one clock **and** moving to the
every-cycle-bus-access model is the genuine fractional/sub-cycle master-clock refactor — it
is what closes the *last* residuals (which all sit OUTSIDE the AccuracyCoin oracle) and is
what cleanly enables full Vs. DualSystem dual-core. That is the v2.0.0 architectural core
(Workstream A below).

the BG-serial-in subpos model); they re-appear in v2.0.0 only as *regression-guard* surface
under the rewrite, not as open work.

## 1. The proposed v2.0.0 timebase (Workstream A core)

### 1.1 Target model — one clock, every cycle a bus access

Adopt the Mesen2/TriCNES/ares-converged model exactly:

- **One monotonic `master_clock: u64`** owned by the CPU, in master-clock units, incremented
  once per CPU cycle *including every DMA cycle*. This already exists as a field; v2.0.0
  makes it the *only* counter and derives everything else from it.
- **Derive, never maintain:** `bus.cycle`, `apu.cpu_cycle`, `apu_phase`, `put_cycle`
  (get/put parity), the DMC-timer phase, the OAM 513/514 alignment, and the reload arm all
  become **pure arithmetic of `master_clock`** (e.g. `put_cycle = ((master_clock / div) + seed) & 1`).
  RW-1 of the cumulative-cycle plan already proved this collapse byte-identical for the
  parity pair (`mc-r1-one-clock`, commit `c1e2a25`) — v2.0.0 finishes it for all five.
- **Every cycle is a bus access.** Replace the `dispatch()`-declares-length + `idle_tick`
  burn-loop with a per-cycle model where each CPU cycle issues a real read, a real write, or
  a dummy read of the held address (the references' `DummyRead`). This removes the
  `dma-cycle-budget` reconciliation hack entirely — a DMC DMA halts the CPU *on a bus access*,
  and now every cycle is one, so the halt insertion point and span are exact.

### 1.2 The divider numbers (already region-exact in the shipping core; preserve them)

These are *load-bearing* and are already correct in the shipping R1 core
(`docs/audit/v2.0-phase7f-r1-default-promotion-2026-06-10.md` §PF-1). v2.0.0 must preserve
them under the one-clock rewrite:

| Quantity | Value | Source |
|---|---|---|
| CPU divider NTSC | **12** master clocks / CPU cycle | `CPU_DIVIDER_NTSC = 12` |
| CPU divider PAL | **16** | `region_dividers().0` → PAL 16 |
| CPU divider Dendy | **15** | Dendy 15 |
| PPU dots / CPU cycle NTSC | 3 (12/4) | `ppu_divider` |
| PPU dots / CPU cycle PAL | **3.2** = 16/5 | exact integer dividers on the common clock |
| Read access split (NTSC) | pre **5** / post **7** mc (`READ_PRE_MC`/`READ_POST_MC`) | `read_split(div)` |
| Write access split (NTSC) | pre **7** / post **5** mc (`WRITE_PRE_MC`/`WRITE_POST_MC`) | `write_split(div)` |
| PPU offset | 1 (`PPU_OFFSET`) | CPU/PPU phase align |
| APU cadence | every other CPU cycle (get/put parity) | `apu_phase` |
| PAL CPU frame | **33,247** cyc (1.662 MHz / 50 Hz / 312 lines) — hardware-true, NOT the legacy 35,464 approximation | PF-1 region_timing |

The φ1/φ2 split (pre/post master-clock around the access, with `run_ppu_to(mc-1)` before and
after) is the mechanism that makes mid-instruction PPU events (sprite-0 at a precise dot,
MMC3 IRQ at dot 260, mid-scanline scroll) visible to subsequent CPU code at the *exact*
master-clock timestamp. PAL's 3.2 falls out automatically: 16 mc/cycle ÷ 4 mc/dot = 4
dots/cycle for some cycles and the DMA-coherence fixup folds the region's true `cpu_divider`
span into `dma_mc_consumed` so the next `run_ppu_to` catches up to the correct 3.2 average.

> **Reference-emulator constants** (ares/higan via libco + nall scheduler; Mesen2's
> `_state.CycleCount` even/odd `getCycle`; TriCNES's `CPUClock` 12-master/cycle + the
> `_6502`→`_EmulateAPU`→`APU_PutCycle`-flip ordering) confirm the 12:4 NTSC and the
> integer-divider PAL approach. (See §6 reference appendix once the source-mining agents
> report; the RustyNES dividers above already match the references.)

### 1.3 Migration from the current substrate (no AccuracyCoin regression)

The non-negotiable: **AccuracyCoin must stay 139/139 through every step, and the rewrite
must *also* close R1–R5.** Strategy, mirroring the proven RW-0…RW-4 gate discipline:

1. **Pin every residual's failing test-ROM expectation FIRST** (ADR-0002 / project policy:
   "pin the failing test ROM, then implement until it passes"). The oracles already exist:
   `irq_trace` golden traces (6 ROMs), the `cpu-instr-cycle-trace` scaffold, the Mesen2
   `EventType::PpuCycle` per-PPU-cycle Lua patch, `vbl_race_window_2002_read_sweep`, the
   `CheckDMATiming` `$4010=$4E` ROM-landmark cross-diff (RW-0b). Wire a strict expected-fail
   probe for each of R1–R5 so the rewrite *flips them loud*.
2. **Collapse the five counters to one behind a feature** (`mc-one-clock-v2`, default-off),
   reproducing the floor config byte-identical at `parity_seed == 0` (RW-1 already did this
   for the parity pair). Gate: AccuracyCoin 139/139 + nestest 0-diff + 60-ROM oracle
   framebuffer byte-identical at every step.
3. **Move to every-cycle-bus-access** behind the same feature: convert `idle_tick`
   burn-fill into per-cycle dummy reads of the held address, retire the `dma-cycle-budget`.
   This is the make-or-break (the audit's "interleaved-DMA model, divergence C"): one DMA
   cycle interleaved per `_6502()` step sharing the per-cycle put-cycle flip-flop.
4. **Make the reload arm invisible to its own cycle** (a `pending_dmc_dma_next` latch
   promoted at the next cycle boundary, reloads only — loads stay GET-first→3): per the
   timing-map this turns the DMC reload span 3→4 (`CheckDMATiming` 1587→1588, Y 3→4),
   closing R5 *naturally* once the every-cycle model holds.
5. **Cycle-accurate reset** (`Nes::reset()` → a real reset *sequence* that runs the
   reset-vector-delay cycles + frame-counter re-arm phase on the master clock): closes R3/R4.
6. **CPU `T_last-1` IRQ sample + bus poll at the M2 phase the bracket needs:** on the
   one-clock + every-cycle model the M2 sub-cycle phase is now *representable* (it was not
   on the integer 3-dots timebase — that is the documented reason R1/R2 were unreachable),
   so the MMC3 sub-test #3 bracket becomes a sample-point choice rather than an impossible
   constraint. Closes R1/R2.
7. **Promote to default** only after all gates hold together AND R1–R5 flip. Re-baseline the
   60-ROM oracle + audio corpus **visually first** (the `feedback_paired_pipeline_oracle_masking`
   discipline: never re-baseline-to-green without a visual diff). Bump the snapshot versions.

### 1.4 Per-residual closure table (effort + risk + the pinning ROM)

| Residual | Pinning oracle (pin FIRST) | Closes via | Effort | Risk | Determinism note |
|---|---|---|---|---|---|
| R1 `mmc3_test_2/4` #3 | `tests/mmc3.rs` strict probe + `irq_trace` golden | one-clock + M2-phase-representable `T_last-1` sample | **L** | **High** (the 17-rollback axis; only reachable on the new timebase) | sample-point is a phase function of `master_clock` — deterministic |
| R2 `mmc3_test_2/4` #2 / MMC6 | `tests/mmc3.rs` strict probes | same axis as R1 | **M** | High | same |
| R3 `apu_reset` len_ctrs | `tests/apu_reset.rs:113` strict | cycle-accurate reset sequence | **M** | Med (reset path touches every chip) | reset must preserve `cpu_phase` alignment (determinism contract) |
| R4 `apu_reset` 4017 | `tests/apu_reset.rs:138` strict | cycle-accurate reset sequence | **M** | Med | same |
| R5 DMC reload Y=3→4 | `CheckDMATiming` `$4010=$4E` landmark cross-diff | every-cycle-bus-access + reload-arm-invisible latch | **XL** | **Very high** (the multi-week DMA core; the make-or-break of the whole rewrite) | the entire AccuracyCoin DMA cluster + C1 trio ride this seam |

The decisive risk is **R5 / step 3-4**: the audit's Phase-3 make-or-break that *failed*
under the substrate (R1's master-clock CPU loop was found "fundamentally incompatible with
the DMASync-based DMA cluster unless the interleaved-DMA model is solved"). v2.0.0's premise
is that the **one-clock + every-cycle-access rewrite removes the incompatibility's source**
(the cumulative drift) rather than band-aiding it — which is exactly why it must be a rewrite
and not a 16th point-fix.

## 2. Determinism + breaking-change / API / save-state impact (Workstream D adjacent)

v2.0.0 is the one release allowed **breaking changes**. Concrete surface:

### 2.1 Determinism contract (must hold; this is sacred)

Same seed + ROM + input ⇒ bit-identical framebuffer + audio. The rewrite *changes cycle
timing by design* (DMC fetch cadence shifts, the reload span moves 3→4, reset becomes a
sequence). Therefore:

- **Cross-version determinism is intentionally broken at the v1.x ↔ v2.0.0 boundary.** A
  v1.5.0 `.rns` save-state, a v1.5.0 `.rnm` TAS movie, and v1.5.0 netplay rollback snapshots
  are NOT bit-compatible with v2.0.0 (the master-clock state layout and the cycle timeline
  differ). This is allowed by ADR-0003's MAJOR-boundary "best-effort" tier and is the
  textbook reason for a major bump.
- **Within v2.0.x determinism is re-fixed** under the new timebase and must be re-proven by
  the byte-identical save-state-continuation tests (the Stage-4 promotion already added
  `save_state_continuation_byte_identical_with_dmc_and_oam_dma_hot` +
  `save_state_re_encode_stable_every_frame_with_dmc_machinery_hot` — extend these to the
  one-clock state).

### 2.2 Save-state / snapshot format

- **CPU section v2 → v3:** the five-counter collapse changes which fields are load-bearing.
  Stage-4 already bumped CPU v1→v2 (`master_clock` + the `mc_*` interrupt-latch sextet).
  v2.0.0's one-clock makes `cpu.cycles`/`bus.cycle`/`apu.cpu_cycle` *derived* (drop them from
  the blob or keep as zeros for layout stability), and adds the single `parity_seed`. Bump
  CPU → v3, APU tail, BUS tail (the `ppu_clock`/`dma_mc_consumed` pair was already added and
  the Stage-4 audit found it MUST travel with `master_clock` or the restore desyncs ~11
  frames — keep that lesson).
- **`.rns` `FORMAT_VERSION` bump** (the container header) is justified: v1.x readers should
  refuse a v2.0.0 slot with `SnapshotError::UnsupportedFormat` (forward-compat already
  enforced). Per ADR-0003, **a v2.x format break warrants a new ADR (0016) defining explicit
  migration** — but the honest call for v2.0.0 is **no migration code**: a v1.x save-state
  cannot be made bit-meaningful on the new timeline, so v2.0.0 should *cleanly reject* v1.x
  slots with a clear user message rather than silently mis-restore. (Slot files are user data,
  not user output; the rewind ring is per-session anyway.)
- **`.rnm` TAS movies:** ADR-0008 keys movies on ROM SHA-256 + optional `.rns` start point +
  per-frame input. A v1.x movie replayed on v2.0.0 will desync (cycle timeline changed).
  Two options for the major bump: (a) **honest break** — v2.0.0 refuses to *verify-replay*
  a v1.x-recorded movie (still plays it as input, but the determinism guarantee is void),
  documented loudly; or (b) a **movie format-version field** + a "recorded-on-engine" tag so
  the player can warn. Recommend (a) + the warn tag; do NOT attempt timeline transcoding.

### 2.3 Public-API breaks worth bundling into v2.0.0

Because additive v1.x releases deliberately deferred cleanups, v2.0.0 is the place to:

- **Retire the legacy integer-lockstep scheduler entirely** — already done in the engine
  line (`docs/performance.md`: "the legacy integer-lockstep scheduler was removed; R1 is the
  only path"; `--no-default-features` no longer reaches a 94.24% path in the shipping crate).
  v2.0.0 formalizes removal of any remaining `cpu-c1-attempt-17-access-reorder` /
  `mc-*` dead-experiment flags (STATUS lists `cpu-c1-attempt-17-access-reorder`,
  `ppu-2002-read-end-flags`, `accuracycoin-sprite-eval-base-from-oamaddr` as dead
  experiments) and the `dmc-get-put-scheduler` / `mc-r1-*` removed-flag stubs.
- **Introduce a `VsDualSystem` top-level enum** (`Emu { Single(Nes), Dual(VsDualSystem) }`)
  — see Workstream C — which is an API addition but reshapes the frontend's single-`Nes`
  assumption (breaking for embedders of `rustynes-core`).
- **Bus-trait consolidation:** the one-clock rewrite lets the `Bus` trait drop the
  `take_dma_mc_consumed` / `dma_mc_consumed` coherence-layer methods that only existed to
  paper over the multi-counter drift. Cleaner public trait for `rustynes-core` embedders.

## 3. "Anything else for v2.0.0" — the big-ticket / breaking items (Workstreams C, D, E)

### 3.1 Full Vs. DualSystem dual-core (Workstream C)

Design fully scoped in `docs/audit/vs-dualsystem-design-2026-06-11.md` (v2.7.1 engine-line
groundwork shipped: a `vs_db.dual_system` flag for the 4 DualSystem carts + a frontend note
instead of a black screen). v2.0.0 is the natural home because:

- It's a structural addition (a `VsDualSystem { main: Nes, sub: Nes, latch: VsCommsLatch }`
  wrapper in `rustynes-core`) that reshapes the single-`Nes` frontend assumption — a
  breaking API change, hence the major bump.
- The **one-clock rewrite makes the two-core soft-lockstep cleaner**: the 5-CPU-cycle
  tolerance interleave (`RunVsSubConsole`, Mesen2) is far easier to make deterministic when
  each `Nes` advances a single master clock than when each juggles five hand-synced counters.
- Authoritative protocol (Mesen2 `VsControlManager.cpp`): the handshake is a single `$4016`
  bit-1 main/sub signal poking the *other* CPU's `/IRQ`; bit 7 read = `is_sub ? 0x80 : 0`
  (the bit RustyNES currently hard-pins to 0); independent DIP banks (`dipSwitches >> 8` for
  sub); shared-WRAM `SwapMemoryAccess`. Two PPUs → two framebuffers side-by-side.
- **Effort ~4–7 focused days (multi-session); risk Medium-High.** Riskiest: the 5-cycle
  soft-lockstep determinism with **no DualSystem test-ROM oracle** (validation is "boots +
  plays", weaker than the byte-identical gate) and the shared-WRAM swap window. Out of scope
  for the dual path: netplay (rollback assumes one state blob) and RetroAchievements (one
  memory map) — state this explicitly.

### 3.2 Long-tail mapper completion (Workstream E)

Current: **123 families at v1.5.0** (Core ~51 oracle-backed + Curated 9 + BestEffort 63),
tiered behind the ADR-0011 honesty gate (`mapper_tier_honesty.rs`: no BestEffort mapper may
back an oracle). Toward the full ~370 puNES set:

- **+UNIF loader** (the v1.6.0 "Studio" plan's E2 item) — opens a large swath of multicarts.
- **Push toward ~150–200 families** (the v1.6.0 plan targets ~150) as honesty-gated
  BestEffort, with per-mapper register-decode unit tests + staged commercial-ROM boot-smoke
  screenshots (the standing ROM-coverage policy: stage per-mapper ROMs in gitignored
  `tests/roms/external/mapper-NNN-NAME/`, commit screenshots/.snap only).
- **Effort M–L per batch; risk Low** (BestEffort is additive + honesty-gated, byte-identical
  to the oracle). This is *not* gated on the master-clock rewrite — it can land in v1.6+/v1.x
  and merely be *included* in the v2.0.0 cut, or be a v2.0.0 headline. Recommend it stays a
  v1.x feature stream and v2.0.0 simply ships whatever count is current.

### 3.3 Deprecated-path removals & architecture cleanup (Workstream D)

- Remove dead experiment flags (§2.3).
- Drop the `dma_mc_consumed` coherence layer once the rewrite obviates it.
- Collapse the FOUR cycle counters in the public/internal API to one (§1.1).
- Re-home any `idle_tick`/burn-loop public surface.
- Re-baseline `docs/scheduler.md` + `docs/architecture.md` + `docs/cpu-6502.md` to describe
  the one-clock every-cycle model (the docs currently describe the dot-lockstep `tick_one_dot`
  primer, which is already engine-lineage-historical relative to the shipping R1 core — a
  doc-debt v2.0.0 should clear).

## 4. Performance (Workstream F)

### 4.1 The measured cost of the master clock (already paid)

`docs/performance.md` headline A/B (i9-10850K, release+thin-LTO):

| Workload | Legacy integer-lockstep | R1 master clock (default) | Δ |
|---|---|---|---|
| `nes_run_frame_nestest` | 3.62 ms (276 fps) | 3.92 ms (255 fps) | **+8.14%** |
| `nes_run_frame_flowing_palette` | 2.34 ms (428 fps) | 2.49 ms (402 fps) | **+6.32%** |

**The R1 CPU core is *faster* in isolation** (`cpu_nop_step_x1000` 1.08 µs → 0.70 µs,
**−34.6%**) because R1 batches `master_clock` advancement (run-to-timestamp) instead of a
per-cycle `on_cpu_cycle` callback. The +6–8% full-frame cost is entirely bus-side:
`run_ppu_to` (master-clock PPU catch-up) + `unified_dma_cycle`. **Both configs clear the
16.639 ms NTSC deadline by 4.25–7.1×** — the ≤2 ms headless target is met on the
flowing_palette path and the nestest path is ~2× that on a 2020 desktop (well inside the CI
absolute ceiling of 10 ms).

### 4.2 The v2.0.0 perf concern: finer timebase = more ticks/frame?

The every-cycle-bus-access model adds a real bus access (dummy read) on cycles that today are
busless `idle_tick`s. Naively that's more work per frame. But:

- **The references prove it stays fast.** ares/higan use libco cooperative threads with
  `synchronize(other)` catch-up (not per-master-clock stepping); Mesen2 uses the
  `run-to-timestamp` `_state.CycleCount` model RustyNES already adopted in R1. **Event-driven
  catch-up, not naive per-clock stepping**, is the key — RustyNES already does this via
  `run_ppu_to(exact mc)` and must keep it.
- A dummy read of the held address is a single `mapper.cpu_read` (the `mapper_dispatch`
  benches measure even the dearest mapper at <1% of frame cost), so the marginal cost of
  converting `idle_tick` to a dummy read is tiny relative to the PPU dot loop (which
  dominates at ~497 µs/frame for 89,342 dots).
- **Risk:** the one-clock collapse removes the `dma_mc_consumed` batching; if the rewrite
  accidentally moves to per-master-clock stepping it could regress. Mitigation: keep the
  run-to-timestamp PPU catch-up; bench `full_frame` against a saved baseline at every gate
  (the CI absolute 10 ms ceiling + local criterion `--baseline`).

### 4.3 Perf re-baselining for v2.0.0

## 6. Ranked risk list

1. **(Very high) A2 make-or-break — the every-cycle / interleaved-DMA rewrite breaking the
   DMASync DMA cluster or the C1 trio.** This exact seam failed under the substrate
   (`v2.0-master-clock-r1-port-plan` Phase-3 FAILED: "R1 fundamentally incompatible with the
   DMASync-based DMA cluster unless the interleaved-DMA model is solved"). v2.0.0's bet is
   that the *rewrite removes the cumulative-drift source* rather than band-aiding it. If the
   bet is wrong, v2.0.0's accuracy headline (closing R5) is unshippable — bank A1 and ship
   v2.0.0 on the other workstreams.
2. **(High) Regressing AccuracyCoin off 100%.** The 100% contract is sacred; every rollback
   in the 18-month engine line came from a point-fix regressing an orthogonal surface. Only
   the full rewrite-with-gates is defensible; ADR-0002 has a standing stop-condition against
   further point-fixes.
3. **(High) The residual axis (R1/R2 MMC3 #3) is the 17-rollback graveyard.** It is only
   *reachable* on the new timebase; if A lands but the M2-phase sample still can't bracket
   sub-test #3 (1-CPU-cycle), keep it `#[ignore]`'d by-design (zero production impact) rather
   than risking a 16th rollback.
4. **(Med-High) Vs. DualSystem has no test-ROM oracle** — validation is "boots+plays", weaker
   than byte-identical; shared-WRAM swap window is where a residual black-screen lurks.
5. **(Med) Save-state/TAS/netplay cross-version break** confuses users who carry v1.x slots/
   movies — mitigate with loud rejection + ADR-0016 + release-note prominence.
6. **(Med) Perf regression if the rewrite drifts to per-master-clock stepping** — mitigate by
   keeping run-to-timestamp catch-up + the bench gate at every step.
7. **(Low) Long-tail mapper breadth** — additive, honesty-gated, lowest risk.

## 7. Top maintainer decisions (the questions to answer before scoping)

1. **Does v2.0.0 even need the master-clock rewrite?** The shipping core is *already* 100%
   AccuracyCoin with the master clock as default. The 4 real residuals (R1–R5) have **zero
   production-ROM impact** and the maintainer's standing decision is "keep deferring." So the
   honest question is: **is v2.0.0 (a) the residual-closure + one-clock rewrite, or (b) a
   "feature-major" bump (Vs.DualSystem + mapper breadth + API cleanup) that ships the
   residuals still `#[ignore]`'d by-design?** Option (b) is far lower risk and still
   legitimately a major version (Vs.DualSystem reshapes the API, save-state format can still
   break for the dual container). **Recommendation: decide (a) vs (b) FIRST — they are very
   different releases.**
2. **Sequencing: does v2.0.0 ship after the v1.6–1.9 feature/mobile releases?** The v1.6.0
   "Studio" plan (TAStudio, .fm2/.bk2 interop, Mesen2 debugger depth, mappers→~150) is
   scope-locked and executing. v2.0.0's rewrite is orthogonal to those features. Recommend:
   **let v1.6–v1.9 ship the feature/mobile work; reserve v2.0.0 for the rewrite + the
   breaking changes that only a major bump licenses.** This also lets the mapper-breadth and
   Vs.DualSystem land incrementally in v1.x and simply be *included* in the v2.0.0 cut.
3. **How aggressive is the residual-closure goal?** Three postures:
   (i) **Full** — close R1–R5 (the whole rewrite, very high risk, the A2 make-or-break).
   (ii) **Partial** — do the one-clock collapse + cycle-accurate reset (closes R3/R4/R5,
   lower risk) but leave the MMC3 #3 bracket (R1/R2) `#[ignore]`'d by-design.
   (iii) **None** — v2.0.0 is feature-major (option 1b); residuals stay deferred forever
   (they have no production impact). **Recommendation: (ii)** — it banks the architectural
   win (one clock, every-cycle access, DMC Y=4, real reset) and the by-design-defers the one
   genuinely-intractable 1-cycle bracket, which has never shipped a regression and never
   affected a real game.
4. **Save-state migration: clean-reject v1.x slots, or write migration code?** Recommend
   **clean-reject** (a v1.x timeline cannot be made bit-meaningful on the new core) + ADR-0016.
5. **Vs. DualSystem in v2.0.0 or its own v1.x point release?** It's self-contained and
   additive; it does *not* need the rewrite. Could ship as v1.8/v1.9. Recommend it ride
   v2.0.0 only if the API reshape (the `Emu` enum) is wanted as a documented breaking change;
   otherwise ship it sooner.

## 8. Reference-emulator appendix (direct source-mining findings)

### 8.1 The universal NES master-clock model (all four references agree)

| Quantity | NTSC | PAL | Dendy | Source |
|---|---|---|---|---|
| Colorburst Hz | 3,579,545.45 (= 315/88 MHz) | 4,433,618.75 | PAL colorburst | ares `ares.hpp:66-69`; higan `higan.hpp:36-40` |
| **Master clock Hz** (= colorburst × 6) | **21,477,272.73** | 26,601,712.5 | 26,601,712.5 | ares `fc/system/system.hpp:41-42` |
| **CPU divider** (mc/cycle) | **12** | **16** | **15** | ares `fc/system/system.cpp:50-73`; higan `fc/cpu/cpu.hpp:21`; Mesen2 `NesCpu.cpp:376-389` |
| **PPU divider** (mc/dot) | **4** | **5** | **5** | ares; higan `fc/ppu/ppu.hpp:21`; Mesen2 NesPpu UpdateTimings |
| APU rate | = CPU divider | = CPU divider | = CPU divider | ares `apu`; higan `fc/apu/apu.hpp:5` |
| CPU:PPU ratio | 12:4 = **3:1** | 16:5 = **3.2:1 exact** | 15:5 = 3:1 | the integer-divider expression |

**RustyNES's shipping R1 core already uses these exact dividers** (NTSC 12 / PAL 16 /
Dendy 15; PPU 4/5/5). v2.0.0 must preserve them under the one-clock rewrite — they are
hardware-correct and reference-validated.

### 8.2 Two scheduler philosophies (RustyNES already chose Mesen2's)

- **ares + higan: libco cooperative coroutines + fractional 64-bit clock accumulator.**
  Each `Thread` stores `_frequency`, `_scalar = Second/_frequency` (`Second = (u64)-1 >> 1`),
  and a 64-bit `_clock`; advance is `_clock += _scalar * clocks` (no float). Components run
  at their `rate()` and call `synchronize(other)` which `co_switch()`es until the other
  thread's clock catches up — **event-driven catch-up, never per-master-clock stepping**.
  (ares `scheduler/thread.{hpp,cpp}`; PPU per-pixel `step(rate()); synchronize(cpu)` at
  `fc/ppu/ppu.cpp:48-81` with mid-cycle NMI at exact `(ly,lx)` coords.) ares supports Dendy;
  higan does not. ares uses a 6-step APU frame-counter lookup table per mode
  (`apu.hpp:205-212`); higan a divider counter (`NtscPeriod=14915`).
- **Mesen2: run-to-timestamp `_state.CycleCount`** — `StartCpuCycle`/`EndCpuCycle` advance
  `_masterClock` and call `GetPpu()->Run(_masterClock - _ppuOffset)` so the PPU runs to the
  exact master clock after every bus op. **This is the model RustyNES R1 adopted**
  (`run_ppu_to(exact mc)`), and it is the right one to keep for performance (batched advance,
  not per-clock).

### 8.3 Mesen2 specifics (the closest model to RustyNES R1)

- **φ1/φ2 split is `_startClockCount`/`_endClockCount`** (`NesCpu.cpp:927-946`): NTSC **6/6**,
  PAL **8/8**, Dendy **7/8** (asymmetric!). The read/write asymmetry is applied as a **±1**
  around the access: `StartCpuCycle` adds `forRead ? start-1 : start+1`; `EndCpuCycle` adds
  `forRead ? end+1 : end-1` (`NesCpu.cpp:584-627`). **NOTE the divergence from RustyNES's R1
  split** (RustyNES uses `READ_PRE=5/POST=7`, `WRITE_PRE=7/POST=5`; Mesen uses `6/6 ± 1`).
  These are equivalent for the read case (5/7 vs 6-1/6+1) — a useful cross-check that
  RustyNES's split is silicon-faithful — but v2.0.0 should document the relationship and
  consider adopting Mesen's `±1` framing for clarity.
- **Every cycle is a bus access** (`NesCpu.h:65-69, 196-282`): `StartCpuCycle → bus op →
  EndCpuCycle` brackets *every* cycle; the op is one of `ExecOpCode / ExecOperand / Read /
  Write / DummyRead / DmaRead`. Addressing-mode dummy reads are explicit per-cycle bus ops
  (zero-page-indexed, abs-indexed page-cross, indirect-indexed). **This is exactly the model
  RustyNES's burn-loop `idle_tick` does NOT implement** — the v2.0.0 conversion target.
- **φ2 interrupt edge detection** (`NesCpu.cpp:591-604`): NMI edge + IRQ sampled in
  `EndCpuCycle` with a 1-cycle delay (`_prevRunIrq`/`_prevNeedNmi`) — the "second-to-last
  cycle" rule. RustyNES R1 already mirrors this (`mc_prev_run_irq`/`mc_prev_need_nmi`).
- **`$2002` vblank race** (`NesCpu.cpp:56-64`): VBL set at scanline 241 dot 1; the read races
  the edge, resolved by master-clock sub-dot precision. (RustyNES closed this — the
  `vbl_race_window_2002_read_sweep` oracle is the regression guard.)
- **Overclock** (NesPpu UpdateTimings): implemented by **extending scanlines before/after
  NMI** (`PpuExtraScanlinesBeforeNmi`/`AfterNmi`), NOT by changing the master-clock divider.
  This is the model the v1.5.0 "Enhancements" overclock (currently staged-but-inert pending
  v2.0) should adopt — it becomes *implementable* on the v2.0.0 master clock.
- **OAM/DMC DMA**: master clock is NOT frozen during DMA (runs coherently); `getCycle =
  (CycleCount & 1)`; a halt consumes the full cycle's master ticks. (RustyNES's legacy froze
  it — the source of the five-counter drift the rewrite removes.)

### 8.4 TriCNES specifics (the every-cycle / invisible-reload-arm model)

- **`CPUClock` counts to 12, `PPUClock` to 4, `MasterClock` to 24** (`Emulator.cs:456-458`).
  NTSC-only (no PAL 3.2:1 — hardcoded 12/4).
- **The per-master-clock loop ordering** (`_EmulatorCore`, `Emulator.cs:868-937`) is the
  load-bearing detail RustyNES's R5 (DMC Y=3→4) closure depends on:
  `_6502()` (at CPUClock==12) → NMI sample (CPUClock==4, φ1) → IRQ + M2 rise (CPUClock==7,
  φ2) → `_EmulatePPU()` (PPUClock==4) → `_EmulateAPU()` + `APU_PutCycle = !APU_PutCycle`
  (CPUClock==0). Because `_EmulateAPU` (which arms the DMC reload, `1158-1184`) runs **after**
  `_6502()`, **the reload arm is invisible to its own cycle** → first DMA service lands on the
  *next* cycle (PUT parity) → span 4. This is precisely the `pending_dmc_dma_next` latch
  RustyNES needs (timing-map §4); the agent confirms the mechanism at the source.
- **Every operationCycle is a Fetch or Store** (`operationCycle`, `Emulator.cs:503`;
  `FetchPRG`/`StorePRG` mapper interface `123-145`) — the every-cycle-bus-access model.
- **Sprite-0 hit is delayed 1.5 dots** via a two-stage pending queue
  (`PendingSpriteZeroHit → PendingSpriteZeroHit2 → SpriteZeroHit`, `Emulator.cs:1387-1389,
  3351-3359`) — the "stale shifter" sub-dot behavior the brief lists. RustyNES already models
  the sprite-0 timing (AccuracyCoin Sprite-0 Hit passes); v2.0.0's finer timebase only makes
  the sub-dot delay cleaner to express.
- **`$2002` VBL dual-latch** (`PPU_VSET_Latch1/2`, `Emulator.cs:1393-1405`) models sub-dot
  propagation delay.

### 8.5 The single design takeaway for RustyNES v2.0.0

The references split into two camps on *how* master-clock advance couples to bus work:
**Mesen2 couples the master-clock delta to the bus-operation type** (read vs write ±1);
**TriCNES decouples it into fixed countdown dividers** with hardcoded sample offsets
(CPUClock==0/4/7). RustyNES R1 is closer to Mesen2. For v2.0.0 the recommendation is to stay
on the **Mesen2 run-to-timestamp model** (which RustyNES already has and which keeps the
perf characteristics) but adopt **TriCNES's "reload arm invisible to its own cycle"** ordering
and the **every-cycle-bus-access** discipline both share — that combination is what closes
R5 (DMC Y=4) and lets R1/R2/R3/R4 fall out, without the libco rewrite ares/higan would imply.
