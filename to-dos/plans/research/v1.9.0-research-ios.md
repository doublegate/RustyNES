# RustyNES v1.9.0 "iOS / iPadOS App" — Research & Release Plan (READ-ONLY)

Status: research deliverable, no code changed. Date: 2026-06-17.
Counterpart to the sibling v1.8.0 Android plan (shared Rust-on-mobile core; this doc
is Apple-specific). Scope: port the pure-Rust RustyNES NES/Famicom emulator to a
complete, shippable iOS + iPadOS application.

---

## 0. TL;DR verdict

**Feasible and de-risked by precedent.** As of April 2024 Apple's App Store Review
Guidelines explicitly permit retro game console emulators worldwide, and a Rust-free
NES emulator (**Delta**) has shipped on the store and topped the charts. RustyNES is a
**pure interpreter** — it needs **no JIT**, which is the one capability iOS withholds
from third-party apps; so the single hardest iOS-emulator constraint simply does not
apply to NES emulation. The Rust stack (winit/wgpu→Metal, cpal→CoreAudio) already has
working iOS support, the determinism contract is platform-independent, and the existing
wasm touch-overlay design is directly reusable.

**Recommended architecture: Option (b) — a thin Rust core library behind a native
SwiftUI shell, bridged with UniFFI, with the emulator video rendered into a
`CAMetalLayer`/`MTKView` via wgpu.** This trades maximal code reuse for a native,
App-Store-grade UX (Files import, controllers, haptics, scene lifecycle, iPadOS
multitasking) that winit-on-iOS cannot deliver well. Reuse stays high because RustyNES
already isolates a `no_std`+alloc core (`rustynes-core`) from the `std` frontend.

## 1. Architecture decision

### The three options

**(a) Reuse `rustynes-frontend` (winit + wgpu + egui) on iOS verbatim.**
winit *does* have an iOS/UIKit backend, wgpu targets Metal, and egui can render on the
wgpu surface. Maximal code reuse — in principle the existing frontend "just runs."

- Pros: lowest initial effort to first-boot; one frontend codebase; egui menus/debugger
  come free.
- Cons (decisive): the Rust-on-iOS ecosystem consensus in 2026 is that winit's mobile
  story is weak — developers "generally prefer to render graphics within a local area
  of the app rather than directly using winit to take over the entire app window"
  (winit DeepWiki / community devlogs). An egui-only UI feels alien on iOS (no native
  navigation, share sheet, document picker, Dynamic Type, Stage Manager behaviors,
  VoiceOver). App Review and users both penalize non-native chrome. Lifecycle
  edge-cases (scene background/foreground, drawable loss, audio interruption) are
  awkward to handle through winit's abstraction. No path to a polished iPad experience.

**(b) Rust core library behind a SwiftUI UI via a Swift/C FFI bridge, with a
`MTKView`/`CAMetalLayer` game surface. ← RECOMMENDED.**
Ship `rustynes-core` (+ a thin `rustynes-ios` shim crate) as a `staticlib`/XCFramework;
drive it from Swift. The NES framebuffer is uploaded to a Metal texture and blitted in
a `CAMetalLayer` (either through wgpu writing into a caller-supplied layer, or a tiny
hand-written Metal blit shader). All chrome — menus, ROM library, settings, controller
config — is SwiftUI.

- Pros: native iOS/iPadOS feel; clean Files/share-sheet/iCloud import; GameController +
  Core Haptics + AVAudioSession handled idiomatically in Swift; scene lifecycle correct;
  the deterministic core is reused **byte-for-byte** (it already compiles `no_std`+alloc
  and is the determinism oracle), so cross-play with desktop/Android netplay is preserved.
- Cons: the egui debugger/settings UI does NOT port — those would be re-authored in
  SwiftUI (or omitted from the consumer app and kept desktop-only). A defined FFI surface
  must be maintained. Two UI codebases (desktop egui + iOS SwiftUI).

**(c) Hybrid: native SwiftUI shell hosting an embedded wgpu+egui surface for the
*game + optional debugger*, SwiftUI for everything else.**
- Pros: reuses egui debugger panels inside a SwiftUI app (debugger as an opt-in "pro"
  view); native shell for library/settings/import.
- Cons: two render/event systems coexisting (egui pointer events vs SwiftUI gestures),
  more lifecycle plumbing, larger binary. Reasonable as a *later* enhancement, not the
  v1.9.0 baseline.

### Recommendation & rationale

**Adopt (b).** It is the architecture every shipping Rust-core iOS app converges on
(Firefox/Mozilla via UniFFI, Ferrostar, LiveKit, Bitwarden-style cores). RustyNES is
already shaped for it: the `rustynes-core` ⇄ `rustynes-frontend` split is exactly the
"deterministic core / platform shell" boundary an FFI bridge wants. We get a native App
Store-quality product and keep the determinism contract intact. Treat **(c)** as a
post-1.9 option (ship the egui debugger as an embedded "Developer" tab once the native
app is stable). Reject **(a)** as the product baseline — keep it only as a throwaway
"does the core even run on-device" smoke test (it is the fastest path to *a* pixel on a
simulator and useful as a Milestone-0 spike).

### iPadOS-specific UX (designed in from day one under Option b)

- **Larger canvas / split-view & Stage Manager:** SwiftUI scenes resize; the Metal
  game view letterboxes to the NES 8:7 PAR (same `letterbox` math as `gfx.rs`) in any
  window size; the touch overlay scales. Support multiple scenes (Stage Manager) but
  pause non-foreground emulation.
- **Hardware keyboard:** map to the existing default keybinds via `UIKeyCommand` /
  SwiftUI `.keyboardShortcut`, routed to the core's controller state. A natural fit for
  iPad-with-Magic-Keyboard.
- **Trackpad/pointer:** drives Zapper/SNES-mouse/Arkanoid Vaus devices (the core already
  models pointer devices).
- **Picture-in-Picture / external display:** out of scope for v1.9.0 (note as future).

## 3. The JIT constraint — confirmed NON-ISSUE (a key "iOS is fine" point)

iOS forbids third-party apps from generating executable pages with an invalid code
signature — i.e. **no JIT / no W^X violation**. This is what historically crippled
*high-end* console emulators on iOS (Wii/Switch/GameCube want a recompiler; the 2025
"JIT enabler / StikDebug" saga is entirely about restoring JIT for those).

**RustyNES needs none of it:**

- The CPU/PPU/APU core is a **pure cycle-stepped interpreter** (`tick_one_dot()` /
  `Cpu::tick`) — no dynamic code generation anywhere. NES at 1.79 MHz is trivially fast
  enough interpreted; the core targets ≤ 2 ms/frame headless on a 2018 laptop, so an
  A-series/M-series chip has enormous headroom.
- **Lua scripting** (`mlua`, Lua 5.4) executes **interpreted bytecode** — no JIT (it is
  *not* LuaJIT). The experimental wasm Lua backend (`piccolo`) is **pure Rust**, also no
  JIT. Both are fine under iOS rules. (Lua via `mlua` would link the C Lua interpreter
  statically — acceptable; it generates no code at runtime.)
- wgpu→Metal shaders are compiled by Apple's Metal toolchain at build/load time, not
  app-generated machine code — not JIT.

**Conclusion:** the entire feature set ports without any entitlement for code
generation. No feature in RustyNES requires JIT. This is a decisive feasibility point —
iOS's headline emulator restriction is irrelevant to an NES interpreter, and Delta
(also interpreter-based for its NES core) proves it ships.

## 5. App lifecycle + storage

- **Scene lifecycle:** SwiftUI `ScenePhase` (`.active`/`.inactive`/`.background`).
  On background: pause the emu loop (`CADisplayLink` invalidate), stop the audio engine,
  drop the Metal drawable. On foreground: rebuild surface, resume. Auto-pause matches the
  desktop "pause when unfocused" behavior.
- **Audio interruptions:** AVAudioSession interruption notifications → pause/resume; route
  changes → reconfigure stream.
- **ROM import (NEVER bundle commercial ROMs — user-provided only):**
  - `UIDocumentPicker` / `.fileImporter` from the **Files app**, share-sheet "Copy to
    RustyNES", **iCloud Drive**, and `.nes`/`.fds`/`.nsf`/`.zip` UTType registration so
    RustyNES appears as an "Open in" target. Copy imported ROMs into the app sandbox
    (`Application Support`), keyed by SHA-256 (same identity scheme as desktop saves).
    This is exactly Delta's documented "+ → Files" import flow.
- **Save-states / SRAM:** app sandbox (`Application Support/RustyNES/saves/<sha256>/`),
  same `.rns` format as desktop (the format is platform-independent → cross-device save
  portability). Battery-backed SRAM likewise.
- **Optional iCloud sync:** `NSUbiquitousKeyValueStore` for small config, iCloud Drive /
  CloudKit for save-states (future enhancement; ship local-first in v1.9.0).
- **Privacy:** declare a privacy manifest (`PrivacyInfo.xcprivacy`); the app collects no
  data → trivially clean nutrition labels (unless RetroAchievements login is enabled,
  which is account auth to a third party — disclose accordingly).

## 7. App Store policy + legal (CRITICAL feasibility analysis)

**The April 2024 guideline change is the green light.** Apple updated **App Review
Guideline 4.7** in April 2024 to allow **retro game console emulator apps worldwide**,
explicitly stating such apps "can offer to download games" (later, Aug 2024, extended to
PC emulators). This reversed Apple's long-standing rejection of emulators. The practical
proof: **Delta — a multi-system emulator with an NES core — launched on the US App Store
April 17, 2024 and hit #1.** RustyNES is squarely in the now-permitted category.

**The constraints RustyNES must satisfy (and does):**

1. **No bundled copyrighted ROMs.** Ship zero commercial ROMs. User-provided content
   only, imported via Files/share-sheet (§5). The repo policy already forbids committing
   commercial ROMs — extend that to "never bundle in the .ipa." Optionally bundle only
   CC0/public-domain/homebrew demos as built-in content.
2. **"You are responsible for the software" / copyright clause (4.7).** Developer is
   responsible for ensuring the app and any downloadable content follow the law. RustyNES
   downloads nothing; it only runs files the user already has. Lowest-risk posture: no
   in-app ROM browser/downloader, no links to ROM sites, a clear in-app notice that users
   must own the games.
3. **No in-app download of games** (avoid 4.7's IAP-for-digital-items entanglement) →
   simplest compliance: RustyNES is a runner only.
4. **Age rating:** NES content → likely 12+/17+ depending on the rating questionnaire
   (cartoon/fantasy violence). RustyNES itself is a tool; rate conservatively.
5. **Privacy nutrition labels / privacy manifest:** none if no telemetry/accounts. If
   RetroAchievements login ships, disclose account + network usage.
6. **Review-risk history:** emulators were historically rejected and even now occasionally
   get extra scrutiny; the *category* is allowed but reviewers check for (a) bundled
   copyrighted content, (b) download mechanisms, (c) misleading metadata. A clean
   "bring-your-own-ROM runner" with no download path is the well-trodden, low-risk path
   (Delta, RetroArch-style apps). EU note: at launch Delta used the App Store in most
   regions; alt-marketplace/notarization is a separate EU track — not required for the
   standard App Store path.

**Verdict:** App Store distribution is **feasible and precedented.** The legal posture is
"user-provided ROMs, no bundled content, no download mechanism, clear ownership notice."
This is the single biggest risk historically, and it is now a *managed, well-understood*
risk rather than a blocker.

## 8. Completion definition — milestones

- **M0 — Spike (does the core run on-device):** build `rustynes-core` for
  `aarch64-apple-ios-sim`; minimal Option-(a) winit+wgpu run OR a trivial Swift app that
  calls `run_frame()` and blits the framebuffer in a `MTKView`. First pixel in Simulator.
- **M1 — First boot in Simulator (Option b):** UniFFI bridge crate + XCFramework; SwiftUI
  app boots a homebrew `.nes`, blits via Metal at 60 Hz, CoreAudio output. Touch D-pad.
- **M2 — On-device:** code signing + provisioning; runs on a physical iPhone + iPad;
  GameController support; AVAudioSession interruptions handled; scene lifecycle correct.
- **M3 — Feature parity (core product):** Files/share-sheet/iCloud ROM import; save-states
  + rewind + run-ahead; cheats; shaders; touch overlay (incl. Power Pad) + hardware
  keyboard + controller; iPadOS layout (split-view/Stage Manager); haptics.
- **M4 — TestFlight beta:** fastlane match + GitHub Actions macOS CI → `.ipa` →
  TestFlight internal, then external testers; crash-free-rate + perf telemetry (or manual).
- **M5 — App Store release:** privacy manifest, age rating, screenshots, review-safe
  metadata (no ROM references), submit; address review feedback; ship v1.9.0.

## 9. Workstream breakdown (with effort, tools/versions, risks)

> Effort key: S ≈ days, M ≈ 1–2 wks, L ≈ 3–5 wks, XL ≈ 6+ wks (solo, part-time).

### WS-A — Foundation: build + bridge + signing (L)
- Add `crates/rustynes-ios` (`staticlib`) exposing the UniFFI surface over `rustynes-core`.
- `rustup target add aarch64-apple-ios aarch64-apple-ios-sim`; UniFFI + `cargo swift`;
  `xcodebuild -create-xcframework`. SwiftUI Xcode project consuming the XCFramework.
- Apple Developer Program, App ID, distribution cert, provisioning profiles; **fastlane
  match** (certs in private git). GitHub Actions `macos-latest` + fastlane (gym/match).
- **Risks:** wasm-bindgen-style version coupling (UniFFI Rust lib vs generated Swift must
  match); XCFramework arch/lipo mistakes; macOS CI 10× minute cost; provisioning churn.

### WS-B — Metal rendering + surface (M)
- wgpu Metal surface from `CAMetalLayer`, or a hand-written Metal nearest-filter blit of
  the 256×240 RGBA texture. Letterbox to 8:7 PAR (reuse `gfx.rs` math). `CADisplayLink`
  pacing honoring ProMotion 120 Hz; emulate at console rate, present 1:1, DRC absorbs beat.
  Surface recreate on resize/scale/drawable-loss.
- **Risks:** ProMotion vs 60.0988 Hz beat (existing display-sync logic mitigates); drawable
  lifecycle on background; wgpu-into-external-CAMetalLayer ergonomics (may favor hand blit).

### WS-C — CoreAudio (M)
- cpal CoreAudio host (iOS, AVAudioSession via `objc2-avf-audio`); reuse SPSC ring + Hermite
  DRC. AVAudioSession category/activation, interruption + route-change + silent-switch +
  background handling, small `preferredIOBufferDuration` for low latency.
- **Risks:** interruption edge-cases causing audio death; buffer-size negotiation; latency
  vs underruns on older devices. **Determinism note:** DRC is a frontend resampler stage —
  core samples untouched; cross-play/audio-oracle preserved.

### WS-D — Touch + controller input (M)
- SwiftUI translucent touch overlay (port the wasm overlay *design*: D-pad/A/B/Start/Select
  + Power Pad mat + target-port selector). GameController (`GCController`) discovery in
  Swift; hardware keyboard (`UIKeyCommand`) + trackpad pointer (Zapper/mouse/Vaus).
- **Determinism note:** all inputs converge on the same `set_controller` late-latch bit-mask
  as desktop/wasm → TAS/netplay-identical.
- **Risks:** controller hot-plug/disconnect handling; gesture-vs-overlay conflicts; on-device
  touch ergonomics (Delta-style verified manual carryover).

### WS-E — Storage + lifecycle (M)
- Document picker / share-sheet / iCloud import; sandbox storage keyed by SHA-256;
  `.rns` save-states + SRAM in `Application Support`; ScenePhase pause/resume; privacy
  manifest. (iCloud save sync = future.)
- **Risks:** security-scoped URL handling for imported files; iCloud entitlement complexity;
  background-task limits.

### WS-F — Feature port (L)
- Save-states/rewind/run-ahead, cheats, shaders, TAS `.rnm`, (optional) Lua console,
  (optional) native RetroAchievements (rc_client arm64 + keychain). **Netplay deferred.**
- **Risks:** shader perf on-device; Lua/RA scope creep — gate both behind "ship-later" flags.

### WS-G — App Store release (M)
- TestFlight (App Store Connect API key in CI); privacy labels; age rating; review-safe
  metadata (no ROM references, no download path, ownership notice); screenshots (iPhone +
  iPad); submit + iterate on review.
- **Risks:** review rejection over perceived ROM facilitation; metadata wording; first-app
  account/agreement setup latency.

## 11. Ranked risk list

1. **App Store review rejection** over perceived copyrighted-ROM facilitation — *managed*
   (no bundled ROMs, no download path, ownership notice; Delta precedent). Highest-impact,
   now low-probability with discipline.
2. **macOS CI cost + signing/provisioning friction** (10× minutes, fastlane match,
   App Store Connect setup) — recurring operational drag; gate iOS CI to tags/manual.
3. **Two UI codebases (SwiftUI iOS + egui desktop) divergence** — feature drift, double
   maintenance for settings/menus; mitigate by keeping the FFI surface = the `Nes` API and
   not re-implementing core logic in Swift.
4. **Audio interruption / lifecycle bugs** (AVAudioSession, drawable loss, background) —
   classic iOS pain; needs on-device testing the headless CI can't do.
5. **Netplay on mobile networks** (NAT/CGNAT, background limits) — why it's deferred.
6. **ProMotion 120 Hz pacing beat** vs 60.0988 Hz — mostly mitigated by existing
   display-sync/DRC, but needs on-device validation across refresh rates.
7. **UniFFI/XCFramework version coupling** (Rust lib vs generated Swift, arch packaging) —
   a CI gotcha mirroring the desktop wasm-bindgen pin discipline.
8. **On-device performance of shaders / older devices** — measure-first; ship plain blit
   baseline.

## 12. Top maintainer decisions

1. **Confirm architecture Option (b)** (SwiftUI + UniFFI + Metal) vs a winit-reuse spike
   only — recommended (b).
2. **UniFFI vs cbindgen** for the bridge — recommended UniFFI (`cargo swift`).
3. **wgpu-into-CAMetalLayer vs a hand-written Metal blit** for the game surface — pick
   after the M0 spike (hand blit may be simpler/lighter for a single textured quad).
4. **v1.9.0 scope cut:** ship **without** netplay (and likely without native RA + Lua) —
   confirm what's in-beta vs deferred.
5. **Apple Developer account ownership** (personal vs an org), bundle ID, and whether
   RetroAchievements/accounts ship at launch (drives privacy labels).
6. **Shared mobile foundation with the v1.8.0 Android plan:** factor the FFI/bridge crate
   (`rustynes-ios` / `rustynes-android`) and the touch-overlay design so both mobile
   targets reuse one core-binding layer.
7. **CI budget:** accept macOS 10× minutes and gate the iOS pipeline to tags + manual
   dispatch (mirror the existing PGO/release gating pattern).
