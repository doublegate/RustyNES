# RustyNES v1.7.0 Research — the NEXT tier of Creator/TAS/Scripting/Tooling (BizHawk + FCEUX)

**Date:** 2026-06-17
**Author task:** READ-ONLY mining of `ref-proj/BizHawk/` (newly added, emphasized) + `ref-proj/fceux/` (+ `ref-proj/tetanes/` for Rust comparisons, `nesdev_wiki/`) for the *next tier* of creator/TAS/scripting/tooling features RustyNES should consider for **v1.7.0** — the release AFTER the in-dev v1.6.0 "Studio" and BEFORE the v2.0 master-clock refactor.

**Why a "next tier" exists.** v1.6.0 "Studio" is scope-locked MAXIMAL (decisions resolved 2026-06-17). It already absorbs the *first* tier of BizHawk/FCEUX tooling: a TAStudio-class piano-roll editor incl. **A4** branches/`.rnmproj`/multitrack/patterns; `.fm2`+`.bk2` movie interop; Lua `frameadvance`/coroutine driving + memory domains/sized reads + a planned `movie.*`/`joypad` surface; Mesen2-class conditional breakpoints/watchpoints + a full hex editor + RAM Watch/Search; A/V recording; FCEUX Code/Data Logger; FDS-proper; mappers→~150 + UNIF; HD-audio; shaders. So this report is deliberately **the residue** — what BizHawk/FCEUX still do that survives the v1.6.0 absorption, plus the depth layers (scriptable TAStudio, IPC, broad format interop, firmware/multi-disk management, ROM-hacking helpers, savestate-engine compression) a serious RustyNES creator/TAS user expects once the v1.6.0 spine exists.

## Workstream A — Scriptable TAStudio (the `tastudio.*` Lua surface)

v1.6.0 builds the *editor*; BizHawk's `TAStudioLuaLibrary` (28 KB) exposes the editor to **scripts**, which is what powers bots, generated-input TASes, and visualization mods. This is the natural "depth" layer once the v1.6.0 piano roll exists.

### A1. `tastudio.*` core control API — **M, frontend-only, determinism-safe**
- **Source:** `BizHawk/src/BizHawk.Client.EmuHawk/tools/Lua/Libraries/TAStudioLuaLibrary.cs`.
- **What:** `engaged()`, `getrecording`/`setrecording`/`togglerecording`, `setplayback(frame|markerName)`, `getseekframe()`, `getselection()`, `islag(f)`/`setlag(f,bool)`, `hasstate(f)`, marker get/set/remove, `submitinputchange`/`applyinputchanges` (the queued atomic-edit pattern), `getbranches`/`getbranchinput`/`loadbranch`, `setbranchtext`.
- **Why v1.7.0:** the v1.6.0 editor is GUI-only. This makes it *programmable* — the single biggest unlock for bots and generated TASes; the v1.6.0 A2/A4 model (input log + branches + markers + lag log) is the exact data this API exposes, so it's "wire the existing model to Lua," not new engine work.
- **Gap today:** RustyNES Lua has no `tastudio.*` namespace at all (`docs/scripting.md` — flat `emu.*` + `memory`/`cart`/`sym` tables, no movie/TAS surface).
- **Determinism/flag:** gated like `emu.write` (no-op under netplay/replay/RA-hardcore for the mutators); behind `scripting`. NOT a v2.0 item (frontend orchestration over the deterministic replay path).

### A2. `tastudio` custom-column + per-cell draw callbacks — **M, frontend-only**
- **Source:** same file — `onqueryitembg`/`onqueryitemtext`/`onqueryitemicon` + `clearIconCache`.
- **What:** let a script add visualization to the piano-roll grid: per-row/per-column background color, custom text, and icon overlays (cached `.ico`). The de-facto way TASers annotate runs (RNG state, subpixel position, route notes) directly on the grid.
- **Why v1.7.0:** turns the grid into a programmable analysis canvas — high value for serious TAS authoring, and a natural extension of the v1.6.0 marker/note system. egui-native (custom cell draw in the piano-roll `TableBuilder`).
- **Gap today:** none (no piano roll until v1.6.0; no scripting hook into it ever).
- **Determinism/flag:** pure overlay (no state mutation); `scripting`. NOT v2.0.

### A3. `tastudio` greenzone/branch event callbacks — **S, frontend-only**
- **Source:** same — `ongreenzoneinvalidated(fn)`, `onbranchload(fn)`.
- **What:** fire a Lua callback when an edit invalidates the greenzone (passing the first modified frame) or when a branch loads. Lets analysis scripts re-derive state lazily and keep custom data structures in sync with edits.
- **Why v1.7.0:** the event the v1.6.0 `InvalidateAfter(frame)` plumbing already raises internally — exposing it is cheap and closes the scriptable-editor loop.
- **Determinism/flag:** observational; `scripting`. NOT v2.0.

## Workstream B — Broad movie-format interop (beyond `.fm2`/`.bk2`)

v1.6.0 does `.fm2` + `.bk2`. BizHawk's `movie/import/` directory imports **15+ formats** via a uniform `IMovieImport` dispatch — and several are NES-relevant or trivially adjacent. This is the "TASVideos archaeology" layer: the historical NES TAS corpus predates `.fm2`/`.bk2`.

### B1. FCEUX `.fcm` (legacy FCE Ultra movie) import — **M, frontend-only**
- **Source:** `BizHawk/.../movie/import/FcmImport.cs`; FCEUX `movie.cpp` legacy path.
- **What:** the pre-`.fm2` binary FCEUX movie format. A large slice of early NES TASes on TASVideos are `.fcm`-only.
- **Why v1.7.0:** completes the FCEUX-lineage import story `.fm2` started; lets RustyNES replay/convert the legacy corpus. Same parse-then-replay path as the v1.6.0 `.fm2` importer.
- **Gap today:** RustyNES reads only `.rnm` (+ `.fm2`/`.bk2` after v1.6.0).
- **Determinism/flag:** same canonical-power-on-seed gotcha as v1.6.0 `.fm2` (decision #5 fixed alignment applies); additive format path; NOT v2.0.

### B2. Famtasia `.fmv` + VirtuaNES `.vmv`/`.fcm`-family + `.mc2` (Mednafen NES) import — **M overall, frontend-only**
- **Source:** `FmvImport.cs`, `Mc2Import.cs`, `VbmImport.cs`-style adjacent; nesdev/TASVideos format notes.
- **What:** the other historical NES movie containers (Famtasia, VirtuaNES, Mednafen). Each is a small binary header + per-frame button bitfield.
- **Why v1.7.0:** these are the formats the *oldest* published NES TASes use; importing them makes RustyNES a credible "play any NES TAS" tool. Each is S individually; the value is the aggregate corpus reach.
- **Determinism/flag:** additive readers; same alignment gotcha; NOT v2.0.

### B3. Movie *export* to `.fm2` round-trip hardening + a generic exporter — **S, frontend-only**
- **Source:** FCEUX `fm2.txt`; BizHawk `MovieConversionExtensions.cs`.
- **What:** v1.6.0 has `.fm2` import/export; v1.7.0 hardens the export (rerecord count, author comment convention, MD5 `romChecksum`, `palFlag`, fourscore layout) and adds a "verify on TASVideos" doc note. Lossy-hash handling (RustyNES SHA-256 vs `.fm2` MD5 vs `.bk2` SHA-1 — store/recompute both).
- **Why v1.7.0:** the contribute-back direction is what makes RustyNES a *submission* tool, not just a player. Small once v1.6.0 lands the writer.
- **Determinism/flag:** file I/O; NOT v2.0.

*(Note: the Doom/Heretic/Hexen `.lmp`, Genesis `.gmv`, SNES `.smv`/`.lsmv`, GBA `.vbm` importers in `movie/import/` are out of scope — non-NES.)*

## Workstream C — IPC / external-tool surface (`comm.*` + external tools)

This is the cluster v1.6.0 explicitly tabled ("`comm`/sockets, `SQL`, `forms`, `client`/canvas — lower ROI"). It is exactly the v1.7.0 power-user tier: it turns RustyNES into a *platform* for external bots, RL agents, stream tools, and randomizers.

### C1. `comm.*` IPC — TCP socket + HTTP + WebSocket + memory-mapped-file — **L, frontend-only**
- **Source:** `BizHawk/src/BizHawk.Client.Common/lua/CommonLibs/CommLuaLibrary.cs` (+ `Api/SocketServer.cs`, `HttpCommunication.cs`, `WebSocketServer.cs`, `MemoryMappedFiles.cs`); FCEUX `documentation/protocol.txt` (the net-server).
- **What:** the full external-process bridge — `socketServerSend/Response`, `httpGet/Post`, `ws_open/send/receive/close`, `mmfWrite/Read/CopyFromMemory/CopyToMemory`. This is how BizHawk drives modern ML/RL training (e.g. the entire "emulator as gym env" use case), live randomizers, and cross-process stream overlays.
- **Why v1.7.0:** RustyNES's Lua sandbox **deliberately removes** `io`/`os`/`package`/network (`docs/scripting.md` §Determinism+safety), so scripts cannot talk to the outside world *at all* today. A *frontend-mediated*, off-by-default `comm.*` (the script never gets raw sockets; the host owns the connection) preserves the sandbox guarantee while unlocking the agent/bot ecosystem. RustyNES's determinism is a *selling point* for RL (reproducible episodes).
- **Gap today:** confirmed zero IPC; total sandbox isolation.
- **Determinism/flag:** **must** be host-mediated + off-by-default behind a new feature (e.g. `script-ipc`), and flagged "non-deterministic input source — disabled in replay/netplay/RA." The *core* synthesis never sees it. NOT a v2.0 item.

### C2. `client.*` automation verbs (tool open/close, window/screenshot, speed) — **M, frontend-only**
- **Source:** `BizHawk/.../lua/CommonLibs/ClientLuaLibrary.cs` — `opentasstudio`/`openhexeditor`/`openramwatch`/`openramsearch`/`opentracelogger`/`opentoolbox`, `screenshot`/`screenshottoclipboard`, `setwindowsize`, `speedmode`/`frameskip`, `reboot_core`, `pause_av`/`unpause_av`, `getversion`, `addcheat`/`removecheat`/`clearautohold`.
- **What:** scripted control of the *host*, not just the emulator — open tools, drive A/V capture, resize/screenshot, set turbo. The glue that lets a single script orchestrate a full capture/analysis session.
- **Why v1.7.0:** pairs with the v1.6.0 A/V recording and the C1 IPC to make scripted, reproducible capture pipelines (record N frames → screenshot → POST result). All frontend.
- **Gap today:** RustyNES Lua has `emu.pause`/`saveState`/`loadState`/`setInput` but no host/tool/screenshot/AV verbs.
- **Determinism/flag:** frontend control; `scripting`. NOT v2.0.

### C3. SQLite + userdata persistence (`SQL.*`, `userdata.*`) — **S, frontend-only**
- **Source:** `SQLiteLuaLibrary.cs`, `UserDataLuaLibrary.cs`.
- **What:** scripts can persist a KV store (`userdata`) across runs and query a SQLite DB (`SQL`) — used for large RAM-map databases, randomizer logic tables, and bot training logs.
- **Why v1.7.0:** low-cost, high-leverage for serious tool scripts; `userdata` alone (a per-script persisted table) is an S win that many randomizers need. SQLite is optional/heavier.
- **Determinism/flag:** off the synthesis path; `userdata` could persist into save-states (FCEUX `loadscriptdata`); `scripting`. NOT v2.0.

## Workstream D — ROM-hacking & cart helpers

FCEUX ships a suite of ROM-hacking tools RustyNES lacks entirely; these are the "I'm a hacker/translator, not a TASer" audience and complement the v1.6.0 CDL/hex editor.

### D1. iNES / NES 2.0 header editor — **M, frontend-only**
- **Source:** `fceux/src/drivers/Qt/iNesHeaderEditor.cpp` (48 KB).
- **What:** a GUI to inspect/edit the 16-byte iNES + NES 2.0 header of a ROM file — mapper number, mirroring, PRG/CHR sizes, battery, region timing, submapper, PRG/CHR-RAM sizes, console type. ROM-hackers and dumpers use it constantly to fix bad headers.
- **Why v1.7.0:** RustyNES *parses* the header (`docs/cartridge-format.md`) but offers no edit/inspect UI. Pairs naturally with the v1.6.0 UNIF loader and the mapper-DB work; a read-only "Cartridge Info" pane is even an S subset worth shipping first.
- **Gap today:** no header inspect/edit UI; `cart:mapper_id()`/`prg_size()`/`chr_size()` exist in Lua only.
- **Determinism/flag:** edits a *file on disk* (not the running core); off-by-default write, read-only-by-default UI. NOT v2.0.

### D2. Game Genie encoder/decoder UI + `.tbl` text tables — **S, frontend-only**
- **Source:** `fceux/src/drivers/Qt/GameGenie.cpp`; FCEUX `.tbl` char-map support in `HexEditor.cpp`.
- **What:** (a) a bidirectional Game Genie code ↔ address/value/compare converter UI (RustyNES already *validates* `GenieCode` internally — just needs the encode-from-raw direction + a UI); (b) `.tbl` character tables that remap the hex editor's ASCII column to a game's custom font (essential for translation hacking).
- **Why v1.7.0:** both are small, high-visibility ROM-hacker affordances on top of the v1.6.0 hex editor. The `.tbl` loader is the standard translation-hacking tool.
- **Gap today:** Game Genie is decode/apply-only (cheat path); no encoder UI; no `.tbl`.
- **Determinism/flag:** frontend; cheats already deterministic via the post-frame poke path. NOT v2.0.

### D3. "Save stripped/unused ROM" + CDL-driven disassembly export — **S, frontend-only (rides v1.6.0 CDL)**
- **Source:** `fceux/src/drivers/Qt/CodeDataLogger.cpp` (25 KB).
- **What:** once the v1.6.0 CDL coverage arrays exist, FCEUX can emit a ROM with unlogged (unreached) bytes stripped, and feed "log only newly-mapped code" into the trace logger / disassembler. Finds dead space for ROM expansion.
- **Why v1.7.0:** a cheap, valuable *consumer* of the v1.6.0 CDL that v1.6.0 itself may not ship (v1.6.0 builds the logger; this is the export/analysis layer).
- **Determinism/flag:** output-only over the existing CDL side-array; `debug-hooks`. NOT v2.0.

## Workstream E — Firmware / multi-disk / batch management

BizHawk's "boring but expected" infrastructure that a serious multi-game/FDS workflow needs.

### E1. Firmware Manager (FDS BIOS + future BIOS) — **M, frontend-only**
- **Source:** `BizHawk/src/BizHawk.Client.Common/fwmanager/FirmwareManager.cs`.
- **What:** a registry that locates, hash-verifies, and resolves required firmware (for NES: the **FDS BIOS** `disksys.rom`), with a UI showing "have / missing / wrong-hash." BizHawk refuses to boot FDS without a verified BIOS and tells you exactly which file it needs.
- **Why v1.7.0:** pairs directly with the v1.6.0 **FDS-proper** workstream — FDS-proper needs a real BIOS, and a firmware manager is the honest way to source/verify it (vs. silently failing). A hash-checked BIOS resolver also future-proofs any later add-on.
- **Gap today:** FDS BIOS handling is ad-hoc; no firmware registry/verify UI.
- **Determinism/flag:** the BIOS is part of the deterministic FDS boot input (already in-scope for v1.6.0 FDS-proper); the *manager* is frontend. NOT v2.0.

### E2. Multi-Disk Bundler (`.xml` multi-disk/multi-game container) — **S–M, frontend-only**
- **Source:** `BizHawk/src/BizHawk.Client.EmuHawk/tools/MultiDiskBundler/`.
- **What:** bundle multiple FDS disk sides / multi-cart games into one logical `.xml` "game" with named slots, so disk-swap (RustyNES F9) and multi-game carts present as a single ROM with a slot picker. BizHawk uses it for FDS multi-disk titles and disk-swap automation.
- **Why v1.7.0:** improves the FDS UX the v1.6.0 FDS-proper work creates (currently disk-swap is manual F9); a named-slot bundle is the BizHawk-standard ergonomics.
- **Gap today:** RustyNES has F9 disk-swap but no bundle/slot abstraction.
- **Determinism/flag:** frontend container over existing disk-image loading. NOT v2.0.

### E3. Batch Runner (headless multi-ROM run/screenshot/movie-verify) — **M, frontend-or-harness**
- **Source:** `BizHawk/src/BizHawk.Client.EmuHawk/tools/BatchRun.cs` + `BatchRunner.cs`.
- **What:** run a list of ROMs (or movies) headless for N frames, capturing screenshots / verifying movies / collecting state — BizHawk's tool for regression sweeps and movie-verification farms.
- **Why v1.7.0:** RustyNES *already* has the bones of this in its screenshot-corpus + mapper-survey scripts (`scripts/screenshots/`, the boot-smoke harness) and `rustynes-test-harness`. A first-class **batch/headless-run mode** in the binary (or a polished harness command) would consolidate the per-mapper screenshot + movie-verify workflows the project keeps re-building ad hoc, and gives TASers a "verify this `.fm2` against this ROM" CLI. Strong synergy with the standing mapper-ROM-coverage policy.
- **Gap today:** ad-hoc Python/harness scripts, no first-class batch mode.
- **Determinism/flag:** headless deterministic runs; reuses existing replay; NOT v2.0.

## Workstream F — Savestate / rewind engine depth (BizHawk Zwinder)

### F1. Compressed, tiered rewind/greenzone buffer (Zwinder/Zelda-winder) — **L, core-adjacent, determinism-safe**
- **Source:** `BizHawk/src/BizHawk.Client.Common/rewind/ZwinderBuffer.cs` (18 KB) + `tasproj/ZwinderStateManager.cs` (16 KB).
- **What:** BizHawk's rewind + TAStudio greenzone share a **Zwinder**: a fixed-RAM-budget ring of states with **delta/zlib compression**, density tiers (current/recent/ancient buffers with different intervals), and reserved anchors. It is what lets TAStudio keep thousands of frames of greenzone in ~1 GB and rewind for hours.
- **Why v1.7.0:** v1.6.0 builds a "density-tiered greenzone with eviction" (A1) but the *compression* + the unified rewind/greenzone state engine is the depth that makes it scale to feature-length TASes. RustyNES's v1.5.0 already shipped an "allocation-only rewind-keyframe-cache" tweak — this is the next, larger step (XOR-delta + zlib, per the v1.6.0 plan's own "opportunistic netplay" note about rewind compression).
- **Gap today:** RustyNES rewind/greenzone is uncompressed keyframe cache.
- **Determinism/flag:** **must** be bit-identical on restore (compression is lossless; the restored state must equal the saved state) — this is the determinism-critical item, gate it with a save→compress→decompress→restore round-trip equality test. Core-adjacent (serialization) but determinism-neutral if round-trip-exact. NOT v2.0 (no timebase change).

## Per-candidate determinism/v2.0 confirmation (summary)

Every item above is **frontend-only or output-only/serialization**, additive, off-by-default-capable, and **does NOT require the v2.0 master-clock refactor**. The two items closest to the core are: **F1 Zwinder compression** (serialization — gated by a lossless round-trip equality test; no timebase change) and **D3/the CDL-consumer** (rides the v1.6.0 output-only CDL side-array behind `debug-hooks`). The IPC (C1) is the one new *non-deterministic input source* — it must be host-mediated, off-by-default behind a feature, and disabled in replay/netplay/RA, exactly like `emu.write`.
