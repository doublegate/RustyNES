## WORKSTREAM A — Accuracy (dot/CPU-cycle granular; NOT v2.0; pin-the-ROM-first)

ares is the ceiling (integer master clock + libco `synchronize()`); higan is the weaker floor.
The big-ticket off-axis items (DMC/OAM-DMA corruption, `$2007` blocking read, sprite-overflow,
MDR decay) are **already in v1.6.0 (D)** — these are the *next, smaller* refinements ares models.
**Two-thirds of the candidate list is already documented in RustyNES as implemented**, so this
workstream splits into **genuinely-new behavior** vs **verify-only audits**. Every item is
dot/CPU-cycle granular and adoptable on the current "3 dots per CPU cycle" timebase.

### A-1. APU length-counter halt/reload race (deferred update on half-frame clock) — **GENUINELY NEW**
- **Source:** ares `fc/apu/length.cpp` (the `delayHalt` / `delayCounter` flags).
- **What:** writing the length-counter halt or reload *during* a half-frame clock cycle defers the
  new value one frame-counter cycle, so a write inside the measurement window isn't immediately visible.
- **Why v1.7.0:** RustyNES's `length.rs` update is atomic (`docs/apu-2a03.md` does **not** document
  the delay) — a real fidelity gap for games toggling channel length/halt at end-of-measure.
- **Effort:** M · **Gap-confirmed:** YES · **Test-ROM/pin:** `blargg_apu_2005.07.30/01.len_ctr.nes`
  (committed) + `apu_test` · **Determinism:** pure deferred state · **NOT v2.0:** fires on an exact
  frame-counter cycle — confirmed.

### A-2. DMC sample-buffer load-DMA delay (2 vs 3 cycles, even/odd APU phase) — **GENUINELY NEW**
- **Source:** ares `fc/apu/dmc.cpp` (`periodCounter & 1 ? 2 : 3`).
- **What:** on `$4015` bit-4 0→1 with an empty buffer, the scheduled load-DMA delay is 2 or 3 cycles
  depending on even/odd APU cycle — interleaves with game code reading `$2007`/`$4015` right after enabling DMC.
- **Why v1.7.0:** `docs/apu-2a03.md` (DMC channel) says "around the second APU cycle" — does **not**
  specify the even/odd distinction. Interacts with the v1.6.0 D DMA-corruption work (a companion, not a dup).
- **Effort:** M · **Gap-confirmed:** YES · **Test-ROM/pin:** `dmc_dma_during_read4.nes` (already wired) +
  `dmc_tests/{latency,status,status_irq}.nes` (committed, unwired) · **Determinism:** CPU-cycle boundary
  · **NOT v2.0:** confirmed (the *exact PPU-dot* the DMC RDY lands on IS v2.0; the even/odd CPU-cycle bulk is not).

### A-3. PPU palette read/write open-bus mirrors + corruption — **PARTIAL (verify/extend)**
- **Source:** ares `fc/ppu/memory.cpp` (palette mirror logic `$3F10/$3F14/$3F18/$3F1C` → `$3F00…`,
  + the open-bus interaction on palette reads/writes).
- **What:** palette reads update the open-bus latch with the would-have-been nametable fetch; palette
  writes drive the high 2 bits from open-bus.
- **Why v1.7.0:** RustyNES models `$2002` open-bus per-bit-group decay (`docs/ppu-2c02.md:198`) and the
  $2007 palette-read buffer-update — but the *palette-specific* open-bus drive may be incomplete.
- **Effort:** M · **Gap-confirmed:** PARTIAL · **Test-ROM/pin:** `ppu_open_bus.nes` (already passes
  tests 7/9 — extend coverage) + `ppu_read_buffer` (committed, unwired — see B-1) · **NOT v2.0:** CPU-cycle reads/writes — confirmed.

### A-4. PPU sprite-0 hit cannot set on dot 255 — **VERIFY (likely documented)**
- **Source:** ares `fc/ppu/render.cpp` sprite-0 path; RustyNES `docs/ppu-2c02.md:172` lists it.
- **What:** the sprite-0 hit flag cannot transition 0→1 on dot 255 (last visible dot).
- **Effort:** S (verify `ppu.rs` checks `dot != 255`) · **Test-ROM/pin:** `sprite_hit_tests_2005.10.05`
  (committed) · **NOT v2.0:** confirmed.

### A-5. OAMADDR rendering-state write glitch + OAMDATA attribute-byte mask — **VERIFY (documented)**
- **Source:** ares `fc/ppu/memory.cpp`; RustyNES `docs/ppu-2c02.md:195,197` already document both
  (high-6-bits-only glitch during rendering; OAM attr-byte bits 2-4 read as 0). **Audit `ppu.rs` honors the gates.**
- **Effort:** S · **Test-ROM/pin:** `oam_stress.nes`, `ppu_open_bus.nes`, `oam_read` (committed, unwired) · **NOT v2.0:** confirmed.

### A-6. VERIFY-ONLY cluster (RustyNES docs already claim these — audit only)
Pulse duty-sequencer phase reset on `$4003`/`$4007` (`docs/apu-2a03.md:181`); sweep negation
off-by-one P1 ones'-complement vs P2 two's-complement (`:184`); sweep mute on target > `$7FF` or
underflow < 8 (`:183`); noise mode bit 7 (`noise.rs:78` — correct); CPU RDY/halt on read cycles only
(`docs/cpu-6502.md:154`); decimal flag settable-but-inert on 2A03 (`docs/cpu-6502.md:46`). **All S,
verify-only; close documentation/implementation drift if any is found, but do not assume a gap.**

> **My calibration:** A-1 and A-2 are the only two clearly-new accuracy behaviors; A-3 is a
> partial extension; A-4/A-5/A-6 are verify-only against already-documented behavior. This
> workstream is therefore SMALL and audit-heavy — accuracy below v2.0 is genuinely near-exhausted,
> as all three v1.6.0 reports concluded. Recommend a single ~3-5 day "APU/PPU edge-case audit + A-1/A-2"
> pass rather than a headline workstream.

## WORKSTREAM H — Performance (measure-first; determinism + byte-identical held)

Each item NAMES the criterion bench that would show the win. Already done (don't re-propose): F1 PPU
scanline-flag cache, F2 MMC5 short-circuit, mapper-caps gating, LTO/codegen-units, 512-entry pixel LUT,
SIMD-shaped BLEP SAXPY, PGO+BOLT gate, H1 triple-buffer present, H3 rewind-cache alloc reuse. ADR 0001 says
don't monomorphize `Box<dyn Mapper>` (dispatch <1% frame cost).

### Tier-1 (low-risk, isolated — commit after A/B)
- **H-1. Unified-DMA cycle fast-path** — **S–M** — early-return gate on CPU cycles with no pending/in-flight
  DMA (the *only* per-cycle dispatch still running unconditionally; others are caps-gated). Directly parallels
  the v1.4.0 mapper-caps win (−4% to −10%). **Bench:** `full_frame` (nestest +3-5%). **Risk:** LOW (pure early
  return). **Proof:** CPU+PPU cycle counts at frame-end + full oracle.
- **H-2. BLEP phase-row index cache** — **S** — cache the kernel-row index across taps within a call (the phase
  advances ~0.024/sample, so consecutive samples share a row). *Distinct from the v1.4.0-rejected version*: cache
  the row in a local, do **not** reorder the f64→f32→i32 quantization. **Bench:** `spectral::blip_square_wave_0_1s_ntsc`
  (+2-5%). **Risk:** LOW (lookup result identical). **Proof:** sample-by-sample byte-exact audio.

### Tier-2 (profile-first, commit only on measured >2% win)
- **H-3. PPU dot-loop batch advancement** — **M–L** — batch-advance idle scanlines (241-260 = ~7.6% of dots,
  zero visual work) and identical 8-dot fetch spans, WITHOUT skipping a visible dot or a mid-scanline PPUMASK/CTRL
  change. **Bench:** `ppu_tick_one_frame` (+5-10%). **Risk:** MEDIUM — must preserve per-scanline pixel ordering +
  interrupt batches on mid-scanline register writes. **Proof:** per-scanline framebuffer hash + full oracle.
- **H-4. PPU struct cache-locality reorder** — M — co-locate the hot per-dot fields (`scanline`/`dot`/rendering
  flags/shift regs/`nmi_pending`) to cut cache-line spread. **Profile FIRST** with `perf stat -e cache-misses`.
  **Bench:** `ppu_tick_one_frame` (+2-5%). **Risk:** LOW (internal layout). **Caveat:** decouple from save-state field order.
- **H-5. Pixel-composition branch elimination** — S–M — replace nested-if BG/sprite-priority with arithmetic masks.
  **Profile FIRST** (`perf stat -e branch-misses` on `flowing_palette`); commit only if mispredict >5%. **Bench:** `flowing_palette` (+1-3%).

### Tier-3 (high-effort, conditional on profile)
- **H-6. Rewind XOR-delta streaming + threaded decompress** — M — RustyNES already uses LZ4 keyframe + XOR delta
  (`rewind.rs`); the next tier is moving compress/decompress off the emu thread + page pre-faulting. **Bench:** a
  `snapshot_restore` bench (may need creating). +10-30% rewind-capture latency when armed. **Risk:** LOW (side effect only).
- **H-7. Decoupled GPU/NTSC-filter thread** — L — Mesen2 runs the composite filter on a separate submit thread.
  **No headless measurement path** (144+ Hz panel only; maintainer-manual like v1.5.0 H6). **Bench:** perf-log
  `gpu_ms`/`present_max_ms`. **Risk:** MEDIUM (timing only; output byte-identical). Gate on >20% `gpu_ms` p99 win.
- **H-8. Run-ahead double-sim batching** — M — a `run_n_frames(2, [false,true])` API amortizing per-`run_frame`
  setup across the hidden + visible run-ahead frames (run-ahead is default-on → every frame does 2× work). **Bench:**
  `full_frame` with run_ahead=1. **Risk:** LOW (each frame's output must match sequential — proof by oracle).
- **H-9. wasm linear-memory layout** — S — align hot structs off 64 KiB page boundaries (marginal native, ~5-10% wasm).
