# RustyNES v1.7.0 — Niche subsystems, test-ROM coverage & creator/power-user tooling

**Research date:** 2026-06-17 (READ-ONLY; no code modified)
**Scope:** the tier *beyond* v1.6.0 "Studio" (MAXIMAL, in dev). v1.6.0 already
includes: full TAStudio piano-roll (greenzone/lag-log/branches/`.rnmproj`/multitrack/
patterns), `.fm2`/`.bk2`(*see correction below*) movie interop, Lua driving
(`emu.frameadvance`+coroutines/memory-domains/`movie.*`/`joypad.get`/richer gui),
Mesen2-class debugger depth (expr/conditional breakpoints, R/W/X watchpoints, hex
editor edit+freeze+heatmap, RAM watch/search, watch window, trace format-strings/
conditional logging, call stack, profiler, CDL, step-back/reverse), off-axis accuracy
cluster (DMC/OAM-DMA corruption, $2007 blocking read, sprite-overflow, MDR decay),
mappers→~150 + UNIF loader, FDS-proper, A/V recording, HD audio + HD-pack spatial
conditions, shaders/.cgp/LMP88959/upscalers.

**Working-tree facts confirmed (v1.6.0-dev on main):**

- UNIF loader EXISTS: `crates/rustynes-mappers/src/unif.rs` (Workstream E2).
- TAStudio EXISTS: `crates/rustynes-frontend/src/tastudio/mod.rs` + `tastudio_panel.rs`
  (piano-roll, greenzone, branches, `.rnmproj` RNMPROJ1).
- `.fm2` interop EXISTS: `crates/rustynes-core/src/movie_interop.rs` (FCEUX v3,
  2-player, PowerOn-start only, round-trip tested). **CORRECTION to the brief:
  `.bk2` IMPORT is ABSENT on the working tree** — only `.fm2` is implemented.
  Treat `.bk2` as v1.6.0-pending OR a v1.7.0 candidate.
- Save-state THUMBNAILS already ship (core `THM ` section + native grid UI
  `save_states_ui.rs`) — NOT a v1.7.0 gap.
- Symbol files load NAMES ONLY (`crates/rustynes-frontend/src/symbols.rs`:
  `.sym`/`.mlb`/`.nl` → `HashMap<u16,String>`); NO source-line mapping, NO `.dbg`.
- Game DB keyed by CRC32 already exists (`crates/rustynes-frontend/src/game_db.rs`,
  `debugger/game_db_panel.rs`). Region override + CRC32-as-secondary-ROM-key are
  listed "Open questions" in `docs/compatibility.md`.

## WORKSTREAM: test-hardening (cheap, high-value; ALL already committed under tests/roms/)

These suites are physically vendored in `tests/roms/nes-test-roms/` (and exercised by
nothing in `crates/rustynes-test-harness/`). They use the same blargg `$6000` status
protocol the harness already drives via `run_until_complete()`/`run_nes_blargg`, so
wiring is small. Licenses are blargg public-domain / Damian-Yerrick-zlib (already the
basis for the committed corpus).

### 1. `ppu_read_buffer` (blargg "test_ppu_read_buffer.nes")
- **Source:** `ref-proj/nes-test-roms/ppu_read_buffer/` (vendored at
  `tests/roms/nes-test-roms/ppu_read_buffer/`); readme on-disk.
- **What it is:** "mammoth" suite centering on the PPU `$2007` read buffer + palette
  read paths; ~many sub-tests, `$6000` protocol + audio progress beeps.
- **Why v1.7.0:** directly validates `$2007` buffered-read semantics — adjacent to the
  v1.6.0 "$2007 blocking read" off-axis work but a DISTINCT, committable oracle that
  RustyNES does NOT run. Highest-value single addition.
- **Effort:** S. **Gap-confirmed:** YES (0 harness refs). **Determinism/perf:** none
  (headless status-ROM). **NOT v2.0:** correct — buffer semantics, not master-clock.

### 2. `vbl_nmi_timing` (blargg, 7 sub-ROMs)
- **Source:** `ref-proj/nes-test-roms/vbl_nmi_timing/{1..7}.*.nes`.
- **What it is:** VBL flag + NMI timing to single-PPU-clock accuracy: frame_basics,
  vbl_timing, even_odd_frames, vbl_clear_timing, **nmi_suppression**, nmi_disable,
  nmi_timing.
- **Why v1.7.0:** RustyNES already passes `ppu_vbl_nmi` (10/10) but this is a SEPARATE
  blargg pack. CAUTION: sub-ROM #5 `nmi_suppression` exercises the `$2002`-NMI-suppression
  race — flagged in the brief as **v2.0**. Wire the suite as strict where it passes and
  `#[ignore]` + `_currently_fails` the suppression sub-ROM (same pattern as mmc3_test).
- **Effort:** S. **Gap-confirmed:** YES (1 ref but only as doc/path; not wired as tests).
  **NOT v2.0:** mostly — pin the one suppression sub-ROM as the documented v2.0 residual.

### 3. `sprdma_and_dmc_dma` + `dmc_tests` (blargg)
- **Source:** `ref-proj/nes-test-roms/sprdma_and_dmc_dma/{,_512}.nes`,
  `ref-proj/nes-test-roms/dmc_tests/{buffer_retained,latency,status,status_irq}.nes`.
- **What it is:** OAM-DMA + DMC-DMA interaction / DMC buffer + status + IRQ timing.
- **Why v1.7.0:** RustyNES models DMC-DMA controller-bit corruption (v1.4.0) and the
  unified per-cycle DMA engine — these ROMs are the canonical regression net for it and
  for the v1.6.0 DMC/OAM-DMA corruption work. Companions to the already-wired
  `dmc_dma_during_read4`.
- **Effort:** S–M (dmc_tests are visual/audio; likely smoke or RAM-probe). **Gap:** YES.
  **NOT v2.0:** correct (DMA engine behavior, not the fractional clock).

### 4. `cpu_exec_space` (blargg, 2 ROMs: apu + ppuio)
- **Source:** `ref-proj/nes-test-roms/cpu_exec_space/test_cpu_exec_space_{apu,ppuio}.nes`.
- **What it is:** executing code from APU/PPU register space (open-bus fetch, RDY).
- **Why v1.7.0:** exercises CPU-fetch-from-IO + open-bus edge cases the unit tests

### 5. `read_joy3` (blargg, 4 ROMs)
- **Source:** `ref-proj/nes-test-roms/read_joy3/{count_errors,count_errors_fast,test_buttons,thorough_test}.nes`.
- **What it is:** controller-read robustness under DMC-DMA "stealing" controller bits.
- **Why v1.7.0:** RustyNES gates DMC-DMA corruption on `read_joy3/count_errors` per
  `docs/compatibility.md` — but only ONE of 4 ROMs. Wire `thorough_test` + `test_buttons`

  **Effort:** S. **NOT v2.0:** correct.

## WORKSTREAM: debugger-source (source-level debugging + assembler)

### 10. ca65/cc65 `.dbg` source-line mapping (Mesen2 DbgImporter parity)
- **Source:** `ref-proj/Mesen2/UI/Debugger/Integration/{DbgImporter.cs,NesDbgImporter.cs}`;
  nesdev/ld65 `.dbg` format; current RustyNES gap = `crates/rustynes-frontend/src/symbols.rs`
  (names only, `HashMap<u16,String>`).
- **What it is:** parse ld65 `--dbgfile` `.dbg` output → address↔(source file, line)
  + scopes + spans, so the disassembler/breakpoints/step show ORIGINAL SOURCE (asm or C),
  not just labels. Mesen2 "Switch to Source View" + "Auto-load DBG/MLB next to ROM".
- **Why v1.7.0:** RustyNES has symbol NAMES (v1.4.0) but NOT source lines — the single
  biggest devtools gap vs Mesen2 for homebrew authors. Pairs with the v1.6.0 CDL/profiler.
- **Effort:** L (full `.dbg` schema: csym/file/line/scope/span/sym sections). **Gap-confirmed:**
  YES (`symbols.rs` is names-only; no `.dbg`/DWARF/ELF). **Determinism/perf:** none (frontend
  display aid, never touches core). **NOT v2.0:** correct (tooling, no timing change).

### 11. In-app 6502 assembler + inline RAM/ROM patching (Mesen2 NesAssembler)
- **Source:** `ref-proj/Mesen2/Core/Debugger/{Base6502Assembler.cpp,IAssembler.h}`.
- **What it is:** assemble 6502 mnemonics in the debugger and patch live RAM/ROM at a
  PC, with undo; export patched ROM. Today RustyNES only does load-time IPS/UPS/BPS
  (`crates/rustynes-frontend/src/patch.rs`), no runtime assembler.
- **Why v1.7.0:** closes the "edit code in the debugger" half of Mesen2 parity (the
  v1.6.0 hex editor edits DATA; this edits CODE). Great for ROM-hackers + accuracy probes.
- **Effort:** M (6502 is a small ISA; reuse the existing disassembler tables in
  `crates/rustynes-cpu/src/disasm.rs`). **Gap-confirmed:** YES (absent). **NOT v2.0:**
  correct (a frontend write into the existing memory bus; determinism intact if gated
  like `emu.write`).

### 12. Additional symbol/debug formats: ELF/DWARF, WLA-DX `.sym`, RGBDS, SDCC `.cdb`
- **Source:** Mesen2 importers; llvm-mos ELF+DWARF (modern cc65 alternative); SDCC `.cdb`.
- **Why v1.7.0:** llvm-mos (DWARF) is the rising NES C toolchain; SDCC `.cdb` and RGBDS
  cover the rest. Extends #10's importer with more front-ends.
- **Effort:** M–L (DWARF is heavy; gimli crate exists). **Gap:** YES. **NOT v2.0:** correct.

## WORKSTREAM: netplay (beyond 2-4p rollback)

Current: `crates/rustynes-netplay/{session.rs,diagnostics.rs}` — GGPO rollback, 2-4p,
input-resend window (64), desync diagnostics + rolling CRC history, UDP+WebRTC.

### 13. Spectator mode (read-only observer)
- **Source:** GGPO `ggpo_add_spectator`; BizHawk/FightCade spectator. RustyNES gap:
  `session.rs` has only `MAX_PLAYERS=4` active peers, no observer role.
- **What it is:** a peer that receives the input stream + state checksums, runs the
  deterministic core, but contributes no input — for streamers/coaches/lobbies.
- **Why v1.7.0:** natural extension of the existing input-broadcast + determinism contract;
  high community value. **Effort:** M. **Gap-confirmed:** YES. **Determinism:** intact
  (observer is a pure consumer; never feeds the rollback). **NOT v2.0:** correct.

### 14. Adaptive input-delay buffer (auto frame-delay tuning)
- **Source:** GGPO frame-delay auto-adjust; the dynamic-rate-control analogue already in
  the frontend pacer.
- **What it is:** measure round-trip + jitter and auto-pick the input-delay frames
  (currently fixed/manual), trading latency for fewer rollbacks.
- **Why v1.7.0:** quality-of-experience win on jittery links. **Effort:** M. **Gap:** YES.
  **Determinism:** intact (delay is a frontend scheduling choice; both peers agree via
  handshake). **NOT v2.0:** correct.

### 15. Auto config/cheat resync + pre-game consistency handshake
- **What it is:** before play, exchange + verify ROM SHA-256, region, cheat list, mapper
  options, run-ahead; refuse or auto-align on mismatch (today a cheat mismatch silently
  desyncs).
- **Why v1.7.0:** the diagnostics monitor already surfaces desync AFTER the fact; this
  prevents the common cause. **Effort:** S–M. **Gap:** YES. **NOT v2.0:** correct.

### 16. Netplay timeline / HistoryViewer-style scrubber
- **Source:** BizHawk HistoryViewer; RustyNES's existing rewind keyframe cache.
- **What it is:** a post-match (or live, paused) frame-scrubber that replays the

- **Why v1.7.0:** review desyncs/matches frame-by-frame; cheap given TAStudio exists.
  **Effort:** M. **Gap:** YES. **NOT v2.0:** correct.

## WORKSTREAM: NSF (finish the music player)

`$5000` driver, 4KiB bankswitch, **NTSC 60Hz only**, expansion bitfield PARSED but
**audio not synthesized**; lines 22-25 + 68 explicitly defer NSFe + expansion + non-60Hz.

### 17. NSF expansion-audio synthesis (VRC6/VRC7/FDS/MMC5/N163/5B)
- **Source:** RustyNES ALREADY HAS all the synth cores for cart audio
  (`crates/rustynes-apu/src/opll.rs` VRC7, plus VRC6/FME-7/N163/MMC5/FDS in
  `mapper-audio`). The NSF player just needs to ROUTE its `$9000…`/`$4040…` register
  writes into those existing oscillators.
- **What it is:** wire the parsed expansion bitfield (`nsf.rs:69`) to the existing
  expansion-audio synth so VRC7/VRC6/FDS/etc NSFs actually play.
- **Why v1.7.0:** unlocks the bulk of the classic Japanese NSF library; near-free given
  the synth cores exist. **Effort:** M (routing + register-window mapping, no new DSP).
  **Gap-confirmed:** YES (`nsf.rs:68` "not yet synthesized"). **Determinism:** N/A (NSF
  is its own deterministic path). **NOT v2.0:** correct.

### 18. NSFe + NSF2 container parsing (metadata + IRQ)
- **Source:** nesdev wiki NSFe / NSF2; nununoisy/nsf-presenter-rs (Rust, RusticNES-based).
- **What it is:** parse the `NSFE` chunked container (track titles, durations, fade,
  author/`auth`, `tlbl`, playlist) and NSF2's feature flags (bit4 IRQ, non-returning
  INIT, appended NSFe metadata).
- **Why v1.7.0:** modern NSF/NSFe rips carry track names + times the player can show; the
  v1.5.0 NSF waveform scope (C3) + a track-list/timeline UI become genuinely usable.
- **Effort:** M. **Gap-confirmed:** YES (classic NESM only). **NOT v2.0:** correct.

### 19. Non-NTSC NSF play rates (PAL/Dendy + custom `play_speed`)
- **Source:** NSF header PAL/NTSC bits (byte 0 / 1) + the explicit `play_speed_ntsc/pal`
  fields; nesdev NSF page.
- **What it is:** honor the NSF's declared play period instead of forcing 60Hz vblank.
- **Why v1.7.0:** PAL-composed tunes currently play at wrong speed (`nsf.rs:25`). Pairs
  with the region work RustyNES already models. **Effort:** S–M. **Gap-confirmed:** YES.
  **NOT v2.0:** correct.

### 20. NSF player UI: track list, seek/timeline, per-track time, loop/fade
- **What it is:** a real playlist UI (Tools→NSF Player) using #18's NSFe metadata, plus
  a seek bar driving fast-forward through the deterministic driver.
- **Effort:** M (frontend). **Gap:** YES (current UI = song-count only). **NOT v2.0:** correct.
