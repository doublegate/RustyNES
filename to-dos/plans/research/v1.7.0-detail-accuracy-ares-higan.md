**Research Date:** 2026-06-17  
**Scope:** Cross-reference ares (`fc/`), higan, and TriCNES against RustyNES v1.5.0 documentation to identify dot/CPU-cycle-granular behaviors NOT yet modeled that do NOT require a v2.0 master-clock refactor.

**Explicitly v2.0-deferred (fractional master-clock required):**
- `$2002` PPUSTATUS / NMI-suppression race (sub-dot)
- `apu_reset` `$4017` odd/even half-cycle reset alignment

## TIER 1: HIGH PRIORITY (CPU-cycle granular, straightforward, high-impact)

### 1. **Pulse Channel Duty Sequencer Phase Reset on `$4003`/`$4007`**
- **Source:** ares `fc/apu/pulse.cpp` lines 22-31
- **What it is:** Writing to `$4003` (Pulse 1 length/timer-high) or `$4007` (Pulse 2) resets the pulse duty sequencer to step 0 AND sets the envelope-restart flag, but does NOT reset the timer divider. RustyNES documents this in apu-2a03.md line 181 but needs verification this is fully implemented.
- **Why v1.7.0-worthy:** Simple CPU-cycle-granular behavior; games like Mega Man 2 rely on duty phase predictability for arpeggiated notes.
- **Effort:** S (Small)
- **RustyNES gap confirmed:** `docs/apu-2a03.md` line 181 lists the behavior but `crates/rustynes-apu/src/pulse.rs` needs verification against the ares implementation (does `write_timer_hi` reset `dutyCounter`?).
- **Committable test-ROM:** `apu_test.nes` (sub-ROM 8, "duty envelope"), `cpu_timing_test6.nes`
- **Determinism:** Yes — pure state machine.
- **Explicit NOT v2.0:** Yes — no fractional timing required; happens on exact CPU cycle of write.

---

### 2. **Sweep Unit Negate Off-by-One (Pulse 1 vs. Pulse 2)**
- **Source:** ares `fc/apu/sweep.cpp` lines 18-19; RustyNES `docs/apu-2a03.md` line 184
- **What it is:** Pulse 1 negates the sweep delta using one's complement (`~target`), while Pulse 2 uses two's complement (`-target`). This produces an audible frequency difference at certain pitch ranges. Mesen2 and ares both model this.
- **Why v1.7.0-worthy:** Affects audio pitch accuracy for sweep-based musical effects; visible in frequency-sweep ROMs.
- **Effort:** S
- **RustyNES gap confirmed:** `docs/apu-2a03.md` §Pulse duty-sequencer phase reset (line 184-185) documents it; `crates/rustynes-apu/src/sweep.rs` needs to verify the negation logic branches per channel.
- **Committable test-ROM:** `apu_mixer/square.nes`, audio-capture regression (Mega Man 3 opening theme uses pulse sweeps)
- **Determinism:** Yes.
- **Explicit NOT v2.0:** Yes.

---

### 3. **Length Counter Halt/Reload Race on Half-Frame Clock**
- **Source:** ares `fc/apu/length.cpp` lines 1-15; delayHalt/delayCounter mechanism
- **What it is:** When length-counter halt or reload is written during a half-frame clock cycle, the new value is delayed by one frame-counter cycle (deferred update). This prevents a write during the measurement window from being immediately visible.
- **Why v1.7.0-worthy:** Affects games that modify APU channel behavior mid-measure (e.g., end-of-measure length reload + halt toggle).
- **Effort:** M (Medium)
- **RustyNES gap confirmed:** `docs/apu-2a03.md` does not document the delay mechanism; `crates/rustynes-apu/src/length.rs` current impl is atomic. Ares's `delayHalt`/`delayCounter` flags model the race.
- **Committable test-ROM:** `blargg_apu_2005.07.30/01.len_ctr.nes` (length counter test)
- **Determinism:** Yes — purely deterministic deferred state.
- **Explicit NOT v2.0:** Yes — happens on exact frame-counter cycle.

---

### 4. **Noise Channel Mode Bit Position (Bit 7 of `$400E`)**
- **Source:** ares `fc/apu/noise.cpp` line 10; RustyNES `docs/apu-2a03.md` line 140
- **What it is:** Mode bit is bit 7, not bit 0. Writing `$400E` with bit 7 set selects short (6-bit LFSR tap) mode; clear selects long (15-bit LFSR tap) mode. RustyNES currently correct; verify it's not bit 0.
- **Why v1.7.0-worthy:** Register encoding verification; documentation accuracy.
- **Effort:** S (verification only)
- **RustyNES gap confirmed:** `crates/rustynes-apu/src/noise.rs` line 78 reads `(value & 0x80)`, which is bit 7. Correct. No gap.
- **Committable test-ROM:** None (intrinsic register test).
- **Determinism:** N/A
- **Explicit NOT v2.0:** N/A

### 5. **DMC Sample Buffer Load DMA Delay (2 vs. 3 cycles)**
- **Source:** ares `fc/apu/dmc.cpp` lines 6-8; RustyNES `docs/apu-2a03.md` line 133
- **What it is:** When `$4015` bit 4 (DMC enable) transitions 0→1 and the sample buffer is empty, a DMA read is scheduled with a 2- or 3-cycle delay depending on whether the write occurred on an even or odd APU cycle (`periodCounter & 1 ? 2 : 3`). This timing interleaves with game code that reads `$2007` or `$4015` immediately after enabling DMC.
- **Why v1.7.0-worthy:** Affects games using DMC to play percussion during rendering (e.g., Castlevania); CPU stall timing interacts with controller reads.
- **Effort:** M
- **RustyNES gap confirmed:** `docs/apu-2a03.md` §DMC channel, line 133 ("Load DMA follows enabling ... scheduled around the second APU cycle") does NOT specify the 2 vs. 3 cycle distinction based on odd/even phase. `crates/rustynes-apu/src/dmc.rs` needs audit.
- **Committable test-ROM:** `dmc_dma_during_read4.nes` (exists in ref-proj; should test stall cycle count)
- **Determinism:** Yes.
- **Explicit NOT v2.0:** Yes — happens on CPU-cycle boundary.

## TIER 2: MEDIUM PRIORITY (Moderate effort; refinement, not core)

### 6. **Pulse Sweep Muting on Target Period > $7FF or Underflow**
- **Source:** ares `fc/apu/pulse.cpp` line 12; RustyNES `docs/apu-2a03.md` line 183
- **What it is:** When the sweep computes a target period that exceeds $7FF (2047) or, on negation, would underflow below 8, the pulse channel is muted (output forced to 0) regardless of the length counter. This silences pitch-sweep "artifacts" that would occur if the period is invalid.
- **Why v1.7.0-worthy:** Affects sweep-heavy games (Mega Man 3, Kirby's Dream Land); prevents audible glitches.
- **Effort:** M
- **RustyNES gap confirmed:** `docs/apu-2a03.md` line 183 mentions the sweep mute but does not detail the underflow threshold (8). `crates/rustynes-apu/src/sweep.rs` needs audit against ares line 12-13: `if(sweep.pulsePeriod < 0x008) result = 0;`.
- **Committable test-ROM:** `apu_mixer/square.nes` (audio capture comparison)
- **Determinism:** Yes.
- **Explicit NOT v2.0:** Yes.

### 7. **PPU `$2007` Blocking Read Window (6-dot delay)**
- **Source:** ares `fc/ppu/memory.cpp` line 72 (`var.blockingRead = 6`); RustyNES **explicitly excluded from v1.6.0** scope
- **What it is:** After any read of `$2007`, the next 6 PPU dots ignore further reads and return the buffered value (stale data). This prevents mid-scanline VRAM address corruption during rendering.
- **Why NOT in v1.7.0:** This is a **sub-dot behavior** (the 6-dot window crosses PPU dots 0-5 following the read cycle). Requires aligning the CPU's bus-access timing (which happens "before" the 3 PPU dots in our scheduler) to a sub-dot granularity. Deferred to v2.0.
- **Explicit v2.0 confirmation:** Yes.

### 8. **PPU Palette Read/Write Quirks (Mirror Address Corruption)**
- **Source:** ares `fc/ppu/memory.cpp` lines 10-20; RustyNES `docs/ppu-2c02.md` line 193
- **What it is:** Palette RAM `$3F00-$3FFF` has internal mirrors at `$3F10`, `$3F14`, `$3F18`, `$3F1C` mirroring `$3F00`, `$3F04`, `$3F08`, `$3F0C`. Additionally, when reading a palette address, the open-bus latch is updated with the nametable data that would have been fetched instead. When writing, the high 2 bits are driven by the open-bus.
- **Why v1.7.0-worthy:** Affects games that read palette memory or rely on open-bus behavior during rendering (rare but documented).
- **Effort:** M
- **RustyNES gap confirmed:** `docs/ppu-2c02.md` line 198 documents open-bus behavior for `$2002`; palette-specific open-bus interaction may not be fully modeled. Ares lines 9-20 show the mirror logic.
- **Committable test-ROM:** `ppu_open_bus.nes` (tests 7, 9)
- **Determinism:** Yes.
- **Explicit NOT v2.0:** Yes — happens on CPU-cycle boundaries (palette read/write).

### 9. **PPU OAMADDR Corruption on Bit 0-5 Glitch During Rendering**
- **Source:** ares `fc/ppu/memory.cpp` lines 124-132; RustyNES `docs/ppu-2c02.md` line 195
- **What it is:** Writing to `$2003` (OAMADDR) during rendering (visible lines 0-239 or pre-render if rendering enabled) only increments the high 6 bits, producing a glitchy address. Non-rendering writes increment the full address.
- **Why v1.7.0-worthy:** Mid-scanline OAMADDR writes used in rare games for OAM manipulation (e.g., status-bar sprite handling).
- **Effort:** M
- **RustyNES gap confirmed:** `docs/ppu-2c02.md` line 195 ("OAMADDR write during rendering: glitches OAMADDR's high 6 bits"). Verify `crates/rustynes-ppu/src/ppu.rs` honors the rendering-state gate.
- **Committable test-ROM:** `oam_stress.nes`, `sprite_overflow_tests`
- **Determinism:** Yes.
- **Explicit NOT v2.0:** Yes.

### 10. **PPU OAMDATA Attribute Byte Unimplemented Bits (2-4 Read-Mask)**
- **Source:** ares `fc/ppu/memory.cpp` lines 138-139; RustyNES `docs/ppu-2c02.md` line 197
- **What it is:** Every 4th byte in OAM (bytes 2 of each sprite, the attribute byte) has bits 2-4 unimplemented on the 2C02G. Reads return 0 in those bits; writes still store the full byte. This is a hardware quirk to prevent accidental palette selection.
- **Why v1.7.0-worthy:** Affects games that directly manipulate OAM for collision detection or visual tricks.
- **Effort:** S
- **RustyNES gap confirmed:** `docs/ppu-2c02.md` line 197 documents it. `crates/rustynes-ppu/src/ppu.rs` register-read path (`cpu_read_register`) needs audit: does the `$2004` read mask off bits 2-4 during sprite-attribute reads?
- **Committable test-ROM:** `ppu_open_bus.nes` (tests 7, 9)
- **Determinism:** Yes.
- **Explicit NOT v2.0:** Yes.

### 11. **PPU Sprite-0 Hit Cannot Set on Dot 255**
- **Source:** ares `fc/ppu/render.cpp` (implicit sprite-0-hit rendering loop); RustyNES `docs/ppu-2c02.md` line 172
- **What it is:** The sprite-0 hit flag cannot transition 0→1 on PPU dot 255 (last visible dot of scanline). This prevents the flag from being set on the edge case dot where rendering ends.
- **Why v1.7.0-worthy:** Affects games using sprite-0 detection for per-scanline effects timing (Battletoads, etc.); the dot-255 race is a known trap.
- **Effort:** S
- **RustyNES gap confirmed:** `docs/ppu-2c02.md` line 172 lists it. Verify `crates/rustynes-ppu/src/ppu.rs` sprite-0-hit path checks `dot != 255` before setting.
- **Committable test-ROM:** `sprite_hit_tests_2005.10.05` (test ROMs 8, 9, 10)
- **Determinism:** Yes.
- **Explicit NOT v2.0:** Yes.

### 12. **CPU RDY/Halt Behavior (Halt on Read Cycles Only)**
- **Source:** ares `fc/cpu/timing.cpp` (DMA halt gate); RustyNES `docs/cpu-6502.md` line 154-157
- **What it is:** The CPU's RDY line (pulled low by DMA) halts the CPU only on read cycles. Write cycles (e.g., in read-modify-write instructions) cannot halt; the DMA request stays pending until a read cycle occurs.
- **Why v1.7.0-worthy:** Affects games using DMA during store instructions (rare but precise timing-dependent).
- **Effort:** S (verification; RustyNES likely correct)
- **RustyNES gap confirmed:** `docs/cpu-6502.md` lines 154-157 document RDY/halt semantics. `crates/rustynes-cpu/src/cpu.rs` bus halt implementation needs audit: does it correctly skip RMW write-cycle halts?
- **Committable test-ROM:** CPU interrupts suite (implicit in `cpu_interrupts_v2`)
- **Determinism:** Yes.
- **Explicit NOT v2.0:** Yes.

### 13. **Decimal Flag (D) Settable but Non-Functional on 2A03 (No-op on ADC/SBC)**
- **Source:** ares `fc/cpu/cpu.cpp` line 52 (`MOS6502::BCD = 0`); RustyNES `docs/cpu-6502.md` line 46-47
- **What it is:** The NES 2A03 has the decimal flag (D bit) in the status register, and it can be set via `SED` and cleared via `CLD`, but ADC/SBC always perform binary arithmetic. This is a 6502 variant quirk.
- **Why v1.7.0-worthy:** Documentation completeness; games don't exploit this (no practical impact), but emulation fidelity requires correctness.
- **Effort:** S
- **RustyNES gap confirmed:** `docs/cpu-6502.md` line 46 says "D is settable but ignored by ADC/SBC". RustyNES's 6502 impl likely already handles this. Verification only.
- **Committable test-ROM:** None (intrinsic).
- **Determinism:** N/A
- **Explicit NOT v2.0:** N/A

## BEHAVIORS REQUIRING v2.0 (NOT CANDIDATES)

### **PPU `$2002` NMI-Suppression Race (Scanline 241 Dot 0)**
- **Source:** ares `fc/ppu/ppu.cpp` lines 65-68 (scanline 241 / NMI timing).
- **Why v2.0:** Reading PPUSTATUS at EXACTLY scanline 241 dot 0 returns 0 (suppressing the VBL bit) AND suppresses NMI for the frame. This is a **sub-dot race** requiring the CPU's bus read to be timed to within a single PPU dot of the NMI latch's edge. RustyNES's scheduler performs CPU reads "before" the 3 PPU dots, so a read cannot land on dot 0 exactly. Requires fractional-clock refactor.

### **APU Reset `$4017` Odd/Even Half-Cycle Alignment**
- **Source:** ares `fc/apu/framecounter.cpp` lines 7-9 (dmaDelayCounter).
- **Why v2.0:** The `$4017` write's reset delay depends on the APU's internal half-cycle state (odd vs. even). This is a sub-cycle detail of the frame counter's master-clock phase. Requires fractional timing.

### **PAL 3.2:1 Exact CPU:PPU Ratio**
- **Source:** ares `fc/fc.hpp` region timing constants.
- **Why v2.0:** PAL's true 3.2:1 ratio (CPU runs 80% speed relative to NTSC) requires a fractional master-clock that can represent 5 CPU cycles per 16 PPU dots. Current "3 dots per CPU cycle" assumes integer boundaries. V1.5.0 uses the corrected timing table; v2.0 will refactor the internal scheduler.
