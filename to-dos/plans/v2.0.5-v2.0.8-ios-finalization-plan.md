# v2.0.5 → v2.0.8 iOS Finalization — sprint plan

> **Scope: the four-release iOS/iPadOS finalization window of the v2.0.x "Harbor"
> mobile-finalization train.** This mirrors, on the iOS side, exactly what
> **v2.0.1 → v2.0.4** did for Android: re-port the frozen v1.9.x app onto the
> **v2.0.0 "Timebase"** core, finalize feature parity, polish/performance, and cut
> an **iOS release candidate** — all while the cycle-accurate emulation core stays
> **byte-identical** (AccuracyCoin **141/141, 100.00%**; nestest 0-diff; the
> `#![no_std]` chip stack untouched). No accuracy / save-state / determinism number
> moves in this window; every change is **host / iOS-side**. The App Store + AltStore
> PAL launch remains **deferred to v2.1.0** (joint with Google Play + F-Droid), and
> the dual-app readiness gate is **v2.0.9**.

## Execution-environment reality (read first)

This plan is authored and landed on a **Linux host** with no Xcode / macOS / Apple
device. As with the **v2.0.4 Android RC** cut, the work that actually lands here is the
**host-authorable** surface — Rust shim (`crates/rustynes-ios`), Swift source
(`ios/RustyNES/`), UniFFI-Swift regeneration wiring, `fastlane` / App Store Connect
listing metadata, `PrivacyInfo.xcprivacy`, version bumps, and docs — with every
Xcode-build / Metal-on-device / TestFlight / App-Review step **explicitly flagged for
the maintainer + the v2.0.9 readiness pass**. Each release below separates
**Host-done (this release)** from **Flagged for maintainer / on-device (v2.0.9 / the RC
upload)**, exactly like `v2.0.4-android-rc-plan.md`. The **desktop / core determinism
gates run in full** on every release and must stay green.

## Plan reconciliation

`v2.0.x-mobile-finalization-plan.md` slots the iOS window as: **v2.0.5** core re-port +
foundation finalize · **v2.0.6** feature parity · **v2.0.7** polish + performance ·
**v2.0.8** iOS RC. This plan keeps that four-way split and grounds each release in the
concrete carryovers the **v1.9.9 "Workshop" readiness gate** (`docs/ios-v1.9.9-readiness.md`)
already enumerated:

- **Forced re-ports by the ADR 0002 / ADR 0028 save-state-format break:** `.rns`
  save-states + the four slots + CloudKit-synced states, `.rnm` TAS movies / imports,
  and cross-platform save + netplay parity must be re-validated on the v2.0.0 core.
- **Genuine iOS-applicable deferrals from v1.9.9:** box-art scraping, an opt-in
  crash-reporting surface, a dedicated external-display output; plus the intentionally
  scoped creator tools (20-band EQ, `.dbg` source maps, a cheats DB, NSF player).
- **Maintainer-manual (not CI-certifiable):** Apple Developer account + bundle ID,
  fastlane match signing + App Store Connect API key, app icon / launch art, and the
  full on-device TestFlight checklist.

## Hard contract (unchanged from the v1.9.x train; applies to every release below)

- **The chip stack + the emitted-frame/sample contract are NEVER touched.** All iOS
  work lives in `crates/rustynes-ios` + `ios/RustyNES/` over the shared
  `rustynes-mobile` UniFFI bridge. Desktop / native / `no_std` / wasm stay
  byte-identical; **AccuracyCoin 141/141** holds on host CI.
- **Determinism holds on ARM.** A state saved on desktop/Android **v2.0.0 core** loads
  on iOS; a `.rnm` TAS replays identically; desktop ⇄ Android ⇄ iOS cross-play stays
  valid. The DRC resampler + run-ahead stay frontend-side (timing-only).
- **No JIT anywhere; no bundled commercial ROMs; user-sourced ROMs only** (§4.7). No
  `--all-features` (mlua vs piccolo mutually exclusive). `unsafe` only in
  `rustynes-ios` + FFI glue, each with a `// SAFETY:` note.
- **No store submission before v2.1.0.** TestFlight is the only iOS channel until then.

## STANDING directive — re-research the Internet at the start of every release

Per the maintainer, the Internet is a tool **throughout** iOS work. Each release
re-checks: App-Store policy drift (§4.7 sub-clauses, IAP/external-link rules), the
**iOS-SDK submission floor**, ATT / ad-SDK rules, and the Swift / UniFFI / wgpu / cpal /
fastlane tooling versions. Each section ends with a **"re-research before starting"**
note. Make any beneficial, contract-respecting enhancement encountered en route.

**Confirmed this planning pass (2026-07-08):**
- **App Store now mandates the iOS/iPadOS 26 SDK built with Xcode 26**, effective
  **April 28, 2026**, for every App Store Connect upload — the deployment target may
  stay iOS 16/17. This is a hard gate in **v2.0.7/v2.0.8** (build settings + the
  tag-gated macOS CI must pin Xcode 26). Source: Apple Developer "Upcoming
  Requirements" — <https://developer.apple.com/news/upcoming-requirements/?id=02032026a>
  (SDK minimum requirements).
- **§4.7 is stable and emulator-friendly.** The Nov-2025 update only added HTML5/JS
  mini-apps to scope; no NES-emulator impact. Delta precedent stands.

## v2.0.5 "Landfall" — core re-port onto Timebase (≈ Android v2.0.1)

The first iOS finalization release: bring the frozen v1.9.9 SwiftUI/Metal shell onto the
v2.0.0 core and re-establish a green iOS build path on it.

### Host-done (this release)

- **Version bump.** Workspace `2.0.4 → 2.0.5` (version-only `Cargo.lock` cascade); iOS
  `MARKETING_VERSION`/`CFBundleShortVersionString → 2.0.5`, `CURRENT_PROJECT_VERSION`
  bump (in `ios/project.yml`).
- **UniFFI-Swift regeneration wiring.** Confirm the `rustynes-mobile` Swift-binding
  generation (`uniffi-bindgen-swift` / `cargo swift`) targets the **v2.0.0 bridge
  surface** — including the v2.0.3 `HostWarning` / `drain_warning_codes()` API and any
  v2.0.x bridge additions — and that the `ios/Generated/` wiring + the build script
  regenerate cleanly. **Host-verifiable** by compiling `rustynes-ios` +
  `cargo run -p rustynes-mobile --bin uniffi-bindgen` (Swift emit), *without* Xcode.
- **Save-state / movie re-port (code + docs).** Point the Swift `SaveStateManager` /
  `MovieManager` / CloudKit sync at the **v2.0.0 `.rns`/`.rnm` epoch** (ADR 0028): a
  pre-v2.0.0 slot must fail with the clear versioned error, not silently misload.
  Author the on-device migration/re-bless note; the format itself is core-owned and
  already broken at v2.0.0, so this is wiring + messaging, not a new format.
- **Determinism re-verify hooks.** Ensure the host determinism gate (save-state
  round-trip + cross-platform parity) exercises the v2.0.0 core through the mobile
  bridge; document the on-device AccuracyCoin/SMB/Zelda smoke as the ARM counterpart.
- **Docs.** `CHANGELOG.md` `[2.0.5]`; `docs/STATUS.md` current-release block (v2.0.4
  demoted); `to-dos/ROADMAP.md`; `.github/release-notes/v2.0.5.md`; `docs/ios.md` +
  `docs/ios-v2.0.5-readiness.md` (new); this plan's v2.0.5 bullet.

### Flagged for maintainer / on-device (v2.0.9 / the RC upload)

Regenerate the `.xcframework` on macOS (Xcode); on-device save-state migration from a
v1.9.x install; the AccuracyCoin + SMB/Zelda determinism smoke on Apple silicon; the
TestFlight boot smoke on iPhone 14+/iPad.

### Verification gates (host)

`cargo check --workspace` (version-only lock diff) + `cargo fmt --all --check`;
`cargo clippy --workspace --all-targets -- -D warnings`; the mobile/iOS crate unit
tests (`rustynes-mobile` + `rustynes-ios`); UniFFI-Swift emit compiles; the determinism
round-trip test; `pre-commit run markdownlint --all-files` on changed docs.

### Re-research before starting

Current `uniffi` / `cargo swift` / `uniffi-bindgen-swift` invocation + the Rust-lib ↔
generated-Swift version coupling; the v2.0.0 bridge surface delta since v1.9.9;
`xcodebuild -create-xcframework` current flags.

## v2.0.6 "Parity" — feature-parity finalization (≈ Android v2.0.2)

Re-point every host feature at the v2.0.0 behaviour and close the iOS-applicable v1.9.9
parity gaps that don't need the store.

### Host-done (this release)

- **Feature re-verify against the new core (code review + wiring):** Game Center cloud
  saves/achievements, iCloud (KV-store config + CloudKit `.rns` sync), MFi/hardware
  controllers (hot-plug, P1–P4), ReplayKit capture/share + PiP, iPad multitasking +
  Stage Manager, VoiceOver / Dynamic Type / colorblind palettes — confirm each still
  maps to the v2.0.0 bridge surface and the re-ported save-state epoch.
- **Close the v1.9.9 iOS-applicable deferrals that are host-authorable:** the **opt-in
  crash-reporting surface** (parity with Android's `CrashReporter`, off by default) and
  **box-art scraping** groundwork (reuse the Android/libretro box-art library design)
  where it needs no device.
- **Docs + version bump** (`2.0.5 → 2.0.6`), `CHANGELOG`/`STATUS`/`ROADMAP`/release-notes,
  `docs/ios-v2.0.6-readiness.md`.

### Flagged for maintainer / on-device

On-device feature verification (Game Center sign-in, CloudKit sync across two devices,
controller hot-plug, PiP); CloudKit container provisioning + quota; the Game Center /
iCloud entitlement wiring in the real signed build.

### Re-research before starting

Current Game Center / CloudKit entitlement setup; `Network.framework` background limits;
ReplayKit/PiP API drift; VoiceOver best practice on the latest iOS.

## v2.0.8 "Harborlight" — iOS release candidate (≈ Android v2.0.4)

The pure iOS **RC** cut: stage everything a maintainer needs to upload to TestFlight /
App Store Connect, flag every on-device / secret step.

### Host-done (this release)

- **App Store Connect listing metadata (files only).** `fastlane/metadata/ios/` (mirror
  the Android `fastlane/metadata/android/{en-US,es-ES}/` structure): name, subtitle,
  description (no ROM references), keywords, release notes / changelog for `2.0.8`,
  within App Store field limits. **No upload.**
- **Signing scaffold with graceful fallback.** The iOS analog of the v2.0.4 Android
  signing scaffold: `fastlane match` / manual-signing config that resolves the
  distribution cert when the maintainer secret is present, else falls back to
  automatic/dev signing so a keyless CI archive still produces an installable (never
  shippable) artifact. Secrets stay gitignored / CI-injected.
- **App-Review self-audit.** A §4.7 posture checklist (no bundled/downloadable ROMs, no
  in-app ROM links, no Nintendo branding, in-app ownership notice, searchable library
  §4.7.4, age rating §4.7.5), the emulator-policy statement, and the age-rating
  questionnaire answers — staged in `docs/ios-v2.0.8-readiness.md`.
- **Docs + version bump** (`2.0.7 → 2.0.8`), the full doc set.

### Flagged for maintainer / on-device (v2.0.9 / v2.1.0)

The actual ASC listing upload; real distribution-cert signing (inject the secret,
confirm App-Store distribution profile); the TestFlight external-beta promotion (up to
10,000 testers); the App-Review submission itself (v2.1.0); the on-device TestFlight
checklist (ROM import, save/rewind, MFi controller, audio interruptions across iOS
16/17/18/26, ProMotion pacing, no crashes on 5–10 ROMs, an accurate privacy label).

### Re-research before starting

Current App Store Connect listing-asset requirements (screenshot sizes, age-rating
questionnaire); the TestFlight external-beta review requirements; `fastlane match` /
`pilot` current flow; §4.7 text drift since v1.9.9.

## What this window does NOT do (explicit non-goals)

- **No store submission / no flag flips.** `APPSTORE_BUILD` / monetization
  (StoreKit 2 + RevenueCat + AppLovin MAX behind ATT) stay **dormant**; the iOS
  `foss`/App-Store flavor split stays inert. All of that flips at **v2.1.0**.
- **No core / accuracy work.** If accuracy work becomes necessary it is a **separate,
  core-owned** effort (like the Android side's v2.0.3 2-cycle-ALE campaign), not folded
  silently into an iOS host release.
- **No FDS/NSF, 20-band EQ, `.dbg` source maps, cheats DB** — post-Timebase carryovers
  re-evaluated at/after v2.1.0 (per v1.9.9 §5).

## Dependencies & the pre-v2.0.5 dep-bump

A **conservative, in-tier dependency-refresh PR** lands **before v2.0.5** (a `chore`
cut), keeping compatibility guaranteed:

- `cargo update` (semver-compatible `Cargo.lock` refresh — currently a 4-package
  transitive delta) + any safe in-tier direct/dev bumps.
- Safe Android Gradle / AndroidX / Compose-BOM and iOS SwiftPM/tooling minor bumps that
  don't cross a major.
- Refresh the `ref-proj/` reference emulators (Mesen2 / ares / tetanes / …) to current
  upstream as research inputs.
- **Held out of this PR (tracked as a separate aggressive follow-up):** the deliberately
  deferred out-of-tier jumps — **wgpu 29 → 30**, **naga 29 → 30**, **getrandom → 0.4**,
  and the latest egui tier — which carry real frontend + mobile-bridge API churn and get
  their own stabilization window (candidate: a dedicated modernization slice around
  v2.0.7 / v2.1.0). The **wasm-bindgen CLI ↔ library pin** stays locked.

## Gates (every release in this window)

- **Desktop / core determinism contract holds:** AccuracyCoin **141/141**, nestest
  0-diff, the `#![no_std]` chip stack untouched, host CI green. iOS changes are
  presentation / host-side only.
- The standard RustyNES host gate battery: `fmt`, `clippy -D warnings` for **every
  feature combo** (incl. `retroachievements`), `RUSTDOCFLAGS="-D warnings" cargo doc`,
  the `no_std` cross-compile, and both wasm clippy gates.
- `markdownlint` (pinned v0.39.0, via `pre-commit`) on changed docs.
- **No store submission before v2.1.0**; TestFlight only.
- The **on-device verification pass is a hard gate on v2.1.0**, not on these releases —
  each iOS release here flags its on-device items into the **v2.0.9** readiness pass.
