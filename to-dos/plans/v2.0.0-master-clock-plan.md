# RustyNES — Plan: v2.0.0 "Timebase" (the one-clock refactor)

> **Scope LOCKED by the maintainer (2026-06-17):** v2.0.0 = **full timebase rewrite +
> AGGRESSIVE residual closure.** Do the one-clock + every-cycle-bus-access collapse +
> cycle-accurate reset + full Vs. DualSystem + the breaking-API / save-state cleanup, AND
> treat the 1-CPU-cycle MMC3 bracket (R1/R2) as a **hard target**, not an accepted
> `#[ignore]` — accepting that this is the historical rollback graveyard. **Sequence v2.0.0
> AFTER v1.7.0–v1.9.0 ship.** A bounded-effort escape hatch (§Risks) applies to R1/R2 only.

maintainer renames freely. The deep source maps backing this plan live in

## Context — read this first (the framing fact that reshapes the release)

**The master clock already shipped.** The engine-line "R1 master clock" (a `master_clock: u64`
on the CPU advancing 12 units/CPU-cycle, φ1/φ2 split access, `run_ppu_to(exact mc)` PPU
catch-up, region dividers NTSC 12 / PAL 16 / Dendy 15 → **exact PAL 3.2:1** as integer
dividers) was promoted to the **default and only scheduler** and is what RustyNES ships today
(release line at v1.5.0 "Lens"). It already reaches **AccuracyCoin 100% (139/139)**, nestest
0-diff, `cpu_interrupts_v2` 5/5 strict, `ppu_vbl_nmi` 10/10. **The brief-era "residuals" the
master clock was supposed to fix — the `$2002`/NMI-suppression race, the C1 IRQ-sample trio,
the sprite-0 stale shifter, exact PAL 3.2:1 — are ALREADY CLOSED by that shipping core.**

So v2.0.0 is **not** "add a master clock." It is the **architectural rewrite that the R1
master clock was built *on top of* but never finished**, and which only a major bump (the one
release allowed breaking changes) can license. Two things remain, and they are the same
rewrite:

1. **The R1 master clock rides on a FIVE-counter substrate that should be ONE clock.** The
   core keeps `Cpu::master_clock`, `Cpu::cycles`, `LockstepBus::cycle`, `LockstepBus::ppu_clock`,
   `Apu::cpu_cycle` **plus** two parity bits (`apu_phase`, `put_cycle`) **plus** the DMC
   byte-timer — kept in lockstep by hand-written `+= 1` statements, never derived from one
   source of truth (some invariants already do not hold). The reference invariant
   (Mesen2 / ares / TriCNES, all three): **one monotonic counter; everything phase-related is
   fixed arithmetic of it.**
2. **The CPU is NOT "every cycle a bus access."** `dispatch()` declares an instruction's
   length and a burn-loop (`cpu.rs while cycles_emitted < cycles { idle_tick }`) fills the
   remainder with busless `idle_tick`s, with a `dma-cycle-budget` hack to reconcile a DMA that
   halts mid-instruction. The references model *every* 6502 cycle as a real read, a write, or
   a dummy read of the held address — no busless cycle.

Collapsing to **one clock** AND moving to **every-cycle-bus-access** is the genuine
sub-cycle refactor. It is what makes the last residuals (R1–R5, all currently OUTSIDE the
AccuracyCoin oracle) *representable* and what cleanly enables full Vs. DualSystem dual-core.

one release allowed breaking changes, and because the evidence supports a *rewrite*, not a
16th point-fix.

## Hard contract (with the explicit v2.0.0 exception)

- **AccuracyCoin 100% (139/139) is STILL sacred** and must hold at *every* beta gate
  (alongside nestest 0-diff, the 60-ROM + extended oracles byte-identical, `ppu_vbl_nmi`
  10/10, visual-regression 7/7). The rewrite must *also* flip R1–R5 — never trade the 100%
  for a residual.
- **Determinism within v2.0.x holds** — same seed + ROM + input ⇒ bit-identical framebuffer +
  audio, re-proven under the new timebase by the save-state-continuation byte-identical tests.
- **THE EXCEPTION — cross-version determinism is intentionally broken at the v1.x ↔ v2.0.0
  boundary.** The rewrite changes the cycle timeline by design (DMC fetch cadence shifts, the
  reload span moves 3→4, reset becomes a real sequence). v1.x `.rns` save-states, `.rnm` TAS
  movies, and netplay rollback snapshots are **NOT** bit-compatible with v2.0.0. This is
  allowed by ADR-0003's MAJOR-boundary tier and is the textbook reason for a major bump.
  **Clean-reject v1.x slots/movies with a clear message — do NOT write timeline-transcoding
  migration code** (a v1.x timeline cannot be made bit-meaningful on the new core).
- **The chip stack stays `#![no_std]` + alloc;** only `rustynes-frontend` + `rustynes-cheevos`
  carry `unsafe`. Hot paths: no allocation, profile before abstracting; keep the
  run-to-timestamp catch-up (§F) so the every-cycle model stays ≤2 ms/frame headless.
- **Pin-the-failing-test-ROM FIRST** for every residual (ADR-0002 / project policy): wire a
  strict expected-fail probe for each of R1–R5 so the rewrite *flips them loud*, before
  implementing.

## STANDING directive (applies through the whole release)

When implementing any workstream, **re-mine the references continually** — ares/higan
(`scheduler/thread`, libco `synchronize`), Mesen2 (`NesCpu.cpp` φ1/φ2 ±1, `ProcessPendingDma`,
`DeltaModulationChannel`), TriCNES (`_EmulatorCore` ordering, the invisible reload arm),
`nesdev_wiki/`, and the in-tree `docs/audit/v2.0-*` maps — and make any beneficial correctness
or performance win encountered en route even if not explicitly listed here, **provided it
holds the hard contract** (AccuracyCoin 139/139 + byte-identical gates).

## Workstreams

### A — One-clock + every-cycle-bus-access timebase  (the core; XL, Very-high risk)

The rewrite. Adopt the Mesen2/TriCNES/ares-converged model:

- **A1. One monotonic `master_clock: u64`, everything derived.** Make `master_clock` (already
  a field) the *only* counter; derive `bus.cycle`, `apu.cpu_cycle`, `apu_phase`, `put_cycle`,
  the DMC-timer phase, the OAM 513/514 alignment, and the reload arm as **pure arithmetic of
  `master_clock`** (e.g. `put_cycle = ((master_clock / div) + seed) & 1`). RW-1 already proved
  this collapse byte-identical for the parity pair (`mc-r1-one-clock`, commit `c1e2a25`);
  v2.0.0 finishes it for all five. **Preserve every divider exactly** (load-bearing, already
  region-correct): NTSC CPU÷12 / PPU÷4; PAL CPU÷16 / PPU÷5 (= exact 3.2:1); Dendy CPU÷15;
  read split pre-5/post-7, write split pre-7/post-5; PPU offset 1; PAL frame 33,247 cyc.
- **A2. Every cycle is a bus access** (the make-or-break). Replace the `dispatch()`-length +
  `idle_tick` burn-loop with a per-cycle model where each CPU cycle issues a real read, a real
  write, or a **dummy read of the held address** (the references' `DummyRead`). Retire the
  `dma-cycle-budget` reconciliation entirely: a DMC/OAM DMA halts the CPU *on a bus access*,
  and now every cycle is one, so the halt point and span are exact. This is the audit's
  "interleaved-DMA model, divergence C": one DMA cycle interleaved per CPU step sharing the
  per-cycle put-cycle flip-flop.
- **A3. Reload arm invisible to its own cycle.** A `pending_dmc_dma_next` latch promoted at
  the next cycle boundary (reloads only; loads stay GET-first → 3), per TriCNES's
  `_EmulateAPU`-after-`_6502()` ordering — turns the DMC reload span 3→4 (closes R5 naturally
  once A2 holds).
- **A4. Cycle-accurate reset.** Replace the function-call `Nes::reset()` with a real reset
  *sequence* that runs the reset-vector-delay cycles + frame-counter re-arm phase on the
  master clock, preserving `cpu_phase` alignment (determinism contract) — the substrate for
  closing R3/R4.

*Determinism:* the rewrite changes cycle timing by design; re-prove byte-identical save-state
continuation; AccuracyCoin 139/139 at every gate. Develop behind a default-off
`mc-one-clock-v2` feature; promote only when every gate holds together AND R1–R5 flip.

### B — Residual closure R1–R5  (L, rides on A; High risk — AGGRESSIVE per the decision)

Pin each ROM's failing expectation FIRST, then flip:

- **B1. R5 — DMC reload Y=3→4** — falls out of A2+A3 (pinning oracle: the `CheckDMATiming`
  `$4010=$4E` landmark cross-diff). This is the structural proof A2 worked.
- **B2. R3/R4 — `apu_reset` ×2** — close via A4's cycle-accurate reset sequence (pinning:
  `tests/apu_reset.rs:113,138` flipped to strict).
- **B3. R1/R2 — `mmc3_test_2/4` #3 + #2/MMC6 — the HARD TARGET.** On the one-clock +
  every-cycle model the M2 sub-cycle phase the 1-CPU-cycle bracket needs is **now
  representable** (it was not on the integer 3-dots timebase — the documented reason R1/R2
  were "unreachable"), so the `T_last-1` IRQ sample + bus poll at the M2 phase becomes a
  *sample-point choice* rather than an impossible constraint. Pin: `tests/mmc3.rs` strict
  probes + the `irq_trace` golden traces. **This is the 17-rollback axis — see the escape
  hatch in Risks: a bounded effort, then fall back to by-design `#[ignore]` rather than risk a
  16th rollback of the sacred 100%.**

*Determinism:* each closure is a phase function of `master_clock` — deterministic by
construction.

### C — Full Vs. DualSystem dual-core  (L, ~4–7 focused days; Med-High risk)

Design fully scoped in `docs/audit/vs-dualsystem-design-2026-06-11.md` (the v2.7.1 engine-line
groundwork shipped a `vs_db.dual_system` flag + a frontend note instead of a black screen).

- A `VsDualSystem { main: Nes, sub: Nes, latch: VsCommsLatch }` wrapper in `rustynes-core`,
  surfaced as a top-level `Emu { Single(Nes), Dual(VsDualSystem) }` enum (the API reshape that
  helps justify the major bump).
- Protocol (Mesen2 `VsControlManager.cpp`): a single `$4016` bit-1 main/sub signal poking the
  *other* CPU's `/IRQ`; `$4016` bit-7 read = `is_sub ? 0x80 : 0` (RustyNES currently hard-pins
  0); independent DIP banks (`dipSwitches >> 8` for sub); shared-WRAM `SwapMemoryAccess`; two
  PPUs → two framebuffers side-by-side.
- The **one-clock rewrite makes the 5-CPU-cycle soft-lockstep cleaner** (each `Nes` advances a
  single master clock vs juggling five counters).
- *Out of scope for the dual path (state explicitly):* netplay (rollback assumes one state
  blob) and RetroAchievements (one memory map). *Risk:* no DualSystem test-ROM oracle —
  validation is "boots + plays" (weaker than byte-identical); the shared-WRAM swap window is
  where a residual black-screen lurks.

### D — Breaking-API + save-state + cleanup + doc re-baseline  (M, Med risk)

- **Save-state:** CPU section v2→v3 (the five-counter collapse makes `cpu.cycles`/`bus.cycle`/
  `apu.cpu_cycle` derived — drop or zero them; add the single `parity_seed`; **keep** the
  Stage-4 lesson that `ppu_clock`/`dma_mc_consumed` must travel with `master_clock`). Bump APU
  + BUS tails. **`.rns` `FORMAT_VERSION` bump** → v1.x readers refuse with
  `SnapshotError::UnsupportedFormat`; v2.0.0 **cleanly rejects v1.x slots** (no migration code).
- **`.rnm` movies:** honest break — v2.0.0 refuses to *verify-replay* a v1.x-recorded movie
  (still plays it as input, but the determinism guarantee is void); add a "recorded-on-engine"
  warn tag. Do NOT attempt timeline transcoding.
- **API cleanup:** retire the dead experiment flags (`cpu-c1-attempt-17-access-reorder`,
  `ppu-2002-read-end-flags`, `accuracycoin-sprite-eval-base-from-oamaddr`, the `mc-r1-*` /
  `dmc-get-put-scheduler` stubs); drop the `take_dma_mc_consumed` / `dma_mc_consumed`
  coherence-layer trait methods (obviated by A); collapse the four cycle counters in the
  public/internal API to one; re-home the `idle_tick`/burn-loop surface.
- **ADRs:** **0016** (save-state break + the no-migration decision), **0017** (the one-clock
  every-cycle timebase, superseding the dot-lockstep description), and an **update to 0002**
  marking the C1/MMC3 axis CLOSED (or by-design-deferred if the escape hatch fires).
- **Doc re-baseline:** rewrite `docs/scheduler.md` + `docs/architecture.md` + `docs/cpu-6502.md`
  to describe the one-clock every-cycle model (they currently describe the `tick_one_dot`
  dot-lockstep primer, already engine-lineage-historical relative to the shipping R1 core).

### E — Long-tail mapper breadth + UNIF  (M–L, Low risk; NOT gated on A)

Toward the full ~370 puNES set, honesty-gated (ADR-0011 `mapper_tier_honesty.rs`: no
BestEffort mapper may back an oracle): the **UNIF loader** + push toward ~150–200 families as
BestEffort, each with register-decode unit tests + staged commercial-ROM boot-smoke
screenshots (standing ROM-coverage policy: stage per-mapper ROMs in gitignored
`tests/roms/external/`, commit screenshots/.snap only — **never the ROMs**). **This is not
gated on the rewrite** — much of it lands in the v1.6–v1.9 streams and v2.0.0 simply ships
whatever count is current. Byte-identical to the oracle.

### F — Perf re-baseline on the new timebase  (S–M, Low-Med risk)

The master clock's +6–8% full-frame cost is *already paid* (both configs clear the 16.639 ms
NTSC deadline by 4.25–7.1×; the R1 CPU core is −34.6% *faster* in isolation via run-to-timestamp
batching). The every-cycle model adds a dummy read on cycles that today are busless — tiny
(a single `mapper.cpu_read`, <1% of frame cost) **IF run-to-timestamp catch-up is kept**
(ares/higan use libco `synchronize`; Mesen2 + RustyNES use run-to-timestamp — keep it; do NOT
drift to naive per-master-clock stepping). Re-capture `full_frame` / `cpu_throughput` /
`ppu_throughput` / `mapper_dispatch` / `spectral` baselines under the one-clock core; re-run
the PGO/BOLT gate (`pgo.yml`, >3% + byte-identical). Target: no worse than the shipped R1
numbers (≤2 ms headless rendering-heavy, ≤~4 ms nestest); ideally the collapse claws back some
of the +6–8%.

## Risks (ranked)

1. **(Very high) A2 make-or-break — the every-cycle / interleaved-DMA rewrite breaking the
   DMASync DMA cluster or the C1 trio.** This exact seam *failed* under the substrate (the R1
   port Phase-3 found the master-clock CPU loop "fundamentally incompatible with the
   DMASync-based DMA cluster unless the interleaved-DMA model is solved"). v2.0.0's bet is that
   the rewrite **removes the cumulative-drift source** rather than band-aiding it.
   **Escape: beta.2 is the stop-or-go gate — if A2 can't hold the DMA cluster + C1 together,
   bank A1 (the one-clock collapse, byte-identical) and ship v2.0.0 on B-subset/C/D/E/F.**
2. **(High) Regressing AccuracyCoin off 100% — sacred.** Every rollback in the 18-month engine
   line came from a point-fix regressing an orthogonal surface. Only the full rewrite-with-gates
   is defensible; the gate after every step is the guardrail.
3. **(High) R1/R2 (MMC3 #3) is the 17-rollback graveyard — and it is a HARD TARGET this
   release.** It is only *reachable* on the new timebase. **Escape hatch (R1/R2 only):** after
   a **bounded effort** (target ≤ the beta.3 window; if the M2-phase sample still can't bracket
   the 1-CPU-cycle sub-test #3 without regressing another surface), fall back to keeping
   R1/R2 `#[ignore]`'d by-design (zero production-ROM impact) and ship v2.0.0 with R3/R4/R5
   closed. **Do not risk a 16th rollback of the 100% for the 1-cycle bracket.** This is the
   one place "aggressive" yields to "sacred."
4. **(Med-High) Vs. DualSystem has no test-ROM oracle** — validation is "boots+plays"; the
   shared-WRAM swap window is the likely residual-bug site.
5. **(Med) Save-state / TAS / netplay cross-version break** confuses users carrying v1.x slots
   — mitigate with loud rejection + ADR-0016 + release-note prominence.
6. **(Med) Perf regression if the rewrite drifts to per-master-clock stepping** — mitigate by
   keeping run-to-timestamp catch-up + the bench gate at every step.
7. **(Low) Long-tail mapper breadth** — additive, honesty-gated.
