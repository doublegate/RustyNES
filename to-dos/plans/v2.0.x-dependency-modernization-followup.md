# Dependency modernization — tracked aggressive follow-up

> **Companion to the conservative pre-v2.0.5 dep refresh** (the
> `chore/deps-refresh-pre-v2.0.5` PR). That PR took only the **semver-compatible,
> build-verified** updates (a `cargo update` lockfile refresh) so the v2.0.x iOS
> finalization train isn't destabilized mid-flight. This document tracks the
> **out-of-tier major jumps** deliberately held back — each carries real API churn
> and needs its own stabilization window with a full build + accuracy re-verify.
> Two-step by maintainer decision (2026-07-08): *conservative now, aggressive tracked.*

## Why these are held out of the conservative PR

The current desktop/mobile graphics tier is pinned as a set — **wgpu 29 / naga 29 /
egui 0.35 / winit 0.30.13** — because bumping one forces the others and cascades through
`rustynes-frontend` (the wgpu present path + egui shell), `rustynes-android` +
`rustynes-ios` (the `SurfaceView` / `CAMetalLayer` handoff), and the WGSL in
`rustynes-gfx-shaders`. `cargo outdated --root-deps-only` shows the *only* available
root-dep upgrades are exactly these out-of-tier majors (the `Compat` column is `---` for
all three — nothing compatible remains to take). They are not safe "in-tier" bumps; they
are a migration.

## The tracked jumps

| Dep | From | To | Blast radius | Notes |
|---|---|---|---|---|
| **wgpu** | 29 | 30 | `rustynes-frontend`, `rustynes-android`, `rustynes-ios`, `egui-wgpu` | The keystone; drives naga + egui-wgpu. Metal/Vulkan/WebGPU surface-config API churn; re-verify the wasm WebGL2 fallback. |
| **naga** | 29 | 30 | `rustynes-frontend` (dev/shader-validation) | Moves in lockstep with wgpu 30. |
| **egui / egui-wgpu / egui-winit** | 0.35 | the wgpu-30-compatible egui tier (pin the exact minor in the migration PR) | `rustynes-frontend` shell + all overlay/debugger panels | Must land with the wgpu 30 bump — `egui-wgpu` tracks the wgpu major, so the target is whichever egui minor pins that wgpu 30 tier (0.36+ at time of writing; the migration PR records the exact version resolved). Re-verify the persistent egui shell + Settings tabs. |
| **getrandom** | 0.2 / 0.3 | 0.4 | wasm32 frontend only | Backend/register API change on wasm; native unaffected. Re-verify the Pages build. |

## Constraints that MUST survive the migration

- **The wasm-bindgen CLI ↔ library pin stays exact** (`web/Trunk.toml` ↔ `Cargo.lock`);
  a wgpu/egui bump that moves the wasm-bindgen library version must move the Trunk pin
  in the same change, or the Pages deploy fails at the wasm-bindgen step.
- **Byte-identity + AccuracyCoin 141/141** must hold — these are presentation-tier deps,
  so the framebuffer/audio oracle must be unchanged after the bump (verify with the full
  `--features test-roms` suite, and re-bless the visual snapshot corpus only if an
  intentional render change is confirmed).
- **`no_std` chip stack untouched** (none of these reach `rustynes-core`).
- **Every clippy feature combo** stays `-D warnings`-clean (incl. `retroachievements`,
  both wasm targets) — re-run the full gate matrix, not just the active feature set.

## Suggested landing window

A dedicated modernization slice, candidate **around v2.0.7 or folded into v2.1.0** (the
joint-launch release), **not** inside the iOS host-scaffolding releases (v2.0.5–v2.0.8),
so an iOS RC is never destabilized by a graphics-tier migration. Sequence: wgpu 30 +
naga 30 + egui-latest together (they are one migration) → verify native + wasm + mobile
build → getrandom 0.4 (wasm) → full accuracy + snapshot re-verify.
