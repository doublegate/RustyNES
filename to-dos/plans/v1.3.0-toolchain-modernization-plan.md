# RustyNES — Plan: v1.3.0 "Toolchain Modernization" (edition 2024 + MSRV bump + dep next-tier)

**Status:** captured 2026-06-15 (analysis). NOT to land in v1.2.0 — a dedicated effort *after* v1.2.0 ships.
**Hard gate (applies to everything):** AccuracyCoin **100% (139/139)** + framebuffer/audio oracles
**byte-identical** + nestest 0-diff must hold after every step. Rust guarantees stable FP codegen
(no fast-math / no FP contraction across versions), so the deterministic core *should* stay
bit-identical — but this MUST be re-proven with the full `--features test-roms,commercial-roms`
suite. That re-verification, not the mechanical edits, is where the risk concentrates.

## Current state (the starting line)

- `rust-toolchain.toml` channel = **1.86.0** (+ targets `thumbv7em-none-eabihf`, `wasm32-unknown-unknown`);
  `[workspace.package] edition = "2021"`, `rust-version = "1.86"`. The 1.86 pin exists only because
  edition2024 transitive deps in the frontend stack need ≥1.85.
- Stack on egui **0.32** / wgpu **25** / naga **25** / rfd **0.14** (the version-locked UI cluster, just
  migrated from 0.29→0.32 in v1.2.0). egui 0.34 / wgpu 26 are the next tier and need MSRV **1.92 / 1.87**.

## Two coupled axes + the dependency cascade

### A. Edition 2021 → 2024 (mostly mechanical; `cargo fix --edition` does the bulk)
Surveyed migration surface (small):
- **`gen` reserved keyword:** `crates/rustynes-frontend/src/audio.rs` uses `gen` as a variable
  (`let gen = queue.eq_gen()`). Rename to `generation` (cleaner than `cargo fix`'s `r#gen`). ~3 lines.
- **`unsafe extern` blocks:** the rcheevos `extern "C" { … }` declarations in
  `crates/rustynes-cheevos/src/ffi.rs` → `unsafe extern "C" { … }`. Isolated to that one
  already-`unsafe`, native-only crate; the chip stack has NO FFI. `cargo fix` handles it.
- **NOT present:** `#[no_mangle]`/`export_name`/`link_section` (no unsafe-attr migration);
  `env::set_var`/`remove_var` (no unsafe-env churn).
- **Determinism-relevant subtle changes to re-verify (not just compile):** tail-expression temporary
  drop scope reversed (temporaries in a block tail expr drop *before* the block's locals) — the one
  change that could theoretically alter Drop-order behavior; the core is `no_std` + mostly Copy/POD
  with no hot-path RAII, so risk is low but must be oracle-verified. Also `!` never-type fallback,
  RPIT lifetime capture (`+ use<…>` opt-out) — usually surface as compile errors, not silent changes.
- Edition is set once in `[workspace.package]`, inherited by all 10 crates; orthogonal to `no_std`.

### B. MSRV 1.86 → latest stable (~1.92+)
- Edit `rust-toolchain.toml` channel + `[workspace.package] rust-version` + the CI MSRV-pin job + matrix.
- Low user impact (we ship binaries; only from-source builders need the newer toolchain).
- **Unlocks the blocked deps:** **rfd 0.17** (the open Dependabot PR — its `*const i8: Default` failure
  is a Rust-version issue a newer toolchain fixes; bump then close the PR) AND the egui 0.34 / wgpu 26 tier.

### C. Dependency cascade (the real work, gated on B)
- **egui 0.32 → 0.34 + wgpu 25 → 26 + naga 26** — another coordinated version-locked UI-stack migration,
  same shape as the 0.29→0.32 one already done (API renames across `gfx.rs`/`crt.rs`/`ntsc*.rs`/the
  debugger panels/`shader_pass.rs`). Needs MSRV 1.92/1.87. Delegate like prior migrations.
- rfd 0.14 → 0.17 (free once MSRV moves); `cargo update` + a Dependabot sweep for the rest.
- Recurring gotcha: `web/Trunk.toml` wasm-bindgen CLI pin must stay byte-equal to the `wasm-bindgen`
  library in `Cargo.lock`, or the Pages build breaks at the wasm-bindgen step.

## Recommended sequencing
1. **MSRV bump** (toolchain.toml + CI) → unblocks rfd + higher-MSRV deps. Full suite green.
2. **Edition 2024:** `cargo fix --edition` per crate → manual cleanup (`gen` rename, `unsafe extern`)
   → **full determinism re-verify** (the tail-drop-scope check is the point of this).
3. **egui 0.34 / wgpu 26 / naga 26** coordinated cluster + rfd 0.17 + `cargo update`.
4. Full gate suite + oracle byte-identical confirmation + wasm size budget + bump the Trunk.toml pin.

## Effort / risk
Edition + MSRV themselves ≈ ½–1 day (mostly `cargo fix` + the re-verify). The egui 0.34/wgpu 26 cluster
is the larger piece (comparable to the migration already in the v1.2.0 lineage). Risk concentrates
entirely in the determinism re-verification, not the mechanical edits. Do it as its own release
(early v1.3.0), never folded into a feature beta.
