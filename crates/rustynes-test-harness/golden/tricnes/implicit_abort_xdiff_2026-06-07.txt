FIX (2026-06-07): mc-r1-dmc-abort-timer-phase — the A->B +4 is CLOSED

ROOT DEFECT (per-cycle DMC-state trace, both emulators; recipe above + the
new `dtimer,dbits,dsil,dbuf,cooldown` cols on the RustyNES anchor dump and
`dtimer=/dbits=/dsil=/dbytes=` on the TriCNES XdiffLog):
  In the Loop3 disable($4015=00)->re-enable($4015=10) context the re-enable
  fires a LOOPING-sample LOAD DMA. TriCNES's LOAD GET completes 3 cyc BEFORE
  the byte-timer boundary (load fetch entry-rel -1109, boundary -1106), so the
  boundary's clock_output consumes the load byte into the shifter AND arms a
  RELOAD (TriCNES does TWO GETs: LOAD -1109 + RELOAD -1103, then 576-spaced).
  RustyNES's LOAD `deliver_sample` lands ON the boundary cycle (-1103), where
  clock_output already ran at cycle-START and took the still-EMPTY buffer
  (silence) — so the boundary misses the load byte, no reload arms there, and
  the reload chain anchors at the NEXT boundary. Net: the LOAD arms off-boundary
  (dtimer=1, 4 cyc before the boundary) while every normal reload arms AT the
  boundary (dtimer=35) => A->B = (4) + 576 = 580. RustyNES did only ONE GET
  (the LOAD), skipping the 4-cyc RELOAD DMA TriCNES inserts (the +4 in the
  $DA03-entry boot offset: 385561 at re-enable -> 385565 at entry).

FIX: on the LOAD `complete_dmc_dma`, when the byte-timer JUST reset this cycle
(timer == timer_period, i.e. clock_output wrapped + reloaded bits to 8 + took
the empty buffer = silence) and the LOAD has now filled the buffer with bytes
still remaining, retroactively load the delivered byte into the shifter
(un-silence) + clear the post-LOAD cooldown so the per-cycle reload-arm fires
promptly — reproducing TriCNES's boundary-coupled reload. The `timer ==
timer_period` gate keeps it OUT of the Loop1/Loop2 mid-byte 1-byte-abort loads
(a looser gate corrupted Key1/$500 -> error 4 regressed to error 2).
